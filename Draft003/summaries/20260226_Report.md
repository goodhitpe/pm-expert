### 2.2 Day 1 — 오리엔테이션, 프로젝트 관리 프레임워크

**현재 잘 된 부분:**
- 프로젝트 3대 특성(일시성·고유성·점진적 구체화) 상세 설명
- Triple Constraint와 품질의 관계
- 5개 프로세스 그룹, 10개 지식 영역 개요
- PM의 3대 역량(기술적 PM·리더십·전략 및 비즈니스 관리)

**누락 또는 부족한 내용:**

**① 포트폴리오 → 프로그램 → 프로젝트 계층 구조**
- 현재 내용: 단순 언급 수준
- 필요 내용: 포트폴리오(전략 정렬) → 프로그램(시너지 관리) → 프로젝트(산출물) 계층의 차이와 PM이 상위 계층을 인식해야 하는 이유
- 예시: "회사의 디지털 전환 전략(포트폴리오) → MES 고도화 프로그램 → 생산 라인 자동화 프로젝트"

**② PMO (프로젝트 관리 조직)의 3가지 유형**
- 현재 내용: PMO의 역할 간략 언급
- 필요 내용:
  - **지원형(Supportive)** — 정책·도구·템플릿 제공, PM 권한 낮음
  - **통제형(Controlling)** — 표준·프로세스 상시 감사, PM 중간 권한
  - **지시형(Directive)** — 프로젝트 직접 관리, PM 최고 권한
- 왜 중요한가: 조직 내 PMO 유형을 모르면 PM이 어떤 수준의 지원을 기대해야 하는지 알 수 없음

**③ OPA(조직 프로세스 자산)와 EEF(기업 환경 요인) 구별**
- 현재 내용: 약어로만 등장, 개념 설명 없음
- 필요 내용:
  - **OPA**: 조직이 축적한 내부 자산(프로세스, 정책, 템플릿, 교훈, 이력 데이터)
    - 예: 회사 표준 WBS 템플릿, 이전 프로젝트 교훈 DB
  - **EEF**: 조직이 통제할 수 없는 외부/내부 환경(규제, 문화, 시장, 기술 인프라)
    - 예: 개인정보보호법, 회사 조직 문화, 클라우드 인프라 표준
  - 왜 중요한가: PMP 시험의 거의 모든 프로세스에 입력물로 등장

**④ 프로젝트 생애주기 유형**
- 현재 내용: 언급 없음
- 필요 내용:
  - **예측형(Predictive)**: 초기에 범위·일정·원가 확정(전통적 워터폴)
  - **반복형(Iterative)**: 반복을 통해 범위를 점진적으로 상세화
  - **점진형(Incremental)**: 기능을 분할 인도
  - **적응형(Adaptive)**: 애자일 방식, 변화에 반응
  - **혼합형(Hybrid)**: 예측형 + 적응형 조합
  - 각 유형에 적합한 프로젝트 특성과 선택 기준

**⑤ Work Performance: Data → Information → Reports 흐름**
- 현재 내용: 각 프로세스 산출물로 나열되지만 흐름 연결 없음
- 필요 내용:
  - **WPD (작업 성과 데이터)**: 실행에서 수집된 원시 측정값 (실제 완료율, 실제 비용)
  - **WPI (작업 성과 정보)**: WPD를 분석한 의미 있는 정보 (SPI, CPI)
  - **WPR (작업 성과 보고서)**: 이해관계자에게 전달할 패키지 (대시보드, 요약 보고서)
  - 이 3단계 흐름이 통합 관리의 핵심 데이터 파이프라인

### 2.3 Day 2 — 통합 관리

**누락 또는 부족한 내용:**

**① 비즈니스 케이스(Business Case)와 이점 관리 계획서(Benefits Management Plan)**
- 현재 내용: 헌장의 입력물로 '비즈니스 케이스' 언급만
- 필요 내용:
  - 비즈니스 케이스 = 프로젝트 투자 타당성 문서 (NPV, IRR, ROI, 회수기간)
  - **이점 관리 계획서**는 PMBOK 6판에서 새로 강조된 문서:
    - 프로젝트 완료 후 실현될 비즈니스 이점(경제적 가치)을 어떻게 측정할지 정의
    - 예: "ERP 도입 후 재고비 30% 절감 → 연간 5억원, KPI: 재고회전율"
  - PM은 프로젝트 종료 후에도 이점 실현 여부를 추적해야 함
  - 기업에서 '프로젝트가 성공했지만 효과가 없다'는 상황을 방지하는 도구

**② 지식 관리(Manage Project Knowledge)의 현대적 해석**
- 현재 내용: 교훈 등록부 작성으로 단순화
- 필요 내용:
  - 묵시지(Tacit Knowledge, 개인에게 내재된 경험)와 명시지(Explicit Knowledge, 문서화된 지식)의 전환
  - 교훈은 종료 시만 아니라 **프로젝트 전 기간** 지속 수집
  - 지식 저장소(Knowledge Repository) 활용법
  - CoP(Communities of Practice, 실천 공동체) 개념

### 2.5 Day 4 — 일정 관리

**현재 잘 된 부분:** CPM, PERT, Crashing/Fast-tracking 매우 상세, AON 다이어그램, 총 여유·자유 여유 구분

**누락 또는 부족한 내용:**

**① 일정 단축 기법에서 누락된 개념**
- **Resource Leveling vs Resource Smoothing 차이**
  - 자원 평준화(Leveling): 과부하 해소 → 일정 연장 가능
  - 자원 평활화(Smoothing): 총 여유 내에서만 조정 → 일정 불변
  - 실무 적용: 팀원이 동시에 두 프로젝트에 투입될 때 어떻게 배분하는가

**② 일정 감시 기법**
- **일정 성과 지수(SPI)와 간트차트 연계**: 간트차트로 보이는 지연이 SPI에 어떻게 반영되는지
- **Critical Chain PM**: 버퍼(Buffer) 개념 — 프로젝트 버퍼·피딩 버퍼
  - Theory of Constraints(제약이론) 기반
  - 왜 안전 여유를 각 활동에 넣지 않고 뒤에 모아야 하는가

**③ 일정 관리 현대 도구**
- 간트차트 도구 실습 (MS Project 기초)는 Day 17에 있으나, Day 4에서 이론과 연결하는 내용 필요

### 2.6 Day 5 — 원가 관리

**현재 잘 된 부분:** 전체 강의안 중 가장 상세하고 완성도 높음. EVM, S-Curve, 원가 분류, 산정 기법 모두 탁월

**누락 또는 부족한 내용:**

**① ETC vs EAC 4가지 공식 비교**
- 현재 내용: EAC = AC + (BAC-EV)/CPI 한 가지 공식만 집중
- 필요 내용 (PMP 시험 빈출):
  - `EAC₁ = BAC / CPI` — CPI가 앞으로도 지속될 것으로 예상
  - `EAC₂ = AC + (BAC - EV)` — 앞으로의 작업은 계획대로 수행
  - `EAC₃ = AC + (BAC - EV) / CPI` — 현재 효율이 유지
  - `EAC₄ = AC + (BAC - EV) / (CPI × SPI)` — 일정과 원가 모두 현재 효율 반영
  - 각 공식의 사용 조건과 차이 설명

**② TCPI (완료성과지수, To-Complete Performance Index)**
- 현재 내용: 언급 없음
- 필요 내용:
  - `TCPI = (BAC - EV) / (BAC - AC)` — 기준선 기반
  - `TCPI = (BAC - EV) / (EAC - AC)` — EAC 기반
  - TCPI > 1이면 앞으로 더 효율적으로 일해야 함
  - TCPI와 CPI 비교로 목표 달성 가능성 판단

- 현재 내용: 자원 평준화/평활화 개념 나옴

- 현재 내용: 일부 프로세스 상세 설명 부족

  2. 리스크 식별
  3. 정성적 리스크 분석

## 3. PMP 7판 차이점 및 추가 지식 분석

### 3.1 PMP 7판과 6판의 핵심 차이

PMP 7판(2021)은 6판과 패러다임이 근본적으로 다릅니다. **6판은 "프로세스 중심"**, **7판은 "원칙·성과 중심"**입니다.

| 구분 | 6판 | 7판 | 현재 강의안 반영 |
|------|-----|-----|---------------|
| 핵심 구조 | 49개 프로세스 | 12개 원칙 + 8개 성과 영역 | ❌ 7판 원칙 구조 없음 |
| 지식영역 | 10개 KA | 폐지 (성과영역으로 통합) | ⚠️ 6판 구조만 존재 |
| 맞춤화 | 부분적 언급 | Tailoring이 핵심 개념 | ⚠️ 미흡 |
| 방법론 | 예측형 중심 | 예측·적응·혼합 통합 | 🟡 Day 17에서 부분 |
| 가치 | 인도물 중심 | 비즈니스 가치 중심 | ⚠️ 미흡 |

### 3.2 PMP 7판 12대 원칙 — 현재 강의안에 없음

7판의 12가지 프로젝트 관리 원칙은 PMP 시험과 현업에서 점점 더 중요해지고 있습니다.

| # | 원칙 | 현재 강의안 관련 내용 | 추가 필요 |
|---|------|---------------------|----------|
| 1 | 스튜어드십(청지기 정신) — 성실하고 책임감 있고 신뢰할 수 있게 행동 | 간접적 언급 | ✅ 추가 필요 |
| 2 | 팀(Team) — 협업 분위기 조성 | 터크만 모델 | 🟡 원칙으로 재설명 |
| 3 | 이해관계자(Stakeholders) — 적극적 참여 | Day 11 | ✅ 추가 필요 |
| 4 | 가치(Value) — 가치에 집중 | 부분적 | ✅ 추가 필요 |
| 5 | 시스템 사고(Systems Thinking) — 전체 시스템 관점 | 없음 | ✅ 추가 필요 |
| 6 | 리더십(Leadership) — 리더십 행동 模範 | PM 역량으로 간략 | ✅ 심화 필요 |

8대 성과 영역은 6판의 지식 영역을 대체하는 것이 아니라, **성과 측정 관점에서 재구성**한 것입니다.

### 3.4 Tailoring (맞춤화) — 핵심 추가 필요 개념

7판에서 특히 강조되는 Tailoring은 "모든 프로젝트에 동일한 방법론을 쓰지 말고, 상황에 맞게 조정하라"는 원칙입니다.

**추가 필요 내용:**
- Tailoring의 4단계: 현황 파악 → 선택 → 조정 → 지속적 개선
- Tailoring 기준 요소: 조직 문화, 프로젝트 규모, 팀 역량, 산업 규제, 불확실성 수준
- 실습: "당신의 현재 또는 예상 프로젝트에 맞는 방법론은 무엇인가?"
- 예시: 소규모 스타트업 앱 개발 → 스크럼 / 정부 系 대형 시스템 구축 → 워터폴 기반 PMF / 제조사 MES → 하이브리드

### 3.5 Value Delivery System (가치 전달 시스템)

- 현재 내용: 없음
- 필요 내용:
  - 프로젝트는 단독으로 가치를 만들지 않고, **포트폴리오·프로그램·프로젝트 생태계** 전체가 조직 목표에 기여함
  - 프로젝트의 성공 기준 = "인도물 완성"에서 "비즈니스 가치 실현"으로 이동
  - 예: ERP 도입 프로젝트가 성공해도 직원이 쓰지 않으면 가치 없음
  - 가치 실현 추적 방법: Benefits Realization Plan

## 4. PMP 8판 / ECO 2023 내용 분석

### 4.1 현행 PMP 시험 기준: ECO 2023

현재 PMP 시험은 2023년도 Exam Content Outline(ECO)을 기반으로 합니다. 이 ECO는 PMBOK 7판 정신 + 실무 반영으로 구성됩니다.

ECO 2023의 세 도메인과 비율:
- **People (리더십·인적 관리)**: 42%
- **Process (프로세스 관리)**: 50%  
- **Business Environment (비즈니스 환경)**: 8%

### 4.2 ECO 2023에서 강조하는 추가 내용

**① 지속가능성 (Sustainability) / ESG**
- 현재 강의안에 없음
- 추가 필요 내용:
  - **ESG(Environmental, Social, Governance)** 프레임워크
  - 프로젝트가 환경에 미치는 영향 최소화 방법
  - 탄소 발자국 관리, 공급망 ESG 평가
  - IT 프로젝트의 에너지 효율 (Green IT: 서버 에너지 소비, 클라우드 그린 존)
  - 왜 PM에게 중요한가: 글로벌 기업·상장사의 ESG 공시 의무화, 조달 이슈
  - 실습 예시: "신규 데이터센터 구축 시 ESG 관점에서 어떰 위험을 고려해야 하는가?"

**② DEI (다양성·형평성·포용성, Diversity, Equity, Inclusion)**
- 현재 강의안에 없음
- 추가 필요 내용:
  - 팀 구성의 다양성이 프로젝트 성과에 미치는 영향
  - 무의식적 편견(Unconscious Bias) 인식
  - 접근성(Accessibility): 소프트웨어·콘텐츠의 장애인 접근성 (WCAG 기준)
  - 글로벌 팀에서의 문화적 포용성
  - 국내 적용: 세대 다양성(MZ세대·X세대 협업), 남녀 균형, 경력 다양성

**③ 조직 변화 관리 (OCM, Organizational Change Management)**
- 현재 강의안에 없음
- 추가 필요 내용:
  - 프로젝트의 가장 큰 실패 원인 중 하나는 **변화 저항**
  - **ADKAR 모델** (Awareness → Desire → Knowledge → Ability → Reinforcement)
  - **코터(Kotter)의 8단계 변화 관리**
  - 왜 PM이 알아야 하는가: 새로운 시스템 도입 시 사용자 저항이 없어야 ROI 실현
  - 예시: "ERP 도입 후 직원 절반이 사용하지 않는 이유는 OCM 실패"
  - 실습: 현업 저항 예상 이해관계자 대상 변화 관리 계획 수립

**④ AI 활용 프로젝트 관리 (AI-Assisted PM)**
- 현재 강의안에 없음
- 추가 필요 내용:
  - AI 도구를 사용한 일정 최적화, 리스크 예측
  - GitHub Copilot, ChatGPT 등 AI 도구가 개발 생산성에 미치는 영향
  - AI 프로젝트의 특수성: 데이터 품질, 모델 드리프트, 설명가능성(XAI)
  - PM 관점 리스크: AI 도구 의존성, 편향성 리스크, 저작권 이슈
  - 예측 분석을 활용한 EVM 예측 (AI-driven EAC)

**⑤ 복잡성 관리 (Complexity Management)**
- 현재 강의안에 없음  
- 추가 필요 내용:
  - **Cynefin 프레임워크**: Simple·Complicated·Complex·Chaotic 4가지 도메인
    - Simple: 최선 실천 적용(워터폴)
    - Complicated: 전문가 판단 필요
    - Complex: 실험·탐색 필요(애자일)
    - Chaotic: 즉각 행동 후 안정화
  - **Stacey Matrix**: 기술 불확실성 × 요구사항 불확실성으로 방법론 선택
  - 왜 중요한가: "이 프로젝트에 어떤 방법론이 맞는가?" 질문에 대한 체계적 답

**⑥ 혜택 실현 (Benefits Realization)**
- 현재 강의안에 없음
- 추가 필요 내용:
  - 프로젝트 종료 ≠ 이점 실현
  - Benefits Realization Plan 구성: 이점 목록, 측정 지표, 책임자, 측정 시점
  - 프로젝트 종료 후 6개월·1년 후 이점 측정 리뷰
  - PM에서 이점 실현 추적자로 역할 확장

  - **서버리스 아키텍처**: AWS Lambda, Azure Functions — 인프라 서버 없이 함수 단위 실행
    - PM 관점: 인프라 비용 절감 vs 벤더 종속성 vs 콜드 스타트 레이턴시

**② DevSecOps (보안 내재화 개발)**

- 추가 필요:

  - Shift-Left 보안: 개발 초기 단계부터 보안 검토
  - SAST (Static Application Security Testing): 코드 작성 단계
  - DAST (Dynamic Application Security Testing): 실행 단계
  - SCA (Software Composition Analysis): 오픈소스 취약점 분석
  - PM 관점: 보안 테스트를 스프린트에 어떻게 통합하는가? 보안 요구사항 → WBS 반영

**③ 기술적 부채(Technical Debt) 관리 전략**

- 추가 필요:

- PM 관점: 스마트 공장·스마트 오피스 프로젝트에서 무선 네트워크 선택 기준
