## 1교시: 조달 관리 개요 <!-- 슬라이드 #1~#2 -->

### 이론 (50분)

#### 1. 조달(Procurement)의 정의

**PMBOK 정의:**  
외부 조직으로부터 제품, 서비스, 결과를 **구매 또는 획득**하는 프로세스

**핵심 개념:**
- 프로젝트 팀이 직접 생산하지 않고 외부에서 조달
- 구매자(Buyer) 관점에서 관리
- 법적 구속력이 있는 계약(Contract) 기반

#### 2. 조달 관리의 3개 프로세스

**기획 프로세스 그룹:**

**1) 조달 관리 계획 수립 (Plan Procurement Management)**
- 구매 결정 문서화 및 조달 접근 방법 정의
- Make or Buy 의사결정
- 산출물: 조달 관리 계획서, 조달 작업 기술서(SOW)

**실행 프로세스 그룹:**

**2) 조달 수행 (Conduct Procurements)**
- 판매자 응답 획득, 선정, 계약 체결
- 산출물: 선정된 판매자, 협약/계약

**감시/통제 프로세스 그룹:**

**3) 조달 통제 (Control Procurements)**
- 조달 관계 관리, 성과 모니터링, 변경 및 시정 조치
- 산출물: 종료된 조달, 작업 성과 정보

#### 3. Make or Buy 의사결정

**Make (자체 개발)**

**장점:**
- 핵심 역량 내재화
- 기밀 유지
- 장기적 비용 절감 가능
- 통제력 강함
- 조직 학습 및 성장

**단점:**
- 초기 투자 큼
- 전문성 부족 시 품질 저하
- 리스크 전체 부담
- 자원 소모

**Buy (외부 조달)**

**장점:**
- 전문성 활용
- 초기 투자 적음
- 리스크 분산
- 일시적 필요 충족
- 빠른 시작 가능

**단점:**
- 장기적 비용 높을 수 있음
- 기밀 유출 위험
- 통제력 약함
- 의존성 증가

**의사결정 기준:**

| 기준 | Make 선택 | Buy 선택 |
|------|---------|----------|
| 전문성 | 내부 보유 | 외부 전문 |
| 전략적 중요도 | 핵심 역량 | 비핵심 |
| 기밀성 | 높음 | 낮음 |
| 필요 기간 | 지속적 | 일시적 |
| 자원 가용성 | 충분 | 부족 |
| 리스크 | 수용 가능 | 분산 희망 |
| 비용 (장기) | Make 유리 | Buy 유리 |

**IT 예시:**

```
Make 사례:
- 핵심 알고리즘 (경쟁력)
- 보안 인증 모듈 (기밀)
- 사용자 경험 (차별화)

Buy 사례:
- 결제 게이트웨이 (전문 업체)
- 클라우드 인프라 (AWS, Azure)
- 이메일 서비스 (Gmail API)
- 외주 개발 (일시적 인력 필요)
```

## 2교시: 계약 유형 (1.5시간) <!-- 슬라이드 #4~#5 -->

### 이론 (50분)

#### 1. 계약의 기본 이해

**계약(Contract)이란:**
- 법적 구속력이 있는 합의
- 양 당사자 모두 의무를 가짐
- 위반 시 손해배상 책임

**계약의 3가지 필수 요소:**
1. 청약(Offer): 한쪽의 제안
2. 승낙(Acceptance): 상대방의 동의
3. 대가(Consideration): 쌍방향 교환 (돈, 서비스)

#### 2. 고정가 계약 (Fixed-Price)

**원칙:** 범위가 명확히 정의될 때 사용
**리스크 분담:** 판매자에게 유리 (구매자 리스크 낮음)

**A. FFP (Firm Fixed Price) - 확정 고정가**

- **구조:** 계약 시 합의한 금액만 지불
- **구매자 입장:**
  - 장점: 예산 확실, 관리 용이
  - 단점: 초기 가격이 높을 수 있음 (판매자 리스크 프리미엄)
- **판매자 입장:**
  - 장점: 효율화하면 이익 극대화
  - 단점: 비용 초과 시 손실 발생
- **IT 적합 케이스:** 요구사항이 명확한 패키지 개발, 유지보수

**예시:**
```
사내 급여 시스템 고도화 (FFP 2억원)
구매자: 2억원만 지불 (성과 무관)
판매자: 실제 개발 비용 1.5억 → 이익 5천만원
        실제 개발 비용 2.5억 → 손실 5천만원
결론: 판매자가 효율적으로 관리해야 유리
```

**B. FPIF (Fixed Price Incentive Fee) - 성과 인센티브**

- **구조:** 목표 비용 달성 시 인센티브 추가 지급
- **목표 비용(Target Cost) 이하 달성:** 절감액을 구매자/판매자가 분배
- **공식:** Profit = Target Profit ± (Sharing Ratio × Cost Variance)
- **적용:** 일정 단축, 품질 초과 달성 장려

**C. FP-EPA (Fixed Price with Economic Price Adjustment) - 경제적 조정**

- **사용:** 장기 계약 (2년 이상)
- **구조:** 인플레이션, 환율에 따라 가격 자동 조정
- **IT 예시:** 3년 클라우드 관리 서비스 계약

#### 3. 원가보상 계약 (Cost-Reimbursable)

**원칙:** 범위가 불명확하고 연구개발적 성격이 있을 때
**리스크 분담:** 구매자가 대부분 부담

**A. CPFF (Cost Plus Fixed Fee) - 실비 + 고정 수수료**

- **구조:** 실제 비용 전액 + 미리 합의한 고정 수수료
- **문제점:** 판매자가 비용 절감 동기 없음 (수수료는 고정)
- **IT 예시:** 신기술 R&D 프로젝트, 탐색 단계 개발

**B. CPIF (Cost Plus Incentive Fee) - 실비 + 인센티브**

- **구조:** 실비 + 목표 달성도에 따른 변동 수수료
- **인센티브 구조:** 비용 절감 시 분배 (단, 최대/최소 한도 설정)
- **장점:** CPFF보다 비용 절감 동기 부여

**C. CPAF (Cost Plus Award Fee) - 실비 + 포상 수수료**

- **구조:** 실비 + 구매자의 주관적 평가에 따른 포상
- **특징:** 포상 기준이 계약에 명시되지 않음
- **리스크:** 판매자 입장에서 불확실성 높음

#### 4. 시간 및 자재 계약 (T&M: Time and Material)

**구조:**
- 시간: 시간당 노무비 (Labor Rate)
- 자재: 실비 정산

**특징:**
- 중간 수준의 리스크 분담
- 범위가 불명확할 때 적합
- **위험:** 비용 상한선이 없으면 무한정 증가 가능
- **대응:** NTE(Not to Exceed) 조항으로 상한 설정

**IT 적합 케이스:**
- 운영/유지보수 계약
- 범위가 계속 변하는 Agile 개발

#### 5. 계약 유형 선택 가이드

| 상황 | 권장 계약 유형 | 이유 |
|------|------------|------|
| 범위 명확, 완성품 납품 | FFP | 구매자 예산 확실 |
| 범위 명확, 성과 장려 | FPIF | 인센티브로 품질/일정 개선 |
| 범위 불명확, 탐색적 | CPFF | 구매자가 리스크 부담 |
| 유지보수/운영 | T&M | 범위가 계속 변함 |
| 장기 계약 (2년+) | FP-EPA | 물가 변동 반영 |

**리스크 스펙트럼:**

```
구매자 위험↑                          판매자 위험↑

CPFF → CPIF → CPAF → T&M → FPIF → FFP → FP-EPA
```

### 예시 (25분)

#### 예시 1: 계약 유형별 손익 계산

**프로젝트:** 전자정부 시스템 개발

**시나리오: 실제 비용 = 8억원 (목표: 7억원)**

**FFP 계약 (계약금 8.5억):**
```
구매자 지불: 8.5억 (고정)
판매자 수익: 8.5억 - 8억 = 5,000만원
→ 판매자 손익: +5,000만원
→ 구매자: 계획대로 지출
```

**CPFF 계약 (목표 비용 7억 + 수수료 7,000만):**
```
구매자 지불: 8억(실비) + 7,000만(수수료) = 8.7억
판매자 수익: 7,000만 (고정, 초과에 무관계)
→ 구매자가 추가 1억 부담
→ 판매자는 비용 관리 동기 없음
```

**CPIF 계약 (목표 비용 7억, 수수료 7,000만, 공유 비율 60:40):**
```
비용 초과: 1억
공유 (40%): 판매자 4,000만 부담
실제 판매자 수수료: 7,000만 - 4,000만 = 3,000만원
구매자 지불: 8억 + 3,000만 = 8.3억

→ 판매자는 비용 절감 동기 있음 (절감 시 이익 공유)
→ 구매자는 CPFF보다 적게 지불
```

#### 예시 2: T&M 계약 함정

## 3교시: 조달 문서 및 RFP 작성 (1.5시간) <!-- 슬라이드 #3 -->

### 이론 (50분)

#### 1. 조달 문서 유형

**A. RFI (Request for Information) - 정보 요청서**

**목적:** 시장 조사, 잠재 공급자 파악
**사용 시점:** 조달 결정 이전 단계
**내용:** 회사 소개, 기술 현황 요청 (가격 아님)

**RFI 예시 사용:**
```
상황: ERP 도입 검토 중, 어떤 솔루션이 있는지 모름
→ RFI 발송: "당사 요건에 맞는 ERP 솔루션 현황 제출 요청"
→ 10개 업체 응답 → 3개 업체를 RFP 대상으로 선정
```

**B. RFQ (Request for Quotation) - 견적 요청서**

**목적:** 규격이 명확한 제품/서비스의 가격 비교
**기준:** 가격이 핵심 (기술 평가 최소)
**예시:** "서버 50대 견적 요청" - 사양 동일, 가격만 비교

**C. RFP (Request for Proposal) - 제안 요청서**

**목적:** 기술 방안 + 가격을 종합 평가하는 경쟁 입찰
**기준:** 기술적 역량 + 가격 + 경험 (종합 평가)
**필수 구성요소:**

| 섹션 | 내용 |
|------|------|
| 1. 사업 개요 | 배경, 목적, 추진 방향 |
| 2. 사업 범위 | 기능 요구사항, 비기능 요구사항 |
| 3. 요구사항 | 기술 사양, 인터페이스, 보안 |
| 4. 관리 요건 | 일정, 인력, 방법론 |
| 5. 제안서 형식 | 페이지 수, 구성, 제출 방법 |
| 6. 평가 기준 | 항목별 배점 |
| 7. 계약 조건 | 기본 계약 골자 |
| 8. 제출 기한 | 마감일, 방법 |

**D. IFB (Invitation for Bid) - 입찰 초대**

**목적:** 공개 입찰, 주로 공공 조달
**기준:** 자격 요건 충족 중 최저가
**예시:** 공공기관 IT 장비 구매

#### 2. 작업 기술서 (SOW: Statement of Work)

**목적:** 조달 대상 작업의 범위, 목표, 산출물을 상세 기술

**IT SOW 구성요소:**

```
1. 개요
   - 사업 배경 및 목적
   - 현황 분석

2. 작업 범위
   - 개발 범위 (기능 목록)
   - 제외 범위 (명확히 기술!)
   - 인터페이스 범위

3. 납품 산출물 (Deliverables)
   - 기능 명세서
   - 소스코드
   - 테스트 결과서
   - 교육 자료
   - 운영 매뉴얼

4. 일정
   - 마일스톤 계획
   - 인도 기한

5. 기술 요구사항
   - 성능 (응답 시간, 가용율)
   - 보안 (인증, 암호화)
   - 호환성

6. 검수 기준
   - 인수 테스트 기준
   - 결함 허용 기준
```

**SOW 작성 원칙:**
- 명확성: 모호한 표현 제거 ("적절한" → "응답 시간 2초 이하")
- 완전성: 범위 누락 없음
- 검증 가능성: 측정 가능한 기준

#### 3. 입찰자 회의 (Bidder Conference)

**목적:** 모든 잠재 공급자에게 동일한 정보 제공 (공정성)

**운영 방법:**
- RFP 발송 후 지정 날짜에 Q&A 회의
- 모든 질문/답변을 서면으로 공개 (특정 업체 혜택 방지)
- 현장 방문 포함 시 모든 업체 동시 방문

**주의 사항:**
- 특정 업체에만 정보 누출 금지
- 명확하지 않은 항목은 공식 보정서(Addendum) 발행

## 4교시: 공급업체 선정 및 협상 (1.5시간) <!-- 슬라이드 #6 -->

### 이론 (50분)

#### 1. 공급업체 선정 프로세스

**단계:**
1. 제안서 접수 및 형식 검토
2. 예비 필터링 (최소 요건 검토)
3. 기술 평가
4. 가격 평가
5. 종합 평가 및 순위 결정
6. 협상
7. 최종 계약 체결

#### 2. 가중 점수 기법 (Weighted Scoring)

**목적:** 여러 기준을 종합적으로 평가하여 주관성을 줄임

**절차:**

**Step 1: 평가 기준 및 가중치 결정**
- 기준은 RFP에 미리 공개 (공정성)
- 가중치는 RFP 발송 전 확정 (업체 영향 차단)

**Step 2: 각 업체별 점수 부여**
- 각 평가위원이 독립적으로 점수 부여
- 평균 산출

**Step 3: 가중 점수 계산**
- 가중 점수 = 기준 점수 × 가중치 비율

**Step 4: 종합 점수 비교**

**예시 평가표:**

| 평가 항목 | 가중치 | A사 점수 | A사 가중점수 | B사 점수 | B사 가중점수 |
|---------|--------|----------|------------|---------|------------|
| 기술 방법론 | 25% | 85 | 21.25 | 78 | 19.50 |
| 기술 인력 | 20% | 90 | 18.00 | 82 | 16.40 |
| 유사 실적 | 20% | 80 | 16.00 | 88 | 17.60 |
| 보안 대책 | 15% | 75 | 11.25 | 80 | 12.00 |
| 제안 가격 | 20% | 82 | 16.40 | 90 | 18.00 |
| **합계** | **100%** | | **82.90** | | **83.50** |

**결론:** B사 선정 (83.50 > 82.90)

#### 3. 독립 산정 (Independent Estimate)

**목적:** 제안 가격이 시장 가격에서 크게 벗어나지 않는지 검토

**방법:**
- 내부 전문가가 사전에 예상 비용 산정
- 실제 제안 가격과 비교
- 크게 차이나는 경우 → 범위 오해, 숨겨진 비용 의심

**예시:**
```
독립 산정: 내부 예상 3억원
A사 제안: 2.5억 → 왜 싼가? (인력 축소, 범위 축소 의심)
B사 제안: 4.5억 → 왜 비싼가? (과잉 제안, 리스크 프리미엄)
C사 제안: 3.1억 → 합리적 (독립 산정과 근접)
```

#### 4. 협상 (Negotiation)

**협상 목표:**
- 공정하고 합리적인 계약 조건
- 양측 Win-Win (장기 관계 유지)

**BATNA (Best Alternative to Negotiated Agreement):**
- "협상이 결렬될 경우 내가 할 수 있는 최선"
- BATNA를 알면 협상에서 흔들리지 않음

**예시:**
```
구매자 BATNA:
→ 2위 업체와 계약 가능 (83점 A사 대신 82.5점 B사)

판매자 BATNA:
→ 다른 계약 3건 대기 중

구매자는 BATNA가 있으므로 1위 업체에 과도하게 의존 안 함
→ 가격 협상에서 유리한 위치
```

**ZOPA (Zone of Possible Agreement) - 합의 가능 구간:**

```
구매자 최대 지불 의사: 4억원
판매자 최소 수용 가격: 3억원

ZOPA: 3억~4억 (합의 가능 구간)
협상 결과: 3.5억 → ZOPA 내에서 합의
```

**협상 전략:**
1. **충분한 사전 조사** (시장 가격, 업체 상황)
2. **우선순위 명확화** (가격 vs 일정 vs 품질 중 뭐가 가장 중요?)
3. **패키지 협상** (단일 이슈보다 패키지로 교환)
4. **앵커링** (첫 제안이 협상 기준점이 됨)
5. **감사와 공감** (장기 관계는 단기 이익보다 중요)

#### 5. IT 계약 특수 조항

**A. SLA (Service Level Agreement)**

- 서비스 품질 수준 명시
- 미달 시 패널티 조항

```
SLA 예시:
가용성: 99.9% (월간 다운타임 < 44분)
응답 시간: 평균 < 1.5초
장애 복구: Critical 4시간, High 8시간

패널티:
- 99.9% 미달 시: 월 서비스 비용의 5% 감액
- 99.5% 미달 시: 월 서비스 비용의 15% 감액
```

**B. 지적재산권 (IPR) 귀속**

- 개발물에 대한 소유권
- 커스터마이징 코드 vs 솔루션 기본 코드
- 일반적: 발주자가 커스터마이징 코드 소유

**C. 소스코드 에스크로 (Source Code Escrow)**

- 판매자 파산/사업 종료 시 소스코드 확보
- 제3자(에스크로 기관)에 소스코드 위탁
- 조건 충족 시 구매자에게 자동 인도

**D. 하자 보수 보증금**

- 계약 금액의 일정 비율 (예: 5%)을 보증금으로 보유
- 하자 발생 시 판매자 수정 의무

### 예시 (25분)

#### 예시 1: 협상 시뮬레이션

**상황:** 모바일 앱 개발 외주 협상

## 5교시: 조달 통제 및 계약 관리 (1.5시간) <!-- 슬라이드 #6~#7 -->

### 이론 (50분)

#### 1. 조달 통제 (Control Procurements)

**목적:** 계약 이행 모니터링, 변경 및 시정 조치, 계약 종료 관리

**주요 활동:**

**A. 성과 모니터링**

| 모니터링 항목 | 방법 | 주기 |
|------------|------|------|
| SLA 준수 현황 | 자동 모니터링 대시보드 | 실시간|
| 마일스톤 달성 | 진행 보고서 검토 | 주간 |
| 품질 기준 충족 | 검수 회의 | 인도물별 |
| 예산 집행 | 청구서 검토 | 월간 |
| 인력 투입 | 타임시트 확인 | 주간 |

**B. 변경 관리**

**변경 관리 절차:**
1. 변경 요청서 작성 (양식 사용)
2. 영향 분석 (일정/비용/범위)
3. 내부 검토 및 승인
4. 계약 수정서(Contract Amendment) 발행
5. 공식 서명 (양측)
6. 기록 보관

**핵심 원칙:**
- 구두로 합의한 변경은 무효 → 반드시 서면
- 변경 이전에 계약 수정서 서명 완료

**C. 클레임 관리 (Claims Administration)**

**클레임 발생 시나리오:**
- 판매자: "구매자가 납기를 지연시켜 우리 비용이 증가했다"
- 구매자: "판매자가 SOW를 위반했다"

**해결 순서:**
1. 계약서 검토 (계약이 우선)
2. 당사자 간 협의
3. 조정(Mediation) - 중립 제3자
4. 중재(Arbitration) - 구속력 있는 결정
5. 소송 (최후 수단)

**D. 대금 지급 관리**

**단계별 지급 구조 (IT 프로젝트):**
```
계약 체결: 10% 선급
요구사항 확정 완료: 15%
설계 완료: 15%
개발 완료: 25%
테스트 완료: 20%
운영 이관 완료: 10%
하자 보수 기간 종료: 5%
총: 100%
```

**대금 지급 원칙:**
- 모든 인도물 검수 후 지급
- 구두 승인은 무효, 서면 검수 확인서 필수

#### 2. 계약 종료 (Contract Closure)

**종류:**
- 완료 종료: 계약대로 완료
- 중도 해지: 편의/채무불이행 해지

**종료 프로세스:**

1. **최종 검수 (Final Acceptance)**
   - 모든 인도물 확인
   - 검수 완료 확인서 발급

2. **재무 정산**
   - 최종 청구서 처리
   - 잔여 보증금 지급

3. **조달 감사 (Procurement Audit)**
   - 조달 프로세스 전반 검토
   - 잘된 점과 개선점 파악

4. **행정 종료**
   - 모든 계약 관련 서류 보관
   - SLA 및 보증 조건 기록

5. **교훈 정리 (Lessons Learned)**
   - 공급업체 평가
   - 차기 조달에 활용

#### 3. IT 조달 특수 이슈

**A. SLA 미달 시 크레딧/패널티 관리**

```
SLA 미달 사례:
- 목표 가용성: 99.9%
- 실제: 99.7% (4월)
- 패널티: 월 서비스 비용 5% 감액

처리:
1. SLA 위반 공식 통보 (문서화)
2. 패널티 계산서 발행
3. 다음 달 청구서에 감액 반영
4. 재발 방지 계획 요구
```

**B. 하자 기간 관리**

- 구축 완료 후 일정 기간 무상 수정
- 일반적: 6개월~1년
- 하자 vs 기능 추가 구분 명확히

```
하자: 계약 범위 내 기능이 요구사항 미충족 → 무상 수정
기능 추가: 계약 범위 외 새 요구사항 → 추가 계약 필요
```

**Q3.** T&M 계약의 위험성과 이를 통제하기 위한 방법을 설명하시오. (5점)

**정답:** 범위가 명확하지 않아 비용이 무한히 증가할 수 있습니다. 이를 통제하기 위해 NTE(Not to Exceed) 조항으로 상한선을 설정하고, 범위 변경 시 PM의 서면 승인을 의무화해야 합니다.

**Q4.** 계약 변경 시 반드시 지켜야 할 원칙은? (5점)

**정답:** 모든 계약 변경은 반드시 서면 계약 수정서로 양측이 서명해야 합니다. 구두 합의나 이메일만으로는 법적 효력이 없으며, 분쟁 발생 시 계약서 내용이 우선 적용됩니다.
