## 1교시: 소프트웨어공학 개요 및 SDLC (1.5시간) <!-- 슬라이드 #1~#3 -->

### 이론 (50분)

#### 1. 소프트웨어공학(Software Engineering)이란?

**정의:**
소프트웨어의 개발, 운용, 유지보수에 대한 **체계적이고 규율적이며 정량화 가능한** 접근 방법

**소프트웨어공학의 필요성:**
- 소프트웨어의 규모와 복잡도 증가
- 프로젝트 실패율 감소 필요
- 품질 보증 및 유지보수성 향상
- 팀 협업 및 의사소통 개선

#### 2. 소프트웨어 위기(Software Crisis)

**1960년대 후반 등장한 개념:**
- 프로젝트 일정 지연 (평균 50% 이상)
- 예산 초과 (평균 100% 이상)
- 낮은 품질 (버그, 성능 문제)
- 유지보수 어려움

**원인:**
- 체계적인 개발 방법론 부재
- 불명확한 요구사항
- 문서화 부족
- 테스트 소홀
- 프로젝트 관리 미흡

**PM이 알아야 하는 이유:**
소프트웨어공학의 체계적 접근 없이는 프로젝트 관리 기법만으로 성공을 보장할 수 없음

#### 3. SDLC (Software Development Life Cycle)

소프트웨어의 **계획, 개발, 테스트, 배포, 유지보수**에 이르는 전체 생명주기

**SDLC 6단계:**

**1) 요구사항 분석(Requirements Analysis)**

**목적:** 고객과 사용자의 니즈를 파악하고 문서화

**주요 활동:**
- 요구사항 수집(Elicitation)
  - 인터뷰, 설문조사, 워크숍
  - 현행 시스템 분석
  - 프로토타입 검토
- 요구사항 분석(Analysis)
  - 실현 가능성 검토
  - 우선순위 결정
  - 갈등 해결
- 요구사항 명세(Specification)
  - SRS(Software Requirements Specification) 작성
  - 기능/비기능 요구사항 정리
- 요구사항 검증(Validation)
  - 이해관계자 리뷰
  - 프로토타입 확인

**산출물:**
- 요구사항 명세서(SRS)
- 유스케이스 다이어그램
- 사용자 스토리
- 요구사항 추적 매트릭스(RTM)

**PM 관점:**
- 이 단계가 전체 프로젝트의 30~40% 결정
- 요구사항 누락 시 이후 변경 비용 10배 이상 증가
- 일정: 전체의 15~20% 할당

**2) 설계(Design)**

**목적:** 요구사항을 구현 가능한 구조로 변환

**설계 레벨:**

**a) 아키텍처 설계(High-level Design)**
- 시스템 구조 정의
- 주요 컴포넌트 식별
- 기술 스택 선택
- 데이터베이스 설계(개념)
- 인터페이스 정의

**b) 상세 설계(Detailed Design)**
- 모듈별 로직 설계
- 알고리즘 정의
- 데이터 구조 설계
- 인터페이스 상세화

**산출물:**
- 아키텍처 설계서
- 상세 설계서
- ERD(Entity-Relationship Diagram)
- 클래스 다이어그램
- 시퀀스 다이어그램

**PM 관점:**
- 설계 품질이 유지보수 비용 결정
- 일정: 전체의 15~20% 할당
- 설계 리뷰 필수 (개발 전 결함 발견 비용 1/10)

**3) 구현(Implementation/Coding)**

**목적:** 설계를 코드로 변환

**주요 활동:**
- 코딩 표준 준수
- 모듈별 구현
- 단위 테스트 작성
- 코드 리뷰
- 형상 관리

**산출물:**
- 소스 코드
- 단위 테스트 코드
- 코드 리뷰 기록
- 빌드 스크립트

**코딩 표준 예시:**
```
- 네이밍 규칙 (camelCase, PascalCase)
- 들여쓰기 (2칸 또는 4칸)
- 주석 작성 규칙
- 함수 길이 제한 (50줄 이내)
- 복잡도 제한 (McCabe Cyclomatic Complexity < 10)
```

**PM 관점:**
- 코드 품질 = 유지보수 비용 직결
- 일정: 전체의 30~40% 할당
- 진척률 측정: 기능 단위, 모듈 완성도

**4) 테스트(Testing)**

**목적:** 결함 발견 및 품질 확인

**테스트 레벨:** (상세 내용은 5교시)
- 단위 테스트(Unit Test)
- 통합 테스트(Integration Test)
- 시스템 테스트(System Test)
- 인수 테스트(Acceptance Test)

**산출물:**
- 테스트 계획서
- 테스트 케이스
- 테스트 결과 보고서
- 결함 보고서

**PM 관점:**
- 테스트 시간 부족 = 품질 저하
- 일정: 전체의 20~30% 할당 (실제 프로젝트에서 자주 압축됨 ⚠️)
- "테스트 없이 배포" 절대 금지

**5) 배포(Deployment)**

**목적:** 운영 환경에 소프트웨어 설치 및 전환

**주요 활동:**
- 배포 계획 수립
- 환경 구성 (서버, 네트워크, DB)
- 데이터 마이그레이션
- 사용자 교육
- 시범 운영(Pilot)
- 본 운영 전환

**배포 전략:**
- 빅뱅(Big Bang): 한 번에 전체 전환
- 단계적(Phased): 기능별, 부서별 순차 전환
- 병행 운영(Parallel): 구/신 시스템 병행 후 전환
- 파일럿(Pilot): 일부 조직 먼저 적용

**PM 관점:**
- 배포 실패 = 프로젝트 실패
- 롤백 계획 필수
- 일정: 전체의 5~10%

**6) 유지보수(Maintenance)**

**목적:** 운영 중인 소프트웨어의 지속적 개선

**유지보수 유형:**

| 유형 | 설명 | 비율 | 예시 |
|------|------|------|------|
| **수정적 유지보수**<br>(Corrective) | 버그 수정 | 20% | 오류 수정, 결함 제거 |
| **적응적 유지보수**<br>(Adaptive) | 환경 변화 대응 | 25% | OS 업그레이드, DB 마이그레이션 |
| **완전적 유지보수**<br>(Perfective) | 기능 개선/추가 | 50% | 신기능 추가, 성능 개선 |
| **예방적 유지보수**<br>(Preventive) | 미래 문제 예방 | 5% | 코드 리팩토링, 최적화 |

**PM 관점:**
- 유지보수 비용 = 개발 비용의 3~4배 (생애주기)
- 초기 설계/코드 품질이 유지보수 비용 결정
- 완전적 유지보수(신기능)는 별도 프로젝트로 관리

#### 4. 요구사항의 유형

**A. 기능 요구사항(Functional Requirements, FR)**

**정의:** 시스템이 **수행해야 할 기능**

**특징:**
- "무엇을(What)" 시스템이 해야 하는가
- 사용자 관점의 기능
- 테스트 가능해야 함

**예시:**
```
전자상거래 시스템:
- 사용자는 회원가입을 할 수 있다
- 사용자는 상품을 검색할 수 있다
- 사용자는 장바구니에 상품을 담을 수 있다
- 사용자는 신용카드로 결제할 수 있다
- 관리자는 주문 현황을 조회할 수 있다
```

**작성 원칙:**
- 명확하고 구체적
- 측정 가능
- 일관성 유지
- 우선순위 부여

**B. 비기능 요구사항(Non-Functional Requirements, NFR)**

**정의:** 시스템의 **품질 속성** 및 제약조건

**주요 카테고리:**

**1) 성능(Performance)**
```
- 응답 시간: 페이지 로딩 2초 이내
- 처리량(Throughput): 초당 1,000건 처리
- 동시 사용자: 10,000명 동시 접속 지원
```

**2) 가용성(Availability)**
```
- 시스템 가동률: 99.9% (연간 다운타임 8.76시간)
- 복구 시간 목표(RTO): 1시간 이내
- 복구 시점 목표(RPO): 데이터 손실 10분 이내
```

**3) 확장성(Scalability)**
```
- 수평 확장: 서버 추가로 처리량 증가
- 데이터 증가: 연 1억 건 증가 대응
- 사용자 증가: 현재 대비 10배 증가 대응
```

**4) 보안(Security)**
```
- 인증: 이중 인증(2FA) 지원
- 암호화: HTTPS 통신, AES-256 데이터 암호화
- 접근 제어: 역할 기반 접근 제어(RBAC)
- 감사: 모든 민감 작업 로그 기록
```

**5) 사용성(Usability)**
```
- 학습 용이성: 30분 교육으로 기본 기능 사용 가능
- 접근성: WCAG 2.1 Level AA 준수
- 다국어: 한국어, 영어, 중국어 지원
```

**6) 유지보수성(Maintainability)**
```
- 코드 커버리지: 80% 이상
- 문서화: API 문서 자동 생성
- 모듈화: 마이크로서비스 아키텍처
```

**7) 이식성(Portability)**
```
- 크로스 플랫폼: Windows, Linux, macOS 지원
- 브라우저: Chrome, Firefox, Safari, Edge 최신 2개 버전
- 모바일: iOS 14+, Android 10+
```

**PM 관점:**
- NFR 누락 시 프로젝트 후반 큰 문제
- NFR은 아키텍처, 기술 선택 결정
- 성능/보안 NFR은 테스트 필수

#### 예시 2: 기능 요구사항 vs 비기능 요구사항

**프로젝트:** 온라인 교육 플랫폼

**기능 요구사항 (FR) 예시:**

```
사용자 관리:
FR-U01 사용자는 이메일로 회원가입을 할 수 있다
FR-U02 사용자는 비밀번호를 재설정할 수 있다
FR-U03 관리자는 사용자 계정을 비활성화할 수 있다

강의 관리:
FR-C01 강사는 강의를 등록할 수 있다 (제목, 설명, 가격)
FR-C02 강사는 동영상 강의 자료를 업로드할 수 있다 (최대 5GB)
FR-C03 학생은 강의를 검색할 수 있다 (키워드, 카테고리, 난이도)
FR-C04 학생은 강의를 장바구니에 담을 수 있다

수강 관리:
FR-L01 학생은 강의를 수강 신청할 수 있다
FR-L02 학생은 진도율을 확인할 수 있다
FR-L03 학생은 강사에게 질문을 남길 수 있다
FR-L04 학생은 수료증을 다운로드할 수 있다

결제:
FR-P01 학생은 신용카드로 결제할 수 있다
FR-P02 학생은 결제 내역을 조회할 수 있다
FR-P03 학생은 환불을 신청할 수 있다 (7일 이내)
```

**비기능 요구사항 (NFR) 예시:**

```
성능 (Performance):
NFR-P01 페이지 초기 로딩: 3초 이내
NFR-P02 동영상 버퍼링: 5초 이내 재생 시작
NFR-P03 검색 응답 시간: 1초 이내
NFR-P04 동시 접속자: 10,000명 지원
NFR-P05 동영상 스트리밍: 1080p 지원

가용성 (Availability):
NFR-A01 시스템 가동률: 99.9% (월 43분 다운타임 허용)
NFR-A02 유지보수: 매주 일요일 오전 2-4시 (사전 공지)
NFR-A03 장애 복구: 1시간 이내 서비스 재개

확장성 (Scalability):
NFR-S01 사용자 증가: 연 50만 명 증가 대응
NFR-S02 콘텐츠 증가: 연 50TB 증가 대응
NFR-S03 Auto-scaling: 부하 80% 초과 시 자동 확장

보안 (Security):
NFR-SE01 통신 암호화: HTTPS (TLS 1.3)
NFR-SE02 패스워드: 8자 이상, 영문/숫자/특수문자 조합
NFR-SE03 세션 타임아웃: 30분 미활동 시 자동 로그아웃
NFR-SE04 결제 정보: PCI-DSS 준수
NFR-SE05 개인정보: 암호화 저장 (AES-256)

사용성 (Usability):
NFR-U01 반응형 웹: 모바일, 태블릿, 데스크탑 지원
NFR-U02 브라우저: Chrome, Safari, Firefox, Edge 최신 2개 버전
NFR-U03 접근성: WCAG 2.1 Level A 준수
NFR-U04 학습 곡선: 30분 내 기본 기능 숙지

유지보수성 (Maintainability):
NFR-M01 코드 커버리지: 80% 이상
NFR-M02 API 문서: Swagger 자동 생성
NFR-M03 로그: 모든 에러 로그 수집 및 알림
NFR-M04 모니터링: Grafana 대시보드로 실시간 모니터링

규정 준수 (Compliance):
NFR-C01 개인정보보호법 준수
NFR-C02 저작권법 준수 (콘텐츠 보호)
NFR-C03 전자상거래법 준수 (환불, 약관)
```

**NFR이 프로젝트에 미치는 영향:**

| NFR | 영향 분야 | 추가 비용/일정 |
|-----|----------|----------------|
| 성능 | 아키텍처, 인프라 | +20% |
| 가용성 | 이중화, 모니터링 | +15% |
| 확장성 | 클라우드, 설계 | +10% |
| 보안 | 인증, 암호화, 테스트 | +25% |
| 접근성 | UI/UX 설계 | +10% |

**결론:**
NFR을 초기에 명확히 하지 않으면 후반부 재작업으로 일정/예산 2배 이상 증가 가능

#### 예시 3: 유지보수 유형별 사례

**프로젝트:** 운영 중인 온라인 쇼핑몰

**1) 수정적 유지보수 (Corrective) - 20%**

**시나리오:** 버그 수정
```
문제: 장바구니에 담은 상품이 종종 사라짐
원인: 세션 타임아웃 처리 오류
해결: 세션 만료 전 장바구니 데이터를 DB에 저장
일정: 2일
비용: 개발자 1명 × 2일 = 120만원
```

**2) 적응적 유지보수 (Adaptive) - 25%**

**시나리오:** 환경 변화 대응
```
상황: 카카오페이 결제 API v2.0 출시, 기존 v1.0 2025년 말 종료
작업: API v2.0로 마이그레이션
영향: 결제 모듈 전체 수정 필요
일정: 2주
비용: 개발자 2명 × 2주 = 480만원
```

**3) 완전적 유지보수 (Perfective) - 50%**

**시나리오 1:** 기능 개선
```
요청: 사용자가 주문 후 배송 추적을 실시간으로 보고 싶음
작업: 택배사 API 연동, 배송 추적 화면 개발
일정: 4주
비용: 기획 1명 + 개발자 2명 + QA 1명 × 4주 = 1,600만원
```

**시나리오 2:** 신기능 추가
```
요청: 정기 구독 기능 추가 (매월 자동 결제)
작업: 정기 구독 프로세스, 자동 결제, 구독 관리 개발
일정: 8주
비용: 3,200만원
→ 이 정도 규모는 별도 프로젝트로 관리
```

**4) 예방적 유지보수 (Preventive) - 5%**

**시나리오:** 성능 최적화
```
상황: 느리지는 않지만, 코드 복잡도 증가로 미래 문제 예상
작업: 코드 리팩토링, 데이터베이스 인덱스 최적화
일정: 2주
비용: 개발자 2명 × 2주 = 480만원
효과: 향후 유지보수 시간 30% 단축 예상
```

**유지보수 비용 추이:**

```
Year 1 (개발 완료 후):
- 수정적: 20건 × 평균 100만원 = 2,000만원
- 적응적: 5건 × 평균 400만원 = 2,000만원
- 완전적: 3건 × 평균 1,500만원 = 4,500만원
- 예방적: 2건 × 평균 300만원 = 600만원
총 유지보수 비용: 9,100만원 (개발비 5억 대비 18%)

Year 2-5:
- 연평균 유지보수 비용: 1억원
- 5년 누적: 5억원 (개발비와 동일)
```

**PM 교훈:**
- 초기 설계/코드 품질이 유지보수 비용 결정
- 완전적 유지보수(신기능)가 50% 차지 → 별도 예산 확보 필요
- 예방적 유지보수 투자 시 장기적으로 비용 절감

기능 요구사항(FR)은 시스템이 "무엇을" 수행해야 하는지를 정의하는 사용자 관점의 기능입니다. 비기능 요구사항(NFR)은 시스템의 품질 속성과 제약조건으로, "어떻게" 수행해야 하는지를 정의합니다.

1. 사용자는 이메일과 비밀번호로 로그인할 수 있다
2. 관리자는 월별 매출 보고서를 조회할 수 있다

비기능 요구사항 예시:
1. 페이지 로딩 시간은 3초 이내여야 한다 (성능)
2. 시스템 가동률은 99.9% 이상이어야 한다 (가용성)
3. 모든 통신은 HTTPS를 사용해야 한다 (보안)
4. 동시 접속자 10,000명을 지원해야 한다 (확장성)

<details>
<summary>모범 답안</summary>

- 소프트웨어는 평균 5~10년 이상 사용되므로 유지보수 기간이 개발 기간보다 훨씬 깁니다
- 완전적 유지보수(신기능 추가)가 전체의 50%를 차지하며, 이는 지속적인 개발과 유사합니다
- 연구에 따르면 생애주기 비용의 70~80%가 유지보수 비용입니다

유지보수 비용 절감 방법:
1. 초기 설계 품질 향상: 높은 응집도와 낮은 결합도의 모듈화 설계, 명확한 인터페이스 정의를 통해 변경 시 영향 범위를 최소화합니다. 문서화를 철저히 하여 유지보수 담당자의 학습 시간을 단축합니다.

2. 코드 품질 관리: 코딩 표준 준수, 코드 리뷰, 단위 테스트 작성(80% 이상 커버리지)을 통해 결함을 조기에 발견하고 코드 가독성을 높입니다. 정적 분석 도구로 코드 복잡도를 관리합니다.

3. 예방적 유지보수 투자: 정기적인 코드 리팩토링, 기술 부채 관리를 통해 장기적으로 유지보수성을 높입니다.
</details>

## 2교시: 개발 방법론 비교 (1.5시간) <!-- 슬라이드 #4~#5 -->

### 이론 (50분)

#### 1. 개발 방법론(Development Methodology)이란?

**정의:**
소프트웨어 개발을 **체계적이고 효율적**으로 수행하기 위한 프로세스, 기법, 도구의 집합

**방법론을 사용하는 이유:**
- 프로젝트 구조화 및 표준화
- 의사소통 개선
- 품질 향상
- 리스크 감소
- 예측 가능성 증가

**PM이 방법론을 알아야 하는 이유:**
- 프로젝트 특성에 맞는 방법론 선택
- 개발팀과의 효과적 협업
- 현실적인 일정 계획
- 적절한 마일스톤 및 산출물 관리

#### 2. 워터폴(Waterfall) 모델

**개념:**
SDLC 단계를 **순차적**으로 진행하며, 이전 단계 완료 후 다음 단계로 진행

**프로세스:**
```
요구분석 → 설계 → 구현 → 테스트 → 배포 → 유지보수
   ↓        ↓      ↓       ↓       ↓
 승인      승인    승인    승인    승인
```

**특징:**
- **문서 중심**: 각 단계마다 상세한 문서 작성
- **단계별 검증**: 단계 완료 시 formal review
- **변경 제한**: 이전 단계로 돌아가기 어려움
- **명확한 마일스톤**: 단계 완료가 마일스톤

**장점:**
✅ 명확한 구조와 진행 상황 파악 용이
✅ 단계별 산출물로 관리 용이
✅ 문서화가 잘 되어 유지보수 유리
✅ 대규모 프로젝트 관리 용이
✅ 계약 기반 프로젝트에 적합

**단점:**
❌ 요구사항 변경 대응 어려움
❌ 변경 비용이 후반부로 갈수록 기하급수적 증가
❌ 실제 작동 소프트웨어는 후반부에야 확인
❌ 고객 피드백 반영 지연
❌ 리스크가 후반부에 발견될 가능성

**적용 상황:**
- ✅ 요구사항이 **명확하고 안정적**
- ✅ 기술이 익숙하고 **위험도가 낮음**
- ✅ 프로젝트 규모가 크고 **복잡도가 높음**
- ✅ 규제/계약으로 **문서화 필수**
- ✅ 팀이 지리적으로 분산

**PM 관점:**
- 일정 계획이 명확 (단계별 기간 산정)
- 진척률 측정 용이 (단계 완료율)
- 변경 통제 엄격 (CCB 필요)
- 리스크: 후반부 통합 시 문제 발견

**예시 프로젝트:**
- 국방/항공 시스템
- 의료 기기 소프트웨어
- 규제가 많은 금융 시스템
- 대규모 ERP 구축

#### 3. V-모델(V-Model)

**개념:**
워터폴의 변형으로, **각 개발 단계에 대응하는 테스트 단계**를 정의

**프로세스:**
```
    요구분석 ←→ 인수 테스트
       ↓             ↑
    시스템설계 ←→ 시스템 테스트
       ↓             ↑
    상세설계 ←→ 통합 테스트
       ↓             ↑
      코딩   ←→ 단위 테스트
```

**대응 관계:**

| 개발 단계 | 테스트 단계 | 검증 내용 |
|----------|------------|-----------|
| 요구분석 | 인수 테스트 | 사용자 요구사항 충족 확인 |
| 시스템설계 | 시스템 테스트 | 전체 시스템 기능/성능 확인 |
| 상세설계 | 통합 테스트 | 모듈 간 인터페이스 확인 |
| 코딩 | 단위 테스트 | 개별 모듈 기능 확인 |

**특징:**
- 왼쪽(개발)과 오른쪽(테스트)이 대칭
- 각 개발 산출물이 테스트 기준이 됨
- 조기 테스트 계획 수립

**장점:**
✅ 테스트를 개발 초기부터 계획
✅ 결함 조기 발견
✅ 각 단계의 검증 명확

**단점:**
❌ 워터폴의 단점 상속 (변경 어려움)
❌ 여전히 순차적 진행

**PM 관점:**
- 테스트 일정/예산을 초기부터 확보
- 각 개발 단계 완료 시 대응 테스트 계획 검토
- 품질 보증에 유리

#### 4. 프로토타입(Prototype) 모델

**개념:**
**시제품을 먼저 만들어** 사용자 피드백을 받고 요구사항을 구체화

**프로세스:**
```
요구사항 수집 → 프로토타입 개발 → 사용자 평가
                      ↑__________________|
                           (반복)
                              ↓
                        최종 시스템 개발
```

**프로토타입 유형:**

**1) 탐색용(Exploratory)**
- 요구사항 불명확 시 사용
- 사용 후 폐기

**2) 진화형(Evolutionary)**
- 프로토타입을 계속 발전시켜 최종 제품으로
- 점진적 개선

**장점:**
✅ 요구사항 불명확 시 유용
✅ 사용자 참여 증대
✅ 조기 피드백
✅ 오해와 누락 감소

**단점:**
❌ 시간/비용 추가
❌ 사용자가 프로토타입을 완제품으로 오해
❌ 개발자가 프로토타입 코드를 그대로 사용하려는 유혹

**적용 상황:**
- 요구사항이 불명확
- 사용자 인터페이스 중심 시스템
- 혁신적/새로운 시스템

#### 5. 나선형(Spiral) 모델

**개념:**
**반복적 개발 + 리스크 분석**을 결합한 모델

**4단계 사이클 (반복):**
```
1. 계획 수립 (Planning)
   ↓
2. 리스크 분석 (Risk Analysis) ← 핵심!
   ↓
3. 개발 (Engineering)
   ↓
4. 평가 (Evaluation)
   ↓_______________|
      (반복)
```

**특징:**
- 각 사이클마다 리스크 평가
- 리스크 높으면 프로토타입 개발
- 프로젝트 진행 중 방법론 조정 가능

**장점:**
✅ 리스크 조기 식별 및 관리
✅ 유연한 프로세스
✅ 대규모 복잡한 프로젝트에 적합

**단점:**
❌ 리스크 분석에 전문성 필요
❌ 관리 복잡도 높음
❌ 소규모 프로젝트에는 과도

**적용 상황:**
- 기술적 리스크가 높음
- 대규모 프로젝트
- 새로운 기술 적용

#### 6. 애자일(Agile)

**배경:**
2001년 "애자일 선언문(Agile Manifesto)" 발표

**애자일 4대 가치:**

1. **프로세스와 도구보다 개인과 상호작용을**
   - 팀 협업, 대면 의사소통 중시
   
2. **포괄적인 문서보다 작동하는 소프트웨어를**
   - 문서도 중요하지만, 작동 코드가 우선
   
3. **계약 협상보다 고객과의 협력을**
   - 고객을 프로젝트 팀의 일원으로
   
4. **계획을 따르기보다 변화에 대응하기를**
   - 계획은 중요하지만, 변화 수용이 더 중요

**애자일 12원칙 (주요 내용):**
- 가치 있는 소프트웨어를 **조기에, 지속적으로** 제공
- **변경 요청 환영** (경쟁 우위 확보)
- 작동하는 소프트웨어를 **2주~2개월** 주기로 제공
- 비즈니스 담당자와 개발자가 **매일 협업**
- **동기 부여된 개인**들로 팀 구성
- **대면 대화**가 가장 효과적
- **작동하는 소프트웨어**가 진척의 주된 척도
- **지속 가능한 개발** 페이스 유지
- **기술적 우수성**과 좋은 설계가 민첩성 향상
- **단순성** (불필요한 작업 최소화)
- 자기 조직화 팀
- 정기적으로 **회고**하고 조정

**애자일 vs 워터폴 비교:**

| 구분 | 워터폴 | 애자일 |
|------|--------|--------|
| **접근** | 순차적, 예측적 | 반복적, 적응적 |
| **요구사항** | 초기에 고정 | 지속적 변경 |
| **문서** | 상세한 문서 | 최소 필요 문서 |
| **고객 참여** | 초기와 후반 | 지속적 참여 |
| **변경** | 어렵고 비용 높음 | 환영하고 빠르게 대응 |
| **테스트** | 후반부 집중 | 매 반복마다 수행 |
| **인도물** | 말미에 한 번 | 주기적으로 여러 번 |
| **팀 구조** | 역할 분리 | 크로스팩셔널 |
| **계획** | 상세한 초기 계획 | 적응적 계획 |
| **적합 상황** | 요구사항 명확,<br>변경 적음 | 요구사항 불명확,<br>변화 많음 |

**PM 관점:**
- 초기 상세 계획 어려움
- 진척률 측정 방식 변경 (기능 단위)
- 이해관계자와의 빈번한 소통 필요
- 범위 관리 방식 변경 (고정 시간, 가변 범위)

#### 7. 스크럼(Scrum)

**개념:**
가장 널리 사용되는 애자일 **프레임워크** (프로세스가 아님)

**스크럼 3대 기둥:**
- **투명성(Transparency)**: 모든 것을 가시화
- **검사(Inspection)**: 주기적으로 점검
- **적응(Adaptation)**: 필요 시 조정

**A. 스크럼 역할(Roles)**

**1) Product Owner (PO)**
- 제품 비전 정의
- Product Backlog 관리 (우선순위)
- 최종 승인 권한
- 1명 (위원회 아님)

**PM과의 관계:**
- PM이 PO 역할을 겸할 수도 있음
- 또는 PM은 여러 스크럼 팀을 총괄

**2) Scrum Master (SM)**
- 스크럼 프로세스 촉진
- 장애물 제거
- 팀 보호 (외부 간섭 차단)
- 코치 역할

**PM과의 관계:**
- PM은 전체 프로젝트 관리
- SM은 팀 내부 프로세스 관리

**3) Development Team (개발팀)**
- 크로스셀팅팀 (개발, 테스트, 설계 모두 가능)
- 자기 조직화
- 3~9명 (이상적: 7±2명)
- 모든 구성원이 "개발자"

**B. 스크럼 산출물(Artifacts)**

**1) Product Backlog (제품 백로그)**
- 제품에 필요한 **모든 기능 목록**
- PO가 우선순위 관리
- 지속적으로 업데이트
- 형식: 사용자 스토리

**사용자 스토리 형식:**
```
As a [역할/사용자]
I want [기능]
So that [가치/이유]

예시:
As a 고객
I want 주문 내역을 조회하고
So that 배송 상태를 확인할 수 있다

As a 관리자
I want 사용자 통계를 보고
So that 비즈니스 의사결정을 할 수 있다
```

**2) Sprint Backlog (스프린트 백로그)**
- 현재 Sprint에서 완료할 **목표 목록**
- Product Backlog에서 선택
- Sprint 중간에 변경 불가 (보호)
- 팀이 작업 단위로 분해

**3) Increment (증분)**
- Sprint 종료 시 **완성된 제품 증분**
- "Done" 기준 충족
- 배포 가능한 상태
- 이전 증분에 새 기능 추가

**C. 스크럼 이벤트(Events)**

**1) Sprint (스프린트)**
- **2~4주 고정 기간** (보통 2주)
- 모든 작업의 컨테이너
- Sprint 중간에 목표 변경 불가
- 연속적으로 진행 (Sprint 1, 2, 3...)

**2) Sprint Planning (스프린트 계획)**
- Sprint 시작 시 수행
- 시간: 2주 Sprint → 4시간
- 참석: PO, SM, Dev Team
- 결정 사항:
  - Sprint 목표 (Sprint Goal)
  - 선택할 Backlog 항목
  - 작업 계획

**3) Daily Standup (일일 스크럼)**
- 매일 같은 시간, 같은 장소
- **15분 타임박스**
- **서서** 진행 (Standing Meeting)
- 3가지 질문:
  1. 어제 무엇을 했나?
  2. 오늘 무엇을 할 것인가?
  3. 장애물이 있는가?

**목적:**
- 진행 상황 동기화
- 장애물 조기 식별
- **상태 보고 회의 아님**

**4) Sprint Review (스프린트 리뷰)**
- Sprint 종료 시 수행
- 시간: 2주 Sprint → 2시간
- 참석: 팀 + 이해관계자
- 활동:
  - 완성된 Increment 시연
  - 이해관계자 피드백
  - Product Backlog 조정

**5) Sprint Retrospective (스프린트 회고)**
- Sprint Review 후, 다음 Sprint 전
- 시간: 2주 Sprint → 1.5시간
- 참석: PO, SM, Dev Team (내부만)
- 토론:
  - 잘한 점 (Keep)
  - 개선할 점 (Improve)
  - 시도할 것 (Try)

**목적:**
- 프로세스 지속 개선
- 팀 성장

**D. 번다운 차트(Burndown Chart)**

**개념:**
남은 작업량을 시간에 따라 표시한 그래프

```
작업량(SP)
   ↑
100|╲
   | ╲            이상적 (Ideal)
 50|  ╲__
   |     ╲____   실제 (Actual)
  0|___________╲→ 시간(일)
   0   5   10  14
```

**해석:**
- 실제선이 이상선보다 **위쪽**: 지연 위험
- 실제선이 이상선보다 **아래쪽**: 진척 양호
- 마지막 날 0 도달: Sprint 목표 달성

**PM 활용:**
- 일일 진척 모니터링
- 조기 경보 (지연 징후)
- 팀 속도(Velocity) 분석

#### 8. 칸반(Kanban)

**개념:**
시각화된 보드로 **작업 흐름**을 관리

**칸반 보드 예:**
```
| To Do | In Progress(3) | Review(2) | Done |
|-------|----------------|-----------|------|
| 작업A | 작업B          | 작업D     | 작업F|
| 작업C | 작업E          | 작업G     |      |
|       | 작업H          |           |      |
```
(숫자 = WIP 제한)

**핵심 원칙:**

**1) 시각화(Visualize)**
- 모든 작업을 보드에 표시
- 작업 흐름 가시화

**2) WIP 제한(Limit WIP)**
- Work In Progress 제한
- 예: "In Progress 최대 3개"
- 병목 현상 방지
- 품질 향상

**3) 흐름 관리(Manage Flow)**
- 작업이 원활히 흐르도록
- 병목 지점 식별 및 해결

**4) 명시적 정책(Make Policies Explicit)**
- "Done"의 정의 명확화
- 작업 이동 규칙 명확화

**5) 개선(Improve Collaboratively)**
- 데이터 기반 개선
- 리드 타임, 사이클 타임 측정

**스크럼 vs 칸반:**

| 구분 | 스크럼 | 칸반 |
|------|--------|------|
| **반복** | 고정 Sprint (2~4주) | 연속적 흐름 |
| **역할** | PO, SM, Dev Team | 역할 정의 없음 |
| **변경** | Sprint 중 불가 | 언제든 가능 |
| **측정** | Velocity, Burndown | 리드 타임, 사이클 타임 |
| **WIP 제한** | Sprint 목표로 간접 | 명시적 제한 |
| **적합** | 반복적 개발 | 운영, 유지보수 |

#### 9. XP (eXtreme Programming)

**개념:**
소프트웨어 품질과 고객 만족에 초점을 둔 애자일 방법론

**주요 프랙티스:**

**1) 페어 프로그래밍(Pair Programming)**
- 2명이 1대 컴퓨터로 작업
- 1명은 코딩(Driver), 1명은 리뷰(Navigator)
- 장점: 코드 품질, 지식 공유
- 단점: 2배 인력

**2) 테스트 주도 개발(TDD)**
- 테스트 먼저 작성 → 코드 작성
- Red → Green → Refactor
- 품질 향상, 리팩토링 자신감

**3) 지속적 통합(CI)**
- 하루에 여러 번 통합
- 자동 빌드 + 테스트
- 통합 문제 조기 발견

**4) 리팩토링(Refactoring)**
- 기능 변경 없이 구조 개선
- 기술 부채 관리

**5) 단순한 설계(Simple Design)**
- YAGNI (You Aren't Gonna Need It)
- 현재 필요한 것만 구현

### 예시 (25분)

#### 예시 1: 방법론 선택 - 프로젝트별 비교

**시나리오 A: 병원 환자 관리 시스템**

**프로젝트 특성:**
- 요구사항이 명확 (법적 규제 많음)
- 의료법, 개인정보보호법 준수 필수
- 안정성과 신뢰성이 최우선
- 변경 발생 시 규제 기관 승인 필요
- 철저한 문서화 필수

**선택된 방법론: 워터폴 + V-모델**

**이유:**
- 규제로 인해 상세한 문서 필요
- 요구사항이 법으로 정해져 변경 적음
- 안전성을 위한 단계별 검증 필수
- 인증 받기 위해 프로세스 준수

**프로젝트 계획:**
요구분석 3개월 → 설계 2개월 → 구현 6개월 → 테스트 3개월 → 배포 1개월

**시나리오 B: 스타트업 모바일 쇼핑앱**

**프로젝트 특성:**
- 요구사항이 불명확 (시장 반응 봐야 함)
- 빠른 출시 필요 (경쟁사 대응)
- 사용자 피드백으로 지속 개선
- 작은 팀 (5~7명)
- 기능 우선순위가 자주 변경

**선택된 방법론: 애자일 스크럼**

**이유:**
- 빠른 MVP(Minimum Viable Product) 출시
- 시장 반응에 따라 기능 조정 가능
- 2주마다 배포하여 사용자 피드백 수집
- 작은 팀에 적합

**프로젝트 계획:**
Sprint 1-2 (4주): MVP (회원가입, 상품 조회, 주문)
Sprint 3-4 (4주): 결제, 리뷰, 위시리스트
Sprint 5-6 (4주): 추천, 알림, 쿠폰
(계속 반복)

**시나리오 C: 레거시 시스템 고도화**

**프로젝트 특성:**
- 요구사항은 대체로 명확
- 하지만 기술적 리스크 높음 (오래된 코드)
- 기존 시스템과의 통합 불확실성
- 단계적 전환 필요
- 운영 중인 시스템이라 장애 발생 시 업무 중단

**선택된 방법론: 나선형 + 프로토타입**

**이유:**
- 각 사이클마다 리스크 분석 및 프로토타입으로 검증
- 점진적 마이그레이션 (모듈별)
- 리스크 높은 부분은 프로토타입으로 사전 검증
- 병행 운영 기간 확보

**프로젝트 계획:**
Cycle 1: 리스크 분석, 아키텍처 검증 (프로토타입)
Cycle 2: 모듈 A 마이그레이션
Cycle 3: 모듈 B 마이그레이션
(계속 반복)

**방법론 선택 기준표:**

| 프로젝트 특성 | 워터폴 | 애자일 | 나선형 |
|---------------|--------|--------|--------|
| 요구사항 명확도 | 높음 | 낮음 | 중간 |
| 요구사항 변경 빈도 | 낮음 | 높음 | 중간 |
| 프로젝트 규모 | 대형 | 중소형 | 대형 |
| 기술 리스크 | 낮음 | 낮음-중간 | 높음 |
| 문서화 요구 | 높음 | 낮음 | 중간 |
| 고객 참여도 | 낮음 | 매우 높음 | 높음 |
| 규제/계약 엄격성 | 높음 | 낮음 | 중간 |

#### 예시 2: 스크럼 Sprint 전체 사이클

**프로젝트:** 온라인 예약 시스템
**Sprint 2주 사이클**

**Sprint 시작 전 (Sprint 0 완료)**
- Product Backlog 정리 완료 (우선순위화)
- 팀 구성 완료
- 개발 환경 구축 완료

---

**Sprint 1 시작 (2주 Sprint)**

**Day 1: Sprint Planning (오전 9시~12시, 3시간)**

**참석자:** PO, SM, Dev Team (7명)

**Part 1: Sprint 목표 및 Backlog 선택 (1.5시간)**
```
PO: "이번 Sprint의 목표는 사용자가 회원가입하여 
     예약을 조회할 수 있는 MVP를 완성하는 것입니다."

선택된 Backlog:
1. 사용자 회원가입 (8 SP)
2. 사용자 로그인 (5 SP)
3. 예약 목록 조회 (13 SP)
4. 예약 상세 조회 (8 SP)
총 34 Story Points (팀 Velocity: 35 SP)
```

**Part 2: 작업 분해 (1.5시간)**
```
"사용자 회원가입" (8 SP)을 작업으로 분해:
- 회원가입 API 개발 (4시간)
```

#### 실습 1: 프로젝트별 방법론 선택

**과제:** 다음 프로젝트에 가장 적합한 방법론을 선택하고 이유를 설명하세요.

| 프로젝트 | 특성 | 선택 방법론 | 이유 |
|----------|------|------------|------|
| **A. 항공기 관제 시스템** | - 인명 안전 필수<br>- 규제 엄격<br>- 요구사항 명확<br>- 변경 시 재인증 | | |
| **B. 소셜 미디어 앱** | - 시장 반응 불확실<br>- 빠른 출시 필요<br>- 사용자 피드백 중요<br>- 경쟁 치열 | | |
| **C. 은행 차세대 시스템** | - 대규모 (100억)<br>- 레거시 통합 리스크<br>- 단계적 전환<br>- 운영 중 업무 중단 불가 | | |
| **D. 사내 문서 관리** | - 소규모 (5명)<br>- 요구사항 불명확<br>- 사용자 참여 가능<br>- 점진적 개선 필요 | | |

**해설 (강사용):**

A. **워터폴 + V-모델**
- 이유: 안전성이 최우선, 규제로 인한 상세 문서 필수, 요구사항이 명확하고 변경 시 재인증 필요로 변경 비용 매우 높음. 각 단계의 철저한 검증이 필요하므로 V-모델 적합.

B. **애자일 스크럼**
- 이유: 시장 반응을 빠르게 확인하고 개선해야 하므로 짧은 Sprint로 MVP를 먼저 출시하고 사용자 피드백을 받아 지속 개선. 경쟁이 심한 시장에서는 빠른 출시가 생명.

C. **나선형**
- 이유: 기술적 리스크가 높고(레거시 통합), 프로젝트 규모가 크며, 단계적 전환이 필요. 각 사이클마다 리스크를 분석하고 프로토타입으로 검증한 후 본격 개발. 운영 중인 시스템이므로 안정성 확보가 중요.

D. **애자일 스크럼 또는 칸반**
- 이유: 소규모 팀, 요구사항 불명확, 사용자가 가까이 있어 즉각 피드백 가능. 2주 Sprint로 빠르게 반복하며 실제 필요한 기능 중심으로 개발. 칸반도 가능 (연속적 흐름).

#### 주관식 문제

**Q6. 워터폴 방법론이 적합한 프로젝트의 특성을 3가지 쓰고, 각각 이유를 설명하세요.**

<details>
<summary>모범 답안</summary>
1. **요구사항이 명확하고 변경이 적은 프로젝트**  
   이유: 워터폴은 초기에 모든 요구사항을 확정하고 순차적으로 진행하므로, 중간에 요구사항이 변경되면 비용이 기하급수적으로 증가합니다. 따라서 요구사항이 명확하고 안정적인 프로젝트에 적합합니다.

2. **규제가 엄격하여 문서화가 필수인 프로젝트**  
   이유: 워터폴은 각 단계마다 상세한 문서를 작성하고 승인을 받습니다. 의료, 국방, 항공 등 규제 산업에서는 인증을 위해 철저한 문서화가 필요하므로 워터폴이 적합합니다.

3. **대규모 프로젝트로 명확한 구조와 계획이 필요한 경우**  
   이유: 대규모 프로젝트는 많은 인원과 자원이 투입되므로 명확한 단계별 계획과 마일스톤이 필요합니다. 워터폴은 단계가 명확하고 진척률 측정이 용이하여 대규모 프로젝트 관리에 유리합니다.
</details>

**Q7. 스크럼의 Sprint Retrospective의 목적을 설명하고, PM이 이 이벤트에서 얻을 수 있는 가치를 2가지 쓰세요.**

<details>
<summary>모범 답안</summary>
**목적:**  
Sprint Retrospective는 팀이 지난 Sprint를 돌아보며 프로세스와 팀워크를 개선하기 위한 회고 회의입니다. "무엇을 개선할까?"에 초점을 두고, 잘한 점은 계속하고(Keep), 개선이 필요한 점은 변경하며(Improve), 새로운 것을 시도(Try)합니다.

**PM이 얻을 수 있는 가치:**

1. **팀의 장애물과 병목 지점 파악**  
   팀원들이 솔직하게 어려움을 공유하므로, PM은 팀이 직면한 조직적/기술적 장애물을 조기에 파악할 수 있습니다. 예를 들어 "코드 리뷰가 늦어져 병목"이라면 PM은 프로세스 개선을 지원할 수 있습니다.

2. **지속적 개선 문화 구축**  
   정기적인 회고를 통해 팀이 스스로 문제를 식별하고 해결책을 찾는 자기 조직화 팀으로 성장합니다. PM은 이를 통해 장기적으로 팀 생산성과 사기를 향상시킬 수 있으며, 동일한 문제의 반복을 방지할 수 있습니다.

3. **실행 가능한 액션 아이템 도출**  
   각 회고마다 구체적인 개선 액션아이템이 나오므로, PM은 다음 Sprint에서 개선 효과를 측정하고 프로젝트 전체의 효율성을 점진적으로 높일 수 있습니다.
</details>

**Q8. 같은 프로젝트를 워터폴과 애자일로 각각 수행할 때, 리스크가 발견되는 시점과 대응 방법이 어떻게 다른지 설명하세요.**

<details>
<summary>모범 답안</summary>
**워터폴의 리스크 관리:**
- **발견 시점:** 리스크는 주로 **후반부(테스트 또는 통합 단계)**에 발견됩니다. 요구사항과 설계 단계에서는 문서로만 검토하므로 실제 구현 시의 문제점을 발견하기 어렵습니다.

- **대응 방법:** 후반부에 발견된 리스크는 대응 비용이 매우 높습니다. 예를 들어 아키텍처 문제가 통합 테스트 시 발견되면 전체 재설계가 필요할 수 있고, 이는 일정 지연과 예산 초과로 이어집니다. 변경이 어렵기 때문에 종종 "차선책"으로 해결하게 되어 품질 저하를 초래합니다.

**애자일의 리스크 관리:**
- **발견 시점:** 리스크는 **초기 Sprint에서 조기에** 발견됩니다. 2주마다 실제 작동하는 소프트웨어를 만들고 통합하므로, 기술적/아키텍처 문제를 빠르게 발견할 수 있습니다.

- **대응 방법:** 조기 발견으로 대응 비용이 낮습니다. Sprint 1에서 성능 문제를 발견하면 Sprint 2에서 아키텍 처를 조정할 수 있으며, 전체 프로젝트에 미치는 영향이 적습니다. 또한 우선순위를 조정하여 고위험 기능을 먼저 개발하는 방식으로 리스크를 관리할 수 있습니다.

**결론:**  
워터폴은 "늦게 발견, 비싼 수정", 애자일은 "빠른 발견, 저렴한 수정"이라는 차이가 있습니다. PM 관점에서 애자일은 리스크를 조기에 가시화하고 프로젝트 후반부의 큰 충격을 방지하는 데 유리합니다.
</details>

## 3교시: 요구사항 분석 상세 (1시간) <!-- 슬라이드 #7 -->

### 이론 (50분)

#### 1. 요구사항 분석의 중요성

**연구 결과:**
- 요구사항 단계의 결함 수정 비용: $1
- 설계 단계 발견 시: $5
- 구현 단계 발견 시: $10
- 테스트 단계 발견 시: $50
- 운영 단계 발견 시: $100~$200

**프로젝트 실패의 주 원인:**
- 불완전/부정확한 요구사항: 39%
- 요구사항 변경 관리 실패: 35%
- 사용자 참여 부족: 26%

#### 2. 유스케이스(Use Case)

**정의:**
시스템이 사용자에게 제공하는 **기능의 단위**를 사용자 관점에서 기술

**구성 요소:**
- **액터(Actor)**: 시스템과 상호작용하는 외부 개체 (사용자, 외부 시스템)
- **유스케이스**: 액터가 시스템을 통해 수행하는 작업
- **관계**: Include, Extend, Generalization

**유스케이스 명세서:**
```
유스케이스명: 상품 주문
액터: 고객
사전조건: 고객이 로그인되어 있고, 장바구니에 상품이 있음
사후조건: 주문이 생성되고 결제 대기 상태가 됨

기본 흐름:
1. 고객이 "주문하기" 버튼을 클릭한다
2. 시스템은 배송지 정보를 표시한다
3. 고객이 배송지를 선택/입력한다
4. 시스템은 결제 수단을 표시한다
5. 고객이 결제 수단을 선택한다
6. 시스템은 주문 요약을 표시한다
7. 고객이 "결제하기"를 클릭한다
8. 시스템은 주문을 생성하고 결제 게이트웨이로 이동한다

대체 흐름:
3a. 새로운 배송지 추가
   3a1. 고객이 "새 주소 추가"를 클릭
   3a2. 시스템은 주소 입력 폼을 표시
   3a3. 고객이 주소를 입력하고 저장
   3a4. 기본 흐름 4번으로 이동

예외 흐름:
7a. 재고 부족
   7a1. 시스템은 재고 부족 메시지 표시
   7a2. 유스케이스 종료
```

#### 3. 사용자 스토리(User Story)

**정의:**
애자일에서 사용하는 간결한 요구사항 표현 방식

**템플릿:**
```
As a [역할/사용자]
I want [기능]
So that [가치/이유]
```

**예시:**
```
US-001: 상품 검색
As a 고객
I want 키워드로 상품을 검색하고
So that 원하는 상품을 빠르게 찾을 수 있다

인수 기준(Acceptance Criteria):
- 검색 키워드는 상품명, 브랜드, 카테고리에서 검색
- 검색 결과는 1초 이내에 표시
- 검색 결과는 관련도 순으로 정렬
- 검색어가 없으면 전체 상품 표시
- 검색 결과가 없으면 "검색 결과 없음" 메시지 표시
```

**INVEST 원칙:**
- **I**ndependent: 독립적 (다른 스토리에 의존하지 않음)
- **N**egotiable: 협상 가능 (상세 내용은 논의 가능)
- **V**aluable: 가치 있음 (사용자에게 가치 제공)
- **E**stimable: 추정 가능 (크기 산정 가능)
- **S**mall: 작음 (한 Sprint에 완료 가능)
- **T**estable: 테스트 가능 (인수 기준 명확)

#### 4. 페르소나(Persona)

**정의:**
실제 사용자를 대표하는 **가상의 인물**

**목적:**
- 사용자 이해도 향상
- 기능 우선순위 결정
- 디자인 결정 지원

**페르소나 예시:**
```
이름: 김영희 (35세, 직장인, 여성)
직업: IT 기업 마케팅 팀장
기술 수준: 중급 (스마트폰 능숙, PC 업무 가능)

목표:
- 출퇴근 시간에 필요한 생필품을 빠르게 주문
- 신뢰할 수 있는 배송 (당일/익일 배송)
- 포인트/쿠폰으로 저렴하게 구매

좌절 요인:
- 복잡한 결제 과정
- 배송 지연
- 고객센터 응답 느림

선호하는 채널:
- 모바일 앱 (80%)
- 웹 (20%)

시나리오:
"아침 출근 전 화장품이 떨어진 것을 발견하고 앱으로 주문. 
저녁 퇴근 시 집에서 받고 싶음."
```

#### 5. 요구사항 우선순위 기법

**A. MoSCoW 기법**

| 분류 | 설명 | 비율 | 예시 |
|------|------|------|------|
| **Must have** | 필수, 없으면 시스템 사용 불가 | 60% | 로그인, 주문, 결제 |
| **Should have** | 중요하지만 없어도 사용 가능 | 20% | 위시리스트, 리뷰 |
| **Could have** | 있으면 좋지만 우선순위 낮음 | 10% | 소셜 공유, 추천 |
| **Won't  have** | 이번 릴리스에는 제외 | 10% | AI 챗봇, VR 쇼핑 |

**B. Kano 모델**

**3가지 요구사항 유형:**

1. **기본(Must-be) 요구사항**
   - 없으면 불만족, 있어도 만족도 증가 안 함
   - 예: 웹사이트가 작동함, 검색 기능

2. **성능(Performance) 요구사항**
   - 많을수록 만족도 증가
   - 예: 배송 속도, 검색 정확도

3. **매력적(Attractive) 요구사항**
   - 없어도 불만 없지만, 있으면 큰 만족
   - 예: 무료 선물 포장, 개인화 추천

**C. 가치 vs 노력 매트릭스**

```
높은 가치, 낮은 노력 → 우선 개발 (Quick Wins)
높은 가치, 높은 노력 → 주요 프로젝트
낮은 가치, 낮은 노력 → 여유시 개발
낮은 가치, 높은 노력 → 제외 (Avoid)
```

#### 6. 요구사항 명세서(SRS)

**Software Requirements Specification**

**IEEE 830 표준 구조:**
```
1. 서론
   1.1 목적
   1.2 범위
   1.3 정의, 약어
   1.4 참조 문서

2. 전반적 설명
   2.1 제품 관점
   2.2 제품 기능
   2.3 사용자 특성
   2.4 제약 조건
   2.5 가정 및 의존성

3. 기능 요구사항
   3.1 기능 1
      3.1.1 설명
      3.1.2 입력
      3.1.3 처리
      3.1.4 출력
   3.2 기능 2
   ...

4. 비기능 요구사항
   4.1 성능
   4.2 안전성
   4.3 보안
   4.4 품질 속성

5. 부록
```

**좋은 SRS의 특성:**
- 정확(Correct): 모든 요구사항이 정확
- 명확(Unambiguous): 단 하나의 해석만 가능
- 완전(Complete): 모든 요구사항 포함
- 일관(Consistent): 요구사항 간 모순 없음
- 검증 가능(Verifiable): 테스트로 확인 가능
- 추적 가능(Traceable): 요구사항 출처 및 구현 추적

```
Epic-001: 사용자가 프로젝트의 태스크를 효과적으로 관리할 수 있다
```

**User Stories:**
```
US-001: 태스크 생성
As a 프로젝트 멤버
I want 새로운 태스크를 생성하고

인수 기준:

- 생성 후 태스크 목록에 즉시 표시
Story Points: 3

US-002: 태스크 상태 변경
As a 프로젝트 멤버
```

**Q7. MoSCoW 기법을 사용할 때 PM이 주의해야 할 점을 2가지 쓰고, 그 이유를 설명하세요.**

<details>
<summary>모범 답안</summary>
1. **Must have의 범위를 넓게 잡지 않도록 주의**
   
   이유: 모든 이해관계자가 자신의 요구사항을 "Must"라고 주장하는 경향이 있습니다. Must have는 "이것 없이는 시스템이 의미 없다"는 기준으로 엄격하게 선정해야 합니다. 일반적으로 전체의 60% 이내로 제한하고, 나머지는 Should/Could로 분류해야 MVP(Minimum Viable Product)를 빠르게 출시할 수 있습니다.

2. **우선순위에 대한 이해관계자 합의를 문서화**
   
   이유: 프로젝트 진행 중 "이 기능이 왜 빠졌나?"라는 질문이 반드시 나옵니다. 초기에 MoSCoW 분류에 대해 주요 이해관계자(스폰서, 고객, 사용자 대표)의 승인을 받고 문서화해야 합니다. 이것이 범위 크리프를 방지하고, 나중에 "Why"에 대한 명확한 근거가 됩니다. 특히 Won't have 항목은 "나중에 하기"를 명확히 하여 현재 프로젝트 범위를 보호합니다.

3. **정기적인 재평가**
   
   이유: 시장 상황, 기술 변화, 비즈니스 우선순위 변화로 인해 요구사항 우선순위가 바뀔 수 있습니다. 특히 애자일 프로젝트에서는 각Sprint 전에 Product Backlog를 재검토하여 MoSCoW를 업데이트해야 합니다.
</details>

**Q8. 요구사항 분석 단계에서 발생하는 일반적인 문제 3가지와 각각의 해결 방안을 제시하세요.**

<details>
<summary>모범 답안</summary>
**문제 1: 요구사항이 모호하고 불명확**

해결 방안:
- 프로토타입을 만들어 사용자에게 보여주고 피드백 수집
- "~ 할 수 있다"가 아닌 구체적 동작 명세 (예: "검색 결과는 2초 이내에 표시된다")
- 인수 기준(Acceptance Criteria)을 명확히 작성하여 테스트 가능하게 만들기
- "좋은", "빠른" 같은 주관적 용어 사용 금지, 정량화 (예: "응답시간 2초 이내")

**문제 2: 이해관계자 간 요구사항 충돌**

해결 방안:
- 이해관계자 회의를 통해 충돌하는 요구사항 논의
- 비즈니스 가치와 기술적 실현 가능성을 기준으로 우선순위 결정
- Trade-off 분석 수행 (A를 선택하면 B는 포기)
- 최종 의사결정권자(스폰서, Product Owner) 확정
- 합의된 내용을 요구사항 명세서에 명확히 문서화

**문제 3: 범위 크리프 (Scope Creep) - 요구사항 지속적 추가**

해결 방안:
- 초기에 범위 기준선 확정 및 승인
- 변경 통제 프로세스 수립 (CCB, Change Request Form)
- 새 요구사항은 다음 릴리스/Sprint로 연기
- "No"라고 말하는 용기 (모든 요구를 수용하면 프로젝트 실패)
- Product Backlog에 추가하되 우선순위 재조정
</details>

## 4교시: 소프트웨어 설계 원칙 (1시간) <!-- 슬라이드 #3~#4 -->

### 이론 (50분)

#### 1. 소프트웨어 설계란?

**정의:**
요구사항을 **구현 가능한 구조**로 변환하는 과정

**설계의 목적:**
- 복잡도 관리
- 변경 용이성
- 재사용성 향상
- 유지보수 비용 감소

**설계 vs 아키텍처:**
- 아키텍처: 시스템의 **큰 그림** (구조, 주요 컴포넌트)
- 설계: 각 컴포넌트의 **상세 구조**

#### 2. 좋은 설계의 원칙

**A. 모듈화(Modularity)**

**정의:** 시스템을 **독립적인 모듈**로 분리

**장점:**
- 이해하기 쉬움 (작은 단위)
- 개발 분담 용이
- 재사용 가능
- 테스트 용이
- 변경 영향 최소화

**예시:**
```
온라인 쇼핑몰을 모듈로 분리:
- 사용자 관리 모듈
- 상품 관리 모듈
- 주문 처리 모듈
- 결제 모듈
- 배송 관리 모듈
```

**B. 결합도(Coupling)**

**정의:** 모듈 간 **상호 의존 정도**

**원칙:** 결합도는 **낮을수록** 좋음

**결합도 유형 (낮음 → 높음):**

1. **데이터 결합(Data Coupling)** - 최선
   - 파라미터로 데이터만 전달
   ```java
   calculateTotal(price, quantity)
   ```

2. **스탬프 결합(Stamp Coupling)**
   - 데이터 구조 전체를 전달
   ```java
   processOrder(Order order)
   ```

3. **제어 결합(Control Coupling)**
   - 제어 플래그 전달
   ```java
   process(data, isSpecialMode)
   ```

4. **공통 결합(Common Coupling)**
   - 전역 변수 공유
   ```java
   global orderList // 여러 모듈이 직접 접근
   ```

5. **내용 결합(Content Coupling)** - 최악
   - 다른 모듈 내부 직접 수정
   ```java
   ModuleA.privateField = value // 절대 금지!
   ```

**낮은 결합도 예:**
```java
// 좋은 예 - 인터페이스를 통한 느슨한 결합
interface PaymentGateway {
    PaymentResult process(Payment payment);
}

class OrderService {
    private PaymentGateway gateway; // 인터페이스에만 의존
    
    public void checkout(Order order) {
        Payment payment = order.getPayment();
        gateway.process(payment); // 구체적 구현 모름
    }
}
```

**높은 결합도 예:**
```java
// 나쁜 예 - 구체 클래스에 직접 의존
class OrderService {
    public void checkout(Order order) {
        CreditCardGateway gateway = new CreditCardGateway();
        gateway.setCreditCard(cardNumber, cvv, expiry);
        gateway.connect("https://payment.com");
        gateway.authenticate(apiKey);
        gateway.process(amount);
        // OrderService가 CreditCardGateway의 모든 세부사항을 알아야 함
    }
}
```

**C. 응집도(Cohesion)**

**정의:** 모듈 내부 요소들의 **관련성 정도**

**원칙:** 응집도는 **높을수록** 좋음

**응집도 유형 (낮음 → 높음):**

1. **우연적 응집(Coincidental)** - 최악
   - 관련 없는 기능이 한 모듈에
   ```java
   class Util {
       void printReport() {}
       void calculateTax() {}
       void sendEmail() {}
       // 아무 관계 없는 기능들
   }
   ```

2. **논리적 응집(Logical)**
   - 논리적으로 비슷한 기능
   ```java
   class InputHandler {
       void handleKeyboard() {}
       void handleMouse() {}
       void handleTouch() {}
   }
   ```

3. **시간적 응집(Temporal)**
   - 같은 시간에 실행
   ```java
   class Initializer {
       void initDB() {}
       void initCache() {}
       void loadConfig() {}
   }
   ```

4. **절차적 응집(Procedural)**
   - 순차적 실행
   ```java
   class OrderProcessor {
       void validateOrder() {}
       void calculateTotal() {}
       void saveOrder() {}
   }
   ```

5. **통신적 응집(Communicational)**
   - 같은 데이터 사용
   ```java
   class CustomerReport {
       void printCustomerInfo() {}
       void printOrderHistory() {}
       // 모두 Customer 데이터 사용
   }
   ```

6. **순차적 응집(Sequential)**
   - 출력이 다음 입력
   ```java
   class DataPipeline {
       Data fetchData() {}
       Data transformData(Data data) {}
       void saveData(Data data) {}
   }
   ```

7. **기능적 응집(Functional)** - 최선
   - 단일 목적, 하나의 책임
   ```java
   class Invoice Generator {
       Invoice generate(Order order) {
           // 오직 청구서 생성만 담당
       }
   }
   ```

**높은 응집도 예:**
```java
// 좋은 예 - 주문 관리만 담당
class OrderService {
    Order createOrder(OrderRequest request) {}
    Order getOrder(String orderId) {}
    void cancelOrder(String orderId) {}
    List<Order> getOrdersByUser(String userId) {}
}
```

**낮은 응집도 예:**
```java
// 나쁜 예 - 여러 책임이 섞여 있음
class Manager {
    void createOrder() {}
    void sendEmail() {}
    void generateReport() {}
    void updateInventory() {}
    void calculateShipping() {}
    // 너무 많은 책임
}
```

**결합도 vs 응집도 요약:**
```
좋은 설계 = 낮은 결합도 + 높은 응집도

[모듈 A] ---- 약한 연결 ---- [모듈 B]
   ↑                              ↑
강한 내부 결속              강한 내부 결속
```

#### 3. 아키텍처 패턴

**A. 모놀리식(Monolithic) 아키텍처**

**특징:**
- 모든 기능이 **하나의 애플리케이션**
- 단일 코드베이스
- 단일 배포 단위

**장점:**
✅ 개발 초기 빠르고 간단
✅ 테스트 용이 (하나의 애플리케이션)
✅ 배포 단순 (하나의 패키지)
✅ 디버깅 상대적으로 쉬움

**단점:**
❌ 규모 커지면 복잡도 급증
❌ 부분 수정 시에도 전체 재배포
❌ 확장성 제한 (전체를 확장해야 함)
❌ 기술 스택 변경 어려움
❌ 팀 간 의존성 높음

**적합한 상황:**
- 초기 스타트업, MVP
- 소규모 애플리케이션
- 단순한 비즈니스 로직

**B. 마이크로서비스(Microservices, MSA) 아키텍처**

**특징:**
- 기능별로 **독립적인 작은 서비스**
- 서비스별 독립 배포
- API를 통한 통신

**구조 예:**
```
          [API Gateway]
                |
    ┌───────────┼───────────┐
    |           |           |
[User Service] [Order Service] [Product Service]
    |           |           |
 [User DB]   [Order DB]  [Product DB]
```

**장점:**
✅ 독립적 배포 및 확장
✅ 기술 스택 자유 (서비스별 다른 언어 가능)
✅ 장애 격리 (한 서비스 문제가 전체에 영향 적음)
✅ 팀 독립성 (서비스별 팀 구성)
✅ 빠른 혁신 및 실험

**단점:**
❌ 초기 설정 복잡
❌ 분산 시스템의 복잡도 (네트워크, 데이터 일관성)
❌ 테스트 어려움 (여러 서비스 통합)
❌ 운영 오버헤드 (모니터링, 로깅)
❌ 트랜잭션 관리 복잡

**적합한 상황:**
- 대규모 애플리케이션
- 빠른 변화와 배포 필요
- 독립적 팀 운영
- 높은 확장성 요구

**모놀리식 vs MSA 비교:**

| 구분 | 모놀리식 | MSA |
|------|----------|-----|
| **구조** | 단일 애플리케이션 | 다수 독립 서비스 |
| **배포** | 전체 배포 | 서비스별 배포 |
| **확장** | 전체 확장 | 서비스별 확장 |
| **개발** | 초기 빠름 | 초기 느림 |
| **유지보수** | 후반 어려움 | 서비스별 독립적 |
| **장애** | 전체 영향 | 격리 가능 |
| **팀 구조** | 기능별 팀 | 서비스별 팀 |
| **데이터** | 공유 DB | 서비스별 DB |

**PM 관점:**
- 모놀리식 → MSA 전환은 큰 프로젝트
- 초기엔 모놀리식, 성장하면 MSA 검토
- MSA는 초기 비용 높지만 장기적 확장성 우수

**C. 3계층(3-Tier) 아키텍처**

**구조:**
```
┌──────────────────┐
│ Presentation     │ ← UI, 사용자 인터페이스
│ Layer            │
└────────┬─────────┘
         │
┌────────▼─────────┐
│ Business Logic   │ ← 비즈니스 규칙, 처리
│ Layer            │
└────────┬─────────┘
         │
┌────────▼─────────┐
│ Data Access      │ ← 데이터베이스 접근
│ Layer            │
└──────────────────┘
```

**각 계층의 역할:**

**1) Presentation Layer (프레젠테이션 계층)**
- 사용자 인터페이스
- 사용자 입력 받기
- 결과 표시
- 예: JSP, React, Angular

**2) Business Logic Layer (비즈니스 로직 계층)**
- 비즈니스 규칙 처리
- 데이터 검증
- 트랜잭션 관리
- 예: Service 클래스, 비즈니스 로직

**3) Data Access Layer (데이터 접근 계층)**
- 데이터베이스 CRUD
- 데이터 영속성
- 예: DAO, Repository

**장점:**
- 관심사 분리 (Separation of Concerns)
- 각 계층 독립 개발/테스트 가능
- 유지보수 용이
- 재사용성 향상

**예시:**
```java
// Presentation Layer (Controller)
@RestController
class OrderController {
    @Autowired
    private OrderService orderService;
    
    @PostMapping("/orders")
    public ResponseEntity<Order> createOrder(@RequestBody OrderRequest request) {
        Order order = orderService.createOrder(request);
        return ResponseEntity.ok(order);
    }
}

// Business Logic Layer (Service)
@Service
class OrderService {
    @Autowired
    private OrderRepository orderRepository;
    @Autowired
    private InventoryService inventoryService;
    
    public Order createOrder(OrderRequest request) {
        // 비즈니스 규칙 검증
        if (request.getQuantity() <= 0) {
            throw new IllegalArgumentException("수량은 0보다 커야 합니다");
        }
        
        // 재고 확인
        if (!inventoryService.isAvailable(request.getProductId(), request.getQuantity())) {
            throw new OutOfStockException();
        }
        
        // 주문 생성
        Order order = new Order(request);
        return orderRepository.save(order);
    }
}

// Data Access Layer (Repository)
@Repository
interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByUserId(String userId);
    List<Order> findByStatus(OrderStatus status);
}
```

**D. MVC (Model-View-Controller) 패턴**

**구조:**
```
     [User]
       ↓
   [Controller] ←─── 입력 처리, 흐름 제어
       ↓    ↑
       ↓    ↑
   [Model] ←──────── 데이터, 비즈니스 로직
       ↓
       ↓
     [View] ←──────── 화면 표시
```

**구성 요소:**

**1) Model (모델)**
- 데이터와 비즈니스 로직
- 상태 변경 시 View에 알림
- Controller와 View에 독립적

**2) View (뷰)**
- 사용자 인터페이스
- Model의 데이터를 표시
- 사용자 입력을 Controller로 전달

**3) Controller (컨트롤러)**
- 사용자 입력 처리
- Model 업데이트
- View 선택

**예시:**
```java
// Model
class Product {
    private String id;
    private String name;
    private int price;
    // getters, setters
}

// Controller
@Controller
class ProductController {
    @Autowired
    private ProductService productService;
    
    @GetMapping("/products")
    public String listProducts(Model model) {
        List<Product> products = productService.getAllProducts();
        model.addAttribute("products", products);
        return "product-list"; // View 이름
    }
    
    @PostMapping("/products")
    public String createProduct(@ModelAttribute Product product) {
        productService.save(product);
        return "redirect:/products";
    }
}

// View (product-list.jsp)
<c:forEach items="${products}" var="product">
    <div>${product.name} - ${product.price}원</div>
</c:forEach>
```

**장점:**
- 관심사 분리
- 병렬 개발 가능 (UI와 로직 독립)
- 재사용성 (Model은 여러 View에서 사용)
- 테스트 용이

#### 4. UML 다이어그램 (PM이 알아야 할 수준)

**A. 유스케이스 다이어그램** (이미 3교시에서 다룸)
- 사용자 관점의 기능

**B. 클래스 다이어그램**

**목적:** 시스템의 **정적 구조** 표현

**구성 요소:**
```
┌─────────────────┐
│   Order         │ ← 클래스명
├─────────────────┤
│ - orderId       │ ← 속성 (private)
│ - orderDate     │
│ - totalAmount   │
├─────────────────┤
│ + createOrder() │ ← 메서드 (public)
│ + cancelOrder() │
└─────────────────┘
```

**관계:**
- 연관(Association): 실선
- 의존(Dependency): 점선 화살표
- 일반화(Generalization): 빈 삼각형 화살표 (상속)
- 실체화(Realization): 점선 + 빈 삼각형 (인터페이스 구현)

**C. 시퀀스 다이어그램**

**목적:** 객체 간 **시간 순서**에 따른 상호작용

**예시: 주문 처리**
```
User    Controller    OrderService    PaymentGateway    Database
 |          |              |                 |             |
 |─주문────→|              |                 |             |
 |          |─생성 요청───→|                 |             |
 |          |              |─재고 확인──────────────────→|
 |          |              |←──────────────────────────---|
 |          |              |─결제 요청──────→|             |
 |          |              |←─결제 완료──────|             |
 |          |              |─주문 저장──────────────────→|
 |          |←─주문 완료───|                 |             |
 |←─성공────|              |                 |             |
```

**PM이 UML을 알아야 하는 이유:**
- 개발팀  설계 문서 이해
- 복잡한 로직 검토 가능
- 이해관계자에게 시스템 설명
- 요구사항과 설계의 일치 확인

### 예시 (25분)

#### 예시 1: 높은 결합도 → 낮은 결합도 개선

**Before (높은 결합도):**
```java
class OrderService {
    public void processOrder(Order order) {
        // 구체적인 결제 게이트웨이에 직접 의존
        CreditCardPayment payment = new CreditCardPayment();
        payment.setApiKey("sk_live_xxxxx");
        payment.setEndpoint("https://stripe.com/api");
        payment.charge(order.getAmount(), order.getCard());
        
        // 이메일 발송도 직접 구현
        SmtpEmailSender email = new SmtpEmailSender();
        email.connect("smtp.gmail.com", 587);
        email.authenticate("user", "pass");
        email.send(order.getCustomerEmail(), "Order Confirmed");
        
        // SMS도 직접
        TwilioSms sms = new TwilioSms();
        sms.setAccountSid("ACxxxxx");
        sms.setAuthToken("xxxxx");
        sms.send(order.getCustomerPhone(), "주문이 접수되었습니다");
    }
}
```

**문제점:**
- `OrderService`가 결제, 이메일, SMS 구현 세부사항에 의존
- 결제 게이트웨이 변경 시 `OrderService` 수정 필요
- 테스트 어려움 (실제 API 호출)

**After (낮은 결합도):**
```java
// 인터페이스 정의
interface PaymentGateway {
    PaymentResult process(Payment payment);
}

interface NotificationService {
    void sendEmail(String to, String subject, String body);
    void sendSms(String phone, String message);
}

// OrderService는 인터페이스에만 의존
class OrderService {
    private final PaymentGateway paymentGateway;
    private final NotificationService notificationService;
    
    // 의존성 주입 (Dependency Injection)
    public OrderService(PaymentGateway paymentGateway, 
                       NotificationService notificationService) {
        this.paymentGateway = paymentGateway;
        this.notificationService = notificationService;
    }
    
    public void processOrder(Order order) {
```

#### 실습 2: 모놀리식 vs MSA 선택

**다음 프로젝트에 적합한 아키텍처를 선택하고 이유를 설명하세요.**

| 프로젝트 | 특성 | 모놀리식 or MSA? | 이유 |
|----------|------|-----------------|------|
| **A. 스타트업 MVP** | - 3명 개발팀<br>- 3개월 개발<br>- 사용자 1만명 예상<br>- 예산 5천만원 | | |
| **B. 금융 플랫폼** | - 100명 개발팀<br>- 사용자 1000만명<br>- 일 10억건 거래<br>- 규제 요구사항 많음 | | |
| **C. 전자상거래** | - 현재 모놀리식<br>- 성능 병목 발생<br>- 특정 기능만 확장 필요<br>- 배포 다운타임 문제 | | |

**해설 (강사용):**

**A. 모놀리식**
- 이유:
  - 초기 개발 속도 중요 (MVP빠른 출시)
  - 소규모 팀으로 MSA 복잡도 감당 어려움
  - 사용자 규모가 아직 작음
  - 예산 제한 (MSA 인프라 비용 높음)
- 전략: 모놀리식으로 시작, 성장하면 MSA 검토

**B. MSA**
- 이유:
  - 대규모 팀 (100명)은 MSA로 독립적 개발 필요
  - 높은 트래픽 (10억건)은 서비스별 확장 필요
  - 금융은 장애 격리 중요 (한 기능 문제가 전체 영향 방지)
  - 규제 요구사항이 서비스별로 다를 수 있음
- 주의: 분산 트랜잭션, 데이터 일관성 관리 필요

**C. MSA (단계적 전환, Strangler Pattern)**
- 이유:
  - 성능 병목은 해당 부분만 MSA로 분리
  - 무중단 배포 필요성
  - 일부 기능의 독립적 확장 필요
- 전략:
  1. 병목 서비스(예: 상품 검색)부터 분리
  2. 나머지는 모놀리식 유지
  3. 점진적으로 확대
- 장점: 리스크 최소화, 학습 곡선

#### 실습 3: 3계층 아키텍처 구현

**과제:** 간단한 "도서 대출 시스템"을 3계층 아키텍처로 설계하세요.

**요구사항:**
- 책 대출
- 책 반납
- 대출 가능 여부 확인
- 연체료 계산

**각 계층에 어떤 클래스와 메서드가 필요한지 작성하세요.**

**해설 (강사용):**

```java
// ===== Presentation Layer =====
@RestController
class BookController {
    @Autowired
    private BookService bookService;
    
    @PostMapping("/books/{bookId}/checkout")
    public ResponseEntity<Loan> checkoutBook(
        @PathVariable String bookId,
        @RequestParam String userId
    ) {
        Loan loan = bookService.checkoutBook(bookId, userId);
        return ResponseEntity.ok(loan);
    }
    
    @PostMapping("/loans/{loanId}/return")
    public ResponseEntity<ReturnResult> returnBook(@PathVariable String loanId) {
        ReturnResult result = bookService.returnBook(loanId);
        return ResponseEntity.ok(result);
    }
    
    @GetMapping("/books/{bookId}/available")
    public ResponseEntity<Boolean> isAvailable(@PathVariable String bookId) {
        boolean available = bookService.isBookAvailable(bookId);
        return ResponseEntity.ok(available);
    }
}

// ===== Business Logic Layer =====
@Service
class BookService {
    @Autowired
    private BookRepository bookRepository;
    @Autowired
    private LoanRepository loanRepository;
    
    public Loan checkoutBook(String bookId, String userId) {
        // 비즈니스 규칙 검증
        Book book = bookRepository.findById(bookId)
            .orElseThrow(() -> new BookNotFoundException());
        
        if (!book.isAvailable()) {
            throw new BookNotAvailableException();
        }
        
        if (hasOverdueLoan(userId)) {
            throw new UserHasOverdueException();
        }
        
        // 대출 생성
        Loan loan = new Loan(bookId, userId, LocalDate.now());
        book.setAvailable(false);
        
        bookRepository.save(book);
        return loanRepository.save(loan);
    }
    
    public ReturnResult returnBook(String loanId) {
        Loan loan = loanRepository.findById(loanId)
            .orElseThrow(() -> new LoanNotFoundException());
        
        loan.setReturnDate(LocalDate.now());
        
        // 연체료 계산 (비즈니스 로직)
        int overdueDays = calculateOverdueDays(loan);
        double fine = overdueDays > 0 ? overdueDays * 1000 : 0;
        
        // 책 상태 업데이트
        Book book = bookRepository.findById(loan.getBookId()).get();
        book.setAvailable(true);
        
        bookRepository.save(book);
        loanRepository.save(loan);
        
        return new ReturnResult(fine, overdueDays);
    }
    
    public boolean isBookAvailable(String bookId) {
        Book book = bookRepository.findById(bookId)
            .orElseThrow(() -> new BookNotFoundException());
        return book.isAvailable();
    }
    
    private boolean hasOverdueLoan(String userId) {
        List<Loan> loans = loanRepository.findByUserId(userId);
        return loans.stream().anyMatch(loan -> 
            loan.getReturnDate() == null && 
            loan.getDueDate().isBefore(LocalDate.now())
        );
    }
    
    private int calculateOverdueDays(Loan loan) {
        LocalDate dueDate = loan.getDueDate();
        LocalDate returnDate = loan.getReturnDate();
        if (returnDate.isAfter(dueDate)) {
            return (int) ChronoUnit.DAYS.between(dueDate, returnDate);
        }
        return 0;
    }
}

// ===== Data Access Layer =====
@Repository
interface BookRepository extends JpaRepository<Book, String> {
    List<Book> findByAvailable(boolean available);
    List<Book> findByCategory(String category);
}

@Repository
interface LoanRepository extends JpaRepository<Loan, String> {
    List<Loan> findByUserId(String userId);
    List<Loan> findByReturnDateIsNull(); // 현재 대출 중인 목록
}
```

**계층별 역할 요약:**
- **Presentation**: HTTP 요청 처리, 응답 반환
- **Business Logic**: 대출 규칙, 연체료 계산
- **Data Access**: DB CRUD

해설: 비즈니스 규칙 검증은 Business Logic Layer의 핵심 역할입니다. Presentation Layer는 UI 처리, Data Access Layer는 DB 접근만 담당합니다.
</details>

#### 주관식 문제

**Q6. "낮은 결합도, 높은 응집도"가 좋은 설계인 이유를 각각 설명하고, 이를 달성하기 위한 방법을 2가지씩 제시하세요.**

<details>
<summary>모범 답안</summary>
**낮은 결합도가 좋은 이유:**
모듈 간 의존성이 낮으면 한 모듈의 변경이 다른 모듈에 미치는 영향이 작아집니다. 이는 유지보수성, 재사용성, 테스트 용이성을 높입니다. 예를 들어 결제 모듈을 변경해도 주문 모듈에 영향이 없어야 안전하게 변경할 수 있습니다.

**달성 방법:**
1. 인터페이스 사용: 구체적 구현이 아닌 인터페이스에 의존하면 구현체를 자유롭게 교체할 수 있습니다.
2. 의존성 주입(DI): 객체를 직접 생성하지 않고 외부에서 주입받아 결합도를 낮춥니다.

**높은 응집도가 좋은 이유:**
하나의 모듈이 단일 책임을 가지면 이해하기 쉽고, 변경 시 해당 모듈만 수정하면 됩니다. 관련 없는 기능이 섞이면 변경 이유가 많아져 유지보수가 어렵습니다.

**달성 방법:**
1. 단일 책임 원칙(SRP): 각 클래스는 하나의 변경 이유만 가져야 합니다. 예: UserService는 사용자 관리만, EmailService는 이메일 발송만.
2. 기능별 모듈화: 관련된 기능끼리 묶고, 관련 없는 기능은 분리합니다.
</details>

**Q7. 모놀리식 시스템을 MSA로 전환하려는 PM에게 주의사항 3가지를 조언하세요.**

<details>
<summary>모범 답안</summary>
1. **명확한 비즈니스 가치 확인**
   - MSA 전환은 비용과 시간이 많이 드는 대형 프로젝트입니다
   - "왜" 전환하는지 명확한 이유가 있어야 합니다 (확장성 문제, 배포 빈도, 팀 독립성 등)
   - 단순히 "유행"이 어서가 아닌, ROI를 계산하고 경영진 승인을 받으세요
   - 작은 시스템은 모놀리식이 더 효율적일 수 있습니다

2. **단계적 전환 (Strangler Pattern)**
   - 한 번에 전체를 전환하지 말고, 문제가 되는 부분부터 분리하세요
   - 예: 성능 병목인 "상품 검색"부터 독립 서비스로
   - 각 단계의 성공을 확인한 후 다음 단계로 진행
   - 실패 시 롤백 계획을 항상 준비

3. **인프라와 운영 역량 확보**
   - MSA는 모니터링, 로깅, 서비스 디스커버리, API Gateway 등 인프라가 필수
   - DevOps 역량이 없으면 MSA 운영이 매우 어렵습니다
   - 분산 트랜잭션, 데이터 일관성 문제를 해결할 기술적 역량 필요
   - 팀 교육과 새로운 도구 도입에 충분한 시간 배정
</details>

**Q8. 3계층 아키텍처에서 각 계층이 독립적이어야 하는 이유를 설명하고, PM이 이를 검증하는 방법을 제시하세요.**

<details>
<summary>모범 답안</summary>
**독립성이 중요한 이유:**

1. **변경 영향 최소화**
   - Presentation Layer(UI) 변경이 Business Logic에 영향 주지 않음
   - 예: 웹에서 모바일 앱으로 변경해도 비즈니스 로직은 재사용

2. **병렬 개발 가능**
   - UI 팀, 백엔드 팀, DB 팀이 독립적으로 작업 가능
   - 인터페이스 만 합의하면 동시 개발

3. **테스트 용이**
   - 각 계층을 독립적으로 테스트 가능
   - 예: Business Logic은 UI 없이도 단위 테스트 가능

**PM이 검증하는 방법:**

1. **설계 리뷰 시 확인**
   - Presentation Layer에서 DB를 직접 접근하는가? (❌ 나쁜 설계)
   - Business Logic에 UI 관련 코드(HTML, CSS)가 있는가? (❌)
   - 각 계층이 인터페이스를 통해 통신하는가? (✅ 좋은 설계)

2. **코드 리뷰 체크리스트**
   - Controller에 SQL 쿼리가 있는가?
   - Service에 HttpServletRequest  같은 웹 관련 객체가 있는가?
   - 계층 간 의존 방향이 올바른가? (Presentation → Business → Data)

3. **테스트 커버리지 확인**
   - Business Logic의 단위 테스트가 UI 없이 가능한가?
   - 각 계층의 테스트 커버리지가 80% 이상인가?

4. **아키텍처 준수 도구 사용**
   - ArchUnit 같은 도구로 계층 간 의존성 규칙을 자동 검증
   - 빌드 시 위반 사항이 있으면 실패하도록 설정
</details>

## 5교시: 소프트웨어 테스트 (1시간) <!-- 슬라이드 #6 -->

### 이론 (50분)

#### 1. 소프트웨어 테스트의 중요성

**테스트의 목적:**
- 결함 발견 및 제거
- 품질 확인
- 리스크 감소
- 고객 신뢰 확보

**테스트 비용 vs 결함 수정 비용:**
```
결함 발견 시점     수정 비용
─────────────────────────
요구사항 단계   →    $1
설계 단계       →    $5
구현 단계       →    $10
테스트 단계     →    $50
운영 단계       →    $100~$200
```

**결론:** 테스트는 비용이 아닌 **투자**

#### 2. 테스트 레벨 (Test Levels)

**A. 단위 테스트 (Unit Test)**

**정의:** 개별 함수/메서드 단위의 테스트

**특징:**
- 가장 작은 테스트 단위
- 개발자가 작성
- 자동화 필수
- 빠른 실행 (수 초 이내)

**예시:**
```java
// 테스트 대상 코드
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
    
    public int divide(int a, int b) {
        if (b == 0) {
            throw new IllegalArgumentException("0으로 나눌 수 없습니다");
        }
        return a / b;
    }
}

// 단위 테스트
@Test
public void testAdd() {
    Calculator calc = new Calculator();
    int result = calc.add(2, 3);
    assertEquals(5, result); // 기대값 5, 실제값 result
}

@Test
public void testDivide() {
    Calculator calc = new Calculator();
    int result = calc.divide(10, 2);
    assertEquals(5, result);
}

@Test
public void testDivideByZero() {
    Calculator calc = new Calculator();
    assertThrows(IllegalArgumentException.class, () -> {
        calc.divide(10, 0);
    });
}
```

**장점:**
✅ 결함 조기 발견
✅ 리팩토링 시 안전망
✅ 문서 역할 (코드 사용법)
✅ 빠른 피드백

**B. 통합 테스트 (Integration Test)**

**정의:** 여러 모듈이 **연동**되는지 테스트

**목적:**
- 모듈 간 인터페이스 검증
- 데이터 흐름 확인

**예시: 주문 서비스 + 재고 서비스**
```java
@Test
public void testOrderCreationWithInventoryCheck() {
    // Given: 재고 10개
    inventoryService.setStock("PRODUCT-001", 10);
    
    // When: 5개 주문
    Order order = orderService.createOrder("PRODUCT-001", 5);
    
    // Then: 주문 성공, 재고 5개 감소
    assertTrue(order.isSuccess());
    assertEquals(5, inventoryService.getStock("PRODUCT-001"));
}

@Test
public void testOrderFailsWhenOutOfStock() {
    // Given: 재고 3개
    inventoryService.setStock("PRODUCT-001", 3);
    
    // When: 5개 주문 시도
    assertThrows(OutOfStockException.class, () -> {
        orderService.createOrder("PRODUCT-001", 5);
    });
    
    // Then: 재고 변동 없음
    assertEquals(3, inventoryService.getStock("PRODUCT-001"));
}
```

**통합 전략:**
- **빅뱅(Big Bang)**: 모든 모듈 한 번에 통합 (위험)
- **상향식(Bottom-Up)**: 하위 모듈부터 통합
- **하향식(Top-Down)**: 상위 모듈부터 통합 (스텁 사용)
- **샌드위치**: 상향식 + 하향식 혼합

**C. 시스템 테스트 (System Test)**

**정의:** **전체 시스템**을 대상으로 한 테스트

**특징:**
- 운영 환경과 유사한 환경에서 수행
- 기능적/비기능적 요구사항 모두 검증
- 테스트팀이 수행

**테스트 유형:**
1. **기능 테스트**: 요구사항 명세서 기반
2. **성능 테스트**: 응답시간, 처리량
3. **보안 테스트**: 취약점 점검
4. **사용성 테스트**: UI/UX
5. **호환성 테스트**: 브라우저, OS

**예시: 전자상거래 시스템**
```
테스트 시나리오: 상품 검색부터 주문 완료까지

1. 사용자가 로그인한다
2. "노트북"을 검색한다
3. 검색 결과가 2초 이내에 표시되는지 확인 (성능)
4. 상품 상세 페이지로 이동한다
5. 장바구니에 추가한다
6. 주문하기를 클릭한다
7. 배송지를 입력한다
8. 결제 수단을 선택한다 (신용카드)
9. 결제를 완료한다
10. 주문 완료 이메일을 받는지 확인 (통합)
11. 관리자 화면에서 주문이 생성되었는지 확인
```

**D. 인수 테스트 (Acceptance Test, UAT)**

**정의:** **사용자가 직접** 테스트하여 인수 여부 결정

**목적:**
- 비즈니스 요구사항 충족 확인
- 실제 업무 시나리오 검증
- 최종 배포 전 승인

**특징:**
- 사용자(고객)가 수행
- 실제 업무 환경
- 합격/불합격 판정

**예시: 사내 인사 시스템 UAT**
```
참여자: 인사팀 담당자 3명

테스트 시나리오:
1. 신입사원 정보 등록
   - 성공: 등록 완료
   - 확인: 조직도에 표시됨

2. 연차 신청 및 승인
   - 사원: 연차 신청
   - 팀장: 승인 처리
   - 확인: 연차 차감됨

3. 급여 명세서 조회
   - 각 사원이 자신의 급여만 조회 가능
   - 타인 급여 조회 시 오류 (보안)

인수 기준:
- 모든 시나리오 통과
- 치명적 결함 0건
- 주요 결함 2건 이하
- 사용성 점수 4점 이상 (5점 만점)
```

#### 3. V-모델과 테스트

**V-모델:** 각 개발 단계와 대응하는 테스트 단계

```
요구사항 분석   ←────→   인수 테스트 (UAT)
    ↓                          ↑
시스템 설계    ←────→   시스템 테스트
    ↓                          ↑
상세 설계      ←────→   통합 테스트
    ↓                          ↑
구현(코딩)     ←────→   단위 테스트
```

**핵심:**
- 왼쪽(개발)에서 정의한 것을 오른쪽(테스트)에서 검증
- 요구사항 정의 시 인수 기준도 함께 작성

#### 4. 블랙박스 vs 화이트박스 테스트

**A. 블랙박스 테스트 (Black Box Testing)**

**정의:** **내부 구조 모르고** 입출력만 검증

**기법:**
1. **동등 분할(Equivalence Partitioning)**
   - 입력을 유효/무효 그룹으로 분할
   - 각 그룹에서 대표값 테스트

**예시: 나이 입력 (0~120 유효)**
```
그룹 1: 유효 (1~120) → 대표값: 50 테스트
그룹 2: 무효 - 음수 → 대표값: -1 테스트
그룹 3: 무효 - 초과 → 대표값: 150 테스트
```

2. **경계값 분석(Boundary Value Analysis)**
   - 경계값 근처에서 결함 많이 발생
   - 최소, 최소-1, 최소+1, 최대-1, 최대, 최대+1 테스트

**예시: 나이 입력 (1~120)**
```
테스트 값: 0, 1, 2, 119, 120, 121
```

3. **결정 테이블(Decision Table)**
   - 조건 조합에 따른 동작 정의

**예시: 할인 규칙**
```
조건                  TC1  TC2  TC3  TC4
────────────────────────────────────
회원인가?            Y    Y    N    N
10만원 이상?         Y    N    Y    N
────────────────────────────────────
할인율               20%  10%  5%   0%
```

**B. 화이트박스 테스트 (White Box Testing)**

**정의:** **내부 구조(코드)** 기반 테스트

**기법:**
1. **문장 커버리지 (Statement Coverage)**
   - 모든 코드 라인 실행
   - 목표: 80% 이상

2. **분기 커버리지 (Branch Coverage)**
   - 모든 if/else 분기 실행
   - 목표: 70% 이상

**예시:**
```java
public String getGrade(int score) {
    if (score >= 90) {          // 분기 1
        return "A";
    } else if (score >= 80) {   // 분기 2
        return "B";
    } else {                    // 분기 3
        return "C";
    }
}

// 100% 분기 커버리지를 위한 테스트 케이스:
// TC1: score=95 → "A" (분기 1)
// TC2: score=85 → "B" (분기 2)
// TC3: score=70 → "C" (분기 3)
```

**PM이 알아야 할 점:**
- 블랙박스: 사용자 관점, 요구사항 검증
- 화이트박스: 개발자 관점, 코드 품질
- 둘 다 필요! (상호 보완)

#### 5. 테스트 케이스 작성

**테스트 케이스 구조:**
```
TC-001: 로그인 성공

전제조건: 
- 테스트 계정 존재 (ID: test@test.com, PW: test1234)

입력:
- 아이디: test@test.com
- 비밀번호: test1234

실행 단계:
1. 로그인 페이지 접속
2. 아이디 입력
3. 비밀번호 입력
4. "로그인" 버튼 클릭

기대 결과:
- 메인 페이지로 이동
- 상단에 "test@test.com님 환영합니다" 표시
- 세션에 사용자 정보 저장

우선순위: 높음
```

**좋은 테스트 케이스:**
- 재현 가능 (누가 해도 같은 결과)
- 독립적 (다른 테스트에 영향 안 받음)
- 명확한 기대 결과
- 하나의 목적만 (여러 검증 X)

#### 6. 테스트 자동화

**자동화가 필요한 이유:**
- 반복 테스트 비용 절감
- 빠른 피드백
- 사람의 실수 방지
- 회귀 테스트(Regression Test) 용이

**자동화 피라미드:**
```
        /\
       /  \    E2E 테스트 (10%)
      /    \   - 느림, 비용 높음, 불안정
     /------\
    /        \   통합 테스트 (20%)
   /          \  - 중간 속도
  /------------\
 /              \ 단위 테스트 (70%)
/______________\ - 빠름, 비용 낮음, 안정적
```

**권장 비율:**
- 단위 테스트 70%
- 통합 테스트 20%
- E2E 테스트 10%

**테스트 대상 코드:**
```java
public class OrderCalculator {
    public double calculateTotal(List<OrderItem> items, String couponCode) {
        double subtotal = items.stream()
            .mapToDouble(item -> item.getPrice() * item.getQuantity())
            .sum();
        
        double discount = 0;
        if (couponCode != null) {
            if (couponCode.equals("WELCOME10")) {
                discount = subtotal * 0.1; // 10% 할인
            } else if (couponCode.equals("VIP20")) {
                discount = subtotal * 0.2; // 20% 할인
            }
        }
        
        double total = subtotal - discount;
        return total;
    }
}
```

```java
@SpringBootTest
@Transactional
public class OrderIntegrationTest {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private InventoryRepository inventoryRepository;
    
    @Autowired
    private PaymentGateway paymentGateway;
    
    @MockBean
    private EmailService emailService; // 실제 이메일 발송은 Mock
    
    @Test
    @DisplayName("주문 생성 성공 시나리오")
    void testSuccessfulOrderCreation() {
        // Given: 초기 재고 설정
        Inventory inventory = new Inventory("PRODUCT-001", 100);
        inventoryRepository.save(inventory);
        
        OrderRequest request = OrderRequest.builder()
            .productId("PRODUCT-001")
            .quantity(5)
            .customerId("CUST-001")
            .paymentMethod("CREDIT_CARD")
            .build();
        
        // When: 주문 생성
        Order order = orderService.createOrder(request);
        
        // Then: 검증
        // 1. 주문이 생성되었는가?
        assertNotNull(order.getOrderId());
        assertEquals(OrderStatus.COMPLETED, order.getStatus());
        
        // 2. 재고가 감소했는가?
        Inventory updatedInventory = inventoryRepository
            .findByProductId("PRODUCT-001").get();
        assertEquals(95, updatedInventory.getQuantity());
        
        // 3. 결제가 처리되었는가?
        // (실제 환경에서는 결제 게이트웨이 조회)
        assertTrue(order.getPayment().isPaid());
        
        // 4. 이메일이 발송되었는가?
        verify(emailService, times(1))
            .sendOrderConfirmation(eq("CUST-001"), any(Order.class));
    }
    
    @Test
    @DisplayName("재고 부족 시 주문 실패")
    void testOrderFailsWhenOutOfStock() {
        // Given: 재고 3개만 있음
        Inventory inventory = new Inventory("PRODUCT-002", 3);
        inventoryRepository.save(inventory);
        
        OrderRequest request = OrderRequest.builder()
            .productId("PRODUCT-002")
            .quantity(5) // 5개 주문 시도
            .customerId("CUST-002")
            .build();
        
        // When & Then: 예외 발생
        assertThrows(OutOfStockException.class, () -> {
            orderService.createOrder(request);
        });
        
        // 재고는 변동 없어야 함
        Inventory unchangedInventory = inventoryRepository
            .findByProductId("PRODUCT-002").get();
        assertEquals(3, unchangedInventory.getQuantity());
        
        // 이메일 발송 안 됨
        verify(emailService, never()).sendOrderConfirmation(any(), any());
    }
    
    @Test
    @DisplayName("결제 실패 시 주문 취소 및 재고 복구")
    void testOrderCancellationOnPaymentFailure() {
        // Given
        Inventory inventory = new Inventory("PRODUCT-003", 50);
        inventoryRepository.save(inventory);
        
        // 결제 게이트웨이가 실패하도록 설정 (Mock)
        when(paymentGateway.process(any()))
            .thenReturn(PaymentResult.failure("카드 한도 초과"));
        
        OrderRequest request = OrderRequest.builder()
            .productId("PRODUCT-003")
            .quantity(10)
            .customerId("CUST-003")
            .paymentMethod("CREDIT_CARD")
            .build();
        
        // When & Then: 결제 실패 예외
        assertThrows(PaymentFailedException.class, () -> {
            orderService.createOrder(request);
        });
        
        // 재고는 원래대로 (롤백)
        Inventory restoredInventory = inventoryRepository
            .findByProductId("PRODUCT-003").get();
        assertEquals(50, restoredInventory.getQuantity());
    }
}
```

**Q4. 테스트 자동화 피라미드에서 가장 많은 비중을 차지해야 하는 테스트는?**

#### 주관식 문제

**Q6. 테스트를 조기에 많이 수행해야 하는 이유를 비용 관점에서 설명하고, PM이 이를 프로젝트 계획에 어떻게 반영해야 하는지 서술하세요.**

<details>
<summary>모범 답안</summary>
**비용 관점:**

결함은 발견 시점이 늦을수록 수정 비용이 기하급수적으로 증가합니다.
- 요구사항 단계: $1
- 설계 단계: $5
- 구현 단계: $10
- 테스트 단계: $50
- 운영 단계: $100~$200

예를 들어, 요구사항 오류를 운영 단계에서 발견하면 100~200배의 비용이 듭니다. 이는 코드 수정뿐 아니라 재설계, 재테스트, 재배포, 고객 보상, 신뢰 손실까지 포함하기 때문입니다.

**PM의 프로젝트 계획 반영:**

1. **각 단계별 테스트 계획 수립**
   - 요구사항 단계: 사용자 스토리 인수 기준 명확화
   - 설계 단계: 설계 리뷰 및 프로토타입 테스트
   - 구현 단계: 단위 테스트 작성 필수화 (TDD)
   - 통합 단계: 통합 테스트 자동화

2. **테스트 시간과 예산 할당**
   - 전체 프로젝트 기간의 25~30%를 테스트에 배정
   - "코딩 완료 = 테스트 완료"가 아님을 이해관계자에게 주지
   - 테스트 전담 인력 또는 QA 팀 투입

3. **테스트 자동화 투자**
   - 초기 비용이 들지만 회귀 테스트 비용 대폭 절감
   - CI/CD 파이프라인에 자동 테스트 통합

4. **결함 조기 발견 인센티브**
   - 개발 단계에서 발견한 결함은 긍정적 평가
   - "버그 은폐" 문화 방지, "조기 발견" 문화 조성
</details>

**Q7. 단위 테스트, 통합 테스트, 시스템 테스트, 인수 테스트의 차이를 표로 정리하고, 각각 누가 수행하는지 명시하세요.**

<details>
<summary>모범 답안</summary>

| 구분 | 범위 | 목적 | 수행자 | 환경 | 예시 |
|------|------|------|--------|------|------|
| **단위 테스트** | 개별 함수/메서드 | 로직 정확성 | 개발자 | 로컬 | `calculateTax()` 함수 검증 |
| **통합 테스트** | 여러 모듈 연동 | 인터페이스 검증 | 개발자/QA | 개발/테스트 환경 | 주문 서비스 + 결제 서비스 연동 |
| **시스템 테스트** | 전체 시스템 | 요구사항 충족 | QA 팀 | 스테이징 환경 | 로그인~주문 완료까지 시나리오 |
| **인수 테스트 (UAT)** | 전체 시스템 | 업무 적합성 | 고객/사용자 | 운영과 유사한 환경 | 실제 업무 흐름 수행 및 승인 |

**핵심 차이:**
- 단위: 개발자가 작은 단위로 빠르게
- 통합: 모듈 간 연결점 검증
- 시스템: QA가 전체 기능을 기술적으로 검증
- 인수: 고객이 비즈니스 관점에서 최종 승인
</details>

**Q8. 테스트 자동화를 도입하려는 PM에게 고려사항 3가지와 각각의 이유를 설명하세요.**

<details>
<summary>모범 답안</summary>

1. **초기 투자 비용과 ROI 계산**
   
   **이유:** 테스트 자동화는 초기에 도구 도입, 스크립트 작성, 팀 교육 비용이 듭니다. 하지만 회귀 테스트를 반복할수록 비용이 절감됩니다.
   
   **PM의 판단:**
   - 프로젝트 기간이 짧고 (3개월 이하) 일회성이면 수동 테스트가 나을 수 있음
   - 장기 프로젝트거나 지속적 개발(CI/CD)이면 자동화 필수
   - ROI = (수동 테스트 비용 절감 - 자동화 구축 비용) / 자동화 구축 비용
   - 일반적으로 6개월~1년 후 ROI 달성

2. **적절한 자동화 범위 선정**
   
   **이유:** 모든 테스트를 자동화하는 것은 비효율적입니다. 반복적이고 안정적인 테스트를 우선 자동화해야 합니다.
   
   **자동화에 적합:**
   - 회귀 테스트 (매 릴리스마다 반복)
   - 단위 테스트 (빠르고 안정적)
   - API 테스트 (UI보다 안정적)
   - 스모크 테스트 (핵심 기능 빠른 검증)
   
   **자동화에 부적합:**
   - 자주 변경되는 UI (스크립트 유지보수 비용 높음)
   - 탐색적 테스트 (사람의 직관 필요)
   - 일회성 테스트
   - 사용성 테스트 (주관적 평가)

3. **팀 역량과 유지보수 계획**
   
   **이유:** 자동화 스크립트도 코드이므로 유지보수가 필요합니다. 방치하면 "깨진 테스트"가 쌓여 신뢰를 잃습니다.
   
   **PM의 준비:**
   - 팀원 교육: Selenium, JUnit, Postman 등 도구 교육
   - 전담 인력: 테스트 자동화 엔지니어 배치
   - 유지보수 시간 할당: Sprint마다 "테스트 스크립트 정비" 시간 확보
   - CI/CD 통합: Jenkins, GitHub Actions에서 자동 실행
   - 실패한 테스트 즉시 수정: 방치하면 팀이 테스트를 무시하게 됨
</details>

## 6교시: 형상관리와 PM의 역할 (1시간) <!-- 슬라이드 #7~#8 -->

### 이론 (50분)

#### 1. 형상관리(Configuration Management)란?

**정의:**
소프트웨어 개발 과정에서 발생하는 **산출물의 변경을 체계적으로 관리**하는 활동

**형상 항목(Configuration Item, CI):**
- 소스 코드
- 설계 문서
- 요구사항 명세서
- 테스트 케이스
- 빌드 스크립트

**형상관리의 목적:**
- 변경 이력 추적
- 버전 관리
- 동시 작업 지원
- 이전 버전으로 복구 가능
- 협업 효율성 향상

**형상관리 4대 기능:**
1. **형상 식별(Identification)**: 관리 대상 정의
2. **형상 통제(Control)**: 변경 요청 검토 및 승인
3. **형상 상태 보고(Status Accounting)**: 현재 상태 기록
4. **형상 감사(Audit)**: 무결성 검증

#### 2. 버전 관리 시스템 (VCS: Version Control System)

**A. 버전 관리의 필요성**

**버전 관리 없이 협업하면:**
```
개발자 A: 파일명: "프로젝트_최종.docx"
개발자 B: 파일명: "프로젝트_최종_진짜최종.docx"
개발자 C: 파일명: "프로젝트_최종_진짜최종_20240315.docx"
개발자 A: 파일명: "프로젝트_최최최종.docx"

→ 어느 것이 최신인가? 누가 무엇을 변경했는가? 혼란!
```

**버전 관리 시스템 사용 시:**
```
커밋 해시: a3f2c1 (2024-03-15 14:30)
작성자: 개발자 A
메시지: "로그인 버그 수정"
변경 파일: login.js (15줄 수정)

→ 명확한 이력, 변경 내용, 책임자 추적 가능
```

**B. Git 기본 개념**

**Git의 3가지 영역:**
```
┌─────────────────────────────────────────┐
│ Working Directory (작업 디렉터리)        │
│ - 실제 파일이 있는 곳                    │
│ - 코드 수정 작업                         │
└──────────────┬──────────────────────────┘
               │ git add
               ▼
┌─────────────────────────────────────────┐
│ Staging Area (스테이징 영역, Index)      │
│ - 커밋 예정인 변경사항                   │
│ - 커밋할 파일 선택                       │
└──────────────┬──────────────────────────┘
               │ git commit
               ▼
┌─────────────────────────────────────────┐
│ Repository (저장소)                      │
│ - 커밋 이력 저장                         │
│ - 프로젝트 역사                          │
└─────────────────────────────────────────┘
```

**주요 명령어:**

1. **git init / git clone**
```bash
# 새 저장소 생성
git init

# 기존 저장소 복제
git clone https://github.com/user/repo.git
```

2. **git add / git commit**
```bash
# 파일을 스테이징
git add login.js
git add .  # 모든 변경사항

# 커밋 (스냅샷 저장)
git commit -m "로그인 기능 구현"
```

3. **git status / git log**
```bash
# 현재 상태 확인
git status

# 커밋 이력 조회
git log
git log --oneline  # 간략하게
```

4. **git branch / git checkout**
```bash
# 브랜치 목록
git branch

# 새 브랜치 생성
git branch feature/login

# 브랜치 전환
git checkout feature/login

# 생성과 전환 동시에
git checkout -b feature/payment
```

5. **git merge**
```bash
# feature 브랜치를 main에 병합
git checkout main
git merge feature/login
```

6. **git push / git pull**
```bash
# 원격 저장소로 푸시
git push origin main

# 원격 저장소에서 가져오기
git pull origin main
```

**C. 브랜치 전략 (Branching Strategy)**

**Git Flow:**

프로젝트에서 가장 많이 사용하는 브랜치 전략

```
main (master)     ─●─────────●──────────●─→ (운영)
                   │         │          │
release           │    ●────●│          │
                   │   /      │          │
develop    ────────●──●───────●──────────●─→ (개발)
                  / \  \       \        /
feature/login  ──●   \  \       \      /
feature/payment    ●──●  \       \    /
hotfix                    ●───────●──●
```

**5가지 브랜치 유형:**

1. **main (master)**
   - 운영 환경 배포
   - 항상 안정적인 상태
   - 직접 커밋 금지 (병합만 허용)

2. **develop**
   - 개발의 중심 브랜치
   - 다음 릴리스 준비
   - feature 브랜치들이 병합됨

3. **feature/**
   - 새 기능 개발
   - develop에서 분기
   - 개발 완료 후 develop에 병합
   - 예: `feature/user-login`, `feature/payment`

4. **release/**
   - 릴리스 준비 (QA, 버그 수정)
   - develop에서 분기
   - 완료 후 main과 develop에 병합
   - 예: `release/1.2.0`

5. **bugfix/**
   - 운영 환경 긴급 버그 수정
   - main에서 분기
   - 수정 후 main과 develop에 병합
   - 예: `hotfix/payment-error`

**워크플로우 예:**
```
1. develop에서 feature/login 브랜치 생성
git checkout develop
git checkout -b feature/login

2. 로그인 기능 개발 및 커밋
git add login.js
git commit -m "로그인 API 구현"

3. 개발 완료 후 develop에 병합
git checkout develop
git merge feature/login

4. 릴리스 준비
git checkout -b release/1.0.0
# QA 및 버그 수정...

5. 릴리스 완료 후 main에 병합
git checkout main
git merge release/1.0.0
git tag v1.0.0  # 버전 태그

6. develop에도 병합 (릴리스 중 수정사항 반영)
git checkout develop
git merge release/1.0.0
```

**GitHub Flow (간소화된 전략):**

더 단순한 워크플로우, 소규모 팀이나 빠른 배포에 적합

```
main    ─●───────●─────────●───────●─→
          \     /           \     /
feature    ●───●      feature ●───●
```

**규칙:**
1. main은 항상 배포 가능 상태
2. 작업 시 main에서 브랜치 생성 (feature, bugfix)
3. 작업 완료 후 Pull Request 생성
4. 리뷰 및 테스트 통과 후 main에 병합
5. main 병합 즉시 배포

#### 3. 형상 통제 (Change Control)

**변경 요청 프로세스:**

```
1. 변경 요청서 작성 (Change Request)
   └→ 변경 이유, 영향 범위, 우선순위

2. CCB(Change Control Board) 검토
   └→ PM, 아키텍트, 개발 리더 참여
   └→ 승인/거부/보류 결정

3. 승인 시 개발 진행
   └→ 브랜치 생성
   └→ 개발 및 테스트

4. 코드 리뷰 (Pull Request)
   └→ 다른 개발자가 검토
   └→ Approve → 병합

5. 형상 상태 보고
   └→ 변경 이력 문서화
```

**Pull Request (PR) / Merge Request (MR):**

**정의:** 코드 변경사항을 검토받기 위한 요청

**PR 템플릿 예:**
```markdown
## 변경 사항
- 사용자 로그인 기능 구현
- JWT 토큰 기반 인증

## 관련 이슈
- Closes #123

## 테스트
- [ ] 단위 테스트 작성 완료
- [ ] 통합 테스트 통과
- [ ] 수동 테스트 완료

## 스크린샷
(UI 변경 시 첨부)

## 체크리스트
- [ ] 코드 스타일 준수
- [ ] 문서 업데이트
- [ ] Breaking Change 없음
```

**코드 리뷰의 장점:**
✅ 결함 조기 발견
✅ 코드 품질 향상
✅ 지식 공유
✅ 팀 코딩 스타일 통일

#### 4. PM이 알아야 할 형상관리

**A. PM의 역할**

1. **브랜치 전략 수립**
   - 프로젝트 특성에 맞는 전략 선택
   - Git Flow vs GitHub Flow
   - 배포 빈도 고려

2. **형상 통제 프로세스 정의**
   - CCB 구성 및 운영
   - 변경 요청 승인 기준
   - 긴급 변경(Emergency Change) 절차

3. **릴리스 관리**
   - 버전 번호 체계 (Semantic Versioning)
   - 릴리스 주기 계획
   - 릴리스 노트 작성

4. **이해관계자 소통**
   - 변경 사항 보고
   - 형상 상태 공유
   - 릴리스 일정 조율

**B. Semantic Versioning (유의적 버전)**

**형식:** `Major.Minor.Patch` (예: 2.4.1)

```
2  .  4  .  1
│     │     │
│     │     └─ Patch: 버그 수정 (하위 호환 O)
│     └─────── Minor: 기능 추가 (하위 호환 O)
└──────────── Major: 큰 변경 (하위 호환 X)
```

**예시:**
- `1.0.0` → `1.0.1`: 버그 수정
- `1.0.1` → `1.1.0`: 새 기능 추가 (기존 기능 유지)
- `1.1.0` → `2.0.0`: API 변경 (기존 코드 수정 필요)

**C. 릴리스 노트**

**목적:** 사용자에게 변경 사항 전달

**내용:**
```markdown
# Version 2.1.0 (2024-03-15)

## 새로운 기능
- 다단계 인증(2FA) 지원
- 다크 모드 테마 추가
- 상품 비교 기능

## 개선 사항
- 검색 속도 30% 향상
- 모바일 UI 반응성 개선

## 버그 수정
- 결제 시 쿠폰 중복 적용 문제 해결
- iOS에서 이미지 업로드 오류 수정

## 알려진 이슈
- Chrome 브라우저에서 PDF 다운로드 느림 (다음 버전 수정 예정)

## Breaking Changes
- API 엔드포인트 변경: `/api/v1/users` → `/api/v2/users`
- 마이그레이션 가이드: https://docs.example.com/migration
```

#### 예시 2: 충돌(Conflict) 해결

**시나리오:** 두 개발자가 같은 파일 수정

**상황:**

```
개발자 B: 같은 price.js의 10번 줄을 다르게 수정하고 커밋

```

**개발자 B의 충돌 해결:**
```bash

git pull origin develop

Auto-merging price.js
CONFLICT (content): Merge conflict in price.js
Automatic merge failed; fix conflicts and then commit the result.

# 충돌 파일 확인
git status

On branch develop
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  both modified:   price.js
```

**price.js 파일 내용:**
```javascript
function calculatePrice(quantity) {
<<<<<<< HEAD
    // 개발자 A의 코드
    return quantity * 10000; // 10% 할인
=======
    // 개발자 B의 코드 (내 코드)
    return quantity * 9500; // 5% 할인
>>>>>>> feature/pricing-update
}
```

**충돌 해결:**
```javascript
// 두 개발자가 논의 후 결정: 10% 할인 채택
function calculatePrice(quantity) {
    return quantity * 10000; // 10% 할인 (개발자 A 안 채택)
}
```

**해결 후 커밋:**
```bash
git add price.js
git commit -m "Resolve merge conflict in price.js

- 개발자 A의 10% 할인 정책 채택
- 개발자 B와 협의 완료"

git push origin develop
```

  - 이유: 릴리스전 충분한 QA 기간 필요, 안정성 최우선, 여러 버전 동시 지원
  
- **패키지 소프트웨어**: Microsoft Office, Adobe Photoshop
  - 이유: 정해진 릴리스 주기 (분기별, 반기별), 고객에게 배포 후 수정 어려움
  
- **모바일 앱**: 앱스토어/플레이스토어 출시 앱
  - 이유: 앱 심사 기간 존재, release 브랜치에서 심사 빌드 생성

**GitHub Flow:**

**특징:**
- 2개의 주요 브랜치 (main + feature 브랜치들)
- main은 항상 배포 가능 상태
- 빠르고 단순한 워크플로우
- 지속적 배포(Continuous Deployment)에 최적화

**적합한 프로젝트:**
- **SaaS 웹 서비스**: Gmail, Slack, Notion
  - 이유: 하루 수십 회 배포, 빠른 피드백 반영, 복잡한 브랜치 전략 불필요
  
- **스타트업 MVP**: 초기 제품 개발
  - 이유: 빠른 변화와 실험, 단순한 프로세스, 작은 팀
  
- **오픈소스 프로젝트**: 기여자가 많은 GitHub 프로젝트
  - 이유: Fork & Pull Request 워크플로우와 잘 맞음, 단순한 기여 프로세스

**비교 표:**

| 구분 | Git Flow | GitHub Flow |
|------|----------|-------------|
| 복잡도 | 높음 (5개 브랜치) | 낮음 (2개 브랜치) |
| 릴리스 주기 | 계획적 (주/월/분기) | 즉시 (하루 여러 번) |
| QA 프로세스 | release 브랜치에서 | PR 리뷰 및 자동화 테스트 |
| 적합한 팀 | 대규모, 명확한 역할 | 소규모, 유연한 팀 |
| 학습 곡선 | 높음 | 낮음 |

**Q8. 형상관리가 없는 프로젝트와 있는 프로젝트의 차이를 3가지 예시로 설명하세요.**

<details>
<summary>모범 답안</summary>

**1. 변경 이력 추적**

**형상관리 없음:**
```
파일명: "프로젝트_최종_진짜최종_v3_수정본_0315.docx"

PM: "이 파일이 최신인가요?"
개발자 A: "저는 어제 보낸 버전이 최신인 줄 알았는데요..."
개발자 B: "제가 오늘 아침에 수정한 게 있어서..."
→ 어느 것이 최신인지 불명확, 작업 중복 및 누락 발생
```

**형상관리 있음:**
```
git log --oneline
a3f2c1 (2024-03-15 14:30) Fix login bug - by 개발자 A
b7e8d9 (2024-03-15 10:20) Add payment feature - by 개발자 B
c9f1a2 (2024-03-14 16:00) Update user profile - by 개발자 C

→ 명확한 이력, 누가 언제 무엇을 변경했는지 추적 가능
→ 필요시 이전 버전으로 즉시 복구 (git checkout c9f1a2)
```

**2. 동시 작업 및 충돌 해결**

**형상관리 없음:**
```
개발자 A와 B가 같은 파일을 각자 수정
→ A가 먼저 저장
→ B가 저장하면 A의 변경사항이 덮어써짐
→ A의 작업이 사라짐! 심각한 문제

실제 사례: 2주간 작업한 코드가 덮어써져 손실 → 프로젝트 2주 지연
```

**형상관리 있음:**
```
개발자 A와 B가 각자 브랜치에서 작업
- A: feature/login 브랜치
- B: feature/payment 브랜치

병합 시 Git이 자동으로 충돌 감지:
CONFLICT (content): Merge conflict in app.js

→ 충돌 부분을 명확히 표시
→ 개발자가 협의 후 해결
→ 두 사람의 작업이 모두 반영됨
```

**3. 배포 및 롤백**

**형상관리 없음:**
```
신규 기능 배포 → 운영 환경에 치명적 버그 발생!

PM: "이전 버전으로 돌려야 합니다!"
개발자: "...이전 버전 코드를 어디에 저장했는지 모르겠습니다"
→ 코드를 찾는 동안 시스템 다운 (1시간)
→ 고객 서비스 중단, 매출 손실, 신뢰 하락
```

**형상관리 있음:**
```
v2.1.0 배포 → 버그 발견!

git log
a3f2c1 (tag: v2.1.0) 최신 릴리스 (버그 있음)
b7e8d9 (tag: v2.0.0) 이전 안정 버전

# 즉시 이전 버전으로 롤백
git checkout v2.0.0
./deploy.sh

→ 5분 이내에 안정 버전으로 복구
→ 서비스 다운타임 최소화
→ 버그 수정 후 재배포 (hotfix 브랜치)
```

**결론:**
형상관리는 프로젝트의 안전망입니다. 형상관리 없이는 변경 추적 불가, 협업 혼란, 배포 리스크가 크게 증가하여 프로젝트 실패 확률이 높아집니다.
</details>
