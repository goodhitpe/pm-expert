> **Day 6 핵심 요점**
> 1. **QA vs QC 핵심 차이**: QA는 프로세스(예방), QC는 결과물(탐지). "잘못 만들어지는 것을 막는 것" vs "잘못 만들어진 것을 찾는 것"
> 2. **7가지 품질 도구**: 인과관계도(어골도) / 산점도 / 파레토 차트 / 히스토그램 / 관리도 / 체크시트 / 흐름도
> 3. **품질 비용(COQ)**: 예방비용 + 평가비용 (적합 비용) vs 내부실패 + 외부실패 (부적합 비용)
> 4. **품질 우선 원칙**: 등급(Grade) ≠ 품질(Quality). 낮은 등급이어도 요구사항을 충족하면 품질은 높을 수 있음

## 1교시: 자원 관리 개요 (1.5시간) <!-- 슬라이드 #1~#2 -->

### 이론 (50분)

#### 1. 자원 관리(Resource Management)란?

**정의:**  
프로젝트를 성공적으로 완료하기 위해 필요한 팀 자원과 물적 자원을 식별, 확보, 관리하는 프로세스

**핵심 개념:**
- 자원은 프로젝트의 가장 중요한 자산
- 사람(Team Resources)과 물적 자원(Physical Resources)을 모두 포함
- 효과적인 자원 관리 = 프로젝트 성공의 핵심

#### 2. 자원의 유형

**A. 팀 자원(Team Resources)**
- 프로젝트 팀원
- 핵심 특징: 감정과 동기가 있음, 커뮤니케이션 필요, 성장 가능
- 관리 복잡도: 높음
- 예: 개발자, QA, PM, 디자이너, DBA

**B. 물적 자원(Physical Resources)**
- 장비, 재료, 시설, 인프라
- 핵심 특징: 예측 가능, 표준화, 수량 관리
- 관리 복잡도: 낮음
- 예: 서버, 개발 PC, 라이센스, 사무실, 네트워크 장비

#### 3. 자원 관리의 6개 프로세스

**기획:**
1. 자원 관리 계획 수립
2. 활동 자원 산정

**실행:**
3. 자원 확보
4. 팀 개발
5. 팀 관리

**감시/통제:**
6. 자원 통제

#### 4. 조직 구조와 PM 권한

**A. 기능형 조직**
- PM 권한: 거의 없음 (0~10%)
- 기능 관리자가 모든 권한 보유

**B. 프로젝트형 조직**
- PM 권한: 매우 높음 (85~100%)
- PM이 전체 권한 보유

**C. 매트릭스 조직**
- PM 권한: 25~75%
- PM과 기능 관리자 권한 공유
- 약한/균형/강한 매트릭스

### 예시 (25분)

#### 예시 1: 조직 구조에 따른 자원 확보의 차이

**기능형 조직:**
- 자원 확보: 2주 소요
- PM 통제: 거의 없음
- 갈등: 팀장과 지속적 협상 필요

**프로젝트형 조직:**
- 자원 확보: 1주 소요
- PM 통제: 완전 통제
- 갈등: 없음

**매트릭스 조직:**
- 자원 확보: 3일 소요
- PM 통제: 중간
- 갈등: 주기적 우선순위 협의 필요

#### 예시 2: 팀 자원 vs 물적 자원 관리

**팀 자원 (개발자):**
- 동기 관리: 필수
- 갈등 해결: 필요
- 성과 관리: 지속적 피드백
- 교체 난이도: 매우 어려움

**물적 자원 (서버):**
- 동기부여: 불필요
- 갈등: 없음
- 성과: 단순 모니터링
- 교체: 쉬움

- 실무 작성 실습

### 3교시: 활동 자원 산정 및 확보 (1.5시간)
- RBS (자원 분류 체계)
- 자원 달력
- 자원 확보 방법 (협상, 사전 배정, 외주)
- 다중 프로젝트 환경에서의 자원 경합 관리

### 4교시: 팀 개발 - 터크만 모델 (1.5시간)
- 형성기(Forming) → 폭풍기(Storming) → 규범기(Norming) → 성과기(Performing) → 해산기(Adjourning)
- 각 단계별 팀 특성과 PM 대응 방안
- 동기부여 이론: 매슬로 욕구단계설, 허즈버그 2요인이론, 맥클리랜드 성취동기이론
- 팀 빌딩 기법: 집중 배치, 가상팀, 인정과 보상

### 5교시: 갈등 해결 (1.5시간)
- 5가지 갈등 해결 기법:
  1. 협력/문제해결 (Win-Win) ★가장 이상적
  2. 타협 (Give & Take)
  3. 완화 (공통점 강조)
  4. 강요 (Win-Lose)
  5. 회피 (문제 회피)
- 상황별 적용 방법
- 리더십 스타일: 지시적 → 코칭 → 지원적 → 위임

### 6교시: 종합 실습 (1.5시간)
- RACI 차트 작성 실습
- 갈등 시나리오 롤플레이
- 팀 발전 단계 분석
- 팀별 발표 및 피드백

## 2교시: 자원 관리 계획 및 RACI 차트 (1.5시간) <!-- 슬라이드 #3 -->

### 이론 (50분)

#### 1. 자원 관리 계획서 (Resource Management Plan)

**목적:** 프로젝트 팀 자원과 물적 자원을 어떻게 범주화, 배분, 관리, 방출할지 지침 제공

**주요 구성요소:**

**A. 역할과 책임 (Roles and Responsibilities)**
- 역할(Role): 직함이 아닌 기능적 역할
- 책임(Responsibility): 팀원이 수행해야 하는 작업
- 권한(Authority): 결정, 서명, 승인 범위
- 역량(Competency): 필요한 기술/경험 수준

**B. 프로젝트 조직도 (OBS: Organizational Breakdown Structure)**
- 프로젝트 팀 구성도
- 보고 관계 명확화
- 의사소통 채널 시각화

**C. 팀 헌장 (Team Charter)**
- 팀의 가치관 및 운영 지침
- 의사결정 기준
- 회의 규칙
- 갈등 해결 절차
- **팀 헌장은 팀원들이 스스로 작성하도록 장려** → 주인의식 향상

**D. 자원 확보 계획**
- 내부 자원 vs 외부 조달
- 인력 확보 타임라인
- 특수 장비/시설 획득 방법

**E. 교육 계획**
- 필요 기술 갭 분석
- 교육 방법 및 일정
- 예산 배분

#### 2. RACI 차트 (Responsibility Assignment Matrix)

**RACI란?**
- 프로젝트 활동에 대한 책임자를 명확히 정의하는 매트릭스
- RAM (Responsibility Assignment Matrix)의 가장 일반적인 형태
- 모든 팀원이 자신의 역할을 명확히 알 수 있어 혼선 방지

**RACI 4가지 역할:**

**R (Responsible) - 실행 책임자**
- 작업을 실제로 수행하는 사람
- 한 활동에 여러 명 가능
- "누가 이 일을 실제로 하는가?"
- 예: 개발 활동의 R = 개발자

**A (Accountable) - 최종 책임자**
- 작업의 최종 승인 및 의사결정 담당
- **반드시 1명만** (두 명이면 아무도 책임지지 않음)
- "누가 이 일에 최종 책임을 지는가?"
- 예: 개발 활동의 A = 개발팀장

**C (Consulted) - 자문**
- 작업 전/중에 의견을 구해야 함
- **양방향(2-way) 소통**
- 지나치게 많으면 의사결정 지연
- 예: IT 보안 담당자, 법무팀

**I (Informed) - 통보**
- 결과만 알려주면 됨
- **단방향(1-way) 소통**
- 과하면 정보 과부하, 부족하면 이해관계자 불만
- 예: 스폰서, 상급 관리자

**RACI 작성 규칙:**

| 규칙 | 설명 |
|------|------|
| A는 1명 | 각 활동에 최종 책임자는 반드시 1명 |
| R은 최소 1명 | R 없는 활동은 수행자가 없음 |
| A 없는 활동 금지 | 승인자 없으면 완료 기준 불명확 |
| C 주의 | C가 너무 많으면 의사결정 지연 |
| I 주의 | I가 너무 많으면 정보 과부하 |

#### 3. RACI 차트 작성 프로세스

**Step 1: 활동/인도물 목록 작성 (행)**
- WBS에서 주요 패키지/활동 추출
- 너무 세부적이면 관리 부담, 너무 광범위하면 의미 없음
- 권장: 10-30개 활동

**Step 2: 이해관계자/역할 목록 작성 (열)**
- 개인이 아닌 역할(Role)로 작성
- 예: "홍길동"이 아닌 "개발팀장"

**Step 3: RACI 코드 입력**
- 각 셀에 R, A, C, I 중 하나 입력
- 빈 셀도 의미 있음 (관여 없음)

**Step 4: 검증**
- 각 행(활동): A가 정확히 1명인지 확인
- 각 열(역할): 과부하 역할 없는지 확인
- 빈 행(R이 없는 활동): 없어야 함

### 예시 (25분)

#### 예시 1: IT 시스템 구축 프로젝트 RACI 차트

**프로젝트:** 인사관리 시스템 고도화 (6개월)

**역할 정의:**

| 약어 | 역할 | 담당자 |
|------|------|--------|
| PM | 프로젝트 관리자 | 김철수 |
| SA | 시스템 아키텍트 | 박영희 |
| DEV | 개발팀장 | 이민우 |
| QA | QA 엔지니어 | 최지연 |
| USR | 현업 담당자 | 정수진 |
| MGR | 스폰서/임원 | 홍길동 부장 |

**RACI 차트:**

| 활동 | PM | SA | DEV | QA | USR | MGR |
|------|----|----|-----|----|----|------|
| 요구사항 분석 | A | C | R | I | R | I |
| 시스템 설계 | I | A | R | C | C | I |
| 개발 환경 구축 | C | A | R | I | - | I |
| 기능 개발 | I | C | A | C | I | - |
| 단위 테스트 | I | - | R | A | - | - |
| 통합 테스트 | C | C | R | A | R | I |
| UAT (사용자 수용 테스트) | A | I | C | C | R | I |
| 교육 자료 개발 | A | I | R | C | C | I |
| 운영 이관 | A | C | R | C | I | I |
| 최종 승인 | C | I | I | I | C | A |

**RACI 검증:**
- 각 활동의 A 확인: ✅ 모두 1명
- R 없는 활동: ✅ 없음
- 스폰서(MGR)는 주요 결정에만 관여 → 적절

#### 예시 2: RACI 오류 사례 및 수정

**잘못된 RACI:**

| 활동 | PM | 개발자1 | 개발자2 | 팀장 |
|------|----|----|-----|------|
| 코드 배포 | A | A | R | R |

**문제점:**
- A가 2명 (PM, 팀장) → 누가 최종 책임?)
- 의사결정 충돌 시 해결 불가

**수정 RACI:**

| 활동 | PM | 개발자1 | 개발자2 | 팀장 |
|------|----|----|-----|------|
| 코드 배포 | I | R | R | A |

- A = 팀장 (technical authority)
- PM은 I만 (결과 보고받음)
- 개발자 2명이 R로 실제 수행

### 실습 (20분)

#### 실습 1: RACI 차트 작성

**과제:** 다음 프로젝트 시나리오에 맞는 RACI 차트를 작성하세요.

- QA (品質 보증)
- CLT (클라이언트/현업)

**작성할 활동:**
1. 요구사항 수집
2. DB 설계

5. 통합 테스트

## 3교시: 활동 자원 산정 및 자원 확보 (1.5시간) <!-- 슬라이드 #3~#4 -->

### 이론 (50분)

#### 1. 활동 자원 산정 (Estimate Activity Resources)

**목적:** 각 프로젝트 활동을 수행하기 위해 필요한 자원의 유형과 수량, 특성 결정

**산정 방법:**

**A. 전문가 판단 (Expert Judgment)**
- 유사 프로젝트 경험자의 의견 활용
- 가장 빠르지만 주관적일 수 있음
- 적용: 초기 단계, 불확실성 높은 경우

**B. 대안 분석 (Alternative Analysis)**
- 같은 작업을 다양한 자원 조합으로 수행하는 방법 비교
- 예: 시니어 개발자 1명 vs 주니어 개발자 2명
- 비용, 기간, 리스크 트레이드오프 고려

**C. 게시된 산정 데이터 (Published Estimating Data)**
- 업계 표준 자원 생산성 지표 활용
- 예: 기능 포인트당 개발 시간 벤치마크

**D. 상향식 산정 (Bottom-Up Estimating)**
- WBS 최하위 수준 활동별 자원 산정 후 합산
- 가장 정확하지만 시간 오래 걸림

**E. 프로젝트 관리 정보 시스템 (PMIS)**
- 자원 관리 소프트웨어 (MS Project, Primavera)
- 자원 수요 vs 공급 분석

#### 2. 자원 분류 체계 (RBS: Resource Breakdown Structure)

**정의:** 자원을 범주별로 계층화한 체계

**IT 프로젝트 RBS 예시:**

```
프로젝트 자원
├── 인력 자원
│   ├── 관리
│   │   ├── 프로젝트 관리자
│   │   └── 품질 관리자
│   ├── 개발
│   │   ├── 프론트엔드 개발자
│   │   ├── 백엔드 개발자
│   │   └── 데이터베이스 개발자
│   ├── 품질
│   │   ├── QA 엔지니어
│   │   └── 테스트 자동화 엔지니어
│   └── 지원
│       ├── 시스템 관리자
│       └── 네트워크 엔지니어
└── 물적 자원
    ├── 하드웨어
    │   ├── 개발 서버
    │   ├── 테스트 서버
    │   └── 개발 PC
    ├── 소프트웨어
    │   ├── 개발 도구 라이센스
    │   └── 테스트 도구
    └── 시설
        ├── 개발 공간
        └── 회의실
```

#### 3. 자원 달력 (Resource Calendar)

**정의:** 각 자원의 가용성 및 가용 기간을 나타내는 달력

**포함 정보:**
- 업무일/공휴일
- 교대 근무 시간
- 가용 시간 비율 (예: 50% 투입)
- 휴가 계획

**IT 프로젝트 예시:**

```
개발자 A (시니어 백엔드) 자원 달력:
- 1월: 100% 가용
- 2월: 50% 가용 (다른 프로젝트 지원)
- 3월: 80% 가용 (2/21~2/28 연차)
- 4월: 100% 가용

의미: 2월에는 할당량의 절반만 기대 가능
→ 일정 계획 시 2월 생산성 50% 적용
```

#### 4. 자원 확보 방법

**A. 사전 배정 (Pre-Assignment)**
- 계약서나 프로젝트 헌장에 특정 자원 명시
- 예: "선임 아키텍트 박영희가 이 프로젝트에 배정됨"
- 장점: 확실한 자원 확보
- 단점: 유연성 낮음

**B. 협상 (Negotiation)**
- 기능 관리자, 다른 PM과 협상
- 가장 일반적인 방법
- 매트릭스 조직에서 빈번

**협상 전략:**
1. 자원 필요 시기와 기간을 구체적으로 제시
2. 프로젝트 중요성과 비즈니스 가치 강조
3. 대체 자원 제안 (win-win)
4. 스폰서의 지원 요청

**C. 가상 팀 (Virtual Team)**
- 지리적으로 분산된 팀원들이 원격 협업
- 장점: 전문 인력 지역 제한 없음, 비용 절감
- 단점: 의사소통 어려움, 시간대 차이, 신뢰 구축 필요

**가상팀 성공 요건:**
- 명확한 역할과 책임
- 강력한 의사소통 도구 (Slack, Teams, Zoom)
- 주기적인 화상 회의
- 문화 차이 인식 및 존중

**D. 다중 기준 의사결정 분석 (MCDA)**
- 가용성, 비용, 경험, 역량 등 여러 기준으로 자원 선정
- 점수 기반 평가

| 기준 | 가중치 | 후보A | 후보B |
|------|--------|-------|-------|
| 기술 역량 | 40% | 9 | 7 |
| 가용성 | 30% | 8 | 9 |
| 비용 | 20% | 6 | 8 |
| 의사소통 | 10% | 8 | 6 |
| **총점** | | **7.9** | **7.5** |

#### 5. 자원 부하 분석 및 평준화

**자원 과다 배정 (Resource Over-allocation):**
- 한 자원에 동시에 여러 활동이 배정된 경우
- 예: 개발자가 하루에 200% 작업 배정됨

**자원 평준화 (Resource Leveling):**
- 자원 과다 배정 해소
- 활동을 재조정하여 균등한 자원 사용
- 결과: 일정이 늘어날 수 있음

**자원 평활화 (Resource Smoothing):**
- 여유(Float) 범위 내에서 자원 사용 최적화
- 주요 경로에 영향 없음

## 4교시: 팀 개발 (1.5시간) <!-- 슬라이드 #4~#6 -->

### 이론 (50분)

#### 1. 터크만의 팀 발전 모델 (Tuckman's Model)

**개요:** 브루스 터크만이 1965년에 제시한 팀 발전 5단계 모델

**핵심: 팀은 예측 가능한 단계를 거쳐 성숙한다**

**📊 터크만 팀 발달 5단계 흐름도:**

```
성과
수준
  ↑
높음│                               ★★★★
    │                          ★★        ★★★
    │                     ★★★                 ★★
    │                 ★★★                        ★★★
중간│            ★★★                                  ★
    │       ★★★
낮음│  ★★★
    └──────────────────────────────────────────────────▶ 시간

   [형성기]  [폭풍기]  [규범기]  [성과기]     [해산기]
  Forming   Storming  Norming  Performing  Adjourning

  서로 알아감  갈등 최고  규칙 확립  최고 성과    프로젝트 종료
  역할 불명확  역할 충돌  팀워크↑    자율 운영    성과 정리
  리더 의존   저생산성   신뢰 형성  高생산성     팀 해산

  PM 스타일: 지시형 →  코칭형  →  지원형  →  위임형  → 종료 지원

  ⚠️ 핵심: 모든 팀은 반드시 폭풍기를 통과해야 한다
         갈등 억압 금지 → 폭풍기를 빠르게 통과하는 것이 목표
```

---

**1단계: 형성기 (Forming)**

**특징:**
- 팀원들이 서로 알아가는 단계
- 독립적으로 행동, 서로 조심스러움
- 역할과 책임이 불명확
- 리더에게 의존하는 경향

**팀원 심리:**
- "내 역할이 뭐지?"
- "이 팀에서 나는 어디에 속하는가?"
- "프로젝트 목표는 무엇인가?"

**PM 역할:**
- 명확한 방향과 목표 제시
- 팀원 소개 및 아이스브레이킹 활동
- 역할과 책임 명확화
- 그라운드 룰 설정

---

**2단계: 폭풍기 (Storming)**

**특징:**
- 의견 충돌과 갈등이 표면화
- 권력 투쟁과 우선순위 충돌
- **가장 어려운 단계**
- 팀이 해체될 위험도 있음

**팀원 심리:**
- "이 방식은 틀렸다"
- "내 의견이 존중받지 못한다"
- 불만과 좌절

**발생하는 갈등 유형:**
- 일정 우선순위 갈등
- 기술적 방향성 충돌
- 권한 범위 분쟁

**PM 역할:**
- 갈등을 억압하지 않고 건설적으로 해결
- 정기적인 1:1 면담
- 갈등 해결 미팅 진행
- 공통 목표 재강조

**중요:** 폭풍기를 건너뛰려 하면 안 됨 → 갈등은 자연스러운 성장통

---

**3단계: 규범기 (Norming)**

**특징:**
- 팀 규칙과 규범이 확립됨
- 신뢰와 응집력 형성
- 효과적인 협업 시작
- 역할 상호 이해 완성

**팀원 심리:**
- "이 팀이 어떻게 돌아가는지 이해한다"
- "팀원들을 신뢰할 수 있다"

**PM 역할:**
- 확립된 규범 강화
- 협업 성공 사례 공유
- 팀 자율성 점진적 부여

---

**4단계: 성과기 (Performing)**

**특징:**
- 높은 자율성과 성과
- 목표 지향적 행동
- 효율적 문제 해결
- 강한 팀 정체성

**특이 사항:** 모든 팀이 여기까지 도달하지는 못함

**팀원 심리:**
- "우리 팀은 잘 기능한다"
- "함께라면 해결할 수 있다"

**PM 역할:**
- 위임 리더십 (Delegating Leadership)
- 팀 성과 인정 및 보상
- 장애 요소 제거에 집중

---

**5단계: 해산기 (Adjourning)**

**특징:**
- 프로젝트 종료로 팀 해체
- 성과 기록 및 교훈 정리
- 정서적 이별의 과정

**PM 역할:**
- 공식적 프로젝트 종료 행사
- 개인별 기여 인정
- Lessons Learned 문서화
- 팀원의 다음 단계 지원

#### 2. 동기부여 이론

**A. 매슬로의 욕구단계설 (Maslow's Hierarchy of Needs)**

```
자아실현 욕구 (Self-Actualization)
        ↑
  존경 욕구 (Esteem)
        ↑
  소속 욕구 (Love & Belonging)
        ↑
  안전 욕구 (Safety)
        ↑
  생리적 욕구 (Physiological)
```

**PM 적용:**
- 하위 욕구가 충족되어야 상위 욕구 동기부여 가능
- 급여(생리/안전) 해결 → 인정(존경) → 성장 기회(자아실현)
- 팀원별 현재 욕구 단계 파악 중요

**IT 프로젝트 예시:**

| 욕구 단계 | 팀원 상황 | PM 대응 |
|---------|---------|--------|
| 생리 | 야근 과로 | 업무량 조정 |
| 안전 | 계약 불안 | 고용 안정 보장 |
| 소속 | 팀에서 소외 | 팀 빌딩, 인클루전 |
| 존경 | 기여 미인정 | 공개 칭찬, 포상 |
| 자아실현 | 단순 반복 업무 | 도전 과제 제공 |

---

**B. 허즈버그의 2요인 이론 (Herzberg's Two-Factor Theory)**

**위생 요인 (Hygiene Factors):**
- 없으면 불만족, 있어도 만족은 아님
- 일종의 "최소 필요 조건"
- 급여, 작업 환경, 감독, 보안, 회사 정책, 동료 관계

**동기 요인 (Motivators):**
- 있으면 만족과 동기부여
- 성취감, 인정, 책임감, 성장, 흥미로운 업무

**PM 적용:**
- 위생 요인 충족 필수 (불만족 제거)
- 동기 요인으로 성과 극대화
- 단순히 급여 올린다고 동기부여가 되지 않음

---

**C. 맥그리거의 X-Y 이론 (McGregor's XY Theory)**

| 구분 | X 이론 | Y 이론 |
|------|--------|--------|
| 인간관 | 게으름, 책임 회피 | 능동적, 자율적 |
| 관리 방식 | 통제, 지시, 위협 | 자율, 위임, 참여 |
| 적합 상황 | 단순 반복 작업 | 창의적·지식 집약적 작업 |

**IT 프로젝트에서는 Y 이론 지향:**
- 개발자는 창의적 문제 해결자
- 자율성과 위임이 더 높은 성과

---

**D. 맥클리랜드의 성취동기이론 (McClelland's Theory)**

**3가지 핵심 욕구:**

**1) 성취 욕구 (Need for Achievement)**
- 목표 달성, 도전적 과제 선호
- 적합 역할: 기술 개발, 문제 해결
- 관리: 도전적 목표와 피드백 제공

**2) 권력 욕구 (Need for Power)**
- 타인에게 영향력 행사 욕구
- 적합 역할: 팀리더, 아키텍트
- 관리: 리더십 역할 부여

**3) 친화 욕구 (Need for Affiliation)**
- 좋은 인간관계, 협력 선호
- 적합 역할: 팀 코디네이터, 고객 대면 역할
- 관리: 팀 협업 기회 제공

#### 3. 팀 개발 기법

**A. 집중 배치 (Co-location / War Room)**
- 팀원을 같은 물리적 공간에 배치
- 비공식 의사소통 극대화
- "Osmotic communication" (자연스러운 지식 전이)
- 코로나 이후: 하이브리드 형태로 변화

**B. 가상 팀 (Virtual Team)**
- 지리적 분산 환경에서의 팀 운영
- 필수 요소: 신뢰, 명확한 프로세스, 도구
- 주의: 시간대, 문화, 언어 차이

**C. 인정과 보상 (Recognition and Rewards)**
- 공개적 칭찬 (팀 회의, Slack)
- 개인 기여 인정
- 다양한 보상 형태: 상장, 교육 지원, 인센티브

**원칙:**
- 개인 성과 인정
- 비금전적 보상도 매우 효과적
- 팀 이익보다 개인을 지나치게 우대하면 팀 결속력 해침

**D. 팀 빌딩 활동**
- 팀 저녁 식사, 아웃도어 활동
- 해커톤, 스터디 그룹
- 목표: 신뢰 구축, 소통 활성화

## 5교시: 팀 관리 및 갈등 해결 (1.5시간) <!-- 슬라이드 #5 -->

### 이론 (50분)

#### 1. 프로젝트에서의 갈등 원인

**연구에 따른 갈등 발생 순위 (IT 프로젝트):**

1. **일정 (Schedules)** - "언제까지 해야 하는가?"
2. **우선순위 (Priorities)** - "무엇이 더 중요한가?"
3. **자원 (Resources)** - "누가 이 일을 하는가?"
4. **기술적 의견 (Technical Opinions)** - "어떤 방식이 맞는가?"
5. **관리 절차 (Administrative Procedures)** - "어떤 프로세스를 따를 것인가?"
6. **비용 (Cost)**
7. **성격 차이 (Personal)**

**긍정적 갈등 vs 부정적 갈등:**

| 구분 | 긍정적 갈등 | 부정적 갈등 |
|------|------------|------------|
| 특징 | 아이디어/방법론 충돌 | 개인 공격, 감정 충돌 |
| 결과 | 혁신, 개선된 의사결정 | 팀 분열, 생산성 저하 |
| PM 역할 | 허용 및 촉진 | 즉시 개입 필요 |

**갈등에 대한 현대적 시각:**
- 과거: 갈등은 나쁜 것, 회피해야 함
- 현재: 관리 가능한 갈등은 팀 성장에 도움
- PMBOK: 갈등은 자연스러운 현상, 중요한 것은 해결 방식

#### 2. 5가지 갈등 해결 기법

**기법 1: 협력/문제해결 (Collaborate/Problem Solve)** ⭐최선

- **특징:** Win-Win 접근
- **방법:** 갈등의 근본 원인을 파악하고 모든 당사자가 만족하는 해결책 도출
- **적용 시점:** 시간이 있고, 장기적 관계가 중요할 때
- **장점:** 관계 강화, 창의적 해결책, 완전한 해결
- **단점:** 시간/에너지 많이 소모

**기법 2: 타협 (Compromise/Reconcile)**

- **특징:** Give & Take, 양쪽 모두 일부 양보
- **방법:** 양쪽이 원하는 것의 일부씩을 얻는 중간 지점 찾기
- **적용 시점:** 시간이 제한적이고, Win-Win이 불가능할 때
- **장점:** 상대적으로 빠름, 양쪽 모두 부분적 만족
- **단점:** 완전한 해결이 아닐 수 있음

**기법 3: 완화/수용 (Smooth/Accommodate)**

- **특징:** 공통점 강조, 차이점 축소
- **방법:** 차이점보다 공통점을 부각시켜 갈등 완화
- **적용 시점:** 관계 유지가 결과보다 중요할 때, 일시적 완화 필요 시
- **장점:** 즉각적 긴장 완화
- **단점:** 근본 문제 해결 안 됨, 재발 가능

**기법 4: 강요 (Force/Direct)**

- **특징:** Win-Lose, 한쪽의 의견 강제 관철
- **방법:** 권력을 이용해 자신의 입장 관철
- **적용 시점:** 긴급 상황, 중요한 마감, 안전 문제
- **장점:** 즉각적 결론
- **단점:** 패배자의 반감, 장기 관계 손상, 최소한으로 사용

**기법 5: 회피 (Withdraw/Avoid)** ⚠️최후수단

- **특징:** 갈등 자체를 외면
- **방법:** 이슈를 다른 사람에게 떠넘기거나 무시
- **적용 시점:** 문제가 사소하거나, 더 중요한 이슈에 집중해야 할 때
- **장점:** 즉각적 불편함 회피
- **단점:** 문제 해결 안 됨, 이슈 악화 가능

**5가지 기법 비교:**

| 기법 | 결과 | 관계 | 속도 | 완전성 |
|------|------|------|------|--------|
| 협력 | Win-Win | 강화 | 느림 | 완전 |
| 타협 | 부분 Win | 유지 | 중간 | 부분 |
| 완화 | 유보 | 유지 | 빠름 | 미완 |
| 강요 | Win-Lose | 손상 | 빠름 | 미완 |
| 회피 | 없음 | 현상유지 | 즉시 | 없음 |

#### 3. 갈등 해결 프로세스

**Step 1: 갈등 조기 인식**
- 비언어적 신호 포착 (침묵, 회피, 긴장)
- 팀 분위기 모니터링
- 1:1 면담 정기 실시

**Step 2: 당사자들과 비공개 면담**
- 각자의 시각 청취
- 사실과 감정 분리
- 근본 원인 파악

**Step 3: 당사자 간 대화 촉진**
- 중립적 공간, 중립적 시간
- 규칙 설정 (말 끊지 않기, 공격하지 않기)
- PM은 촉진자(Facilitator)로 역할

**Step 4: 해결책 합의**
- 모든 당사자가 동의하는 해결책
- 구체적인 action item
- 추후 검토 일정 설정

**Step 5: 모니터링**
- 합의된 해결책 이행 확인
- 재발 여부 감시

#### 4. 리더십 스타일과 팀 관리

**상황적 리더십 (Situational Leadership)**

팀/개인의 성숙도에 따라 리더십 스타일 조정:

| 팀원 성숙도 | 리더십 스타일 | 특징 |
|------------|-------------|------|
| 낮음 (신입) | 지시적(Directing) | 명확한 지시, 상세한 설명 |
| 중간 (성장 중) | 코칭(Coaching) | 질문하고 설명, 설득 |
| 높음 (경험 있음) | 지원적(Supporting) | 아이디어 지원, 협의 |
| 매우 높음 (전문가) | 위임(Delegating) | 책임 완전 위임 |

**PM의 대인관계 기술:**

1. **능동적 경청 (Active Listening)**
   - 말하는 내용과 감정 모두 이해
   - 판단 보류, 확인 질문

2. **공감 (Empathy)**
   - 상대방 관점에서 이해
   - 감정적 연결

3. **코칭/멘토링**
   - 답을 주기보다 스스로 찾도록 유도
   - 질문 기술 활용

4. **영향력 (Influencing)**
   - 공식 권한 없이도 결과 달성
   - 설득, 데이터, 관계 활용

#### 5. 팀 성과 평가

**팀 성과 지표:**

| 지표 | 측정 방법 | 목표 |
|------|---------|------|
| 스프린트 속도 | Story points/sprint | 증가 추세 |
| 버그 발생률 | 결함/LOC | 감소 추세 |
| 일정 준수율 | 마일스톤 달성 | > 90% |
| 팀원 만족도 | 분기별 설문 | > 4.0/5.0 |
| 자발적 제안 수 | 개선 제안 건수 | 증가 추세 |

**개인 성과 평가:**
- 기여도 (코드 커밋, 리뷰, 문서)

#### 예시 1: 갈등 해결 시나리오 - 일정 충돌

**상황:**
```
프로젝트: 온라인 쇼핑몰 리뉴얼 (3개월)
상황: 5주차, 일정 2주 지연

개발팀장(이민우): 
"범위를 줄이거나 일정을 연장해야 합니다.
현재 속도로는 계획된 기능을 모두 구현할 수 없습니다."

영업팀(정수진):
"절대로 일정은 변경 불가! 이미 고객에게 약속했습니다.
야근이라도 해서 해결하세요."

PM 김철수:
→ 두 당사자 사이에서 갈등 해결 필요
```

**분석:**
- 갈등 원인: 일정 vs 범위 트레이드오프
- 개발팀: 현실적 일정 필요
- 영업팀: 고객 신뢰 유지 필요
- 공통 목표: 프로젝트 성공

**갈등 해결 프로세스:**

```
Step 1: 개별 면담
- 개발팀: 정확한 지연 원인 파악 (요구사항 추가가 원인)
- 영업팀: 고객과의 계약 조건 (핵심 기능 명시 여부)

Step 2: 데이터 분석
- EVM 분석: SPI = 0.78 (일정 지연 22%)
- 현재 속도로 완료 예상: +3주

Step 3: 협력/문제해결 미팅
참석: PM, 개발팀장, 영업팀

해결책 도출:
Option A: 일정 2주 연장 + 비핵심 기능 다음 릴리즈로 이동
Option B: 팀원 2명 추가 투입 (비용 증가)
Option C: MVP로 출시 + 2주 후 전체 기능 업데이트

선택: Option C (고객과 협의 후)
→ 고객도 동의 (초기 버전 먼저 사용, 빠른 피드백)
```

**결과:** Win-Win-Win
- 고객: 예상 날짜에 MVP 사용 시작
- 영업: 약속 유지 (형태 변경)
- 개발: 현실적 일정

#### 예시 2: 기술적 갈등 해결

**상황:**
```
이슈: 데이터베이스 선택 - MySQL vs PostgreSQL

주니어 개발자(박지성):
"MySQL이 더 빠릅니다. 우리 팀도 잘 압니다."

시니어 DBA(이영수):
"복잡한 쿼리와 트랜잭션 요구사항을 고려하면
PostgreSQL이 훨씬 적합합니다."
```

**PM의 대응:**

```
1. 갈등 파악: 기술적 의견 충돌 (긍정적 갈등)

2. 데이터 기반 해결:
- 각자 자신의 선택에 대한 근거 자료 작성 요청
- POC(Proof of Concept) 진행: 두 DB에 실제 쿼리 테스트

3. 결과 발표 미팅:
- MySQL: 단순 조회는 빠름, 복잡한 조인에서는 느림
- PostgreSQL: 단순 조회 약간 느림, 복잡 쿼리에서 훨씬 빠름
- 현 프로젝트: 복잡한 집계 쿼리 많음 → PostgreSQL 선택

4. 성장 기회:
- 박지성에게 PostgreSQL 학습 시간과 멘토링 제공
- 의사결정 기준 문서화 → 팀 자산으로 활용
```

**정답:** 위생 요인(급여, 근무 환경)은 불만족을 제거하는 요소로, 충족되어도 적극적 만족이나 동기부여로 이어지지 않습니다. 진정한 동기부여를 위해서는 성취감, 인정, 도전적 과제, 성장 기회 등 동기 요인이 필요합니다.

**Q4.** 다음 상황에서 가장 적합한 갈등 해결 기법과 이유를 쓰시오. (10점)

**상황:** 개발 팀장과 현업 담당자가 요구사항 범위에 대해 갈등 중. 프로젝트 최종 납품이 한 달 후이며, 양쪽 모두의 우려가 합리적임.

**정답:** 협력/문제해결(Collaborate). 시간적 여유가 있고, 양쪽의 요구가 모두 타당하므로 근본 원인을 파악하여 Win-Win 해결책을 도출하는 것이 최선입니다. 타협도 가능하지만 협력이 더 완전한 해결을 제공합니다.
