## 1교시: 원가 관리 개요 (이론 + 예시 + 실습 + 퀴즈) <!-- 슬라이드 #1~#3 -->

### 이론 (50분)

#### 1. 원가 관리(Cost Management)란?

**정의:**
프로젝트 원가 관리는 프로젝트를 **승인된 예산 내에서 완료**하기 위해 원가를 계획, 산정, 예산 수립, 통제하는 프로세스입니다.

**핵심 개념:**
- "원가"는 프로젝트 실행에 투입되는 모든 자원의 금전적 가치
- 범위, 일정과 더불어 **Triple Constraint**의 핵심 요소
- 원가 관리 실패 = 프로젝트 실패로 직결
- 예산 초과는 조직의 재무 건전성에 직접 영향

#### 2. 원가 관리의 4개 프로세스

PMBOK 6판 기준, 원가 관리는 **4개 프로세스**로 구성됩니다:

**기획 프로세스 그룹:**
1. **원가 관리 계획 수립 (Plan Cost Management)**
   - 원가를 어떻게 산정, 예산 수립, 통제할지 정의
   - 산출물: 원가 관리 계획서

2. **원가 산정 (Estimate Costs)**
   - 프로젝트 활동 완료에 필요한 자원의 원가 추정
   - 산출물: 원가 산정치, 산정 근거

3. **예산 수립 (Determine Budget)**
   - 활동별 원가를 합산하여 승인된 원가 기준선 설정
   - 산출물: 원가 기준선, 프로젝트 자금 요구사항

**감시/통제 프로세스 그룹:**
4. **원가 통제 (Control Costs)**
   - 원가 기준선 대비 실적 모니터링 및 편차 관리
   - 산출물: 작업 성과 정보, 원가 예측, 변경 요청

#### 3. 원가의 유형

**A. 직접비(Direct Cost) vs 간접비(Indirect Cost)**

**직접비(Direct Cost):**
- 프로젝트에 **직접 귀속**되는 비용
- 프로젝트가 없으면 발생하지 않는 비용
- 예시:
  - 프로젝트 팀원 인건비
  - 프로젝트 전용 장비 구매비
  - 개발용 라이센스 비용
  - 외주 용역비
  - 출장비 (프로젝트 관련)

**간접비(Indirect Cost):**
- **여러 프로젝트가 공유**하는 비용
- 특정 프로젝트에 명확히 배분하기 어려움
- 예시:
  - 사무실 임대료
  - 전기/수도/통신비
  - 공용 인프라 유지보수
  - 관리부서 인건비 (HR, 재무팀)
  - 회사 전체 소프트웨어 라이센스

**간접비 배분 방법:**
- 프로젝트 인건비 비율로 안분
- 프로젝트 예산 비율로 안분
- 인원 수 기준으로 균등 배분

**B. 고정비(Fixed Cost) vs 변동비(Variable Cost)**

**고정비(Fixed Cost):**
- 수량/규모에 **관계없이 일정**한 비용
- 단기적으로 변경 어려움
- 예시:
  - 사무실 임대료 (월 300만원 고정)
  - 소프트웨어 라이센스 (연간 계약)
  - 프로젝트 관리 인건비 (PM 급여)

**변동비(Variable Cost):**
- 작업량/수량에 **비례**하여 변하는 비용
- 프로젝트 규모에 따라 조정 가능
- 예시:
  - 시간당/일당 계약 인력 비용
  - 클라우드 사용료 (사용량 기준)
  - 재료비 (수량에 비례)
  - 출장비 (횟수에 비례)

#### 4. 특수한 원가 개념

**A. 매몰비용(Sunk Cost)**

**정의:**
- 이미 지출되어 **회수 불가능한 비용**
- 과거에 발생했으며 미래 의사결정으로 변경 불가

**중요 원칙:**
- 매몰비용은 향후 의사결정의 기준이 되어서는 **안 됨**
- "이미 많이 투자했으니 계속해야 한다" → **잘못된 판단** (Sunk Cost Fallacy)
- 미래 비용과 편익만으로 의사결정해야 함

**예시:**
```
상황: CRM 시스템 개발 프로젝트
- 현재까지 투자: 3억원
- 완료까지 추가 필요: 5억원
- 예상 편익: 4억원

잘못된 판단: "3억을 이미 투자했으니 끝까지 가야 한다"
올바른 판단: "앞으로 5억을 더 투자해 4억 편익 → 순손실 1억"
→ 프로젝트 중단 고려 (3억은 이미 매몰비용)
```

**B. 기회비용(Opportunity Cost)**

**정의:**
- 특정 대안을 선택함으로써 포기하게 되는 **다른 대안의 가치**
- 명시적으로 지출되지 않지만 고려해야 할 비용

**예시:**
```
선택지:
A 프로젝트: 예상 수익 5억원, 소요 자원 10명×6개월
B 프로젝트: 예상 수익 3억원, 소요 자원 10명×6개월

→ A 선택 시, B의 3억원이 기회비용
→ A의 순편익 = 5억 (실제 수익) - 3억 (기회비용) = 2억원
```

**프로젝트 선택 시 고려사항:**
- 자원(인력, 예산, 시간)이 제한적일 때
- 여러 프로젝트 중 우선순위를 정할 때
- 가장 높은 순편익을 제공하는 프로젝트 선택

#### 5. 생애주기 원가(Life Cycle Cost, LCC)

**정의:**
제품/시스템의 **전체 수명 동안** 발생하는 총비용

**구성 요소:**
1. **개발 비용** (Development Cost)
   - 요구사항 분석, 설계, 구현, 테스트
   - 초기 장비 구매, 인프라 구축
   
2. **운영 비용** (Operating Cost)
   - 인건비 (운영 인력)
   - 클라우드/서버 운영비
   - 라이센스 갱신비
   
3. **유지보수 비용** (Maintenance Cost)
   - 버그 수정
   - 기능 개선
   - 보안 패치
   
4. **폐기 비용** (Disposal Cost)
   - 시스템 종료 비용
   - 데이터 마이그레이션
   - 계약 종료 비용

**LCC의 중요성:**
- 일반적으로 **운영/유지보수 비용 > 개발 비용**
- 연구에 따르면 LCC 중 운영/유지보수가 70~80% 차지
- 초기 개발비만 고려하면 장기적으로 손실

**예시:**
```
웹 애플리케이션 개발 프로젝트
- 개발 비용: 5억원 (1회성)
- 연간 운영비: 1억원 × 10년 = 10억원
- 연간 유지보수: 0.5억원 × 10년 = 5억원
→ LCC = 20억원 (개발비는 25%에 불과)
```

**PM이 LCC를 고려해야 하는 이유:**
- 기술 선택이 LCC에 큰 영향
  - 예: 오픈소스 vs 상용 라이센스
  - 예: 클라우드 vs 온프레미스
- 초기 비용 절감이 장기적으로 더 큰 비용 발생 가능
- 스폰서에게 전체 비용 명확히 제시해야 승인 가능

### 예시 (25분)

#### 예시 1: IT 프로젝트의 원가 분류

**프로젝트:** 신규 전자상거래 플랫폼 구축 (예산 10억원, 6개월)

**직접비 (총 7억원):**
| 항목 | 금액 | 설명 |
|------|------|------|
| 프로젝트 인건비 | 4.5억원 | 개발자 6명, 디자이너 2명, QA 2명 (6개월) |
| 외주 용역비 | 1.5억원 | UI/UX 디자인, 보안 컨설팅 |
| 개발 도구 라이센스 | 0.3억원 | IDE, 테스트 도구, 형상관리 |
| 클라우드 초기 비용 | 0.5억원 | 개발/테스트 환경 6개월 |
| 교육 훈련비 | 0.2억원 | 신기술 교육 (React, Kubernetes) |

**간접비 (총 1.5억원, 프로젝트 규모로 안분):**
| 항목 | 금액 | 설명 |
|------|------|------|
| 사무실 임대료 | 0.6억원 | 전체 임대료의 프로젝트 점유 비율 |
| 관리부서 인건비 | 0.5억원 | HR, 재무, 총무 등 |
| 공용 인프라 | 0.3억원 | 네트워크, 전기, 통신비 등 |
| 기타 경비 | 0.1억원 | 비품, 시설관리비 등 |

**우발 예비 (Contingency Reserve):** 1억원 (알려진 리스크 대비)
**관리 예비 (Management Reserve):** 0.5억원 (미지의 리스크 대비)

**총 프로젝트 예산:** 10억원

#### 예시 2: 매몰비용의 함정 - 실패 프로젝트 사례

**프로젝트:** 차세대 ERP 시스템 구축
**타임라인:**

**Month 0 (프로젝트 시작):**
- 초기 계획: 12개월, 예산 20억원
- 기대 효과: 연간 5억원 비용 절감

**Month 6 (중간 점검):**
- 진행 상황: 30% 완료 (계획: 50%)
- 이미 지출: 12억원 (예산의 60%)
- 남은 작업 예상: 추가 15억원 필요
- 총 예상 비용: 27억원 (초과: 7억원)

**Month 6 의사결정 회의:**

❌ **잘못된 판단 (매몰비용 오류):**
```
CFO: "이미 12억을 투자했습니다. 이제 와서 포기하면 
     12억이 모두 낭비됩니다. 끝까지 진행해야 합니다."
CIO: "여기까지 온 것이 아까워서 멈출 수 없습니다."
→ 계속 진행 결정
```

✅ **올바른 판단 (매몰비용 무시):**
```
분석:
- 이미 지출한 12억은 회수 불가 (매몰비용)
- 앞으로 필요한 비용: 15억
- 예상 편익: 연간 5억 절감 × 5년 = 25억
- 순현재가치(NPV) 분석: 15억 투자로 25억 회수 → 긍정적
- 하지만, 연간 5억 절감이 불확실함 (리스크 높음)

대안 검토:
1. 계속 진행: 추가 15억 투자
2. 프로젝트 중단: 손실 12억에서 멈춤
3. 범위 축소: 핵심 기능만 7억 투자로 완성

→ 범위 축소 안 선택 (총 19억 투자, 최소 기능으로 효과 확보)
```

**Month 18 (프로젝트 종료):**
- 실제 결과: 범위 축소로 19억 투입
- 핵심 기능만 구현하여 연간 3억 절감 달성
- 초기 목표 대비 축소되었으나, 무리한 진행보다 나은 선택

#### 예시 3: 기회비용 고려한 프로젝트 선택

**상황:** IT 부서의 자원(개발팀 10명)으로 한 프로젝트만 수행 가능

**프로젝트 A: 모바일 앱 리뉴얼**
- 예상 투자: 인력 10명 × 6개월
- 예상 수익: 매출 증가로 8억원/년
- NPV (3년): 15억원

**프로젝트 B: 물류 시스템 고도화**
- 예상 투자: 인력 10명 × 6개월
- 예상 수익: 물류비 절감 5억원/년
- NPV (3년): 10억원

**프로젝트 C: 고객 데이터 분석 플랫폼**
- 예상 투자: 인력 10명 × 6개월
- 예상 수익: 마케팅 효율 증대 3억원/년
- NPV (3년): 6억원

**의사결정:**
```
프로젝트 A 선택 시:
- 실제 수익: 15억원
- 기회비용: 10억원 (B를 포기한 대가)
- 순편익: 15억 - 10억 = 5억원

프로젝트 B 선택 시:
- 실제 수익: 10억원
- 기회비용: 15억원 (A를 포기한 대가)
- 순편익: 10억 - 15억 = -5억원 (손실)

→ 프로젝트 A 선택이 최적
```

**추가 고려사항:**
- 리스크 평가: A는 시장 반응 불확실, B는 내부 프로세스로 안정적
- 전략적 정렬: 회사 전략이 "고객 경험 개선"이라면 A가 유리
- 자원 가용성: 6개월 후 자원 확보 가능하다면 순차 진행 고려

#### 예시 4: 생애주기 원가(LCC) 분석

**프로젝트:** 사내 협업 시스템 구축
**비교 대안:**

**대안 1: 상용 SaaS 솔루션 (예: Slack, Microsoft Teams)**

3. 순현재가치(NPV) 관점에서 프로젝트를 계속해야 하는가? (할인율 10%)

4. 대안: 범위 축소하여 7억 투자로 핵심 기능만 구현 (연 2.5억 절감)

## 2교시: 원가 산정 기법 (1.5시간) <!-- 슬라이드 #2~#3 -->

### 이론 (50분)

#### 1. 원가 산정(Cost Estimating)이란?

**정의:**
프로젝트 활동을 완료하는 데 필요한 **자원의 근사 비용**을 추정하는 프로세스

**산정의 중요성:**
- 예산 수립의 기초
- 프로젝트 승인/기각 결정의 근거
- 자금 조달 계획 수립
- 성과 측정의 기준 (EVM)

**산정 vs 견적:**
- 산정(Estimate): 내부 기획 단계의 추정
- 견적(Quotation): 외부에 제시하는 공식 가격

#### 2. 원가 산정 기법 비교

프로젝트 상황과 정보 수준에 따라 적절한 기법을 선택해야 합니다.

| 구분 | 유사 산정 | 모수 산정 | 상향식 산정 | 3점 산정 |
|------|----------|----------|-----------|---------|
| **영문명** | Analogous | Parametric | Bottom-up | Three-point |
| **접근** | Top-down | 통계적 관계 | Bottom-up | 확률 분포 |
| **정확도** | 낮음 (-25%~+75%) | 중간 (-10%~+25%) | 높음 (-5%~+10%) | 중간~높음 |
| **시간/비용** | 빠름/저렴 | 중간 | 느림/비쌈 | 중간 |
| **필요 정보** | 최소 | 과거 데이터 | 상세 WBS | 전문가 판단 |
| **적용 시점** | 착수/초기 | 기획 중반 | 기획 후반 | 전 단계 |

#### 3. 유사 산정 (Analogous Estimating)

**정의:**
과거 유사 프로젝트의 **실제 원가**를 기반으로 현재 프로젝트의 원가를 추정

**특징:**
- **Top-down** 접근 (전체 → 부분)
- 빠르고 간단
- 정확도 낮음 (ROM 수준: -25% ~ +75%)
- 프로젝트 초기 단계에서 활용

**적용 조건:**
- 과거 프로젝트와 **유사성**이 높을 것
- 과거 데이터가 **정확**할 것
- PM이 해당 영역의 **전문 지식** 보유

**공식:**
```
현재 프로젝트 원가 = 과거 프로젝트 원가 × 조정 계수
```

**조정 요소:**
- 규모 차이 (기능 수, 사용자 수 등)
- 복잡도 차이
- 기술 차이 (신기술 vs 검증된 기술)
- 팀 역량 차이

**예시:**
```
과거 프로젝트 A: 온라인 쇼핑몰 (20개 기능, 5억원)
현재 프로젝트 B: 온라인 교육 플랫폼 (30개 기능)

산정:
- 기능 수 비율: 30/20 = 1.5배
- 복잡도 조정: 1.2배 (결제 모듈이 더 복잡)
- B 원가 = 5억 × 1.5 × 1.2 = 9억원 (ROM)
```

**장점:**
- 빠르고 저렴
- 프로젝트 초기에 빠른 의사결정 가능
- 큰 그림 파악에 유용

**단점:**
- 정확도 낮음
- 주관적 판단에 의존
- 고유한 특성 반영 어려움
- 과거 프로젝트가 유사하지 않으면 부정확

#### 4. 모수 산정 (Parametric Estimating)

**정의:**
**통계적 관계식**을 활용하여 원가를 계산

**공식:**
```
원가 = 단가 × 수량
```

**특징:**
- 과거 프로젝트 데이터 기반의 **단가** 산출
- 정량적이고 객관적
- 정확도는 과거 데이터의 품질에 좌우

**적용 사례:**

**소프트웨어 개발:**
```
원가 = 기능점수(Function Point) × FP당 단가
원가 = 코드 라인 수(LOC) × LOC당 단가
원가 = 화면 수 × 화면당 개발 단가
```

**건설:**
```
원가 = 면적(㎡) × ㎡당 공사비
```

**예시:**
```
프로젝트: 웹 애플리케이션 개발
- 화면 수: 80개
- 과거 데이터 분석 결과: 화면 1개당 평균 500만원
- 복잡도 조정 계수: 1.2 (평균보다 복잡)

산정:
원가 = 80개 × 500만원 × 1.2 = 4.8억원
```

**COCOMO (Constructive Cost Model) 예:**
```
노력(인월) = a × (KLOC)^b
a, b: 프로젝트 유형별 계수
KLOC: 1000 라인의 코드

예: KLOC=50, a=2.4, b=1.05
노력 = 2.4 × 50^1.05 = 약 130인월
원가 = 130인월 × 월 800만원 = 10.4억원
```

**장점:**
- 객관적이고 일관성 있음
- 중간 수준의 정확도
- 규모 변화에 따른 원가 추정 용이

**단점:**
- 신뢰할 만한 과거 데이터 필요
- 단가 산출이 어려울 수 있음
- 프로젝트 고유 특성 반영 제한적

#### 5. 상향식 산정 (Bottom-up Estimating)

**정의:**
**작업 패키지 또는 활동 수준**에서 개별 산정 후 **합산**

**프로세스:**
1. WBS를 작업 패키지까지 분해
2. 각 작업 패키지의 원가 개별 산정
3. 상위 수준으로 합산
4. 프로젝트 전체 원가 도출

**특징:**
- **Bottom-up** 접근 (부분 → 전체)
- **가장 정확** (Definitive: -5% ~ +10%)
- 시간과 노력이 많이 소요
- 상세한 WBS 필요

**산정 단위:**
```
작업 패키지 원가 = Σ(자원별 원가)
자원별 원가 = 투입 시간 × 시간당 단가
```

**예시:**
```
프로젝트: 고객 관리 시스템 개발
WBS 작업 패키지:

1.1 로그인 모듈
  - 화면 개발: 2일 × 60만원/일 = 120만원
  - API 개발: 3일 × 70만원/일 = 210만원
  - 테스트: 1일 × 50만원/일 = 50만원
  소계: 380만원

1.2 고객 등록 모듈
  - 화면 개발: 4일 × 60만원/일 = 240만원
  - API 개발: 5일 × 70만원/일 = 350만원
  - DB 설계: 2일 × 70만원/일 = 140만원
  - 테스트: 2일 × 50만원/일 = 100만원
  소계: 830만원

... (계속 합산)

프로젝트 전체: 4.8억원
```

**장점:**
- 가장 정확함
- 각 작업의 원가 근거 명확
- 리스크 식별 용이
- 자원 배정 및 일정 계획과 통합 가능

**단점:**
- 시간과 비용이 많이 듦
- 상세한 WBS 필요 (초기 단계에는 불가)
- "나무만 보고 숲을 못 보는" 위험

#### 6. 3점 산정 (Three-point Estimating)

**정의:**
**낙관치, 최빈치, 비관치** 3개 값을 활용하여 **불확실성**을 반영한 산정

**3가지 시나리오:**
- **O (Optimistic, 낙관치):** 최상의 경우 (모든 것이 잘 될 때)
- **M (Most Likely, 최빈치):** 가장 가능성 높은 경우 (정상 진행)
- **P (Pessimistic, 비관치):** 최악의 경우 (문제 발생)

**공식:**

**1) 삼각 분포 (Triangular Distribution):**
```
기대값 (Expected Value) = (O + M + P) / 3
표준편차 (SD) = (P - O) / 6
```
- 단순 평균
- 3가지 시나리오가 동등한 확률

**2) 베타 분포 (Beta Distribution, PERT):**
```
기대값 = (O + 4M + P) / 6
표준편차 (SD) = (P - O) / 6
```
- 최빈치에 **4배 가중치**
- 현실적으로 M이 발생할 확률이 높음
- **PERT에서 사용** (PMI 권장)

**표준편차 활용:**
- 1σ (68%): 기대값 ± 1·SD
- 2σ (95%): 기대값 ± 2·SD
- 3σ (99.7%): 기대값 ± 3·SD

**예시:**
```
활동: API 개발 모듈
- 낙관치 (O): 5일 (모든 것이 순조로움)
- 최빈치 (M): 8일 (정상 진행)
- 비관치 (P): 14일 (기술 문제 발생)

베타 분포 계산:
기대값 = (5 + 4×8 + 14) / 6 = 51 / 6 = 8.5일
표준편차 = (14 - 5) / 6 = 1.5일

해석:
- 68% 확률: 7~10일 (8.5 ± 1.5)
- 95% 확률: 5.5~11.5일 (8.5 ± 3)
```

**원가 산정 예:**
```
활동: 보안 모듈 개발
- 낙관 원가 (O): 600만원
- 최빈 원가 (M): 1,000만원
- 비관 원가 (P): 1,800만원

베타 분포:
기대 원가 = (600 + 4×1,000 + 1,800) / 6 
         = 6,400 / 6 = 1,067만원
표준편차 = (1,800 - 600) / 6 = 200만원

예산 책정:
- 50% 신뢰: 1,067만원
- 95% 신뢰: 1,467만원 (1,067 + 2×200)
```

**장점:**
- 불확실성 정량화
- 리스크 인식 제고
- 범위 산정으로 의사결정 개선

**단점:**
- 3가지 값 산정에도 전문가 판단 필요
- 복잡도 증가
- 정확도는 전문가 역량에 좌우

#### 7. 산정 정확도 등급

프로젝트 단계에 따라 요구되는 산정 정확도가 다릅니다.

| 등급 | 정확도 범위 | 적용 시점 | 목적 |
|------|------------|----------|------|
| **ROM**<br>(Rough Order of Magnitude) | -25% ~ +75% | 착수 단계 | Go/No-Go 결정<br>예비 타당성 |
| **예산 산정**<br>(Budget Estimate) | -10% ~ +25% | 기획 단계 | 예산 승인<br>자금 조달 계획 |
| **확정 산정**<br>(Definitive Estimate) | -5% ~ +10% | 실행 직전 | 계약 체결<br>원가 기준선 |

**예시:**
```
프로젝트: 전자상거래 플랫폼 구축

착수 단계 (ROM):
- 유사 산정: 5억원
- 범위: 3.75억 ~ 8.75억원 (-25% ~ +75%)
- 의사결정: 10억 예산 범위 내 → 진행

기획 단계 (예산 산정):
- 모수 산정: 6.5억원
- 범위: 5.85억 ~ 8.13억원 (-10% ~ +25%)
- 예산 승인: 8억원으로 승인

실행 직전 (확정 산정):
- 상향식 산정: 7.2억원
- 범위: 6.84억 ~ 7.92억원 (-5% ~ +10%)
- 원가 기준선: 7.2억 + 우발예비 0.5억 = 7.7억
```

#### 주관식 문제

**Q5. 유사 산정과 상향식 산정의 차이를 설명하고, 각각 어느 프로젝트 단계에 적합한지 쓰세요.**

<details>
<summary>모범 답안</summary>
유사 산정은 과거 유사 프로젝트의 실제 원가를 기반으로 Top-down 방식으로 추정하는 기법으로, 빠르지만 정확도가 낮습니다 (ROM 수준). 프로젝트 착수 단계에서 빠른 의사결정을 위해 사용합니다.

상향식 산정은 작업 패키지 수준에서 개별 산정 후 합산하는 Bottom-up 방식으로, 가장 정확하지만 시간과 노력이 많이 소요됩니다 (Definitive 수준). 상세한 WBS가 완성된 기획 후반부 또는 실행 직전 단계에서 사용하며, 계약 체결이나 원가 기준선 설정에 활용됩니다.
</details>

**Q6. 3점 산정에서 표준편차를 왜 계산하는지, 그리고 어떻게 활용하는지 설명하세요.**

<details>
<summary>모범 답안</summary>
표준편차는 원가 산정의 불확실성 정도를 나타냅니다. 공식은 (P - O) / 6이며, 값이 클수록 불확실성이 높습니다.

활용 방법:
1. 신뢰 구간 계산: 68% 신뢰(1σ), 95% 신뢰(2σ), 99.7% 신뢰(3σ)
2. 우발 예비 책정: 일반적으로 2σ 수준으로 예비비 설정
3. 리스크 평가: 표준편차가 큰 활동을 고위험으로 식별하여 집중 관리

예: 기대값 2,000만원, 표준편차 500만원
- 95% 신뢰 예산 = 2,000 + 2×500 = 3,000만원
- 우발 예비 = 1,000만원
</details>

**Q7. 모수 산정 기법이 부정확해질 수 있는 상황을 2가지 설명하세요.**

<details>
<summary>모범 답안</summary>
1. 과거 데이터의 품질이 낮거나 신뢰할 수 없는 경우: 과거 프로젝트의 원가 기록이 부정확하거나, 프로젝트 환경이 크게 달라진 경우 (예: 5년 전 데이터를 현재에 적용) 단가 자체가 부정확해집니다.

2. 프로젝트의 고유한 특성이 강한 경우: 모수 산정은 통계적 평균을 사용하므로, 현재 프로젝트가 평균과 크게 다른 특성을 가질 때 (예: 매우 높은 보안 요구사항, 신기술 적용) 정확도가 떨어집니다. 이런 경우 복잡도 조정 계수를 추가해야 합니다.
</details>

## 3교시: 예산 수립 (1.5시간) <!-- 슬라이드 #4 -->

### 이론 (50분)

#### 1. 예산 수립(Determine Budget)이란?

**정의:**
개별 활동 또는 작업 패키지의 **원가 산정치를 합산**하여 **승인된 원가 기준선**을 설정하는 프로세스

**목적:**
- 성과 측정의 기준 제공
- 프로젝트 진행 중 원가 추적 및 통제
- 자금 조달 계획 수립
- 이해관계자에 대한 재무적 투명성 확보

**Input:**
- 원가 산정치 (활동별/작업 패키지별)
- 원가 관리 계획서
- 범위 기준선 (WBS)
- 일정 기준선 (시간 배분 필요)
- 리스크 등록부

**Output:**
- **원가 기준선 (Cost Baseline)**
- 프로젝트 자금 요구사항
- 프로젝트 문서 업데이트

#### 2. 프로젝트 예산 구조

프로젝트 예산은 **3단계 구조**로 구성됩니다:

```
[활동 원가 산정 합계]
        ↓
    + 우발 사태 예비 (Contingency Reserve)
        ↓
    = 원가 기준선 (Cost Baseline) ← 성과 측정 기준
        ↓
    + 관리 예비 (Management Reserve)
        ↓
    = 프로젝트 예산 (Project Budget) ← 최종 승인 예산
```

**단계별 설명:**

**레벨 1: 활동 원가 산정 합계**
- WBS 작업 패키지별 원가 산정을 합산
- 모든 직접비와 배분된 간접비 포함
- 산정 기법: 유사/모수/상향식/3점 산정 활용

**레벨 2: 우발 사태 예비 추가 → 원가 기준선**
- 우발 예비를 추가하여 원가 기준선 설정
- PM이 승인 없이 사용 가능
- 기준선은 성과 측정의 기준 (EVM 적용)

**레벨 3: 관리 예비 추가 → 프로젝트 예산**
- 관리 예비를 추가하여 최종 예산
- 경영진 승인 필요
- 계약서에 명시되는 최종 금액

#### 3. 우발 사태 예비 (Contingency Reserve)

**정의:**
**알려진 리스크**(Identified Risks)에 대응하기 위해 원가 기준선에 포함하는 예비비

**특징:**
- 원가 기준선의 일부
- PM이 승인 없이 사용 가능
- 리스크 등록부의 리스크에 대응
- EVM 측정 시 포함됨

**산정 방법:**

**1) 비율 기반 (Percentage)**
```
우발 예비 = 활동 원가 합계 × 비율
비율: 5~15% (프로젝트 복잡도/리스크 수준에 따라)
```

**2) 리스크 기반 (Risk-based)**
```
우발 예비 = Σ (리스크 영향 금액 × 발생 확률)
```

**3) 표준편차 기반 (3점 산정 활용)**
```
우발 예비 = 2 × 표준편차 합계 (95% 신뢰도)
```

**예시:**
```
활동 원가 합계: 8억원

방법 1 (비율): 8억 × 10% = 0.8억원
방법 2 (리스크):
  - 리스크 A: 5,000만 × 40% = 2,000만
  - 리스크 B: 8,000만 × 30% = 2,400만
  - 리스크 C: 3,000만 × 50% = 1,500만
  합계: 5,900만원 ≈ 0.6억원

방법 3 (표준편차): 2 × 0.4억 = 0.8억원

→ 우발 예비: 0.7억원으로 책정
→ 원가 기준선: 8.7억원
```

**사용 원칙:**
- 리스크가 발생하면 우발 예비에서 지출
- 리스크가 발생하지 않으면 남은 예비는 조직에 반환
- 정기적으로 리스크 재평가 후 예비 조정

#### 4. 관리 예비 (Management Reserve)

**정의:**
**미지의 리스크**(Unknown Unknowns, Unknown Risks)에 대비한 예비비

**특징:**
- 원가 기준선에 **불포함**
- 프로젝트 예산에는 포함
- **경영진 승인 후** 사용 가능
- EVM 측정 시 제외됨
- 범위 변경 시에도 활용 가능

**산정 방법:**
```
관리 예비 = 원가 기준선 × 5~10%
```

**우발 예비 vs 관리 예비 비교:**

| 구분 | 우발 사태 예비 | 관리 예비 |
|------|---------------|----------|
| **대상** | 알려진 리스크 | 미지의 리스크 |
| **원가 기준선** | 포함 | 불포함 |
| **프로젝트 예산** | 포함 | 포함 |
| **EVM 측정** | 포함 | 제외 |
| **사용 승인** | PM 권한 | 경영진 승인 |
| **비율** | 5~15% | 5~10% |
| **리스크 등록부** | 연계 있음 | 연계 없음 |

**예시:**
```
프로젝트: 전자상거래 플랫폼 구축

활동 원가 합계: 8억원
우발 사태 예비: 0.8억원 (10%)
→ 원가 기준선: 8.8억원

관리 예비: 0.6억원 (기준선의 약 7%)
→ 프로젝트 예산: 9.4억원

계약서 명시: "프로젝트 예산 9.4억원"
PM 관리 대상: "원가 기준선 8.8억원"
```

#### 5. 원가 기준선 (Cost Baseline)

**정의:**
시간에 따라 **배분된(Time-phased)** 승인된 프로젝트 예산으로, **성과 측정의 기준**

**특징:**
- 활동 원가 합계 + 우발 예비
- 시간 기반으로 배분 (월별/주별)
- 승인 받은 후에는 정식 변경 통제 절차 필요
- EVM의 PV(Planned Value) 기준

**시간 배분 방법:**
1. 일정 기준선과 통합
2. 각 활동의 계획 기간에 원가 배분
3. 월별/분기별로 누적 원가 산출

**형태:**
- S-Curve로 표현 (다음 섹션 참조)

**변경:**
- 범위 변경 시: 변경 통제 절차를 거쳐 기준선 조정
- 리스크 발생 시: 우발 예비 사용 (기준선 변경 불필요)
- 미지의 리스크 시: 관리 예비 사용 후 기준선 재설정 (승인 필요)

#### 6. S-Curve (S자 곡선)

**정의:**
프로젝트 **누적 원가**를 시간에 따라 그래프로 표현한 곡선 (S자 형태)

**S자 형태의 이유:**

```
초기 (기획/설계): 완만한 증가
  - 소수 인력 투입
  - 계획 수립 단계
  
중기 (실행): 급격한 증가
  - 대규모 인력/자원 투입
  - 개발/구축 본격화
  
후기 (종료): 완만한 증가
  - 테스트, 최종 정리
  - 인력 점진적 철수
```

**S-Curve의 용도:**

**1) 원가 기준선 설정**
- 시간에 따른 계획 지출을 시각화
- PV(Planned Value) 곡선

**2) 성과 모니터링**
- 계획 vs 실제 비교
- PV, EV, AC 3개 곡선 중첩

**3) 자금 조달 계획**
- 각 시점별 필요 자금 파악
- 현금 흐름 관리

**4) 진행 상황 보고**
- 이해관계자에게 시각적 전달

**S-Curve 예시:**
```
     누적 원가 (억원)
     ↑
  10 |                    ___---***  (목표선)
     |              ___---
   8 |         __--
     |      _-
   6 |    _/
     |   /
   4 | _/
     | /
   2 |/
     |________________________→ 시간
        M1  M2  M3  M4  M5  M6

초기 (M1-M2): 완만 (2억 → 4억)
중기 (M2-M4): 급증 (4억 → 8억)
후기 (M4-M6): 완만 (8억 → 10억)
```

**3개 곡선 비교 (EVM):**
```
     누적 원가
     ↑
  10 |              PV (계획)
     |         
   8 |        AC (실제 지출)
     |      
   6 |    EV (획득 가치)
     |  
   4 |
     |
   2 |
     |________________________→ 시간
        M1  M2  M3  M4  M5  M6

해석:
- EV < PV: 일정 지연
- AC > EV: 예산 초과
```

#### 7. 자금 조달 한도 (Funding Limit Reconciliation)

**정의:**
조직의 **자금 가용성**과 프로젝트 **자금 요구**를 조정하는 프로세스

**문제 상황:**
- 프로젝트는 S-Curve대로 지출 필요
- 조직은 분기별/연도별 **예산 집행 한도** 보유
- 두 가지가 일치하지 않을 수 있음

**조정 방법:**

**1) 일정 조정**
- 자금이 부족한 시기의 활동을 연기
- 자금이 여유로운 시기로 이동
- 단, 크리티컬 패스는 최대한 유지

**2) 범위 조정**
- 단계별 구현 (Phased Implementation)
- 우선순위 높은 기능 먼저 구현

**3) 추가 자금 확보**
- 경영진에게 추가 예산 요청
- 외부 차입 고려

**예시:**
```
프로젝트 자금 요구:
- Q1: 2억
- Q2: 4억 (누적 6억)
- Q3: 3억 (누적 9억)
- Q4: 1억 (누적 10억)

조직의 분기별 한도:
- Q1: 2억
- Q2: 2.5억 (누적 4.5억) ← 부족!
- Q3: 3억 (누적 7.5억)
- Q4: 3억 (누적 10.5억)

문제: Q2에 4억 필요하지만 2.5억만 가능

해결책:
- Q2의 일부 활동을 Q3로 연기
- 또는 외부 용역 대신 내부 인력 활용 (지출 분산)
```

**자금 흐름 그래프:**
```
     자금 (억원)
     ↑
   5 |        ___
     |       |   |___
   4 |   ___|       |___
     |  |               |___
   3 | _|                   
     |_________________________→ 분기
        Q1   Q2   Q3   Q4

조직 자금 한도 (단계형)
프로젝트 요구 (S-Curve)
→ 조정 필요
```

### 예시 (25분)

#### 예시 1: 프로젝트 예산 구조 상세 사례

**프로젝트:** 온라인 학습 플랫폼 구축 (12개월)

**1단계: 활동 원가 산정 합계**

| WBS | 작업 패키지 | 원가 산정 (억원) |
|-----|------------|-----------------|
| 1.1 | 요구사항 분석 | 0.5 |
| 1.2 | 시스템 설계 | 0.8 |
| 2.1 | 회원 관리 모듈 | 1.2 |
| 2.2 | 강의 관리 모듈 | 1.5 |
| 2.3 | 결제 모듈 | 1.0 |
| 2.4 | 콘텐츠 관리 시스템 | 1.3 |
| 3.1 | 통합 테스트 | 0.6 |
| 3.2 | 인프라 구축 | 0.9 |
| 4.1 | 교육 훈련 | 0.3 |
| 4.2 | 문서화 | 0.2 |
| | **활동 원가 합계** | **8.3억** |

**2단계: 우발 사태 예비 추가**

**리스크 기반 산정:**

| 리스크 ID | 리스크 내용 | 영향 (억원) | 확률 | 기대값 |
|-----------|------------|------------|------|--------|
| R-001 | 결제 PG 연동 지연 | 0.8 | 40% | 0.32 |
| R-002 | 클라우드 비용 초과 | 0.5 | 50% | 0.25 |
| R-003 | 핵심 인력 이탈 | 1.2 | 20% | 0.24 |
| R-004 | 요구사항 변경 | 1.0 | 60% | 0.60 |
| R-005 | 보안 취약점 발견 | 0.6 | 30% | 0.18 |
| | | | **우발 예비** | **1.59억** |

**반올림: 1.6억원**

```
원가 기준선 = 8.3억 + 1.6억 = 9.9억원
```

**3단계: 관리 예비 추가**
```
관리 예비 = 9.9억 × 7% = 0.7억원
프로젝트 예산 = 9.9억 + 0.7억 = 10.6억원
```

**최종 예산 구조:**
```
활동 원가 합계:     8.3억 (78%)
우발 사태 예비:     1.6억 (15%)
------------------------
원가 기준선:        9.9억 (93%) ← PM 관리 대상
관리 예비:          0.7억 (7%)
------------------------
프로젝트 예산:     10.6억 (100%) ← 계약 금액
```

4. 알 수 없음

<details>
<summary>정답 보기</summary>
정답: 2번
해설: 관리 예비 = 프로젝트 예산 - 원가 기준선 = 10억 - 9.2억 = 0.8억원. 프로젝트 예산은 원가 기준선(활동 원가 + 우발 예비)에 관리 예비를 더한 값입니다.
</details>

#### 주관식 문제

**Q5. 우발 사태 예비(Contingency Reserve)와 관리 예비(Management Reserve)의 차이를 3가지 측면에서 비교 설명하세요.**

<details>
<summary>모범 답안</summary>
1. **대상 리스크:**
   - 우발 예비: 알려진 리스크(Identified Risks) - 리스크 등록부에 기록된 리스크

   - 관리 예비: 원가 기준선에 불포함 → EVM 측정 시 제외

3. **사용 승인 권한:**
   - 우발 예비: PM이 독자적으로 사용 결정 가능
   - 관리 예비: 경영진(스폰서) 승인 후 사용 가능

예시: 10억 프로젝트에서 활동 원가 8억 + 우발 예비 1.5억 = 원가 기준선 9.5억, 관리 예비 0.5억 추가 = 프로젝트 예산 10억
</details>

**Q6. S-Curve가 전형적인 S자 모양을 보이는 이유를 프로젝트 생애주기 단계와 연결하여 설명하세요.**

<details>
<summary>모범 답안</summary>
S-Curve는 프로젝트 생애주기의 자원 투입 패턴을 반영합니다:

1. **초기 (착수/기획 단계): 완만한 증가**
   - 소수의 인력만 투입 (PM, 분석가 등)
   - 요구사항 분석, 계획 수립 중심
   - 대규모 자원 투입 전 단계
   - 예: 전체 예산의 10-20% 사용

2. **중기 (실행 단계): 급격한 증가**
   - 대규모 개발팀 투입
   - 본격적인 개발/구축 작업
   - 자재, 장비 구매 집중
   - 예: 전체 예산의 50-70% 사용
   - 이 시기의 급증이 S자의 중간 가파른 부분 형성

3. **후기 (종료 단계): 완만한 증가**
   - 테스트, 검수, 문서화
   - 인력 점진적 철수
   - 마무리 작업 중심
   - 예: 전체 예산의 10-20% 사용

이러한 3단계 패턴이 S자 곡선을 만들며, 이는 대부분의 프로젝트에서 공통적으로 나타나는 자연스러운 현상입니다.
</details>

<details>
<summary>모범 답안</summary>
1. **일정 조정 (Schedule Adjustment):**
   - 자금이 부족한 시기의 활동을 자금 여유가 있는 시기로 이동
   - 비크리티컬 패스 활동 우선 조정
   - 단, 크리티컬 패스 활동은 최대한 유지하여 전체 일정 영향 최소화
   - 예: Q2에 집중된 활동 일부를 Q1 또는 Q3로 분산

2. **지출 방식 변경 (Payment Method Change):**
   - 외주 용역을 내부 인력으로 전환 (일시 지출 → 분산 지출)
   - 장비 구매를 리스로 전환 (초기 비용 감소)
   - 단계적 납품으로 지불 시기 분산
   - 예: 한 번에 5억 지출 대신 월 5천만원씩 10개월 지출

3. **범위 조정 또는 추가 자금 확보:**
   - 단계별 구현 (Phased Implementation): 우선순위 높은 기능 먼저 구현
   - 경영진에게 추가 예산 요청 (비즈니스 케이스 제시)
   - 외부 차입 또는 다른 프로젝트에서 자금 이전 협상
   - 예: 전체 10개 모듈 중 핵심 6개를 1차로 구현, 나머지는 2차로 연기
</details>

## 4교시: 원가 통제 및 EVM 개요 (1시간) <!-- 슬라이드 #5 -->

### 이론 (50분)

#### 1. 원가 통제(Control Costs)란?

**정의:**
프로젝트 원가 **기준선 대비** 실적을 모니터링하고, 승인되지 않은 변경을 방지하며, 변경 사항을 관리하는 프로세스

**목적:**
- 원가 편차(Cost Variance) 조기 발견
- 승인되지 않은 변경 방지
- 기준선 대비 성과 측정
- 완료 시 원가 예측
- 시정 조치 수행

**원가 통제의 핵심 활동:**
1. 실제 원가(AC) 측정
2. 기준선(PV) 대비 비교
3. 획득 가치(EV) 계산
4. 편차 분석 (Variance Analysis)
5. 추세 분석 (Trend Analysis)
6. 완료 시 예측 (Forecasting)
7. 시정 조치 권고
8. 변경 요청 처리
9. 원가 기준선 업데이트 (승인 시)

**원가 통제 vs 일정 통제:**
- 원가 통제: "돈"을 측정 (예산 대비 지출)
- 일정 통제: "시간"을 측정 (계획 대비 진행)
- **EVM은 두 가지를 통합 측정**

#### 2. 전통적 원가 관리의 한계

**문제 상황: 단순 예산 대비 지출 비교**

**예시:**
```
프로젝트: 웹 애플리케이션 개발
- 총 예산: 10억원
- 계획 기간: 10개월
- 현재: 5개월 경과

현재 상태:
- 지출액 (AC): 4억원
- 예산 대비: 4억 / 5억(50% 시점) = 80% 수준

일반적인 판단: "예산 절감 중! 잘하고 있다!"
```

**진실은?**
```
실제 진행률 확인:
- 계획 진행률: 50% (5/10개월)
- 실제 진행률: 30% (작업 지연)

재해석:
- 50% 진행 계획이었지만 30%만 완료
- 4억을 써서 30% 완료 → 전체 완료 시 약 13.3억 예상
- 실제로는 예산 초과 위험!
```

**문제점:**
❌ 지출액만 보면 진행 상황을 알 수 없음
❌ "적게 쓴다 = 잘한다"는 착각
❌ 일정 지연과 원가 초과를 동시에 파악 불가

**해결책: EVM (획득가치 관리)**

#### 3. EVM (Earned Value Management) 개요

**정의:**
프로젝트의 **범위, 일정, 원가**를 통합하여 성과를 측정하고 진행 상황을 평가하는 기법

**EVM의 핵심 가치:**
- 범위와 일정을 **금액(원가)**으로 통일하여 측정
- 객관적이고 정량적인 성과 지표
- 조기 경보 시스템 (Early Warning System)
- 완료 시 예측 가능

**EVM의 역사:**
- 1960년대: 미국 국방부에서 개발
- 1990년대: PMBOK에 포함
- 현재: 대형 프로젝트의 표준 관리 기법

#### 4. EVM의 3대 핵심 변수

**기본 개념: "가치"로 통합**

모든 측정값을 **"얼마의 가치(돈)"**로 표현:
- 계획한 작업 → PV (얼마의 가치를 계획했나?)
- 완료한 작업 → EV (얼마의 가치를 획득했나?)
- 실제 지출 → AC (얼마를 실제로 썼나?)

#### **A. PV (Planned Value, 계획가치)**

**정의:**
**현 시점까지** 계획된 작업에 할당된 **승인 예산**

**다른 이름:**
- BCWS (Budgeted Cost of Work Scheduled) ← 구 용어
- 계획된 작업의 예산

**계산:**
```
PV = 원가 기준선 × 계획 진행률(%)
```

**특징:**
- 원가 기준선에서 도출
- S-Curve의 계획 곡선
- 시간에 따라 누적 증가
- 프로젝트 완료 시 PV = BAC

**예시:**
```
프로젝트: CRM 시스템 구축
- BAC (총 예산): 10억원
- 계획 기간: 10개월
- 원가 기준선 (S-Curve):
  M1: 5%, M2: 15%, M3: 30%, M4: 50%, M5: 70%, ...

Month 4 시점:
- 계획 진행률: 50%
- PV = 10억 × 50% = 5억원

해석: "4개월 차에 5억원 가치의 작업을 완료할 계획이었다"
```

#### **B. EV (Earned Value, 획득가치)**

**정의:**
**현 시점까지** 실제로 완료한 작업에 할당된 **승인 예산**

**다른 이름:**
- BCWP (Budgeted Cost of Work Performed) ← 구 용어
- 완료된 작업의 예산 가치

**계산:**
```
EV = BAC × 실제 진행률(%)
```

**핵심 개념:**
- **"완료한 작업의 가치"**를 예산 기준으로 측정
- 실제 지출액(AC)과는 무관
- **성과 측정의 핵심** (진짜 얼마나 했는가?)

**예시:**
```
Month 4 시점:
- BAC: 10억원
- 실제 진행률: 40% (계획보다 지연)
- EV = 10억 × 40% = 4억원

해석: "4개월 차에 4억원 가치의 작업을 실제로 완료했다"
```

**EV 측정 방법:**

**1) 물리적 진행률 (Physical % Complete)**
- 실제 완료된 작업량을 측정
- 가장 정확하나 측정 어려움
- 예: 코드 라인 수, 기능 개수

**2) 0-100 규칙**
- 시작 전: 0%
- 완료 후: 100%
- 작업 중: 0% (보수적)

**3) 50-50 규칙**
- 시작 시: 50%
- 완료 시: 100%
- 중간 규모 작업에 적합

**4) 20-80 규칙**
- 시작 시: 20%
- 완료 시: 100%

**예시 (50-50 규칙):**
```
작업 A: 로그인 모듈 개발 (예산 2,000만원)
- Week 1: 작업 시작 → EV = 2,000만 × 50% = 1,000만원
- Week 2: 작업 완료 → EV = 2,000만 × 100% = 2,000만원
```

#### **C. AC (Actual Cost, 실제원가)**

**정의:**
**현 시점까지** 완료한 작업에 **실제로 지출한 총비용**

**다른 이름:**
- ACWP (Actual Cost of Work Performed) ← 구 용어
- 실제 지출액

**계산:**
```
AC = Σ 실제 지출액
```

**특징:**
- 회계 시스템에서 집계
- 가장 측정하기 쉬움
- 인건비, 재료비, 외주비 등 모든 지출 포함

**예시:**
```
Month 4 시점:
- 개발자 인건비: 2.5억원
- 외주 용역비: 1.8억원
- 라이센스 비용: 0.5억원
- 기타: 0.4억원
- AC = 5.2억원

해석: "4개월 차까지 실제로 5.2억원을 지출했다"
```

#### 5. PV, EV, AC 통합 분석

**3개 변수 비교**

| 변수 | 의미 | 질문 | 기준 |
|------|------|------|------|
| **PV** | 계획가치 | 얼마를 계획했나? | 원가 기준선 × 계획% |
| **EV** | 획득가치 | 얼마를 완료했나? | BAC × 실제% |
| **AC** | 실제원가 | 얼마를 썼나? | 실제 지출 합계 |

**📊 EVM 3곡선 그래프:**

```
비용
(원가)
  ↑
  │         ......PV (계획가치: 우리가 이 시점까지 쓰기로 계획한 비용)
  │      ....
  │   ..../....
  │  /.......  ....
  │ / ..EV       ....
  │/.(획득가치)        ...
  │ \(이 시점까지 완료한                PV
  │  \ 작업의 가치)
  │   ─ ─ ─ AC (실제원가: 현재 시점까지 실제 지출)      EV
  │          (AC가 EV보다 위 = 예산 초과)              AC
  └─────────────────────────────────▶ 시간
            현재 ↑

  SV (일정편차) = EV - PV  → 음수면 일정 지연
  CV (원가편차) = EV - AC  → 음수면 예산 초과

  이상적:  PV = 기준선 / EV = PV 위 (일정 앞섬) / AC = EV 아래 (예산 절감)
  위험:    EV < PV (일정 뒤처짐) + AC > EV (예산 초과) → 이중 위기!
```

**프로젝트 상태 진단:**

**1) EV vs PV → 일정 성과**
- EV > PV: 계획보다 빠름 (일정 앞섬)
- EV = PV: 계획대로 진행
- EV < PV: 계획보다 느림 (일정 지연)

**2) EV vs AC → 원가 성과**
- EV > AC: 예산 대비 적게 씀 (예산 절감)
- EV = AC: 예산대로 지출
- EV < AC: 예산 대비 많이 씀 (예산 초과)

**4가지 프로젝트 상태:**

| 상태 | EV vs PV | EV vs AC | 해석 |
|------|----------|----------|------|
| **이상적** | EV > PV | EV > AC | 일정 앞서고, 예산 절감 ✅ |
| **주의** | EV > PV | EV < AC | 일정 앞서나, 예산 초과 ⚠️ |
| **주의** | EV < PV | EV > AC | 일정 지연, 예산 절감 ⚠️ |
| **위험** | EV < PV | EV < AC | 일정 지연, 예산 초과 ❌ |

#### 6. BAC (Budget at Completion)

**정의:**
프로젝트 **완료 시 총 계획 예산** = 원가 기준선 총액

**특징:**
- 프로젝트 시작 시 설정
- 승인된 변경이 있으면 업데이트
- EVM 계산의 기준값

**예시:**
```
프로젝트: 물류 시스템 개발
- 활동 원가 합계: 8억
- 우발 예비: 1억
- BAC = 9억원 (원가 기준선)
```

### 예시 (25분)

#### 예시 1: PV, EV, AC 계산 기본

**프로젝트:** 모바일 앱 개발 (6개월, 6억원)

**원가 기준선 (월별 계획):**

| 월 | M1 | M2 | M3 | M4 | M5 | M6 |
|----|----|----|----|----|----|----|
| 월별 예산 (백만) | 50 | 80 | 120 | 150 | 130 | 70 |
| 누적 PV (백만) | 50 | 130 | 250 | 400 | 530 | 600 |
| 계획 진행률 (%) | 8% | 22% | 42% | 67% | 88% | 100% |

**Month 3 시점 실적:**
- 실제 진행률: 35% (계획 42%보다 낮음)
- 실제 지출: 280백만원

**EVM 계산:**
```
BAC = 600백만원

PV (계획가치):
- 원가 기준선 M3 누적 = 250백만원

EV (획득가치):
- EV = BAC × 실제 진행률
- EV = 600 × 35% = 210백만원

AC (실제원가):
- AC = 280백만원 (주어짐)
```

**상태 분석:**
```
PV = 250백만, EV = 210백만, AC = 280백만

1. EV vs PV:
```

#### 주관식 문제

**Q5. PV, EV, AC 각각의 의미를 설명하고, 어떻게 측정하는지 쓰세요.**

<details>
<summary>모범 답안</summary>
**PV (Planned Value, 계획가치):**
- 의미: 현 시점까지 계획된 작업에 할당된 승인 예산
- 측정: 원가 기준선(S-Curve)에서 현 시점의 누적 계획 예산
- 공식: PV = BAC × 계획 진행률
- 예: 10억 프로젝트에서 50% 완료 계획 시 PV = 5억

**EV (Earned Value, 획득가치):**
- 의미: 현 시점까지 실제로 완료한 작업의 예산 가치
- 측정: 작업 완료율을 측정하여 BAC에 곱함
- 공식: EV = BAC × 실제 진행률
- 예: 실제 40% 완료 시 EV = 10억 × 40% = 4억

**AC (Actual Cost, 실제원가):**
- 의미: 현 시점까지 실제로 지출한 총비용
- 측정: 회계 시스템에서 집계된 실제 지출 합계
- 공식: AC = 모든 실제 지출의 합
- 예: 인건비 + 외주비 + 라이센스 = 4.5억
</details>

**Q6. EV가 EVM에서 가장 중요한 이유를 설명하세요.**

<details>
<summary>모범 답안</summary>
EV는 "실제로 완료한 작업의 가치"를 나타내는 핵심 지표로, EVM의 중심입니다.

**중요한 이유:**
1. **성과의 객관적 측정:** 단순히 "돈을 얼마 썼는가(AC)"가 아닌 "얼마의 가치를 만들었는가"를 측정하여 진짜 진행 상황을 파악합니다.

2. **일정과 원가의 통합:** EV를 기준으로 PV와 비교하면 일정 성과를, AC와 비교하면 원가 성과를 동시에 측정할 수 있습니다.

3. **조기 경보:** EV가 PV나 AC와 괴리되면 문제를 조기에 발견할 수 있습니다.

예시: AC=5억을 지출했을 때,
- EV=6억이면: 5억으로 6억 가치 창출 → 효율적 ✅
- EV=4억이면: 5억으로 4억 가치만 창출 → 비효율적 ❌

EV 없이는 단순히 "5억을 썼다"는 사실만 알 뿐, 그것이 좋은지 나쁜지 판단할 수 없습니다.
</details>

**Q7. 전통적 원가 관리(지출액만 추적)의 문제점을 EVM 관점에서 설명하세요.**

<details>
<summary>모범 답안</summary>
전통적 원가 관리는 AC(실제 지출액)만 추적하여 예산 대비 비교하는 방식으로, 다음과 같은 문제점이 있습니다:

**문제점 1: 진행 상황 파악 불가**
- AC만으로는 "얼마나 완료했는지" 알 수 없음
- 예: AC=4억 지출 시, 50% 완료인지 30% 완료인지 모름
- EVM: EV를 측정하여 실제 완료 정도 파악

**문제점 2: 잘못된 판단 유도**
- "적게 쓴다 = 잘한다"는 착각
- 예: AC가 예산 대비 80% → 절감으로 보이지만, 실제로는 진행이 느려서 적게 쓴 것일 수 있음
- EVM: EV=30%라면 실제로는 초과 지출 (30% 완료에 80% 지출)

**문제점 3: 일정과 원가 통합 불가**
- AC는 원가만 측정, 일정 성과는 별도 추적 필요
- EVM: EV 하나로 일정(EV vs PV)과 원가(EV vs AC) 모두 측정

**결론:**
전통적 방식은 "돈"만 보고 "가치"를 보지 못합니다. EVM은 "얼마를 써서 얼마의 가치를 만들었는가"를 통합 측정하여 프로젝트의 진짜 상태를 파악합니다.
</details>

### 이론 (50분)

#### 1. 편차 분석 (Variance Analysis)

**편차(Variance):**
계획 vs 실적의 **차이**를 정량화한 지표

**EVM의 2가지 핵심 편차:**
1. **SV (Schedule Variance, 일정 차이)** - 일정 성과
2. **CV (Cost Variance, 원가 차이)** - 원가 성과

#### **A. SV (Schedule Variance, 일정 차이)**

**정의:**
획득 가치와 계획 가치의 **차이** (일정 성과를 금액으로 표현)

**공식:**
```
SV = EV - PV
```

**해석:**
- **SV > 0:** 일정 앞섬 (Ahead of Schedule) ✅
  - 계획보다 더 많이 완료
  - EV가 PV보다 큼
  
- **SV = 0:** 계획대로 진행 (On Schedule) ✅
  
- **SV < 0:** 일정 지연 (Behind Schedule) ❌
  - 계획보다 덜 완료
  - EV가 PV보다 작음

**단위:**
- 금액 (원, 억원 등)
- 일정 차이를 **돈의 가치**로 표현

**예시:**
```
프로젝트: 모바일 앱 개발
BAC = 8억원
현재 시점 (Month 4):
- PV = 4억원 (50% 계획)
- EV = 3.2억원 (40% 완료)

SV = EV - PV = 3.2억 - 4억 = -0.8억원

해석:
- SV = -0.8억원 (마이너스)
- 일정 지연
- 0.8억원 가치만큼의 작업이 계획보다 덜 완료됨
- 진행률: 계획 50% vs 실제 40% (-10%p)
```

**주의사항:**
- SV는 프로젝트 완료 시 항상 0이 됨 (EV = PV = BAC)
- 프로젝트 후반부에는 일정 지표로서 의미 약화
- SPI (Schedule Performance Index)와 함께 사용

#### **B. CV (Cost Variance, 원가 차이)**

**정의:**
획득 가치와 실제 원가의 **차이** (원가 성과)

**공식:**
```
CV = EV - AC
```

**해석:**
- **CV > 0:** 예산 대비 절감 (Under Budget) ✅
  - 완료한 작업 대비 적게 지출
  - EV가 AC보다 큼
  
- **CV = 0:** 예산대로 지출 (On Budget) ✅
  
- **CV < 0:** 예산 초과 (Over Budget) ❌
  - 완료한 작업 대비 많이 지출
  - EV가 AC보다 작음

**단위:**
- 금액 (원, 억원 등)

**예시:**
```
프로젝트: 모바일 앱 개발
현재 시점 (Month 4):
- EV = 3.2억원 (40% 완료)
- AC = 4.5억원 (실제 지출)

CV = EV - AC = 3.2억 - 4.5억 = -1.3억원

해석:
- CV = -1.3억원 (마이너스)
- 예산 초과
- 3.2억 가치를 만들기 위해 4.5억 지출
- 1.3억원 초과 지출
- 효율성: 3.2 / 4.5 = 71% (29% 비효율)
```

**CV의 중요성:**
- 프로젝트 끝까지 의미 유지 (SV와 차이)
- 최종 예산 초과/절감 직접 예측
- 시정 조치의 우선순위 결정

### 2. 성과 지수 (Performance Indexes)

**지수(Index):**
비율로 표현한 성과 지표 (백분율 또는 배율)

**편차(Variance) vs 지수(Index):**
- 편차: 절대값 (금액)
- 지수: 상대값 (비율)
- 지수가 더 직관적이고 비교하기 쉬움

#### **A. SPI (Schedule Performance Index, 일정 성과 지수)**

**정의:**
획득 가치 대비 계획 가치의 **비율** (일정 효율성)

**공식:**
```
SPI = EV / PV
```

**해석:**
- **SPI > 1.0:** 일정 앞섬 ✅
  - 계획보다 빠름
  - 예: SPI = 1.2 → 20% 빠름
  
- **SPI = 1.0:** 계획대로 진행 ✅
  
- **SPI < 1.0:** 일정 지연 ❌
  - 계획보다 느림
  - 예: SPI = 0.8 → 20% 지연

**의미:**
```
SPI = 현재 진행 속도 / 계획 속도
```

**예시:**
```
프로젝트: CRM 시스템 구축
현재 시점:
- PV = 5억원
- EV = 6억원

SPI = EV / PV = 6억 / 5억 = 1.2

해석:
- SPI = 1.2 (> 1.0)
- 계획 대비 120% 속도로 진행
- 20% 빠름
- 진행률: 계획 50% vs 실제 60%
```

**인버스 해석:**
```
1 / SPI = 예상 완료 기간의 배율

SPI = 0.8일 때:
1 / 0.8 = 1.25
→ 계획 기간의 1.25배 소요 예상 (25% 지연)
```

#### **B. CPI (Cost Performance Index, 원가 성과 지수)**

**정의:**
획득 가치 대비 실제 원가의 **비율** (원가 효율성)

**공식:**
```
CPI = EV / AC
```

**해석:**
- **CPI > 1.0:** 예산 절감 ✅
  - 효율적으로 지출
  - 예: CPI = 1.25 → 25% 절감
  
- **CPI = 1.0:** 예산대로 지출 ✅
  
- **CPI < 1.0:** 예산 초과 ❌
  - 비효율적 지출
  - 예: CPI = 0.75 → 25% 초과

**의미:**
```
CPI = 1원당 획득한 가치
CPI = 투자 대비 효율성
```

**예시:**
```
프로젝트: 빅데이터 플랫폼
현재 시점:
- EV = 4억원
- AC = 5억원

CPI = EV / AC = 4억 / 5억 = 0.8

해석:
- CPI = 0.8 (< 1.0)
- 1원 투자로 0.8원의 가치 획득
- 25% 비효율 (1/0.8 = 1.25)
- 4억 가치를 만들기 위해 5억 지출 (1억 초과)
```

**CPI의 중요성:**
- **프로젝트 초반 CPI는 최종까지 유지되는 경향** (통계적 사실)
- 조기에 CPI 1.0 이하면 회복 어려움
- 완료 시 예산 예측의 핵심 지표

**경험 법칙:**
```
프로젝트 20% 진행 시점의 CPI는 최종 CPI와 90% 이상 상관관계
→ 조기 발견 및 조치가 중요!
```

### 3. 완료 시 예측 (Forecasting)

**Forecasting:**
현재까지의 성과를 바탕으로 **프로젝트 완료 시 지표를 예측**

**4가지 핵심 지표:**
1. **EAC** (Estimate at Completion) - 완료 시 총 원가 예측
2. **ETC** (Estimate to Complete) - 완료까지 필요한 추가 원가
3. **VAC** (Variance at Completion) - 완료 시 예산 대비 편차
4. **TCPI** (To-Complete Performance Index) - 목표 달성 위한 필요 CPI

#### **A. EAC (Estimate at Completion, 완료 시 산정치)**

**정의:**
프로젝트 완료 시 **예상 총 원가**

**4가지 계산 공식 (상황별):**

**공식 1: 현재 추세 유지 (가장 일반적)**
```
EAC = BAC / CPI
```

**사용 조건:**
- 현재 원가 성과가 계속 유지된다고 가정
- 특별한 이유 없이 CPI가 지속

**예시:**
```
BAC = 10억원
CPI = 0.8 (예산 초과 중)

EAC = 10억 / 0.8 = 12.5억원

해석: 현재 효율(CPI 0.8)이 계속되면 12.5억 소요 예상
→ 2.5억 초과 예상
```

**공식 2: 비전형적 편차 (일회성 문제)**
```
EAC = AC + (BAC - EV)
```

**사용 조건:**
- 현재까지의 편차가 **일회성**
- 향후는 계획대로 진행 예상
- 예: 초기 설계 오류는 해결됨

**예시:**
```
BAC = 10억원
AC = 5억원 (실제 지출)
EV = 4억원 (40% 완료)

EAC = 5억 + (10억 - 4억) = 11억원

해석: 지금까지 1억 초과했지만, 나머지는 계획대로 6억이면 완료
```

**공식 3: 현재 SPI와 CPI 모두 유지**
```
EAC = AC + [(BAC - EV) / (CPI × SPI)]
```

**사용 조건:**
- 일정과 원가 모두 영향 고려
- 복합 지수 적용

**예시:**
```
BAC = 10억원
AC = 5억원
EV = 4억원
CPI = 0.8
SPI = 0.85

EAC = 5억 + [(10억 - 4억) / (0.8 × 0.85)]
    = 5억 + [6억 / 0.68]
    = 5억 + 8.82억
    = 13.82억원

해석: 일정과 원가 모두 나빠 더 큰 초과 예상
```

**공식 4: 새로운 산정치 (리베이스라인)**
```
EAC = AC + 새로운 ETC
```

**사용 조건:**
- 과거 성과가 미래 예측에 부적절
- 프로젝트를 재계획함 (Rebaseline)

**예시:**
```
AC = 5억원 (지금까지 지출)
새로운 ETC = 7억원 (재산정)

EAC = 5억 + 7억 = 12억원
```

**EAC 공식 선택 가이드:**

| 상황 | 사용 공식 | 근거 |
|------|----------|------|
| 일반적 상황 | BAC / CPI | 현재 추세 유지 가능성 높음 |
| 초기 문제 해결됨 | AC + (BAC - EV) | 일회성 편차 |
| 일정/원가 모두 문제 | AC + [(BAC-EV)/(CPI×SPI)] | 복합 영향 |
| 대폭 변경/재계획 | AC + 새로운 ETC | 과거 무의미 |

#### **B. ETC (Estimate to Complete, 완료까지 산정치)**

**정의:**
프로젝트 완료까지 **추가로 필요한 원가**

**공식:**
```
ETC = EAC - AC
```

**의미:**
- "앞으로 얼마가 더 필요한가?"
- 남은 작업의 예상 비용

**예시:**
```
EAC = 12억원 (완료 시 예상)
AC = 5억원 (현재까지 지출)

ETC = 12억 - 5억 = 7억원

해석: 프로젝트 완료까지 7억원 추가 필요
```

**ETC 활용:**
- 남은 예산 확인
- 추가 자금 조달 필요성 판단
- 자원 재배분 계획

#### **C. VAC (Variance at Completion, 완료 시 편차)**

**정의:**
프로젝트 완료 시 **예산 대비 편차** (초과/절감 금액)

**공식:**
```
VAC = BAC - EAC
```

**해석:**
- **VAC > 0:** 예산 절감 예상 ✅
- **VAC = 0:** 예산대로 완료 예상 ✅
- **VAC < 0:** 예산 초과 예상 ❌

**예시:**
```
BAC = 10억원 (원래 예산)
EAC = 12억원 (완료 시 예상)

VAC = 10억 - 12억 = -2억원

해석:
- VAC = -2억원 (마이너스)
- 예산 2억원 초과 예상
- 추가 예산 확보 필요
```

**VAC 활용:**
- 경영진 보고 (예산 초과/절감 예측)
- 변경 요청 (추가 예산 요구)
- 프로젝트 중단/계속 의사결정

#### **D. TCPI (To-Complete Performance Index, 완료 성과 지수)**

**정의:**
프로젝트를 **목표 예산 내에 완료**하기 위해 **남은 작업에서 달성해야 할 CPI**

**2가지 공식:**

**공식 1: BAC 기준 (원래 예산)**
```
TCPI = (BAC - EV) / (BAC - AC)
```

**의미:**
- 원래 예산(BAC) 내에 완료하려면 필요한 CPI

**공식 2: EAC 기준 (수정 예산)**
```
TCPI = (BAC - EV) / (EAC - AC)
```

**의미:**
- 수정된 예산(EAC) 내에 완료하려면 필요한 CPI

**해석:**
- **TCPI > 현재 CPI:** 목표 달성 어려움 (성과 개선 필요) ❌
- **TCPI = 현재 CPI:** 현상 유지하면 목표 달성 ✅
- **TCPI < 현재 CPI:** 목표 달성 가능 (여유 있음) ✅

**예시:**
```
프로젝트: 데이터 분석 플랫폼
BAC = 10억원
EV = 4억원 (40% 완료)
AC = 5억원 (실제 지출)
현재 CPI = 4억 / 5억 = 0.8

TCPI (BAC 기준):
TCPI = (10억 - 4억) / (10억 - 5억)
     = 6억 / 5억
     = 1.2

해석:
- 남은 작업: 6억원 가치
- 남은 예산: 5억원
- 필요 CPI: 1.2
- 현재 CPI: 0.8

결론:
- TCPI(1.2) > 현재 CPI(0.8)
- 현재보다 50% 더 효율적으로 작업해야 BAC 내 완료 가능
- 현실적으로 매우 어려움 → 예산 초과 불가피
```

**TCPI 활용:**
- 목표 달성 가능성 판단
- 필요한 성과 개선 수준 정량화
- 현실적 목표(EAC) 재설정

### 4. EVM 지표 종합 정리

**기본 변수 (3개):**

| 변수 | 의미 | 측정 |
|------|------|------|
| PV | 계획가치 | BAC × 계획% |
| EV | 획득가치 | BAC × 실제% |
| AC | 실제원가 | 실제 지출 합계 |
| BAC | 완료 시 예산 | 원가 기준선 |

**편차 지표 (2개):**

| 지표 | 공식 | 의미 | 해석 |
|------|------|------|------|
| SV | EV - PV | 일정 차이 (금액) | > 0: 앞섬 / < 0: 지연 |
| CV | EV - AC | 원가 차이 (금액) | > 0: 절감 / < 0: 초과 |

**성과 지수 (2개):**

| 지표 | 공식 | 의미 | 해석 |
|------|------|------|------|
| SPI | EV / PV | 일정 효율성 (비율) | > 1: 앞섬 / < 1: 지연 |
| CPI | EV / AC | 원가 효율성 (비율) | > 1: 절감 / < 1: 초과 |

**예측 지표 (4개):**

| 지표 | 공식 | 의미 |
|------|------|------|
| EAC | BAC / CPI 등 | 완료 시 예상 총 원가 |
| ETC | EAC - AC | 완료까지 추가 필요 원가 |
| VAC | BAC - EAC | 완료 시 예산 대비 편차 |
| TCPI | (BAC-EV)/(BAC-AC) | 목표 달성 위한 필요 CPI |

### 5. EVM 그래프 해석

**기본 EVM 그래프:**

```
     누적 원가 (억원)
     ↑
  12 |                  AC (실제)
     |              __--
  10 |          __-  BAC
     |      __--   
   8 |  __--  EAC
     | /  PV (계획)
   6 |/     
     |  EV (획득)
   4 |
     |
   2 |
     |_____________________________→ 시간
     현재
```

**현재 시점 분석:**
- PV (계획선): 예정된 경로
- EV (획득선): 실제 진행 경로 (PV보다 아래 = 지연)
- AC (지출선): 실제 지출 (EV보다 위 = 초과)

**미래 예측:**
- BAC (계획 종점): 10억
- EAC (예상 종점): 12억 (초과 예상)

**6가지 대표 패턴:**

**패턴 1: 이상적 프로젝트 ✅**
```
         $
         ↑
     EV |     ●●●    (가장 위)
        |   ●
     PV |  ●●       (중간)
        | ●
     AC | ●         (가장 아래)
        |____________→ t
```
- EV > PV: 일정 앞섬
- EV > AC: 예산 절감

**패턴 2: 일정 지연, 예산 절감 ⚠️**
```
         $
         ↑
     PV |     ●●●    (가장 위)
        |   ●
     EV |  ●●       (중간)
        | ●
     AC | ●         (가장 아래)
        |____________→ t
```
- PV > EV: 일정 지연
- EV > AC: 예산 절감

**패턴 3: 일정 앞섬, 예산 초과 ⚠️**
```
         $
         ↑
     AC |     ●●●    (가장 위)
        |   ●
     EV |  ●●       (중간)
        | ●
     PV | ●         (가장 아래)
        |____________→ t
```
- EV > PV: 일정 앞섬
- AC > EV: 예산 초과
- 급하게 진행하느라 비용 증가

**패턴 4: 위험 상태 ❌**
```
         $
         ↑
     AC |     ●●●    (가장 위)
        |   ●
     PV |  ●●       (중간)
        | ●
     EV | ●         (가장 아래)
        |____________→ t
```
- PV > EV: 일정 지연
- AC > EV: 예산 초과
- 최악의 상황

### 예시 (25분)

#### 예시 1: 종합 EVM 계산

**프로젝트:** 차세대 물류 시스템 구축 (12개월)

**Month 6 시점:**
- BAC: 12억원
- 계획 진행률: 55%
- 실제 진행률: 45%
- 실제 지출 (AC): 7억원

**1단계: 기본 변수 계산**
```
PV = BAC × 계획%
   = 12억 × 55% = 6.6억원

EV = BAC × 실제%
   = 12억 × 45% = 5.4억원

AC = 7억원 (주어짐)
```

**2단계: 편차 계산**
```
SV = EV - PV
   = 5.4억 - 6.6억 = -1.2억원
   → 일정 지연 (1.2억 가치만큼 뒤처짐)

CV = EV - AC
   = 5.4억 - 7억 = -1.6억원
   → 예산 초과 (1.6억 초과 지출)
```

**3단계: 성과 지수 계산**
```
SPI = EV / PV
    = 5.4억 / 6.6억 = 0.818
    → 계획 대비 81.8% 속도 (18.2% 지연)

CPI = EV / AC
    = 5.4억 / 7억 = 0.771
    → 1원당 0.77원 효율 (22.9% 비효율)
```

**4단계: 완료 시 예측**
```
EAC = BAC / CPI
    = 12억 / 0.771 = 15.6억원
    → 현재 효율 유지 시 15.6억 소요 예상

ETC = EAC - AC
    = 15.6억 - 7억 = 8.6억원
    → 완료까지 8.6억 추가 필요

VAC = BAC - EAC
    = 12억 - 15.6억 = -3.6억원
    → 3.6억 초과 예상

TCPI (BAC 기준):
TCPI = (BAC - EV) / (BAC - AC)
     = (12억 - 5.4억) / (12억 - 7억)
     = 6.6억 / 5억
     = 1.32
     → BAC 내 완료하려면 CPI 1.32 필요
```

**5단계: 종합 진단**
```
현재 상태:
✗ 일정: 18.2% 지연 (SPI 0.82)
✗ 원가: 22.9% 비효율 (CPI 0.77)
✗ 예측: 3.6억 (30%) 초과 예상

TCPI 분석:
- 필요 CPI: 1.32
- 현재 CPI: 0.77
- 격차: 55% 성과 개선 필요 (거의 불가능)

권고 사항:
1. 경영진에게 상황 보고 (Red Alert)
2. 추가 예산 3.6억 요청
3. 범위 축소 검토 (우선순위 기능만)
4. 전문 인력 투입 (CPI 개선)
5. 프로젝트 재계획 (리베이스라인)
```

#### 예시 2: 4가지 EAC 공식 비교

**프로젝트:** 전사 ERP 통합 (18개월, 20억원)

**Month 9 시점:**
- BAC: 20억원
- PV: 12억원 (60% 계획)
- EV: 9억원 (45% 완료)
- AC: 11억원 (실제 지출)
- CPI = 9억 / 11억 = 0.818
- SPI = 9억 / 12억 = 0.75

**공식 1: 현재 추세 유지**
```
EAC = BAC / CPI
    = 20억 / 0.818
    = 24.4억원

해석: 현재 효율(CPI 0.82)이 계속되면 24.4억 소요
초과: 4.4억 (22%)
```

**공식 2: 비전형적 편차**
```
EAC = AC + (BAC - EV)
    = 11억 + (20억 - 9억)
    = 11억 + 11억
    = 22억원

해석: 초기 문제는 해결, 향후 계획대로 진행 가정
초과: 2억 (10%)
```

**공식 3: CPI × SPI 복합**
```
EAC = AC + [(BAC - EV) / (CPI × SPI)]
    = 11억 + [(20억 - 9억) / (0.818 × 0.75)]
    = 11억 + [11억 / 0.614]
    = 11억 + 17.9억
    = 28.9억원

해석: 일정과 원가 모두 나쁜 상황 반영
초과: 8.9억 (44.5%)
```

**공식 4: 재산정 (리베이스라인)**
```
상황: 프로젝트 재검토 후 범위 조정
- 핵심 모듈만 구현 결정
- 새로운 ETC: 10억원

EAC = AC + 새로운 ETC
    = 11억 + 10억
    = 21억원

해석: 범위 축소로 21억에 완료 계획
초과: 1억 (5%)
```

**어떤 공식 선택?**

| 상황 판단 | 선택 공식 | 예상 EAC | 근거 |
|----------|----------|---------|------|
| 현재 추세 계속 | 1번 | 24.4억 | CPI 0.82가 유지 |
| 초기 문제만 | 2번 | 22억 | 일회성 어려움 |
| 모든 문제 계속 | 3번 | 28.9억 | 일정+원가 악화 |

**PM 결정:**
```
상황 분석:
- 초기 레거시 통합 어려움 (해결됨)
- 팀 역량은 향상 중
- 일정 지연은 불가피하나 원가는 통제 가능

선택: 공식 2번 (비전형적 편차)
EAC = 22억원
```

#### 주관식 문제

<details>
<summary>모범 답안</summary>
**공식 1: EAC = BAC / CPI**
- 상황: 현재 원가 성과가 계속 유지될 것으로 예상
- 가장 일반적으로 사용
- 예: CPI 0.8이 지속되면 BAC/0.8 소요

**공식 2: EAC = AC + (BAC - EV)**
- 상황: 현재까지의 편차가 비전형적(일회성)
- 향후는 계획대로 진행 예상
- 예: 초기 설계 오류는 해결됨, 이후 정상 진행

**공식 3: EAC = AC + [(BAC - EV) / (CPI × SPI)]**
- 상황: 일정과 원가 성과 모두 고려
- 복합적 영향 반영
- 예: 일정 지연과 예산 초과가 함께 진행

**공식 4: EAC = AC + 새로운 ETC**
- 상황: 프로젝트를 재계획(Rebaseline)
- 과거 성과가 미래 예측에 부적절
- 예: 범위 변경, 대폭 수정 후 재산정

선택 가이드: 일반적으로 공식 1을 사용하며, 특수한 상황에서 다른 공식을 선택합니다.
</details>

**Q6. TCPI가 현재 CPI보다 높을 때의 의미와 PM이 취해야 할 조치를 설명하세요.**

<details>
<summary>모범 답안</summary>
**의미:**
TCPI > 현재 CPI는 "목표를 달성하려면 지금보다 더 좋은 성과가 필요하다"는 뜻입니다악화됩니다.

예: 현재 CPI 0.8, TCPI 1.2
→ 앞으로 1.2 CPI를 달성해야 BAC 내 완료 가능
→ 현재보다 50% 성과 개선 필요

**PM의 조치:**

1. **격차가 작을 때 (TCPI가 현재 CPI보다 10-20% 높음):**
   - 달성 가능으로 판단
   - 프로세스 개선
   - 비효율 요소 제거
   - 팀 동기부여 및 교육
   - 외주 단가 재협상

2. **격차가 클 때 (TCPI가 현재 CPI보다 30% 이상 높음):**
   - 현실적으로 불가능 판단
   - BAC 포기, EAC 재설정
   - 경영진에게 추가 예산 요청
   - 범위 축소 검토
   - 프로젝트 재계획 (리베이스라인)

3. **지속적 모니터링:**
   - 주간 CPI 추적
   - TCPI 재계산
   - 목표 달성 가능성 재평가

예시: 
TCPI 1.32, 현재 CPI 0.77 (격차 71%)
→ 현실적으로 불가능
→ 추가 예산 요청 또는 범위 축소
</details>

**Q7. SPI와 SV의 차이점을 설명하고, 각각의 장단점을 쓰세요.**

<details>
<summary>모범 답안</summary>
**정의:**
- SV (Schedule Variance) = EV - PV (금액 차이)
- SPI (Schedule Performance Index) = EV / PV (비율)

**차이점:**
1. **표현 방식:**
   - SV: 절대값 (예: -2억원)
   - SPI: 상대값 (예: 0.8)

2. **의미:**
   - SV: "얼마나 많이" 지연/앞섬 (금액 기준)
   - SPI: "얼마나 빠르게/느리게" 진행 (속도 기준)

**SV의 장단점:**
- 장점: 구체적 금액으로 편차 파악 (2억 지연)
- 단점: 프로젝트 간 비교 어려움 (10억 프로젝트의 2억 vs 100억 프로젝트의 2억)
- 단점: 프로젝트 완료 시 항상 0 (후반부 의미 약화)

**SPI의 장단점:**
- 장점: 비율로 직관적 (0.8 = 20% 느림)
- 장점: 프로젝트 간 비교 가능
- 장점: 완료 예상 기간 계산 가능 (기간 / SPI)
- 단점: 프로젝트 후반부에는 SPI도 의미 약화

**활용:**
두 지표를 함께 사용하는 것이 효과적입니다.
- SV로 구체적 편차 파악
- SPI로 전체적 건강도 및 추세 파악

예: SV = -2억, SPI = 0.8
→ "2억 가치만큼 지연되었으며, 계획 대비 80% 속도로 진행 중"
</details>

**해설 (강사용):**

**Part 1: EVM 지표 계산**

```
1. 기본 변수:
   BAC = 30억원
   
   PV = BAC × 계획%
      = 30억 × 68%
      = 20.4억원
   
   EV = BAC × 실제%
      = 30억 × 60%
      = 18억원
   
   AC = 22억원 (주어짐)

2. 편차:
   SV = EV - PV
      = 18억 - 20.4억
      = -2.4억원
   
   해석: 2.4억 가치만큼 지연 (8% 지연)
   
   CV = EV - AC
      = 18억 - 22억
      = -4억원
   
   해석: 4억 초과 지출 (22% 초과)

3. 성과 지수:
   SPI = EV / PV
       = 18억 / 20.4억
       = 0.882
   
   해석: 계획 대비 88.2% 속도 (11.8% 지연, Yellow)
   
   CPI = EV / AC
       = 18억 / 22억
       = 0.818
   
   해석: 1원당 0.82원 효율 (18.2% 비효율, Red)

4. EAC:
   EAC = BAC / CPI
       = 30억 / 0.818
       = 36.7억원
   
   해석: 현재 효율 유지 시 36.7억 소요 예상

5. ETC, VAC:
   ETC = EAC - AC
       = 36.7억 - 22억
       = 14.7억원
   
   해석: 완료까지 14.7억 추가 필요
   
   VAC = BAC - EAC
       = 30억 - 36.7억
       = -6.7억원
   
   해석: 6.7억 초과 예상 (22.3% 초과)

6. TCPI:
   TCPI = (BAC - EV) / (BAC - AC)
        = (30억 - 18억) / (30억 - 22억)
        = 12억 / 8억
        = 1.5
   
   분석:
   - 필요 CPI: 1.5
   - 현재 CPI: 0.82
   - 격차: 83% 성과 개선 필요
   
   판단: 거의 불가능 → BAC 포기, 추가 예산 필요
```

**Part 2: 상태 진단**

```
7. 일정 상태:
   - SPI = 0.882 (< 0.95)
   - SV = -2.4억
   - 판단: 지연 (Yellow)
   - 근거: 계획 68% vs 실제 60% (8%p 차이)

8. 원가 상태:
   - CPI = 0.818 (< 0.85)
   - CV = -4억
   - 판단: 심각한 초과 (Red)
   - 근거: 18억 가치를 만들기 위해 22억 지출

9. 전체 상태:
   - 판단: 🔴 RED (위기)
   - 이유: CPI가 Red 수준 (0.818 < 0.85)
   - 일정과 원가 모두 문제

10. 주요 문제점:
    1. 보안 인증 추가 요구사항 (미지의 리스크 발생)
       → 1.5억 추가 비용, BAC 초과의 주 원인
    
    2. 외부 API 연동 지연 (2개월)

    3. 핵심 인력 교체로 생산성 저하
       → CPI 악화 (재작업, 인수인계 시간 손실)
       → 팀 사기 저하 우려
```

**Q2. EVM 보고서에서 가장 중요한 지표는?**
1. SV (Schedule Variance)
2. SPI (Schedule Performance Index)
3. CV (Cost Variance)
4. CPI (Cost Performance Index)

<summary>정답 보기</summary>
정답: 4번
해설: CPI는 원가 효율성을 나타내는 가장 중요한 지표입니다. 이유는: 1) 프로젝트 전체에서 의미 유지 (SPI는 후반부 약화), 2) 최종 예산 초과/절감을 직접 예측 (EAC = BAC/CPI), 3) 조기 CPI는 최종 CPI와 높은 상관관계. 통계적으로 프로젝트 20% 시점의 CPI가 최종 성과를 90% 예측합니다.
</details>

#### 주관식 문제

**Q5. EVM을 적용할 때 진행률 측정이 중요한 이유를 설명하고, 정확한 측정을 위한 방법 3가지를 제시하세요.**

<details>
<summary>모범 답안</summary>
**진행률 측정이 중요한 이유:**

EV(획득가치) = BAC × 실제 진행률이므로, 진행률 측정의 정확도가 EVM 전체 신뢰도를 결정합니다. 진행률이 부정확하면:
- EV가 왜곡되어 SV, CV 모두 부정확
- CPI, SPI 지표가 의미 없어짐
- 잘못된 의사결정 초래
- "90% 완료 증후군" 발생 (실제는 70%인데 90%로 보고)

**정확한 측정 방법:**

1. **명확한 측정 규칙 수립:**
   - 0-100 규칙: 완료 시에만 100% 인정 (보수적, 단순 작업)
   - 50-50 규칙: 시작 50%, 완료 100% (중간 규모 작업)
   - 20-80 규칙: 시작 20%, 완료 100% (큰 작업)
   - 규칙을 일관되게 적용

2. **작업 패키지 크기 축소:**
   - 작업 기간을 2주 이하로 제한
   - 작을수록 완료/미완료 판단이 명확
   - 중간 진행 애매함 최소화
   - 예: "개발" (×) → "로그인 화면 개발" (○)

3. **객관적 완료 기준 정의:**
   - 인도물(Deliverable) 기반 측정
   - 체크리스트 활용 (5개 중 3개 완료 = 60%)
   - 코드 리뷰/테스트 통과 등 검증 가능 기준
   - 주관적 판단 배제

예시:
- 나쁜 예: "70% 정도 진행된 것 같다" (주관적)
- 좋은 예: "10개 화면 중 7개 개발 완료, 테스트 통과" (객관적)
</details>

**Q6. CPI가 0.75인 프로젝트의 PM이 되었습니다. CPI를 개선하기 위한 구체적인 조치 5가지를 제시하고, 각각의 예상 효과를 설명하세요.**

<details>
<summary>모범 답안</summary>
CPI 0.75는 매우 심각한 비효율 상태(25% 초과)입니다. 다음 조치를 통해 개선 가능합니다:

**1. 프로세스 개선 및 낭비 제거**
- 조치: 재작업 원인 분석, 품질 관리 강화, 불필요한 회의 축소
- 효과: 재작업 30% 감소 → CPI 0.05~0.10 개선
- 예: 코드 리뷰 체계 도입으로 버그 조기 발견

**2. 외주 단가 재협상**
- 조치: 경쟁 입찰, 장기 계약 조건 제시, 성과급 도입
- 효과: 단가 10~15% 절감 → CPI 0.03~0.05 개선
- 예: 4개 외주사 경쟁 입찰로 평균 12% 단가 인하

**3. 내부 자원 활용 증대**
- 조치: 외주 대신 사내 인력 활용, 공용 컴포넌트 재사용
- 효과: 외주비 20% 절감 → CPI 0.05 개선
- 예: 인증 모듈을 타 프로젝트에서 재사용

**4. 클라우드/인프라 최적화**
- 조치: 사용량 모니터링, 불필요 자원 제거, RI/Savings Plan 활용
- 효과: 인프라 비용 25% 절감 → CPI 0.02~0.04 개선
- 예: 개발 환경을 야간 자동 종료로 30% 절감

**5. 생산성 향상**
- 조치: 자동화 도구 도입, 교육, 시니어 멘토링
- 효과: 개발 속도 20% 향상 → CPI 0.10~0.15 개선
- 예: CI/CD 파이프라인 자동화로 배포 시간 75% 단축

**종합 예상 효과:**
- 현재 CPI: 0.75
- 개선 합계: +0.25~0.40
- 목표 CPI: 1.00~1.15 (예산 내 또는 절감)

**주의사항:**
모든 조치를 동시에 적용하되, 품질 저하 없이 진행해야 합니다. 급하게 CPI를 올리려다 품질이 떨어지면 나중에 더 큰 비용이 발생할 수 있습니다.
</details>

**Q7. 프로젝트 후반부(80% 진행)에 SPI 0.85, CPI 0.90인 상황입니다. 프로젝트를 성공적으로 마무리하기 위한 전략을 수립하세요.**

- 진행: 80% (후반부)
