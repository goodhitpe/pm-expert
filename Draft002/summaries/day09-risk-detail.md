## 1교시: 리스크 관리 개요 <!-- 슬라이드 #1~#2 -->

### 이론 (50분)

#### 1. 리스크(Risk)의 정의

**PMBOK 정의:**  
발생하면 프로젝트 목표에 긍정적/부정적 영향을 미치는 **불확실한 사건 또는 조건**

**핵심 키워드:**
- **불확실성** (Uncertainty): 발생할 수도, 안 할 수도 있음
- **미래** (Future): 아직 발생하지 않음
- **영향** (Impact): 목표(범위, 일정, 원가, 품질)에 영향

#### 2. 리스크 vs 이슈(Issue)

| 구분 | 리스크 (Risk) | 이슈 (Issue) |
|------|--------------|-------------|
| **발생 시점** | 미래 (아직 안 일어남) | 현재 (이미 일어남) |
| **불확실성** | 있음 (발생 가능성 < 100%) | 없음 (100% 확실) |
| **대응** | 예방, 완화, 우발 계획 | 즉각적 조치, 문제 해결 |
| **관리 접근** | 사전적(Proactive) | 사후적(Reactive) |
| **예시** | "AWS 장애 발생 가능" | "AWS 지금 다운됨" |

**리스크가 이슈화:**
```
리스크: "핵심 개발자가 퇴사할 수 있다" (확률 30%)
→ 대응: 지식 문서화, 백업 인력 준비

이슈: "핵심 개발자가 어제 퇴사했다" (확률 100%)
→ 조치: 즉시 대체 인력 투입, 인수인계
```

#### 3. 리스크의 유형

**A. 영향에 따른 분류**

**1) 위협 (Threat) - 부정적 리스크**
- 일정 지연, 비용 증가, 품질 저하 초래
- 대응: 회피, 전가, 완화, 수용
- 예: 보안 사고, 핵심 인력 퇴사, 기술 실패

**2) 기회 (Opportunity) - 긍정적 리스크**
- 일정 단축, 비용 절감, 품질 향상 가능
- 대응: 활용, 공유, 증대, 수용
- 예: 신기술 도입 성공, 조기 완료, 예산 절감

**B. 범위에 따른 분류**

**1) 개별 리스크 (Individual Risk)**
- 특정 활동, 패키지, 인도물에 영향
- 예: "데이터 마이그레이션 실패 가능"

**2) 전체 프로젝트 리스크 (Overall Project Risk)**
- 프로젝트 전체에 영향
- 모든 개별 리스크의 종합 효과
- 예: "프로젝트가 완전 실패할 수 있음"

#### 4. 리스크 관리의 7개 프로세스

**기획 프로세스 그룹:**

**1) 리스크 관리 계획 수립 (Plan Risk Management)**
- 리스크 관리 활동 수행 방법 정의
- 산출물: 리스크 관리 계획서

**2) 리스크 식별 (Identify Risks)**
- 개별 프로젝트 리스크 식별 및 문서화
- 산출물: 리스크 등록부 (Risk Register)

**3) 정성적 리스크 분석 (Perform Qualitative Risk Analysis)**
- 리스크 우선순위화 (확률 × 영향)
- 산출물: 리스크 등록부 업데이트

**4) 정량적 리스크 분석 (Perform Quantitative Risk Analysis)**
- 리스크의 수치적 영향 분석 (EMV, 시뮬레이션)
- 산출물: 리스크 등록부 업데이트

**5) 리스크 대응 계획 수립 (Plan Risk Responses)**
- 리스크 대응 전략 개발
- 산출물: 리스크 등록부 업데이트, 변경 요청

**실행 프로세스 그룹:**

**6) 리스크 대응 실행 (Implement Risk Responses)**
- 합의된 리스크 대응 계획 실행
- 산출물: 변경 요청

**감시/통제 프로세스 그룹:**

**7) 리스크 감시 (Monitor Risks)**
- 리스크 대응 계획 이행 모니터링
- 새로운 리스크 식별
- 산출물: 작업 성과 정보, 변경 요청

### 예시 (25분)

#### 예시 1: 리스크 vs 이슈 구분 실전

**IT 시스템 구축 프로젝트 사례:**

| 항목 | 리스크 or 이슈? | 이유 | 대응 방법 |
|------|--------------|------|---------|
| "AWS 리전 장애가 발생할 수 있다" | 리스크 | 아직 발생하지 않음 (불확실) | 멀티 리전 구성 (완화) |
| "오늘 오전 AWS 장애로 배포 불가" | 이슈 | 이미 발생함 (확실) | 즉각 비상 배포 계획 실행 |
| "핵심 개발자가 이직할 수도 있다" | 리스크 | 아직 발생 안 함 | 지식 문서화, 백업 인력 준비 |
| "DB 팀장이 어제 퇴사했다" | 이슈 | 발생함 | 즉시 대체 인력 투입 |
| "요구사항이 변경될 가능성이 있다" | 리스크 | 불확실한 미래 사건 | 변경 관리 절차 강화 |

#### 예시 2: 리스크 유형별 분류

**프로젝트:** 의료 정보 시스템 구축

**위협 (Threats) - 부정적 리스크:**
```
1. 의료법 규정 변경 (확률: 20%, 영향: 매우 높음)
2. 핵심 시스템 설계자 이직 (확률: 30%, 영향: 높음)
3. 타 병원 통합 API 연동 실패 (확률: 40%, 영향: 높음)
4. 의료 데이터 보안 사고 (확률: 10%, 영향: 치명적)
```

**기회 (Opportunities) - 긍정적 리스크:**
```
1. AI 진단 보조 기능 조기 완성 시 특허 출원 기회 (확률: 25%)
2. 클라우드 비용 협상 성공 시 예산 절감 (예상 절감: 20%)
3. 프로젝트 완료 조기 달성 시 다음 계약 우선권 확보 (확률: 30%)
```

**핵심:** 기회도 리스크입니다. PM은 위협뿐만 아니라 기회도 적극적으로 관리해야 합니다.

### 퀴즈 (15분)

**Q1. 다음 중 리스크(Risk)가 아닌 것은?**

A) 핵심 인력이 프로젝트 도중 이직할 수 있다  
B) 현재 인프라가 성능 요구사항을 충족하지 못한다  
C) 범위가 변경될 가능성이 있다  
D) 신기술 도입 시 개발 기간이 단축될 수 있다

**정답: B**

**해설:** B는 이미 발생한 확인된 사실이므로 이슈입니다. A, C, D는 아직 발생하지 않은 불확실한 사건으로 리스크입니다.

**Q2. 긍정적 리스크(기회)의 대응 전략이 아닌 것은?**

A) 활용 (Exploit)  
B) 공유 (Share)  
C) 전가 (Transfer)  
D) 증대 (Enhance)

**정답: C**

**해설:** 전가(Transfer)는 위협 대응 전략입니다. 기회 대응은 활용, 공유, 증대, 수용입니다.

## 2교시: 리스크 식별 (1.5시간) <!-- 슬라이드 #3 -->

### 이론 (50분)

#### 1. 리스크 식별의 중요성

**공식:** "식별되지 않은 리스크는 관리할 수 없다"

**특징:**
- 반복적(Iterative) 프로세스: 프로젝트 내내 계속
- 모든 이해관계자 참여 권장
- "위협과 기회" 모두 식별

**리스크 식별 정보 수집 기법:**

#### 2. 브레인스토밍 (Brainstorming)

**일반 브레인스토밍:**
- 자유롭게 아이디어 제시
- 비판 없음 (수렴 전 발산)
- 모든 아이디어 기록

**명목 집단 기법 (NGT):**
- 각자 독립적으로 리스크 목록 작성
- 익명 제출 → 투표
- 편향 방지 효과

**델파이 기법 (Delphi Technique):**
- 전문가 집단의 익명 의견 수렴
- 반복 설문 및 피드백
- 합의 도달까지 반복
- 객관적 판단, 권위자 영향 차단

#### 3. 체크리스트 (Checklist)

**과거 프로젝트 기반 리스크 목록 활용:**

**IT 프로젝트 리스크 체크리스트:**

**기술 리스크:**
- [ ] 새 기술/플랫폼 도입 (경험 부족)
- [ ] 시스템 통합 복잡성
- [ ] 성능/확장성 요구사항 불명확
- [ ] 레거시 시스템 연동 문제
- [ ] 보안 취약점

**관리 리스크:**
- [ ] 요구사항 불명확 또는 변경 빈도
- [ ] 이해관계자 참여 부족
- [ ] PM 경험 부족
- [ ] 자원 가용성 불확실

**외부 리스크:**
- [ ] 법규 변경
- [ ] 외주업체 성과 미달
- [ ] 시장 환경 변화
- [ ] 하드웨어/소프트웨어 가격 변동

**조직 리스크:**
- [ ] 경영진 지원 약화
- [ ] 변화 저항
- [ ] 우선순위 경쟁

#### 4. SWOT 분석

**리스크 관리에 활용:**

| 구분 | 내부 | 외부 |
|------|------|------|
| 긍정적 | 강점(S) → 기회 활용 발판 | 기회(O) → 긍정적 리스크 |
| 부정적 | 약점(W) → 위협 취약점 | 위협(T) → 부정적 리스크 |

**IT 프로젝트 SWOT 리스크 예시:**
```
강점: AI 개발 전문 팀 보유
→ 기회: AI 기능 조기 완성 가능 (긍정적 리스크)

약점: 클라우드 경험 부족
→ 위협: 클라우드 마이그레이션 실패 (부정적 리스크)
```

#### 5. 리스크 등록부 (Risk Register)

**정의:** 식별된 모든 리스크와 분석 결과, 대응 계획을 기록하는 문서

**필수 필드:**

| 필드 | 설명 | 예시 |
|------|------|------|
| 리스크 ID | 고유 번호 | R-001 |
| 범주 | 기술/관리/외부/조직 | 기술 |
| 리스크 설명 | 구체적 사건 기술 | "API 연동 지연" |
| 원인(Cause) | 왜 발생하는가? | 외주업체 개발 역량 부족 |
| 영향(Effect) | 목표에 미치는 영향 | 일정 4주 지연 |
| 확률(P) | 발생 가능성 (0-1) | 0.4 |
| 영향도(I) | 목표에 미치는 크기 | 높음(0.8) |
| P×I 점수 | 우선순위 | 0.32 |
| 대응 전략 | 회피/전가/완화/수용 | 완화 |
| 리스크 소유자 | 책임자 | 개발팀장 |
| 잔여 리스크 | 대응 후 남은 리스크 | 낮음 |
| 상태 | 활성/종결/발생 | 활성 |

#### 6. 리스크 분류 체계 (RBS: Risk Breakdown Structure)

**목적:** 리스크를 체계적으로 범주화하여 빠짐없이 식별

**IT 프로젝트 RBS:**
```
프로젝트 리스크
├── 기술 리스크
│   ├── 복잡성
│   ├── 인터페이스
│   ├── 성능
│   └── 보안
├── 관리 리스크
│   ├── 요구사항 관리
│   ├── 일정 관리
│   └── 자원 관리
├── 외부 리스크
│   ├── 협력업체
```

## 3교시: 정성적 리스크 분석 (1.5시간) <!-- 슬라이드 #4 -->

### 이론 (50분)

#### 1. 정성적 리스크 분석 개요

**목적:** 주관적 판단으로 리스크 우선순위 결정 (빠르고 효율적)

**방법:** 확률(Probability) × 영향(Impact) = 리스크 점수

**차별점:**
- 정성적: 전문가 판단, 빠름, 모든 프로젝트에 적용
- 정량적: 수치, 시간/비용 많이 필요, 복잡한 프로젝트에 적용

#### 2. 확률-영향 매트릭스 (P-I Matrix)

**구성:**
- X축: 영향도 (Impact) - 낮음/보통/높음/매우 높음
- Y축: 확률 (Probability) - 낮음/보통/높음

**색상 구분:**
- 🔴 Red (High): P×I > 0.25 → 즉각 대응 필요
- 🟡 Yellow (Medium): 0.10 ≤ P×I ≤ 0.25 → 모니터링 + 대응 계획
- 🟢 Green (Low): P×I < 0.10 → 수용 (기록만)

**📊 P-I 매트릭스 (확률-영향 매트릭스) 시각화:**

```
  확률(P)
    ↑
    │
高  │  🟡 중간    │  🔴 높음    │  🔴 높음
0.7 │ 0.07~0.21  │ 0.49        │ 0.63
    │  모니터링   │  즉각 대응  │  즉각 대응
    ├────────────┼─────────────┼────────────
中  │  🟢 낮음    │  🟡 중간    │  🔴 높음
0.4 │  0.04~0.12  │ 0.12~0.28  │ 0.36
    │  수용 가능  │  계획 수립  │  즉각 대응
    ├────────────┼─────────────┼────────────
低  │  🟢 낮음    │  🟢 낮음    │  🟡 중간
0.2 │  0.02~0.06  │ 0.06       │ 0.14~0.18
    │  수용 가능  │  수용 가능  │  모니터링
    └────────────┴─────────────┴────────────▶ 영향도(I)
           低(0.1)       中(0.3)       高(0.7~0.9)

  색상 기준:
  🔴 Red = P×I ≥ 0.25  → 리스크 대응 계획 즉시 수립
  🟡 Yellow = 0.10 ≤ P×I < 0.25  → 모니터링 + 대응 계획 준비
  🟢 Green = P×I < 0.10  → Watch List에 등록, 수용

  수험 팁: P×I 점수 계산 후 0.25 기준으로 Red/Yellow/Green 판단
```

**P-I 매트릭스 예시:**

```
영향도:  낮음(0.1) | 보통(0.3) | 높음(0.7) | 매우높음(0.9)
확률
높음(0.7):  0.07🟢   0.21🟡    0.49🔴     0.63🔴
보통(0.4):  0.04🟢   0.12🟡    0.28🔴     0.36🔴 
낮음(0.2):  0.02🟢   0.06🟢    0.14🟡     0.18🟡
```

#### 3. 리스크 우선순위화

**순위 결정 기준:**

1. **P×I 점수**: 기본 우선순위 지표
2. **근접성(Proximity)**: 언제 발생하는가? 가까울수록 우선
3. **긴급성(Urgency)**: 얼마나 빨리 대응해야 하는가?
4. **탐지 가능성(Detectability)**: 발생을 얼마나 빨리 알 수 있는가?
5. **관리 가능성(Manageability)**: 팀이 영향을 줄일 수 있는가?

**Watch List:**
- P×I 점수가 낮아 Watch List에 등록된 리스크
- 주기적으로 재평가
- 변화 시 즉시 재분석

#### 4. 데이터 품질 평가

**리스크 분석의 신뢰도 체크:**

| 질문 | 신뢰도 낮음 | 신뢰도 높음 |
|------|-----------|-----------|
| 데이터 출처? | 추측 | 검증된 데이터 |
| 전문가 합의? | 1인 의견 | 다수 합의 |
| 유사 프로젝트 경험? | 없음 | 있음 |
| 항목 명확성? | 모호함 | 명확함 |

**낮은 신뢰도 리스크:**
- 확률 범위를 넓게 설정 (예: 30~50%)
- 추가 정보 수집 계획 수립
- 정기 재평가 강화

## 4교시: 정량적 리스크 분석 (1.5시간) <!-- 슬라이드 #5 -->

### 이론 (50분)

#### 1. 정량적 리스크 분석 개요

**목적:** 수치적 분석으로 리스크의 프로젝트 목표에 대한 영향을 정량화

**적용 대상:**
- 높은 우선순위 리스크 (Red 등급 위주)
- 복잡하고 대규모 프로젝트
- 추가 분석 가치가 있는 경우

**주요 기법:**

#### 2. 기대화폐가치 (EMV: Expected Monetary Value)

**공식:**
$$EMV = 확률(P) \times 영향(I)$$

**단위:** 확률은 %, 영향은 금액

**적용 예시:**

```
리스크 A: AWS 서비스 중단
- 발생 확률: 20%
- 발생 시 손실: 5,000만원 (복구 비용)
- EMV = 0.20 × 5,000만 = 1,000만원

해석: 이 리스크에 대한 대응 비용 예산은 1,000만원 이하로 설정 권장
→ 멀티 리전 구성 비용이 800만원이면 → 채택
→ 멀티 리전 구성 비용이 2,000만원이면 → 미채택 (수용)
```

**기회(Opportunity)의 EMV:**
```
기회 B: 클라우드 비용 협상 성공
- 발생 확률: 30%
- 성공 시 절감: 2,000만원
- EMV = 0.30 × 2,000만 = 600만원 (긍정적 가치)
```

**총 EMV (포트폴리오):**
- 전체 리스크 EMV 합산
- 수용 가능 리스크 예비비 산정의 근거

#### 3. 의사결정 나무 (Decision Tree)

**구조:**
- 의사결정 노드 (□): PM이 선택
- 확률 노드 (○): 불확실한 가능성 분기
- 최종 결과값 (→): 금전적 결과

**작성 절차:**
1. 의사결정 선택지 정의
2. 각 선택에 따른 결과 시나리오 정의
3. 확률 및 결과값 배정
4. EMV 계산 (뒤에서 앞으로)
5. 최적 선택 결정

**의사결정 나무 예시:**

**상황:** IT 시스템 자체 개발 vs 패키지 도입 의사결정

```
총 예산: 10억원

Option A: 자체 개발 (비용 7억 예상)
  ├─ 성공 (확률 60%): 추가 비용 없음 → 총 비용 7억
  └─ 실패/재작업 (확률 40%): +3억 추가 → 총 비용 10억

Option B: 패키지 도입 (비용 5억)
  ├─ 요구사항 충족 (65%): 커스터마이징 1억 → 총 비용 6억
  └─ 요구사항 불충족 (35%): 추가 개발 4억 → 총 비용 9억

EMV 계산:
Option A EMV = (0.6 × 7억) + (0.4 × 10억) = 4.2 + 4.0 = 8.2억
Option B EMV = (0.65 × 6억) + (0.35 × 9억) = 3.9 + 3.15 = 7.05억

결론: Option B (패키지 도입)가 EMV가 낮음 → 더 유리
```

#### 4. 몬테카를로 시뮬레이션 (Monte Carlo Simulation)

**개념:**
- 랜덤 시뮬레이션을 수천~수만 번 반복
- 가능한 모든 결과의 분포 도출
- 확률적 결과 제공

**활용:**
- "이 프로젝트가 50% 확률로 완료되는 날짜는?"
- "90% 확률로 완료하려면 예산이 얼마 필요한가?"

**입력:** 각 활동의 낙관/최빈/비관 기간(PERT)
**출력:** 완료 날짜/비용의 분포 곡선

**장점:**
- 불확실성의 범위를 보여줌
- "만일 ~~하면" 시나리오 분석 가능
- 의사결정자에게 리스크 시각화 제공

**단점:**
- 전문 소프트웨어 필요 (Oracle Primavera, @Risk)
- 입력 데이터 품질 의존
- 복잡한 해석 필요

#### 5. 민감도 분석 (Sensitivity Analysis)

**목적:** 어느 리스크가 프로젝트 목표에 가장 큰 영향을 주는가? 파악

**토네이도 차트 (Tornado Chart):**
- 각 불확실 요소의 영향 범위 시각화
- 영향이 큰 순서로 정렬 (위에서 아래로)
- 최우선 관리 대상 식별

**예시:**
```
원가 목표 영향 분석 (토네이도):
                    ← 좌(비용 감소)  우(비용 증가) →
인력 비용 변동      ████████████|████████████████████  가장 큼
하드웨어 가격 변동  ███████████|█████████████          중간
외주 계약 조건      ████████|████████████              중간
라이센스 비용       ███|████                           가장 작음
```

**결론:** 인력 비용이 가장 큰 리스크 → 인력 리스크 집중 관리

### 예시 (25분)

| # | 리스크 | 유형 | 확률 | 영향(만원) | EMV(만원) |
|---|--------|------|------|---------|---------|
| 1 | 클라우드 과금 초과 | 위협 | 40% | +2,000 | +800 |
| 2 | 핵심 개발자 이직 | 위협 | 25% | +5,000 | +1,250 |
| 3 | 성능 테스트 실패 | 위협 | 30% | +3,000 | +900 |
| 4 | 빠른 시장 출시 성공 | 기회 | 35% | -1,500 | -525 |
| 5 | 협력업체 비용 협상 | 기회 | 40% | -2,000 | -800 |

**총 리스크 EMV:**

#### 예시 2: 의사결정 나무 활용

**결정 문제:** 주요 기능을 자체 개발할 것인가, 외부 API를 사용할 것인가?

```
의사결정:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

□ Option A: 자체 개발 (개발 비용 3,000만원)
  ○ 성공 (P=0.7) → 추가 비용 0 → 총비용 3,000만
  ○ 기술 실패 (P=0.3) → 재설계 2,000만 → 총비용 5,000만

       = 2,100 + 1,500 = 3,600만원

□ Option B: 외부 API 도입 (초기 비용 5,500만원/년)

EMV(B) = 0.8 × 5,500 + 0.2 × 9,000
       = 4,400 + 1,800 = 6,200만원

결론: Option A (자체 개발, EMV 3,600만) 선택
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 5교시: 리스크 대응 계획 수립 (1.5시간) <!-- 슬라이드 #6 -->

### 이론 (50분)

#### 1. 위협 대응 전략 (Negative Risk)

**전략 1: 회피 (Avoid)**
- **정의:** 리스크 원인을 제거하거나 프로젝트를 변경하여 리스크 발생 가능성을 0으로 만듦
- **사용 시점:** P×I가 매우 높고, 다른 방법으로는 리스크를 줄일 수 없을 때
- **예시:**
  - "신기술 X는 불안정하므로 검증된 기술 Y로 교체" → 기술 리스크 완전 제거
  - "리스크가 높은 A 기능 개발 중단" → 범위에서 제외
- **주의:** 범위 변경, 비용, 시간 증가 가능

**전략 2: 전가 (Transfer)**
- **정의:** 리스크 영향을 제3자에게 이전 (책임은 이전, 소유권은 유지)
- **사용 시점:** 제3자가 더 잘 관리할 수 있을 때
- **예시:**
  - **보험 (Insurance):** 데이터 유출 사고 보험 가입
  - **계약 조항:** "지연 시 위약금" 조항으로 업체에 전가
  - **외주:** 특수 기술 리스크를 전문 업체에 이전
  - **보증/보증금:** 하자 보수 보증금 요구
- **주의:** 전가 비용 발생 (보험료, 위약금 등)

**전략 3: 완화 (Mitigate)**
- **정의:** 리스크 발생 확률 또는 영향을 허용 가능한 수준으로 줄임
- **사용 시점:** 가장 일반적인 전략, 회피가 불가능하고 전가가 불합리할 때
- **두 가지 접근:**

| 접근 | 목표 | 예시 |
|------|------|------|
| 확률 완화 | 발생 가능성 ↓ | 교육, 프로세스 개선, 테스트 강화 |
| 영향 완화 | 발생 시 피해 ↓ | 백업, 이중화, 조기 경보 시스템 |

- **IT 예시:**
  - 성능 리스크 완화: 부하 테스트 조기 실시 (확률↓)
  - 데이터 손실 완화: 3중 백업 구성 (영향↓)
  - 개발자 이직 완화: 지식 문서화 의무화 (영향↓)

**전략 4: 수용 (Accept)**
- **정의:** 리스크를 인식하고 의도적으로 별도 대응 안 함
- **사용 시점:** P×I가 낮거나, 대응 비용이 피해보다 클 때

**능동적 수용 (Active Acceptance):**
- 우발 사태 예비비 (Contingency Reserve) 설정
- 리스크 발생 시 예비비에서 처리
- 예: 마이너 버그 수정 예비비 1,000만원 확보

**수동적 수용 (Passive Acceptance):**
- 아무것도 하지 않고 발생 시 대처
- 낮은 P×I 리스크에 한함

#### 2. 기회 대응 전략 (Positive Risk)

**전략 1: 활용 (Exploit)**
- **정의:** 기회 발생이 확실해지도록 불확실성 제거
- **위협의 "회피"에 대응**
- **예시:**
  - "AI 기능 조기 완성" 기회 → 최고 실력 팀원 배정하여 확실히 달성
  - 조기 완료 기회 → 추가 자원 투입으로 확실히 조기 완성

**전략 2: 공유 (Share)**
- **정의:** 기회 활용 능력이 뛰어난 제3자에게 소유권 부여
- **위협의 "전가"에 대응**
- **예시:**
  - 신기술 특허 기회 → 해당 분야 전문 기업과 공동 개발 협약
  - 새 시장 진출 기회 → 현지 파트너와 JV(합작 투자) 설립

**전략 3: 증대 (Enhance)**
- **정의:** 기회 발생 확률이나 영향을 높이는 활동
- **위협의 "완화"에 대응**
- **예시:**
  - "MVP 조기 출시" 기회 → 범위 축소, 자원 집중 (확률 증가)
  - "고객 만족도 향상" 기회 → UX 투자 확대 (영향 증가)

**전략 4: 수용 (Accept)**
- 기회가 발생하면 활용하지만 특별한 준비는 하지 않음

**위협 vs 기회 전략 비교:**

| 위협 | 기회 | 의미 |
|------|------|------|
| 회피 | 활용 | 확실성 조작 |
| 전가 | 공유 | 제3자 활용 |
| 완화 | 증대 | 확률/영향 조정 |
| 수용 | 수용 | 방관 |

#### 3. 우발 사태 대응 계획 (Contingency Plan)

**정의:** 특정 리스크가 발생했을 때 실행하는 사전 정의된 대응 계획

**작성 요소:**
- 트리거 (Trigger): 언제 계획을 실행하는가?
- 구체적 조치: 무엇을, 누가, 얼마만에?
- 필요 자원: 예산, 인력, 장비

**예시:**
```
리스크: 외주 업체 개발 지연 (2주 이상)
트리거: 마일스톤 기한 +14일 초과 시

우발 사태 계획:
1. PM이 외주업체 임원 직접 접촉 (Day 1)
2. 지연 원인 공식 조사 착수 (Day 1-3)
3. 대체 업체 목록 연락 (Day 2-3)
4. 계약 위약금 조항 검토 (Day 3)
5. 스폰서 보고 및 자원 추가 요청 (Day 3)

예비비 사용: 대체 업체 투입 시 최대 2,000만원
```

#### 4. 잔여 리스크와 2차 리스크

**잔여 리스크 (Residual Risk):**
- 대응 전략 실행 후에도 남아있는 리스크
- 항상 0이 될 수 없음
- 관리 예비비(Management Reserve)로 대응

**2차 리스크 (Secondary Risk):**
- 리스크 대응으로 인해 새로 발생하는 리스크
- 예: 보안 강화(완화) → 성능 저하(2차 리스크)
- 항상 식별 및 등록 필요

### 예시 (25분)

#### 예시 1: 4가지 위협 전략 적용

**프로젝트:** 공항 수하물 처리 시스템 구축

**리스크:** "핵심 외주 업체가 개발 도중 파산할 수 있다" (P=0.15, I=0.9, EMV=1,350만원)

**각 전략 검토:**

```
전략 A: 회피 (Avoid)
- 방법: 외주 전면 취소, 내부 개발로 전환
- 비용: + 2억 (내부 팀 충원 및 기간 연장)
- 효과: 리스크 완전 제거
- 판단: EMV 1,350만원 < 전환 비용 2억 → 불합리

전략 B: 전가 (Transfer)

          제3자 에스크로(escrow) 계좌 활용
- 비용: 법무 비용 + 프리미엄 (약 300만원)
- 효과: 재정적 손실 이전
- 판단: 합리적 (EMV 1,350 > 비용 300)

전략 C: 완화 (Mitigate)
- 방법: 
  ① 업체 재무 상태 분기 모니터링 (확률 완화)
  ② 소스코드 에스크로 + 단계별 인도물 확보 (영향 완화)

- 판단: 채택 권장

전략 D: 수용 (Accept)
- 능동적: 우발 사태 예비비 1,500만원 설정
- 수동적: 아무것도 안 함
- 판단: P×I = 0.135 (Yellow), 완화와 병행 수용 가능

최종 선택: 전략 B + C 병행 (전가로 재정 보호 + 완화로 확률/영향 감소)
```

### 실습 3: 대응 전략 및 EMV 계산 (20분)

**과제:** Red 리스크에 대해:
1. 대응 전략 선택 및 조치
2. EMV 계산 (비용 단위: 만원)
3. 우발 사태 예비비 산정

**예시 계산:**

```
Red 리스크 1: AI 모델 정확도 기준 미달 (P=0.4, I=0.8)
- 영향 금액: 4,000만원 (재트레이닝 비용)
- EMV = 0.4 × 4,000 = 1,600만원

대응 전략: 완화
- 조치 1: 초기 MVP에서 A/B 테스트 (확률 0.4→0.25)
- 조치 2: 외부 AI 전문가 컨설팅 (영향 0.8→0.5)
- 완화 후 EMV = 0.25 × 0.5 × 4,000 = 500만원
- 완화 비용: 800만원
- 순 절감: 1,600 - 500 - 800 = 300만원 → 채택

전체 우발 사태 예비비 = 모든 Red/Yellow EMV 합산
```
