# Day 16: 신기술 트렌드 - 상세 강의안

## 1교시: AI / 머신러닝 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. AI 기본 개념

**약한 AI (Narrow AI):**
- 특정 작업에 특화 (이미지 인식, 번역)
- 현재 일반적인 AI

**강한 AI (General AI):**
- 인간 수준의 범용 지능
- 아직 미실현

#### 2. 머신러닝 유형

**지도학습 (Supervised Learning):**
- 정답이 있는 데이터로 학습
- 예: 스팸 분류, 부동산 가격 예측

**비지도학습 (Unsupervised Learning):**
- 정답 없이 패턴 발견
- 예: 고객 세그먼트, 이상 탐지

**강화학습 (Reinforcement Learning):**
- 보상/패널티로 학습
- 예: 게임 AI, 자율주행

#### 3. 생성형 AI (Generative AI)

**정의:** 콘텐츠를 생성하는 AI

**종류:**
- LLM: ChatGPT, Claude (텍스트)
- 이미지: Midjourney, DALL-E
- 코드: GitHub Copilot

**활용:**
- 문서 작성, 번역
- 코드 생성
- 요약, 분석

#### 4. 모델 성능 지표

**정확도 (Accuracy):** 맞춘 비율  
**정밀도 (Precision):** True Positive / (True + False Positive)  
**재현율 (Recall):** True Positive / (True + False Negative)  
**F1 Score:** Precision과 Recall의 조화평균

#### 5. PM 관점 AI 프로젝트

**성공 요인:**
- 명확한 비즈니스 문제
- 충분한 데이터
- 측정 가능한 목표

**리스크:**
- 데이터 부족/품질
- 과적합 (Overfitting)
- 편향 (Bias)

#### 실습 2: 모델 성능 지표 비교

**과제:** 의료진단 AI의 성능 지표를 계산하세요.

**데이터:**
- 환자 100명
- AI 예측: 암 20명 (실제 암 15명, 오진 5명)
- AI 정상: 80명 (실제 정상 78명, 놓친 암 2명)

<details>
<summary>해설</summary>

**혼동 행렬 (Confusion Matrix):**
|  | 실제 암 | 실제 정상 |
|--|--------|---------|
| 예측 암 | 15 (TP) | 5 (FP) |
| 예측 정상 | 2 (FN) | 78 (TN) |

**지표 계산:**

**정확도 (Accuracy):**
= (TP + TN) / 전체
= (15 + 78) / 100
= 93%

**정밀도 (Precision):**
= TP / (TP + FP)
= 15 / (15 + 5)
= 75%

**재현율 (Recall):**
= TP / (TP + FN)
= 15 / (15 + 2)
= 88%

**F1 Score:**
= 2 × (Precision × Recall) / (Precision + Recall)
= 2 × (0.75 × 0.88) / (0.75 + 0.88)
= 0.81

**PM 해석:**

**정밀도 75%:**
- 암으로 진단한 환자 중 25%는 오진
- 괜찮은 환자에게 불필요한 검사 (비용, 스트레스)

**재현율 88%:**
- 실제 암 환자 중 12%를 놓침
- 치명적 (조기 발견 실패)

**의사결정:**
- 의료는 재현율이 더 중요 (놓치면 위험)
- 목표: 재현율 95% 이상
- 정밀도가 낮아도 수용 (추가 검사로 확인)

**개선 방안:**
- 임계값 조정 (예측 확률 30% 이상 → 암으로 분류)
- 더 많은 데이터 수집
- 모델 개선 (앙상블)

</details>

## 2교시: 클라우드 컴퓨팅 심화 (1.5시간)

### 이론 (50분)

#### 1. 클라우드 컴퓨팅이란?

**정의:**
인터넷을 통해 IT 리소스(서버, 스토리지, 네트워크, 소프트웨어)를 **온디맨드**로 제공받는 서비스

**특징:**
- 필요할 때 즉시 사용 (On-Demand)
- 사용한 만큼 과금 (Pay-as-you-go)
- 탄력적 확장 (Scalability)
- 셀프 서비스 (Self-service Portal)
- 광범위한 네트워크 접근

**온프레미스 vs 클라우드:**

| 구분 | 온프레미스 (On-Premise) | 클라우드 (Cloud) |
|------|------------------------|------------------|
| **초기 비용** | 높음 (서버, 네트워크 장비 구매) | 낮음 (초기 투자 불필요) |
| **운영 비용** | 고정 (전기, 공조, 인건비) | 변동 (사용량 기반) |
| **확장성** | 어려움 (장비 구매 및 설치) | 쉬움 (클릭 몇 번) |
| **구축 기간** | 길음 (수주~수개월) | 짧음 (수분~수일) |
| **유지보수** | 자체 (IT 팀 필요) | 클라우드 제공자 |
| **보안** | 자체 통제 | 공유 책임 모델 |
| **재해 복구** | 자체 구축 (고비용) | 백업/복제 서비스 제공 |
| **적합한 경우** | 규제, 데이터 민감, 예측 가능한 부하 | 스타트업, 변동 많은 트래픽 |

#### 2. 클라우드 서비스 모델 (SPI 모델)

**A. IaaS (Infrastructure as a Service, 인프라 서비스)**

**정의:** 가상화된 컴퓨팅 리소스를 제공

**제공 내용:**
- 가상 서버 (VM)
- 스토리지
- 네트워크
- 로드 밸런서

**사용자 책임:**
- OS 설치 및 관리
- 미들웨어, 런타임
- 애플리케이션
- 데이터

**클라우드 제공자:**
- AWS: EC2, S3, VPC
- Azure: Virtual Machines, Blob Storage
- GCP: Compute Engine, Cloud Storage

**예시:**
```
시나리오: 웹 서비스 구축

IaaS 사용:
1. AWS EC2 인스턴스 생성 (Ubuntu)
2. 웹 서버 설치 (Apache, Nginx)
3. 데이터베이스 설치 (MySQL)
4. 애플리케이션 배포

→ 사용자가 OS부터 모든 것을 관리
```

**장점:**
✅ 완전한 제어 가능
✅ 기존 온프레미스 환경을 그대로 옮기기 쉬움 (Lift & Shift)
✅ 유연성

**단점:**
❌ 관리 부담 (OS 패치, 보안 업데이트)
❌ 전문 인력 필요

**B. PaaS (Platform as a Service, 플랫폼 서비스)**

**정의:** 애플리케이션 개발 및 실행 플랫폼 제공

**제공 내용:**
- OS
- 런타임 (Java, Python, Node.js)
- 미들웨어 (웹 서버, 데이터베이스)
- 개발 도구

**사용자 책임:**
- 애플리케이션 코드
- 데이터

**클라우드 제공자:**
- AWS: Elastic Beanstalk, RDS
- Azure: App Service, SQL Database
- GCP: App Engine, Cloud SQL
- Heroku

**예시:**
```
시나리오: 웹 애플리케이션 개발

PaaS 사용:
1. Heroku에 Git push
2. 자동으로 빌드 및 배포
3. 데이터베이스도 자동 프로비저닝

→ 개발자는 코드만 작성, 인프라는 신경 안 씀
```

**장점:**
✅ 빠른 개발 및 배포
✅ 인프라 관리 불필요
✅ 자동 스케일링

**단점:**
❌ 제한된 제어 (플랫폼 제공 범위 내에서만)
❌ 벤더 종속(Vendor Lock-in)

**C. SaaS (Software as a Service, 소프트웨어 서비스)**

**정의:** 완성된 애플리케이션을 서비스로 제공

**제공 내용:**
- 모든 것 (인프라, 플랫폼, 애플리케이션)

**사용자 책임:**
- 데이터 입력 및 설정

**예시:**
- Google Workspace (Gmail, Drive, Docs)
- Microsoft 365 (Outlook, Word, Excel)
- Salesforce (CRM)
- Slack (협업 도구)
- Zoom (화상회의)

**장점:**
✅ 즉시 사용 가능
✅ 관리 불필요
✅ 저렴한 비용 (구독 모델)
✅ 자동 업데이트

**단점:**
❌ 커스터마이징 제한
❌ 데이터 통제 약함
❌ 인터넷 의존

**비교표:**

```
┌─────────────────────────────────────────┐
│           사용자 관리 영역              │
│ SaaS:      [데이터]                      │
│                                         │
│ PaaS:      [애플리케이션] [데이터]      │
│                                         │
│ IaaS:      [OS] [런타임] [앱] [데이터]  │
│                                         │
│ On-Premise: [전부]                      │
└─────────────────────────────────────────┘
```

| 구분 | IaaS | PaaS | SaaS |
|------|------|------|------|
| **제어** | 높음 | 중간 | 낮음 |
| **유연성** | 높음 | 중간 | 낮음 |
| **관리 부담** | 높음 | 중간 | 낮음 |
| **구축 속도** | 중간 | 빠름 | 매우 빠름 |
| **비용** | 중간 | 중간~높음 | 낮음 |
| **적합 사용자** | IT 전문가 | 개발자 | 일반 사용자 |

#### 3. 클라우드 배포 모델

**A. 퍼블릭 클라우드 (Public Cloud)**

**정의:** 다수의 고객이 공유하는 클라우드 인프라

**특징:**
- 불특정 다수에게 개방
- 멀티 테넌트 (Multi-tenant)
- 가장 저렴

**제공자:**
- AWS
- Microsoft Azure
- Google Cloud Platform (GCP)
- 네이버 클라우드
- 카카오 클라우드

**장점:**
✅ 저렴한 비용
✅ 빠른 확장
✅ 구축 즉시 가능
✅ 글로벌 인프라

**단점:**
❌ 데이터 통제 약화
❌ 보안 우려 (공유 환경)
❌ 규제 산업에 부적합

**B. 프라이빗 클라우드 (Private Cloud)**

**정의:** 단일 조직 전용 클라우드

**특징:**
- 자체 데이터센터 또는 전용 호스팅
- 격리된 환경
- 높은 통제

**방식:**
1. On-Premise Private Cloud: 자체 데이터센터에 구축
2. Hosted Private Cloud: 호스팅 업체의 전용 인프라

**장점:**
✅ 높은 보안 및 통제
✅ 규제 준수 용이 (금융, 의료)
✅ 성능 예측 가능

**단점:**
❌ 고비용
❌ 확장 어려움
❌ 운영 인력 필요

**적합:**
- 금융기관
- 정부기관
- 의료기관
- 민감한 데이터 다루는 기업

**C. 하이브리드 클라우드 (Hybrid Cloud)**

**정의:** 퍼블릭과 프라이빗을 조합

**아키텍처 예:**
```
┌────────────────────┐         ┌──────────────────┐
│ On-Premise         │         │ Public Cloud     │
│ (Private Cloud)    │         │ (AWS/Azure)      │
│                    │         │                  │
│ - DB (고객 정보)   │◄───────►│ - Web Servers    │
│ - 핵심 시스템      │ VPN     │ - 캐시           │
│ - 결제 시스템      │         │ - CDN            │
└────────────────────┘         └──────────────────┘
```

**사용 시나리오:**
1. **데이터 지역성**: 민감 데이터는 온프레미스, 나머지는 클라우드
2. **버스트 처리**: 평상시 온프레미스, 피크 시 클라우드로 확장
3. **재해 복구**: 온프레미스 운영, 클라우드에 백업
4. **단계적 마이그레이션**: 점진적으로 클라우드 전환

**장점:**
✅ 유연성 (각 장점 취합)
✅ 규제 준수 + 확장성
✅ 비용 최적화

**단점:**
❌ 복잡한 관리
❌ 통합 어려움
❌ 보안 정책 이원화

#### 4. 클라우드 네트워킹

**A. VPC (Virtual Private Cloud)**

**정의:** 클라우드 내 격리된 가상 네트워크

**특징:**
- IP 주소 범위 정의 (CIDR)
- 서브넷 생성 (Public/Private)
- 라우팅 테이블 관리
- 인터넷 게이트웨이

**예시: AWS VPC**
```
VPC: 10.0.0.0/16

  ┌─ Public Subnet: 10.0.1.0/24
  │   - 웹 서버 (인터넷 접근 가능)
  │   - 로드 밸런서
  │
  └─ Private Subnet: 10.0.2.0/24
      - 데이터베이스 (외부 접근 불가)
      - 애플리케이션 서버
```

**B. 로드 밸런서 (Load Balancer)**

**역할:** 트래픽을 여러 서버로 분산

**유형:**
1. **L4 (Transport Layer)**: IP, PORT 기반
2. **L7 (Application Layer)**: HTTP 헤더, URL 기반

**예시:**
```
                [Load Balancer]
                /      |      \
               /       |       \
          [Web1]   [Web2]   [Web3]

요청이 균등하게 분산
→ 서버 1대 장애 시에도 서비스 유지
```

**C. CDN (Content Delivery Network)**

**역할:** 콘텐츠를 사용자와 가까운 위치에 캐싱

**동작:**
```
사용자 (서울) → CDN 엣지 (서울) → 오리진 서버 (미국)
                     ↑
                   캐싱됨
                     ↓
다음 요청 → CDN 엣지에서 바로 응답 (빠름!)
```

**장점:**
- 빠른 응답 속도
- 오리진 서버 부하 감소
- 대역폭 절감

**서비스:**
- AWS CloudFront
- Azure CDN
- Cloudflare

#### 5. SLA (Service Level Agreement)

**정의:** 클라우드 제공자가 보장하는 **서비스 수준 합의**

**주요 지표:**

**A. 가용성 (Availability)**

**계산:**
```
가용성 = (총 시간 - 다운타임) / 총 시간 × 100%
```

**수준별 의미:**

| 가용성 | 월 다운타임 | 연 다운타임 | 등급 |
|--------|------------|-------------|------|
| 99% | 7.2시간 | 3.65일 | 낮음 |
| 99.5% | 3.6시간 | 1.83일 | 보통 |
| 99.9% (쓰리나인) | 43분 | 8.76시간 | 높음 |
| 99.95% | 22분 | 4.38시간 | 매우 높음 |
| 99.99% (포나인) | 4분 | 52분 | 미션 크리티컬 |
| 99.999% (파이브나인) | 26초 | 5.26분 | 거의 절대 |

**B. 성능 (Performance)**

- 응답 시간 (Response Time)
- 처리량 (Throughput)
- 동시 사용자 수

**C. 지원 (Support)**

- 응답 시간
  - P1 (긴급): 15분 이내
  - P2 (높음): 1시간 이내
  - P3 (중간): 4시간 이내
- 해결 시간

**D. 보상 (Credits)**

AWS EC2 SLA 예:
```
99.99% 이상: 보상 없음
99.0~99.99%: 10% 크레딧
95.0~99.0%: 25% 크레딧
95% 미만: 100% 크레딧
```

**PM이 확인할 사항:**
- SLA 보장 수준이 요구사항을 충족하는가?
- Multi-AZ 배포 시 SLA 향상
- 보상 절차 및 기준 이해
- SLA는 "다운타임 없음"을 보장하지 않음 (99.9% = 월 43분 다운 허용)

#### 6. 클라우드 비용 관리

**A. 가격 모델**

**1) On-Demand (온디맨드)**
- 필요할 때 사용, 시간당 과금
- 가장 비쌈
- 예: $0.10/시간

**2) Reserved Instance (예약 인스턴스)**
- 1년 또는 3년 약정
- 최대 75% 할인
- 예: $0.05/시간 (1년 약정)

**3) Spot Instance (스팟 인스턴스)**
- 경매 방식, 가격 변동
- 최대 90% 할인
- 예고 없이 회수 가능
- 예: $0.03/시간
- 적합: 배치 작업, 분석

**비교 예:**
```
워크로드: 1대 서버, 24/7 운영

On-Demand: $0.10/h × 730h/월 = $73/월
Reserved (1년): $0.05/h × 730h = $36.5/월 (50% 절감)
Spot: $0.03/h × 730h = $21.9/월 (70% 절감, 단 불안정)
```

**B. 비용 최적화 전략**

**1. 적정 사이징 (Right Sizing)**
```
현재: t3.xlarge (4 vCPU, 16GB RAM) - $120/월
실제 사용률: CPU 20%, RAM 30%

최적화: t3.large (2 vCPU, 8GB RAM) - $60/월
→ 월 $60 절감
```

**2. 자동 스케일링 (Auto Scaling)**
```
고정: 10대 서버 24/7 = 7,300 서버-시간/월
자동 스케일링: 
  - 평상시 3대 (18시간) = 54
  - 피크 10대 (6시간) = 60
  - 총 3,240 서버-시간/월

절감: (7,300 - 3,240) / 7,300 = 56%
```

**3. 예약 인스턴스 활용**
```
베이스 로드는 예약 인스턴스
변동 부분은 On-Demand/Spot

비용 절감: 30~50%
```

**4. 불필요한 리소스 제거**
```
체크리스트:
- 사용하지 않는 VM 삭제
- 테스트용 인스턴스 정리
- 오래된 스냅샷 삭제
- 미사용 EBS 볼륨 삭제
```

### 예시 (25분)

#### 예시 1: 클라우드 전환 의사결정 (On-Premise vs Cloud)

**시나리오:** 중소기업 쇼핑몰, 서버 노후화로 교체 검토

**현재 상황:**
- 온프레미스 서버 5년 사용 (수명 만료)
- 직원 50명
- 월 방문자 10만 명
- 피크: 블랙프라이데이 (평소의 10배)

**옵션 1: 온프레미스 서버 교체**

**초기 비용:**
- 서버 3대: 3,000만 원
- 네트워크 장비: 500만 원
- 설치 및 구축: 500만 원
- **총: 4,000만 원**

**월 운영 비용:**
- 전기: 50만 원
- 인터넷: 30만 원
- 유지보수: 20만 원
- IT 담당자 (일부): 100만 원
- **총: 200만 원/월**

**5년 총비용 (TCO):**
```
초기: 4,000만 원
운영: 200만 원 × 60개월 = 1억 2,000만 원
──────────────────────────────────
총합: 1억 6,000만 원
```

**단점:**
- 피크 시 성능 부족 (블랙프라이데이)
- DR (재해 복구) 비용 별도
- 확장 어려움

**옵션 2: 클라우드 (AWS)**

**기본 구성:**
- EC2 인스턴스 (t3.medium) 3대
- RDS (데이터베이스)
- S3 (이미지 저장)
- CloudFront (CDN)

**월 비용:**
**평상시:**
- EC2 (예약 인스턴스): $60 × 3 = $180
- RDS: $50
- S3 + CloudFront: $100
- **총: $330/월 ≈ 43만 원/월**

**피크 시 (자동 스케일링):**
- EC2 10대로 증가 (7일간)
- 추가 비용: $500
- **총: $830 ≈ 108만 원 (한 달 중 7일)**

**5년 총비용:**
```
평상시: 43만 원 × 11개월 × 5년 = 2,365만 원
피크: 108만 원 × 1개월 × 5년 = 540만 원
──────────────────────────────────
총합: 2,905만 원
```

**추가 장점:**
- 초기 투자 0원
- 자동 확장 (피크 대응)
- DR 자동 구성 (Multi-AZ)
- 글로벌 CDN
- 보안 자동 업데이트

**의사결정:**

| 구분 | 온프레미스 | 클라우드 |
|------|-----------|---------|
| 초기 비용 | 4,000만 원 | 0원 |
| 5년 총비용 | 1억 6,000만 원 | 2,905만 원 |
| 피크 대응 | 어려움 | 자동 확장 |
| DR | 별도 구축 필요 | 포함 |
| 유연성 | 낮음 | 높음 |

**결론:** **클라우드 선택** (5년 1억 3,000만 원 절감!)

**PM의 판단:**
- 스타트업/중소기업은 클라우드가 유리
- 대기업이라도 변동이 큰 워크로드는 클라우드
- 규제/보안 이슈가 없다면 클라우드 우선 고려

#### 예시 2: IaaS vs PaaS vs SaaS 선택

**시나리오:** 사내 협업 도구 도입

**요구사항:**
- 직원 100명
- 파일 공유, 메신저, 화상회의
- 초기 비용 최소화

**옵션 1: IaaS (자체 구축)**

**구성:**
- AWS EC2에 오픈소스 협업 도구 설치 (Nextcloud, Rocket.Chat)
- 직접 커스터마이징

**비용:**
- EC2: $100/월
- 스토리지: $50/월
- **총: $150/월 ≈ 20만 원**

**개발 인력:**
- 구축: 1개월 (개발자 1명)
- 유지보수: 주 4시간
- **인건비: 초기 500만 원 + 월 50만 원**

**총비용 (1년):**
```
인프라: 20만 원 × 12 = 240만 원
인건비: 초기 500만 원 + 월 50만 원 × 12 = 1,100만 원
───────────────────────────────────────
총합: 1,340만 원
```

**장점:**
✅ 완전한 통제
✅ 커스터마이징 가능

**단점:**
❌ 구축 시간 (1개월)
❌ 유지보수 부담
❌ 전문 인력 필요

**옵션 2: PaaS (Heroku + 오픈소스)**

**구성:**
- Heroku에 협업 앱 배포
- 데이터베이스 자동 관리

**비용:**
- Heroku Dyno: $250/월
- Add-ons (DB, 스토리지): $100/월
- **총: $350/월 ≈ 46만 원**

**개발 인력:**
- 구축: 1주일
- 유지보수: 주 1시간

**총비용 (1년):**
```
인프라: 46만 원 × 12 = 552만 원
인건비: 초기 100만 원 + 월 10만 원 × 12 = 220만 원
───────────────────────────────────────
총합: 772만 원
```

**장점:**
✅ 빠른 구축
✅ 인프라 관리 불필요
✅ 자동 스케일링

**단점:**
❌ 비용 높음
❌ 벤더 종속

**옵션 3: SaaS (Google Workspace / Microsoft 365)**

**구성:**
- Google Workspace Business Standard
- 즉시 사용 가능

**비용:**
- 사용자당: $12/월
- 100명: $1,200/월 ≈ 156만 원

**총비용 (1년):**
```
구독료: 156만 원 × 12 = 1,872만 원
인건비: 0원 (관리 불필요)
───────────────────────────────────────
총합: 1,872만 원
```

**장점:**
✅ 즉시 사용 (구축 0분)
✅ 관리 불필요
✅ 자동 업데이트
✅ 모바일 앱 제공
✅ 높은 안정성 (99.9% SLA)

**단점:**
❌ 커스터마이징 제한
❌ 데이터 통제 약함

**의사결정:**

| 고려사항 | IaaS | PaaS | SaaS |
|----------|------|------|------|

**PM의 판단:**
- 일반 협업 도구 → **SaaS** (Google Workspace) 선택
  - 이유: 즉시 사용, 안정적, 관리 불필요
  - 비용은 높지만 인건비 고려 시 합리적
  
- 특수 요구사항 있을 때만 IaaS/PaaS 고려
  - 예: 금융/의료 (데이터 통제 필요)

#### 실습 2: 클라우드 vs 온프레미스 비용 계산

**시나리오:** 스타트업 웹 서비스

**요구사항:**
- 서버 2대 (웹, DB)
- 24/7 운영
- 예상 트래픽: 안정적 (변동 적음)

**옵션 1: 온프레미스**
- 서버 구매: 1,000만 원
- 월 운영비: 50만 원 (전기, 인터넷, 유지보수)

**옵션 2: 클라우드 (AWS)**
- EC2 t3.medium 2대
- On-Demand: $0.0416/h
- Reserved (1년 약정, 선결제): $0.025/h

**과제:** 1년 총비용을 계산하고 어느 것이 유리한지 판단하세요.

**해설 (강사용):**

**온프레미스:**
```
초기: 1,000만 원
운영: 50만 원 × 12 = 600만 원
────────────────────────────
총합: 1,600만 원
```

**클라우드 (On-Demand):**
```
시간: 24h × 365일 = 8,760시간/년
서버 2대: 8,760 × 2 = 17,520 서버-시간

비용: $0.0416 × 17,520 = $729.12/년
한화: $729 × 1,300원 = 약 95만 원/년
```

**클라우드 (Reserved, 1년):**
```
비용: $0.025 × 17,520 = $438/년
한화: $438 × 1,300원 = 약 57만 원/년

+ 선결제 금액 (일반적으로 20-30% 추가)
≈ 74만 원/년
```

**결론:**
- 클라우드 On-Demand: **95만 원** (가장 저렴!)
- 클라우드 Reserved: **74만 원** (약정 시)
- 온프레미스: **1,600만 원**

**판단:** 클라우드 압도적 우위!
- 초기 투자 없음
- 1년 비용이 1/17
- 스타트업은 클라우드 필수

**주의:** 트래픽이 급증하거나 장기 운영(5년+) 시 재계산 필요

#### 실습 3: SLA 계산

**시나리오:** 클라우드 업체의 SLA 검토

**AWS SLA:** 99.95% (Monthly Uptime Percentage)

**과제:**
1. 월 허용 다운타임을 분 단위로 계산하세요.
2. 실제 가용성이 99.5%였을 때 보상은? (SLA: 99.0~99.95% = 10% credit)

**해설 (강사용):**

**1. 허용 다운타임:**
```
월 총 시간: 30일 × 24시간 = 720시간 = 43,200분

가용률: 99.95%
다운타임: 43,200분 × (1 - 0.9995) = 43,200 × 0.0005 = 21.6분

→ 월 21.6분 다운타임 허용
```

**2. 보상 계산:**
```
실제 가용성: 99.5%
SLA: 99.95%

→ SLA 미달!

보상 기준:
- 99.0% ~ 99.95%: 10% credit

월 사용료가 $1,000이었다면:
보상: $1,000 × 10% = $100 credit
```

**PM의 판단:**
- 보상은 "다운타임 없음"을 보장하지 않음
- 99.95% = 월 21분 다운 허용
- 미션 크리티컬 시스템은 Multi-AZ, Multi-Region으로 가용성 향상
- SLA는 최소 보장, 실제는 더 높을 수 있음

1. **레거시 마이그레이션**
   - 시나리오: 온프레미스 시스템을 클라우드로 이전
   - 이유: 기존 아키텍처를 그대로 옮길 수 있음 (Lift & Shift)
   - 예: 은행의 20년 된 COBOL 시스템을 IaaS VM에 그대로 이전

2. **특수 소프트웨어**
   - 시나리오: PaaS가 지원하지 않는 소프트웨어 사용
   - 이유: OS 수준의 통제 필요
   - 예: 특정 버전의 Oracle DB, 커스텀 컴파일된 소프트웨어

3. **완전한 제어 필요**
   - 시나리오: 보안/규제로 인해 모든 설정을 제어해야 함
   - 이유: OS, 네트워크, 보안 정책 완전 통제
   - 예: 금융 거래 시스템, 의료 기록 시스템

**PaaS (Platform as a Service):**

**제공:** OS, 런타임, 미들웨어, 개발 도구
**사용자 관리:** 애플리케이션 코드, 데이터
**제어:** 중간
**예:** Heroku, AWS Elastic Beanstalk, Google App Engine

**적합한 프로젝트:**

1. **신규 웹/모바일 앱 개발**
   - 시나리오: 스타트업의 MVP 개발
   - 이유: 인프라 신경 쓰지 않고 코드에만 집중
   - 예: Node.js 기반 REST API 백엔드 → Heroku에 Git Push만으로 배포

2. **CI/CD 자동화**
   - 시나리오: 빠른 배포 주기 필요
   - 이유: Git push → 자동 빌드 → 자동 배포
   - 예: 코드 커밋 즉시 스테이징 환경에 배포되는 개발 프로세스

3. **프로토타입/POC**
   - 시나리오: 아이디어 검증 단계
   - 이유: 빠른 구축 (1주일 이내)
   - 예: 챗봇 서비스 POC → AWS Lambda + API Gateway로 1주일에 데모

**SaaS (Software as a Service):**

**제공:** 완성된 애플리케이션
**사용자 관리:** 데이터 입력 및 설정
**제어:** 낮음
**예:** Gmail, Office 365, Salesforce, Slack

**적합한 프로젝트:**

1. **사내 협업 도구**
   - 시나리오: 100명 규모 회사의 메일/문서 공유
   - 이유: 즉시 사용, 관리 불필요, 모바일 지원
   - 예: Google Workspace (Gmail, Drive, Docs) → 가입 즉시 사용

2. **CRM/ERP**
   - 시나리오: 영업팀의 고객 관리
   - 이유: 검증된 소프트웨어, 커스터마이징 필요 적음
   - 예: Salesforce → 영업 프로세스에 최적화된 CRM

3. **화상회의/커뮤니케이션**
   - 시나리오: 원격 근무 지원
   - 이유: 별도 인프라 불필요, 글로벌 접속
   - 예: Zoom, Microsoft Teams → 링크 공유만으로 회의

**선택 가이드:**

| 고려사항 | IaaS | PaaS | SaaS |
|----------|------|------|------|
| **개발 리소스** | 많음 (인프라 전문가) | 중간 (개발자) | 없음 (사용만) |
| **커스터마이징** | 완전 자유 | 코드 수준 | UI 설정만 |
| **출시 속도** | 느림 (수개월) | 빠름 (수주) | 즉시 |
| **비용** | 중간 | 중간~높음 | 낮음~중간 |
| **유지보수** | 자체 (높은 부담) | 제한적 (앱만) | 없음 |

**PM의 원칙:**
- 기본은 **SaaS** (즉시 사용 가능, 검증된 소프트웨어)
- 커스터마이징 필요 시 **PaaS** (개발 집중)
- 완전한 통제 필요 시 **IaaS** (최후의 수단)

**실제 사례:**
```
온라인 쇼핑몰 구축:

SaaS: Shopify (즉시 개설, 월 $29, 커스터마이징 제한)
PaaS: Heroku + 오픈소스 (개발 1개월, 월 $500, 자유로운 기능)
IaaS: AWS EC2 (개발 3개월, 월 $200, 완전 통제)

선택: 초기 MVP는 Shopify → 성장 후 PaaS/IaaS로 이전
```

**Q8. 클라우드 비용을 최적화하기 위해 PM이 할 수 있는 방법 5가지를 구체적으로 설명하세요.**

<details>
<summary>모범 답안</summary>

**1. 적정 사이징 (Right Sizing)**

**문제:**
과도하게 큰 인스턴스를 사용하여 비용 낭비

**방법:**
- 모니터링으로 실제 리소스 사용률 확인 (CPU, RAM)
- 70% 이하 사용 시 다운그레이드 고려

**예:**
```
현재: t3.2xlarge (8 vCPU, 32GB RAM) - $300/월
실제 사용률: CPU 20%, RAM 40%

최적화: t3.large (2 vCPU, 8GB RAM) - $75/월
절감: $225/월 = 75%
```

**PM 액션:**
- 월간 리소스 사용 리포트 검토
- 개발팀과 협의하여 적정 크기 결정
- 테스트 후 변경 (부하 테스트 필수)

**2. 자동 스케일링 (Auto Scaling) 활용**

**문제:**
피크 대비 24/7 최대 용량 유지 → 비용 낭비

**방법:**
- AutoScaling Group 설정
- 트래픽에 따라 인스턴스 수 자동 조절
- 최소/최대 인스턴스 수 정의

**예:**
```
고정 운영: 10대 서버 × 24시간 = 240 서버-시간/일

자동 스케일링:
- 00:00~08:00 (저녁): 2대
- 08:00~18:00 (주간): 5대
- 18:00~24:00 (피크): 10대

총: (2×8) + (5×10) + (10×6) = 126 서버-시간/일
절감: (240-126)/240 = 47.5%
```

**PM 액션:**
- 트래픽 패턴 분석 (시간대별, 요일별)
- 적정 임계값 설정 (CPU 70% 시 증설)
- 정기 리뷰 (트래픽 패턴 변화 반영)

**3. 예약 인스턴스 / Savings Plans 활용**

**문제:**
24/7 돌아가는 서버를 On-Demand로 운영 → 비용 초과

**방법:**
- 1년/3년 약정으로 할인 (최대 75%)
- 베이스 로드는 예약, 변동은 On-Demand

**예:**
```
워크로드: 베이스 5대 + 피크 시 추가 5대

옵션 A: 전부 On-Demand
10대 × $0.10/h × 8,760h = $8,760/년

옵션 B: 베이스는 예약, 피크는 On-Demand
예약 5대 × $0.05/h × 8,760h = $2,190
On-Demand 5대 × $0.10/h × 2,000h = $1,000
총: $3,190/년
절감: 64%
```

**PM 액션:**
- 6개월 사용 패턴 분석 후 예약 결정
- 재무팀과 협의 (선불 vs 후불)
- 계약 만료 3개월 전 갱신 검토

**4. 불필요한 리소스 정리**

**문제:**
사용하지 않는 리소스가 계속 요금 부과

**정리 대상:**
- 중단된 프로젝트의 VM
- 개발자 개인 테스트 인스턴스
- 오래된 스냅샷/백업
- 연결되지 않은 EBS 볼륨
- 미사용 Elastic IP (AWS는 연결 안 하면 과금)

**예:**
```
정리 전:
- 프로덕션 서버: 5대 - $500/월
- 테스트 서버: 10대 - $1,000/월 (방치됨)
- 스냅샷: 5TB - $250/월 (2년 전 백업)
총: $1,750/월

정리 후:
- 프로덕션: 5대 - $500/월
- 활성 테스트: 3대 - $300/월
- 스냅샷 (3개월분만): 1TB - $50/월
총: $850/월
절감: $900/월 = 51%
```

**PM 액션:**
- 월 1회 리소스 정기 감사
- 태그 정책 (프로젝트명, 담당자, 삭제 예정일)
- 90일 미사용 리소스 자동 알림
- 삭제 전 담당자 확인

**5. 스팟 인스턴스 활용 (비중요 워크로드)**

**문제:**
배치 작업, 빅데이터 분석 등에 비싼 On-Demand 사용

**방법:**
- 스팟 인스턴스 (경매 방식, 최대 90% 할인)
- 중단 가능한 워크로드에만 사용
- 중단 시 자동 재시작 로직 구현

**적합한 워크로드:**
- 데이터 분석 (Spark, Hadoop)
- 이미지 처리 배치
- CI/CD 빌드 서버
- 테스트 환경

**부적합:**
- 데이터베이스
- 웹 서버 (사용자 영향)
- 실시간 처리

**예:**
```
빅데이터 분석 작업 (10시간 소요):

On-Demand: $1.00/h × 10인스턴스 × 10시간 = $100
Spot: $0.30/h × 10인스턴스 × 10시간 = $30
절감: 70%

주의: 중단 가능성 있으므로 체크포인트 구현 필요
```

**PM 액션:**
- 워크로드 분류 (중단 가능 vs 불가)
- 개발팀과 협의 (체크포인트 구현)
- 하이브리드 전략 (On-Demand + Spot)

**추가 팁:**

**6. 리전/가용영역 최적화**
- 가격이 저렴한 리전 선택 (지연시간 고려)
- 예: 미국 오하이오 리전이 버지니아보다 5% 저렴

**7. S3 스토리지 클래스 최적화**
- 자주 접근: S3 Standard
- 가끔 접근: S3 Infrequent Access (50% 저렴)
- 아카이브: S3 Glacier (80% 저렴)

**8. CDN 활용으로 오리진 비용 절감**
- CloudFront 사용 시 데이터 전송 비용 감소

**9. 모니터링 및 Cost Explorer 활용**
- AWS Cost Explorer, Azure Cost Management
- 이상 지출 알람 설정 (예산 초과 시)
- Tagging으로 프로젝트별 비용 추적

**10. 정기적인 리뷰**
- 월간 비용 리뷰 회의
- 부서별 비용 할당 (Chargeback)
- 분기별 최적화 목표 설정

**ROI 계산 예:**
```
최적화 전: 월 $10,000
최적화 후: 월 $6,000
절감: 40% = $4,000/월 = $48,000/년

최적화 작업 비용: 엔지니어 1주일 = $5,000
ROI: $48,000 / $5,000 = 960%
회수 기간: 1.25주
```

**PM의 역할:**
클라우드 비용 최적화는 **지속적인 프로세스**입니다. 한 번 하고 끝이 아니라, 정기적으로 검토하고 개선해야 합니다.
