# Day 4: 프로젝트 일정 관리 - 상세 강의안

## 1교시: 일정 관리 개요 및 활동 정의 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 일정 관리의 중요성

**프로젝트 성공의 3대 제약조건**
- 범위(Scope) - 무엇을 만들 것인가
- 일정(Schedule) - 언제 완료할 것인가 ⭐️
- 원가(Cost) - 얼마나 들 것인가

**일정 지연의 파급 효과**
- 직접 영향: 인건비 증가 (팀원 투입 기간 연장)
- 간접 영향: 시장 기회 상실, 고객 신뢰 저하, 계약 위약금
- 조직 영향: 리소스 충돌 (다른 프로젝트 지연), 팀 사기 저하

**통계적 사실**
- Standish Group 보고서: IT 프로젝트 64%가 일정 지연 경험
- 평균 지연율: 계획 대비 +89%
- 일정 지연의 1위 원인: 부정확한 초기 추정 (43%)
- 2위 원인: 범위 크리프 (38%)

#### 2. 일정 관리의 6개 프로세스

**6.1 일정 관리 계획 수립 (Plan Schedule Management)**
- 입력물: PMPP, 프로젝트 헌장, EEF, OPA
- 도구: 전문가 판단, 데이터 분석, 회의
- 출력물: 일정 관리 계획서
- 핵심: 일정을 어떻게 계획, 개발, 관리, 통제할지 정의

**6.2 활동 정의 (Define Activities)**
- 입력물: PMPP, 범위 기준선(WBS, WBS 사전), EEF, OPA
- 도구: 분해(Decomposition), 상향식 계획, 전문가 판단
- 출력물: 활동 목록, 활동 속성, 마일스톤 목록
- 핵심: WBS 작업 패키지를 실행 가능한 활동으로 분해

**6.3 활동 순서 배열 (Sequence Activities)**
- 입력물: PMPP, 활동 목록, 활동 속성, 마일스톤 목록
- 도구: PDM(선행 관계 다이어그램법), 의존성 결정, 리드/래그
- 출력물: 프로젝트 일정 네트워크 다이어그램
- 핵심: 활동 간 논리적 관계 설정

**6.4 활동 기간 산정 (Estimate Activity Durations)**
- 입력물: PMPP, 활동 목록, 자원 요구사항, 자원 캘린더
- 도구: 유사 산정, 모수 산정, 3점 산정, 상향식 산정
- 출력물: 기간 산정치, 산정 근거
- 핵심: 각 활동을 완료하는 데 필요한 작업 기간 추정

**6.5 일정 개발 (Develop Schedule)**
- 입력물: 활동 목록, 네트워크 다이어그램, 기간 산정치
- 도구: CPM(주공정선법), 자원 최적화, 일정 단축
- 출력물: 일정 기준선, 프로젝트 일정, 간트차트
- 핵심: 주공정을 식별하고 프로젝트 종료일 결정

**6.6 일정 통제 (Control Schedule)**
- 입력물: PMPP, 프로젝트 일정, 작업 성과 데이터
- 도구: 데이터 분석, 주공정선법, 일정 압축
- 출력물: 작업 성과 정보, 일정 예측, 변경 요청
- 핵심: 일정 기준선 대비 진척 모니터링 및 편차 관리

#### 3. WBS 작업 패키지 vs 활동

**작업 패키지 (Work Package)**
- WBS의 최하위 수준
- "인도물" 중심
- 책임 배정의 단위
- 예: "로그인 모듈 개발"

**활동 (Activity)**
- 작업 패키지를 더 분해한 "작업" 단위
- "동사" 중심
- 일정 계획의 기본 단위
- 예: "로그인 화면 설계", "로그인 API 개발", "로그인 단위 테스트"

**분해 기준**
- 산정 가능성: 기간과 자원을 추정할 수 있는 수준
- 할당 가능성: 1명 또는 소규모 팀에 배정 가능
- 측정 가능성: 진척률을 측정할 수 있는 수준
- 일반적 크기: 8~80시간 (1일~2주)

#### 4. 활동 목록 (Activity List)

**필수 정보**
1. **활동 ID:** ACT-001, ACT-002 (고유 식별자)
2. **활동명:** "사용자 요구사항 인터뷰 수행"
3. **활동 설명:** 구체적인 작업 내용 (2~3문장)
4. **WBS 연결:** 어느 작업 패키지에서 분해되었는가
5. **담당자:** 역할 또는 이름
6. **선행 활동:** 이 활동 전에 완료되어야 할 활동
7. **기간 산정치:** 작업 기간 (일 단위)
8. **노력 산정치:** 작업량 (인시 또는 인일)

**활동 속성 (Activity Attributes)**
- 활동 목록의 상세 정보를 보완하는 메타데이터
- 자원 요구사항, 제약조건, 가정사항
- 지리적 위치, 작업 범주
- 프로젝트 진행 중 점진적으로 상세화

#### 5. 마일스톤 (Milestone)

**정의**
- 프로젝트의 중요한 시점 또는 이벤트
- **기간이 0인 활동** (시작일 = 종료일)
- 중요한 인도물 완성이나 의사결정 시점

**특징**
- Duration = 0 (작업이 아닌 시점)
- 자원이 소비되지 않음
- 프로젝트 진척 측정의 기준점
- 고객 또는 경영진 보고에 사용

**마일스톤 유형**

**필수 마일스톤 (Mandatory):**
- 계약서 명시: "요구사항 승인 (3/31)", "베타 테스트 완료 (10/15)"
- 규제 요구: "보안 인증 취득 (6/30)"
- 외부 의존: "외부 API 제공 시점 (5/1)"

**선택적 마일스톤 (Optional):**
- 내부 관리용: "설계 리뷰 완료", "통합 테스트 시작"
- 의사결정 시점: "Phase Gate Review", "Go/No-Go 결정"

**명명 규칙**
- 완료형으로 작성: "요구사항이 승인됨", "시스템이 배포됨"
- 측정 가능하게: "UAT 통과" (○) vs "테스트 완료" (모호함 ✗)

#### 6. 일정 관리 계획서 (Schedule Management Plan)

**필수 구성 요소**

1. **일정 모델 및 도구**
   - 사용 방법론: CPM, CCM, Agile
   - 소프트웨어 도구: MS Project, Primavera, JIRA
   - 일정 단위: 시간, 일, 주

2. **정확도 및 측정 단위**
   - 산정 정확도: ±10% (계획 phase), ±5% (실행 phase)
   - 측정 단위: 인일, 인시
   - 반올림 규칙: 0.5일 단위

3. **일정 기준선 설정 및 유지**
   - 기준선 승인 절차
   - 재기준선 승인 권한 (PM vs CCB)
   - 기준선 변경 문서화 방법

4. **진척 측정 방법**
   - 0/100%, 50/50%, 20/80% 규칙
   - Earned Value 측정
   - 마일스톤 달성 여부

5. **일정 변경 통제 절차**
   - 변경 요청 제출 → 영향 분석 → CCB 승인 → 기준선 업데이트
   - 승인 권한: 3일 이내 지연 (PM), 3일 초과 (CCB)

6. **보고 형식 및 주기**
   - 주간 보고: 간트차트 + 주요 이슈
   - 월간 보고: Milestone 달성 현황, SPI
   - 경영진 보고: 분기별, Dashboard 형식

**예시 3: 활동 정의 실패 사례 - 전자정부 프레임워크 전환 프로젝트**

**프로젝트 배경:**
- 목표: 레거시 시스템을 전자정부 프레임워크로 전환
- 예산: 12억원, 기간: 8개월
- 팀: PM 1명, 개발자 10명, QA 2명

**활동 정의 시 문제점:**

1. **과도한 분해**
   - "로그인 버튼 생성" (30분 작업)
   - "로그인 버튼 색상 변경" (10분 작업)
   - → 총 활동 수 3,500개 (관리 불가)

2. **불충분한 분해**
   - "시스템 개발" (120일)
   - → 진척 측정 불가, 리스크 조기 발견 불가

3. **명확하지 않은 활동명**
   - "개발 작업" (무엇을?)
   - "테스트" (어떤 테스트?)

4. **WBS와 연결 부재**
   - 활동이 어느 작업 패키지에서 나왔는지 불명확
   - 범위 추적 불가

**결과:**
- 프로젝트 4개월 차 진척률 "50%"라고 보고했으나 실제 30%
- 주공정 식별 실패로 중요하지 않은 작업에 자원 집중
- 프로젝트 기간 12개월로 연장 (+50%)
- 예산 16억원 소진 (+33%)

**교훈:**
- 활동 크기: 8~80시간 (1일~2주) 유지
- 활동명: "동사 + 명사" 구조로 명확하게
- WBS 코드로 추적성 확보
- 활동 수: 프로젝트 기간(월) × 50~100개 정도가 적정
  (8개월 프로젝트 → 400~800개 활동)

**과제 4: 일정 관리 계획서 초안**

- 예: "일정 모델 및 도구: CPM 사용, MS Project로 관리, 일 단위로 계획"

### 퀴즈 (10분)

**문제 1:**
일정 관리의 6개 프로세스를 순서대로 나열하고, 각 프로세스의 핵심 출력물을 쓰시오.

**모범답안:**
1. 6.1 일정 관리 계획 수립 → 일정 관리 계획서
2. 6.2 활동 정의 → 활동 목록, 활동 속성, 마일스톤 목록
3. 6.3 활동 순서 배열 → 프로젝트 일정 네트워크 다이어그램
4. 6.4 활동 기간 산정 → 기간 산정치, 산정 근거
5. 6.5 일정 개발 → 일정 기준선, 프로젝트 일정, 간트차트
6. 6.6 일정 통제 → 작업 성과 정보, 일정 예측, 변경 요청

프로세스 1~5는 계획 프로세스 그룹, 6은 감시 및 통제 프로세스 그룹에 속합니다.

**문제 2:**
마일스톤의 특징 3가지를 쓰고, 마일스톤과 일반 활동의 차이를 설명하시오.

**모범답안:**

마일스톤의 특징:
1. **기간이 0인 활동:** 시작일 = 종료일, 작업이 아닌 시점을 표시
2. **자원 미소비:** 인력이나 비용이 배정되지 않음
3. **중요 시점 표시:** 중요한 인도물 완성, 의사결정 시점, Phase 전환점

차이점:

| 구분 | 마일스톤 | 일반 활동 |
|------|---------|----------|
| 기간 | 0 (시점) | 1일 이상 (구간) |
| 자원 | 없음 | 인력, 장비 등 배정 |
| 표현 | 완료형 ("승인됨") | 동사형 ("개발한다") |
| 목적 | 진척 측정, 보고 | 실제 작업 수행 |
| 간트차트 | ◆ 마름모 | ▬ 막대 |

**문제 3:**
WBS의 작업 패키지와 활동의 차이를 3가지 측면에서 비교하시오.

**모범답안:**

| 측면 | 작업 패키지 | 활동 |
|------|------------|------|
| 소속 체계 | WBS의 최하위 수준 | 작업 패키지를 더 분해 |
| 중심 개념 | 인도물 중심 ("무엇") | 작업 중심 ("어떻게") |
| 관리 목적 | 책임 배정, 원가 집계 | 일정 계획, 진척 측정 |
| 예시 | "로그인 모듈" | "로그인 화면 설계", "로그인 API 개발" |
| 크기 | 일반적으로 더 큼 | 8~80시간 (1일~2주) |

작업 패키지는 "무엇을 만들 것인가"에 초점이 있고, 활동은 "그것을 만들기 위해 무엇을 할 것인가"에 초점이 있습니다.

**문제 4:**
활동 정의 시 흔히 발생하는 오류 3가지를 쓰고, 각각의 문제점을 설명하시오.

**모범답안:**

1. **과도한 분해 (Over-decomposition)**
   - 오류: 활동을 너무 작게 쪼갬 ("버튼 생성", "색상 변경" 등)
   - 문제점: 활동 수 과다 (수천 개)로 관리 오버헤드 증가, PM이 행정 업무에 매몰
   - 해결: 8~80시간 크기로 유지

2. **불충분한 분해 (Under-decomposition)**
   - 오류: 활동이 너무 큼 ("시스템 개발" 120일)
   - 문제점: 진척률 측정 불가 (2개월 지나도 "진행 중"), 리스크 조기 발견 불가
   - 해결: 측정 가능한 크기로 분해, 2주 이내 완료 가능하게

3. **모호한 활동명**
   - 오류: "개발", "테스트", "작업" 등 불명확한 표현
   - 문제점: 담당자가 정확히 무엇을 해야 하는지 불명확, 진척 측정 기준 모호
   - 해결: "동사 + 구체적 명사" 구조로 작성 ("사용자 로그인 API 개발", "결제 모듈 단위 테스트")

## 2교시: 활동 순서 배열 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 활동 순서 배열의 중요성

**정의**
- 활동 간 논리적 관계(의존성)를 파악하여 실행 순서 결정
- 네트워크 다이어그램으로 시각화
- 주공정(Critical Path) 식별의 기반

**중요성**
- 비현실적 일정 방지: "설계 전에 개발" 같은 불가능한 순서 차단
- 자원 최적화: 병렬 수행 가능한 활동 식별
- 리스크 관리: 외부 의존성 조기 파악
- 일정 단축 기회: 병렬화 가능 구간 발견

#### 2. PDM (Precedence Diagramming Method)

**정의**
- 선행 관계 다이어그램법
- AON (Activity-On-Node): 노드가 활동, 화살표가 의존관계
- PMBOK의 표준 기법

**4가지 의존 관계**

**1) FS (Finish-to-Start) - 가장 일반적 (90%)**
- 선행 활동이 완료되어야 후행 활동 시작 가능
- 예:
  - 설계 완료 → 개발 시작
  - 콘크리트 양생 완료 → 철근 작업 시작
  - 요구사항 승인 → 개발 작업 시작
- 표기: A --FS--> B

**2) SS (Start-to-Start)**
- 선행 활동이 시작되면 후행 활동도 시작 가능
- 동시 진행 가능한 작업에 사용
- 예:
  - 설계 시작 → 문서 작성 시작
  - 시스템 테스트 시작 → 사용자 매뉴얼 작성 시작
  - 백엔드 개발 시작 → 프론트엔드 개발 시작 (API 명세 기반)
- 표기: A ==SS==> B

**3) FF (Finish-to-Finish)**
- 선행 활동이 완료되어야 후행 활동도 완료 가능
- 종료 시점을 맞춰야 하는 경우
- 예:
  - 데이터 입력 완료 → 데이터 검증 완료
  - 코딩 완료 → 코드 리뷰 완료
  - 품질 검사 완료 → 포장 완료
- 표기: A ==FF==> B

**4) SF (Start-to-Finish) - 매우 드물게 사용 (<1%)**
- 선행 활동이 시작되어야 후행 활동이 완료 가능
- 실무에서 거의 사용 안 함
- 예: 
  - 야간 경비 시작 → 주간 경비 종료
  - 신규 시스템 가동 시작 → 구 시스템 운영 종료
- 표기: A --SF--> B

**PDM 다이어그램 예시:**

```
[A. 요구사항 분석]
       |
       | FS
       ↓
[B. 설계]
   |     \
   |FS    \SS
   |       \
   ↓        ↓
[C. DB구현] [D. 문서작성]
   |          |
   |FS        |FF
   |          |
   ↓          ↓
  [E. 백엔드 개발]
         |
         |FS
         ↓
    [F. 테스트]
```

#### 3. 의존성 (Dependencies)의 4가지 유형

**1) 강제적 의존성 (Mandatory / Hard Logic)**
- 물리적 또는 법적으로 반드시 지켜야 하는 순서
- 변경 불가
- 예:
  - 기초 공사 완료 → 골조 공사 시작
  - 코딩 완료 → 컴파일 수행
  - 보안 인증 취득 → 시스템 오픈
  - 계약 체결 → 프로젝트 착수

**2) 임의적 의존성 (Discretionary / Soft Logic / Preferred Logic)**
- 모범 사례(Best Practice)나 선호에 의한 순서
- 필요시 변경 가능
- 일정 단축 시 재검토 대상
- 예:
  - "UI 개발 완료 후 백엔드 개발 시작" (동시 진행도 가능하지만 효율적)
  - "통합 테스트 전 단위 테스트 100% 완료" (일부 병행 가능)
  - "문서 작성은 개발 완료 후" (병행 가능하지만 재작업 리스크)

**3) 외부 의존성 (External Dependencies)**
- 프로젝트 팀이 통제할 수 없는 외부 요인
- 리스크가 높음 → 리스크 등록부에 등록 필수
- 예:
  - 고객사의 서버 제공 지연
  - 외부 API 제공사의 작업 완료 대기
  - 정부 인허가 승인 대기
  - 외주 업체의 납품 지연
  - 하드웨어 도착 지연

**4) 내부 의존성 (Internal Dependencies)**
- 프로젝트 팀이 통제 가능한 의존성
- 관리 가능
- 예:
  - 팀 내 활동 간 의존성
  - 설계팀 → 개발팀 작업 순서

**의존성 결정 시 질문:**
1. "이 순서는 물리적으로 필수인가?" (강제적 vs 임의적)
2. "우리 팀이 통제 가능한가?" (내부 vs 외부)
3. "병렬 진행이 가능한가?" (FS vs SS)
4. "일정 단축 시 순서 변경 가능한가?" (임의적 의존성 재검토)

#### 4. 리드(Lead)와 래그(Lag)

**래그 (Lag) - 대기 시간**

**정의:**
- 선행 활동과 후행 활동 사이의 "대기 시간" (지연)
- 양수 값 (+)

**예시:**
- **FS + 3일 래그:** "콘크리트 타설 완료" → (3일 대기, 양생 기간) → "철근 작업 시작"
- **FS + 2일 래그:** "페인트칠 완료" → (2일 건조) → "검사 시작"
- **SS + 5일 래그:** "설계 시작" → (5일 후) → "문서 작성 시작" (설계가 어느 정도 진행되어야 문서화 가능)

**사용 사례:**
- 물리적 대기 시간: 건조, 양생, 배송
- 선행 작업의 일부 완료 후 후행 시작: SS + 래그
- 승인 대기 기간
- 학습 곡선: 새 기술 습득 기간

**리드 (Lead) - 앞당김**

**정의:**
- 후행 활동을 선행 활동 완료 전에 "미리" 시작
- 음수 값 (-)
- 실무에서 래그보다 덜 사용됨

**예시:**
- **FS - 2일 리드:** "개발" 완료 2일 전에 "테스트 계획 수립" 시작 가능
- **FS - 3일 리드:** "문서 작성" 완료 3일 전에 "문서 리뷰" 시작 (초안 기반)

**리드 사용 시 주의사항:**
- 재작업 리스크 증가 (선행 활동 변경 시 후행 활동도 수정)
- CPM 계산이 복잡해짐
- 가능하면 SS 관계로 대체 권장
  - "FS - 2일 리드" → "SS 관계"로 변경하면 더 명확

**표기법:**
- FS + 3 (3일 래그)
- FS - 2 (2일 리드)
- SS + 5 (5일 래그)

#### 5. 프로젝트 일정 네트워크 다이어그램

**정의**
- 프로젝트 활동 간 논리적 관계를 시각화한 도표
- CPM 계산의 기반

**구성 요소**
- **노드 (상자):** 활동
- **화살표 (선):** 의존 관계
- **관계 표시:** FS, SS, FF, SF
- **리드/래그:** 화살표 위에 표기

**작성 규칙**
1. **시작 노드와 종료 노드:** 프로젝트 시작점과 종료점
2. **Dangling 방지:** 모든 활동은 시작과 종료에 연결
3. **루프 방지:** 순환 의존성 금지 (A→B→C→A ✗)
4. **명확한 레이블:** 활동 ID와 이름 명시

**다이어그램 종류**

**AON (Activity-On-Node) - PMBOK 표준**
- 노드 = 활동
- 화살표 = 의존관계
- 현대 소프트웨어(MS Project)가 사용

**AOA (Activity-On-Arrow) - 구식**
- 화살표 = 활동
- 노드 = 이벤트 (시작/종료)
- Dummy 활동 필요
- 거의 사용 안 함

#### 6. 활동 순서 배열 프로세스

**입력물**
1. 일정 관리 계획서
2. 활동 목록
3. 활동 속성
4. 마일스톤 목록
5. 프로젝트 범위 기술서
6. EEF: 정부 규제, 업계 표준
7. OPA: 과거 프로젝트의 네트워크 다이어그램

**도구 및 기법**
1. **선행 관계 다이어그램법 (PDM):** FS, SS, FF, SF
2. **의존성 결정:** 강제적, 임의적, 외부, 내부
3. **리드와 래그:** 활동 간 시간 조정
4. **프로젝트 관리 정보 시스템 (PMIS):** MS Project, Primavera

**출력물**
1. **프로젝트 일정 네트워크 다이어그램**
2. **프로젝트 문서 업데이트:**
   - 활동 속성 (의존 관계 추가)
   - 가정 사항 로그 (의존성 관련 가정)
   - 마일스톤 목록

### 예시 (25분)

**활동 목록 (주어짐):**

            /          \

         /    \            |
       FS      FS          FS
       /        \          |

        \                   FS
         \                  /
          \     [F. 백엔드, 10일]
           \      /
            FS  FS
             \ /
        [G. 통합, 3일]
              |
              FS
              |

- 다음 단계: CPM으로 주공정 식별 → E-F-G 경로일 가능성 높음

**예시 2: 리드/래그 적용 - 건설 프로젝트**

**시나리오:** 아파트 건설 프로젝트

**활동:**

1. **기초 공사 → 골조 공사**
   - 관계: FS + 7일 래그
   - 이유: 콘크리트 양생 기간 7일 필요
   - 기초 공사 완료 후 7일 대기 → 골조 시작

2. **골조 공사 → 전기 배관**
   - 관계: SS + 10일 래그
   - 이유: 골조 10일 진행 후 (1층 완료) 전기 배관 시작 가능
   - 골조와 전기 배관 병행 작업

3. **내부 마감 → 외부 마감**
   - 관계: FF
   - 이유: 내부 마감이 완료되어야 외부 마감도 완료 (동시 종료)

4. **페인트칠 → 최종 검사**
   - 관계: FS + 2일 래그
   - 이유: 페인트 건조 기간 2일

**일정 계산 예시:**

```
활동: 기초 공사 (10일) → [7일 래그] → 골조 공사 (30일)

기초 공사: Day 1 ~ Day 10
대기 기간: Day 11 ~ Day 17 (7일 래그)
골조 공사: Day 18 ~ Day 47

총 소요: 47일
```

**모범답안:**

   - 설명: 선행 활동이 완료되어야 후행 활동 시작 가능
   - 예: "설계 완료" → "개발 시작"

2. **SS (Start-to-Start) - 사용 빈도: 8%**
   - 설명: 선행 활동이 시작되면 후행 활동도 시작 가능 (병행 작업)
   - 예: "개발 시작" → "테스트 계획 수립 시작"

3. **FF (Finish-to-Finish) - 사용 빈도: 2%**
   - 설명: 선행 활동이 완료되어야 후행 활동도 완료 가능
   - 예: "코딩 완료" → "코드 리뷰 완료"

4. **SF (Start-to-Finish) - 사용 빈도: <1%**
   - 설명: 선행 활동이 시작되어야 후행 활동이 완료 가능 (매우 드묾)
   - 예: "신규 시스템 가동 시작" → "구 시스템 운영 종료"

실무에서는 FS가 압도적으로 많이 사용되며, SS는 일정 단축을 위한 병행 작업 시 사용됩니다.

**문제 2:**
의존성의 4가지 유형을 설명하고, 각 유형의 예시를 2개씩 드시오. 외부 의존성이 프로젝트에 미치는 리스크를 설명하시오.

**모범답안:**

**1. 강제적 의존성 (Mandatory)**
- 설명: 물리적/법적으로 반드시 지켜야 하는 순서
- 예: "기초 공사 → 골조 공사", "계약 체결 → 프로젝트 착수"

**2. 임의적 의존성 (Discretionary)**
- 설명: 모범 사례나 선호에 의한 순서, 변경 가능
- 예: "설계 100% 완료 → 개발 시작", "단위 테스트 → 통합 테스트"

**3. 외부 의존성 (External)**
- 설명: 프로젝트 팀이 통제할 수 없는 외부 요인
- 예: "고객사 서버 제공 대기", "외부 API 제공사 작업 완료"

**4. 내부 의존성 (Internal)**
- 설명: 프로젝트 팀이 통제 가능한 의존성
- 예: "설계 → 개발", "개발 → 테스트"

**외부 의존성의 리스크:**
- 프로젝트 팀이 통제할 수 없어 일정 지연 위험 높음
- 외부 업체/고객의 우선순위 변경에 영향받음
- 대응 방법:
  - 리스크 등록부에 등록
  - 계약서에 책임 명시
  - 대체 계획 수립
  - 빈번한 커뮤니케이션

**문제 3:**
리드(Lead)와 래그(Lag)의 차이를 설명하고, 각각의 예시를 2개씩 드시오. 리드 사용 시 주의사항을 쓰시오.

**모범답안:**

**래그 (Lag) - 대기 시간**
- 정의: 선행 활동과 후행 활동 사이의 대기 시간 (양수 +)
- 예시:
  1. "콘크리트 타설 완료" + 3일 래그 → "철근 작업 시작" (양생 기간)
  2. "요구사항 분석 시작" + 5일 래그 (SS) → "문서 작성 시작" (분석이 어느정도 진행되어야 문서화 가능)

**리드 (Lead) - 앞당김**
- 정의: 후행 활동을 선행 활동 완료 전에 미리 시작 (음수 -)
- 예시:
  1. "개발" - 2일 리드 (FS) → "테스트 계획 수립" (개발 완료 2일 전 시작)
  2. "문서 작성" - 3일 리드 (FS) → "문서 리뷰" (초안 기반 리뷰)

**리드 사용 시 주의사항:**
1. 재작업 리스크 증가: 선행 활동 변경 시 후행 활동도 수정 필요
2. CPM 계산 복잡도 증가
3. 팀원들이 리드 개념을 이해하기 어려움
4. 해결책: 가능하면 SS 관계로 대체 권장
   - "FS - 2일 리드" → "SS 관계"로 변경하면 더 명확

**문제 4:**
프로젝트 일정 네트워크 다이어그램 작성 시 흔히 발생하는 오류 3가지를 쓰고, 각각의 해결 방법을 제시하시오.

**모범답안:**

**1. Dangling (매달린 활동)**
- 오류: 시작 노드나 종료 노드에 연결되지 않은 활동
- 예: 중간에 고립된 활동이 있음
- 문제: 주공정 계산 불가, 일정 산정 오류
- 해결: 모든 활동을 시작 노드와 종료 노드에 연결

**2. 순환 의존성 (Loop)**
- 오류: A → B → C → A 식의 순환 구조
- 예: "설계" → "개발" → "설계 수정" → "개발"
- 문제: 논리적 모순, CPM 계산 불가
- 해결: 반복 작업은 별도 활동으로 분리
  - "초기 설계" → "개발" → "설계 재검토" (별개 활동)

**3. 의존 관계 과다 (Over-dependency)**
- 오류: 모든 활동을 순차적으로 연결 (과도한 FS)
- 예: A → B → C → D (실제로는 B와 C 병행 가능)
- 문제: 일정 불필요하게 길어짐, 자원 활용도 낮음
- 해결: 병행 가능한 작업 식별 → SS 관계 활용

## 3교시: 활동 기간 산정 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 활동 기간 산정의 중요성

**기간 (Duration) vs 노력 (Effort)**

**기간 (Duration):**
- 활동 시작부터 완료까지의 "경과 시간"
- 달력 기준 (영업일 또는 달력일)
- 예: "5일" (월~금)

**노력 (Effort):**
- 활동을 완료하는 데 필요한 "작업량"
- 인시(person-hour) 또는 인일(person-day)
- 예: "40 인시" = 1인이 5일 또는 2인이 2.5일

**관계:**
```
기간 = 노력 / (자원 수 × 생산성)
```

**예시:**
- 코딩 작업: 40 인시 (노력)
- 개발자 1명 배정 → 기간: 5일 (1일 8시간 근무)
- 개발자 2명 배정 → 기간: 2.5일 (이론상, 실제로는 커뮤니케이션 오버헤드)

**주의: Brooks' Law**
> "지연되는 소프트웨어 프로젝트에 인력을 추가하는 것은 프로젝트를 더 지연시킨다."

- 이유: 신규 인력의 학습 기간, 커뮤니케이션 오버헤드
- 9명의 여성이 1개월에 아기를 낳을 수 없다

#### 2. 활동 기간 산정 기법

**2.1 유사 산정 (Analogous Estimating)**

**정의:**
- 과거 유사 프로젝트/활동의 실제 데이터를 현재 프로젝트에 적용
- Top-down 방식

**장점:**
- 빠르고 비용 효율적
- 정보가 제한적일 때 유용 (프로젝트 초기)
- 경험 많은 PM이 수행 시 정확도 높음

**단점:**
- 프로젝트 간 차이를 고려하지 못하면 부정확
- 과거 데이터 품질에 의존
- 주관적 판단 개입

**사용 시기:**
- 프로젝트 초기 (상세 정보 부족)
- ROM (Rough Order of Magnitude) 산정
- 유사 프로젝트 데이터가 있을 때

**정확도:**
- ±50% (프로젝트 초기)
- ±25% (유사 프로젝트가 많을 때)

**예시:**
- 과거 "고객 관리 모듈 개발": 30일
- 현재 "제품 관리 모듈 개발": 30일로 산정 (복잡도 유사)
- 조정: 제품 관리가 더 복잡 → 30일 × 1.2 = 36일

**2.2 모수 산정 (Parametric Estimating)**

**정의:**
- 통계적 관계를 사용한 산정
- 수식: 기간 = 단위당 생산성 × 수량

**장점:**
- 높은 정확도 (데이터 기반)
- 정량적, 객관적
- 대규모 프로젝트에 적합

**단점:**
- 파라미터와 모델 개발 필요
- 과거 데이터가 충분해야 함
- 프로젝트 특성이 크게 다르면 부정확

**정확도:**
- ±10~15% (데이터 충분 시)

**예시:**

**소프트웨어 개발:**
```
Function Point 산정:
- 화면 수: 20개
- 1개 화면 개발: 2 인일
- 총 기간 = 20 × 2 = 40 인일
- 개발자 2명 배정 → 기간: 20일
```

**건설:**
```
- 벽돌 쌓기: 1㎡당 0.5 인일
- 벽면적: 100㎡
- 총 노력 = 100 × 0.5 = 50 인일
```

**문서 작성:**
```
- 1페이지당 1시간
- 매뉴얼 50페이지
- 총 시간: 50시간 = 6.25일
```

**2.3 3점 산정 (Three-Point Estimating)**

**정의:**
- 낙관치(O), 최빈치(M), 비관치(P) 3가지 산정치 사용
- 불확실성을 정량화

**산정 값:**
- **O (Optimistic):** 최상의 경우 (모든 것이 완벽)
- **M (Most Likely):** 가장 가능성 높은 경우
- **P (Pessimistic):** 최악의 경우 (모든 것이 잘못됨)

**계산 공식:**

**삼각 분포 (Triangular Distribution) - 단순 평균:**
```
TE (Expected Time) = (O + M + P) / 3
```

**베타 분포 (Beta Distribution / PERT) - 가중 평균:**
```
TE = (O + 4M + P) / 6
```
- 최빈치에 4배 가중치
- PERT (Program Evaluation and Review Technique)에서 사용
- PMBOK 권장

**표준편차 (Standard Deviation):**
```
SD = (P - O) / 6
```
- 불확실성의 정도
- SD가 클수록 리스크 높음

**장점:**
- 불확실성 반영
- 리스크 정량화
- 더 정확한 산정

**단점:**
- 3개 값 산정 필요 (시간 소요)
- 여전히 주관적 판단

**예시:**

활동: "결제 모듈 개발"
- O (낙관): 5일 (경험 많은 개발자, 재사용 코드 많음)
- M (최빈): 8일 (일반적 경우)
- P (비관): 14일 (신규 API 연동 문제, 요구사항 변경)

**베타 분포 (PERT):**
```
TE = (5 + 4×8 + 14) / 6
   = (5 + 32 + 14) / 6
   = 51 / 6
   = 8.5일
```

**표준편차:**
```
SD = (14 - 5) / 6 = 9 / 6 = 1.5일
```

**해석:**
- 기대 기간: 8.5일
- 불확실성: ±1.5일
- 68% 확률로 7~10일 사이 (1 SD)
- 95% 확률로 5.5~11.5일 사이 (2 SD)

**2.4 상향식 산정 (Bottom-Up Estimating)**

**정의:**
- 활동을 더 작은 세부 작업으로 분해
- 각 세부 작업을 산정 후 합산
- Bottom-up 방식

**장점:**
- 가장 정확 (상세도 높음)
- 담당자의 참여 → 책임감
- 누락 방지

**단점:**
- 시간과 비용이 많이 소요
- 과도한 분해 시 관리 오버헤드

**정확도:**
- ±5~10% (가장 높음)

**사용 시기:**
- 프로젝트 실행 단계
- 정확한 산정이 필요할 때
- WBS가 충분히 상세화되었을 때

**예시:**

활동: "사용자 로그인 기능 개발" (상향식 분해)

| 세부 작업 | 기간 |
|-----------|------|
| 1. 로그인 화면 UI 설계 | 0.5일 |
| 2. 로그인 화면 HTML/CSS 코딩 | 1일 |
| 3. 로그인 API 설계 | 0.5일 |
| 4. 로그인 API 백엔드 개발 | 2일 |
| 5. 세션 관리 로직 구현 | 1일 |
| 6. 프론트-백 연동 | 1일 |
| 7. 단위 테스트 작성 | 1일 |
| 8. 통합 테스트 | 0.5일 |
| 9. 코드 리뷰 및 수정 | 0.5일 |
| **합계** | **8일** |

**2.5 기법 선택 가이드**

| 기법 | 정확도 | 비용 | 사용 시기 | 적합 프로젝트 단계 |
|------|--------|------|-----------|--------------------|
| 유사 산정 | ±25~50% | 낮음 | 정보 부족 | 착수, 계획 초기 |
| 모수 산정 | ±10~15% | 중간 | 파라미터 확보 | 계획 단계 |
| 3점 산정 | ±10~20% | 중간 | 불확실성 높음 | 계획 단계 |
| 상향식 산정 | ±5~10% | 높음 | 상세 정보 확보 | 실행 단계 |

#### 3. 예비 분석 (Reserve Analysis)

**정의:**
- 불확실성을 대비한 "버퍼" 시간
- 리스크 대응 전략

**3.1 우발 예비 (Contingency Reserve)**

**정의:**
- "알려진 리스크"에 대비한 예비 시간
- Known Unknowns
- 리스크 등록부의 식별된 리스크 대응

**특징:**
- PM의 권한으로 사용
- 일정 기준선에 **포함됨**
- 고객에게 보고됨

**산정 방법:**
- 리스크 발생 확률 × 영향
- 일반적으로 전체 기간의 5~10%
- 3점 산정의 표준편차 활용

**예시:**
- 프로젝트 총 기간: 100일
- 우발 예비: 10일 (10%)
- 일정 기준선: 110일
- 리스크 발생 시: 우발 예비 사용, PM 승인

**3.2 관리 예비 (Management Reserve)**

**정의:**
- "알려지지 않은 리스크"에 대비한 예비 시간
- Unknown Unknowns
- 예측하지 못한 변경 사항 대응

**특징:**
- 경영진/스폰서 승인 필요
- 일정 기준선에 **포함되지 않음**
- 고객에게 일반적으로 보고 안 함

**산정 방법:**
- 전체 기간의 5~10%
- 조직 정책에 따름

**예시:**
- 일정 기준선: 110일 (활동 100일 + 우발 예비 10일)
- 관리 예비: 10일
- 프로젝트 총 예산 기간: 120일
- 사용: 고객이 갑자기 새로운 법규 준수 요구 → 관리 예비 사용

**비교:**

| 구분 | 우발 예비 | 관리 예비 |
|------|----------|----------|
| 대상 | 알려진 리스크 | 알려지지 않은 리스크 |
| 기준선 포함 | O | X |
| 승인 권한 | PM | 경영진/스폰서 |
| 고객 공개 | O | 보통 X |
| 사용 예시 | 식별된 리스크 발생 | 예측 못한 범위 변경 |

#### 4. 기타 고려 사항

**4.1 자원 캘린더 (Resource Calendar)**

**정의:**
- 자원의 가용성 정보
- 근무일, 휴무일, 가용 시간

**고려 사항:**
- 공휴일, 주말
- 휴가 계획
- 타 프로젝트 투입 시간
- 교육, 출장

**예시:**
- 활동: 8 인일
- 개발자: 주 3일만 투입 가능 (타 프로젝트 50% 투입)
- 기간: 8일 / 0.6 = 13.3일 → 14일

**4.2 생산성 요인**

**영향 요인:**
- 팀원 경험 수준 (주니어 vs 시니어)
- 도구 및 자동화 수준
- 작업 환경 (재택 vs 사무실)
- 팀 사기
- 프로젝트 복잡도

**경험 규칙:**
- 주니어: 시니어 대비 2~3배 시간 소요
- 신규 기술: 기존 기술 대비 1.5~2배
- 분산 팀: 동일 장소 팀 대비 1.2~1.5배

### 예시 (25분)

**예시 1: 3점 산정 (PERT) - 모바일 앱 기능 개발**

**시나리오:**
"푸시 알림 기능" 개발 기간 산정

**전문가 의견 수렴:**
- **O (낙관):** 3일
  - 이유: 라이브러리 활용, 유사 경험 있음, 요구사항 명확
- **M (최빈):** 5일
  - 이유: 일반적인 경우, 약간의 커스터마이징 필요
- **P (비관):** 10일
  - 이유: 서버 API 지연, iOS/Android 플랫폼별 이슈, 테스트 단말 부족

**PERT 계산:**
```
TE = (O + 4M + P) / 6
   = (3 + 4×5 + 10) / 6
   = (3 + 20 + 10) / 6
   = 33 / 6
   = 5.5일
```

**표준편차:**
```
SD = (P - O) / 6
   = (10 - 3) / 6
   = 7 / 6
   = 1.17일
```

**결론:**
- 활동 기간 산정: **6일** (반올림)
- 불확실성: ±1.17일
- 68% 확률: 4.3~6.7일
- 95% 확률: 3.2~7.8일

```
TE = (O + 4M + P) / 6

TE: 기대 기간 (Expected Time)
O: 낙관치 (Optimistic)
M: 최빈치 (Most Likely)
P: 비관치 (Pessimistic)
```

**삼각 분포 공식:**
```
TE = (O + M + P) / 3
```

**차이점:**
- PERT: 최빈치(M)에 4배 가중치 → 가장 가능성 높은 값에 비중
- 삼각 분포: 3개 값을 동등하게 평균

**PERT가 최빈치에 가중치를 두는 이유:**
1. **현실 반영:** 최빈치가 발생할 확률이 가장 높음
2. **극단값 완화:** 낙관치/비관치는 드물게 발생하는 극단값
3. **베타 분포 특성:** 통계적으로 베타 분포가 프로젝트 활동의 불확실성을 더 잘 표현
4. **경험적 검증:** 실제 프로젝트 데이터와 비교 시 PERT가 더 정확

**표준편차 공식:**
```
SD = (P - O) / 6
```
- 불확실성의 크기를 정량화
- SD가 클수록 리스크 높음

**문제 2:**
상향식 산정의 장단점을 3가지씩 쓰고, 어떤 프로젝트 단계에서 사용하는 것이 적합한지 설명하시오.

**모범답안:**

**장점:**
1. **높은 정확도:** 세부 작업 수준까지 분해하므로 ±5~10% 정확도 (가장 높음)
2. **담당자 참여:** 실제 작업 수행자가 산정 → 책임감, 현실성, 동기부여
3. **누락 방지:** 상세 분해로 빠진 작업 발견, 숨겨진 작업 식별

**단점:**
1. **시간/비용 소요:** 세부 분해와 산정에 많은 시간 필요
2. **관리 오버헤드:** 너무 상세하면 추적 및 관리 부담 증가
3. **나무만 보고 숲을 못 봄:** 전체 그림을 놓칠 위험

**적합한 프로젝트 단계:**
- **프로젝트 실행 단계:** WBS가 충분히 상세화되고, 정확한 산정이 필요한 시점
- **계획 단계 후반:** 요구사항이 확정되고, 팀이 구성된 후
- **예산/일정이 타이트한 프로젝트:** 정확도가 매우 중요할 때

**부적합한 경우:**
- 프로젝트 초기 (정보 부족)
- 빠른 ROM (Rough Order of Magnitude) 산정 필요 시

**문제 3:**
우발 예비(Contingency Reserve)와 관리 예비(Management Reserve)의 차이를 5가지 측면에서 비교하시오.

**모범답안:**

| 구분 | 우발 예비 | 관리 예비 |
|------|----------|----------|
| **대상 리스크** | Known Unknowns (식별된 리스크) | Unknown Unknowns (예측 못한 리스크) |
| **일정 기준선 포함** | ○ 포함됨 | ✗ 포함 안 됨 |
| **승인 권한** | PM 권한으로 사용 가능 | 경영진/스폰서 승인 필요 |
| **고객 공개** | 고객에게 보고됨 | 보통 공개하지 않음 |
| **사용 예시** | 리스크 등록부의 리스크 발생 시 | 예상 못한 범위 변경, 신규 법규 |

**산정 방법:**
- 우발 예비: 리스크 발생 확률 × 영향 또는 전체 기간의 5~10%
- 관리 예비: 전체 기간의 5~10% (조직 정책)

**일정 구조:**
```
활동 기간 합계: 100일
+ 우발 예비: 10일
= 일정 기준선: 110일 ← PM 관리
+ 관리 예비: 10일
= 프로젝트 총 예산 기간: 120일 ← 경영진 승인
```

**문제 4:**
다음 활동의 기간을 PERT로 계산하시오. O=4일, M=7일, P=16일. 그리고 이 활동의 리스크 수준을 표준편차로 판단하시오.

**모범답안:**

**PERT 계산:**
```
TE = (O + 4M + P) / 6
   = (4 + 4×7 + 16) / 6
   = (4 + 28 + 16) / 6
   = 48 / 6
   = 8일
```

**표준편차:**
```
SD = (P - O) / 6
   = (16 - 4) / 6
   = 12 / 6
   = 2일
```

**확률 구간:**
- 68% 확률: 8 ± 2 = 6~10일 (1 SD)
- 95% 확률: 8 ± 4 = 4~12일 (2 SD)
- 99.7% 확률: 8 ± 6 = 2~14일 (3 SD)

**리스크 수준 판단:**
- SD = 2일은 비교적 **높은 불확실성**
- P와 O의 차이가 12일 (300% 차이)로 매우 큼
- 비관치(16일)가 최빈치(7일)의 2.3배 → 하방 리스크 존재
- 대응: 
  - 우발 예비 추가 배정 (2일 이상)
  - 리스크 완화 계획 수립
  - 자주 진척 모니터링 필요

**결론:**
- 활동 기간: **8일**
- 우발 예비: **2일** (1 SD)
- 일정 배정: **10일** (안전 여유 확보)

## 4교시: 일정 개발 - CPM (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. CPM (Critical Path Method) - 주공정선법

**정의:**
- 프로젝트 네트워크에서 가장 긴 경로를 식별하는 기법
- 주공정(Critical Path): 프로젝트 최소 완료 기간을 결정하는 경로
- 주공정 활동의 지연 = 전체 프로젝트 지연

**목적:**
1. 프로젝트 최소 완료 기간 파악
2. 주공정 활동 식별 (집중 관리 대상)
3. 여유 시간(Float/Slack) 계산
4. 일정 단축 기회 발견
5. 자원 배분 우선순위 설정

**기본 개념:**
- **주공정 (Critical Path):** 여유 시간이 0인 활동들의 경로
- **주공정 활동 (Critical Activity):** 주공정상의 활동, 지연 시 전체 프로젝트 지연
- **비주공정 활동:** 여유 시간이 있는 활동, 일정 범위 내 지연 가능
- **여유 시간 (Float/Slack):** 활동을 지연시켜도 프로젝트 완료일에 영향 없는 시간

#### 2. CPM 계산 용어

**ES (Early Start) - 최조 시작일:**
- 활동을 가장 빨리 시작할 수 있는 날짜
- 모든 선행 활동이 완료된 직후
- 전진 계산(Forward Pass)으로 산출

**EF (Early Finish) - 최조 완료일:**
- 활동을 가장 빨리 완료할 수 있는 날짜
- 공식: **EF = ES + Duration**
- 전진 계산으로 산출

**LS (Late Start) - 최지 시작일:**
- 프로젝트를 지연시키지 않고 활동을 시작할 수 있는 가장 늦은 날짜
- 후진 계산(Backward Pass)으로 산출

**LF (Late Finish) - 최지 완료일:**
- 프로젝트를 지연시키지 않고 활동을 완료할 수 있는 가장 늦은 날짜
- 후진 계산으로 산출
- 공식: **LS = LF - Duration**

**Duration (D):**
- 활동 기간 (일, 주 단위)

#### 3. 여유 시간 (Float/Slack)

**총 여유 (Total Float - TF)**

**정의:**
- 활동을 지연시켜도 프로젝트 완료일에 영향을 주지 않는 최대 시간

**공식:**
```
TF = LS - ES  또는  TF = LF - EF
```

**특징:**
- 주공정 활동의 TF = 0
- TF > 0 → 비주공정 활동 (지연 가능)
- TF < 0 → 일정 문제 (이미 지연됨)

**활용:**
- 자원 배분 우선순위: TF가 작을수록 우선
- 일정 여유 판단
- 리스크 평가

**자유 여유 (Free Float - FF)**

**정의:**
- 활동을 지연시켜도 후행 활동의 ES에 영향을 주지 않는 시간
- "내 것만의 여유"

**공식:**
```
FF = (후행 활동의 최소 ES) - EF
```

**특징:**
- FF ≤ TF (항상)
- FF = 0 → 후행 활동에 바로 영향
- FF > 0 → 후행 활동에 영향 없이 지연 가능

**총 여유 vs 자유 여유:**

| 구분 | 총 여유 (TF) | 자유 여유 (FF) |
|------|-------------|------------|
| 영향 범위 | 프로젝트 완료일 | 후행 활동 ES |
| 크기 | 더 큼 | 더 작음 |
| 의미 | "프로젝트 차원의 여유" | "내 활동만의 여유" |
| 사용 시 영향 | 후속 활동의 여유 감소 가능 | 후속 활동에 영향 없음 |

**예시:**
- 활동 A: TF = 5일, FF = 2일
- 해석: A를 2일 지연시켜도 후행 활동 시작일에 영향 없음
- A를 3~5일 지연시키면 후행 활동이 늦게 시작하지만 프로젝트 완료일엔 영향 없음

#### 4. CPM 계산 절차

**Step 1: 전진 계산 (Forward Pass) - ES, EF 계산**

**목적:** 프로젝트 최소 완료 기간 산출

**규칙:**
1. 시작 활동: ES = 0 (또는 프로젝트 시작일)
2. EF = ES + Duration
3. 후행 활동의 ES = 선행 활동들의 EF 중 **최대값** (max)
   - 예: 활동 C의 선행이 A(EF=5)와 B(EF=8)이면 → C의 ES = 8
4. 프로젝트 완료일 = 종료 활동의 EF

**전진 계산 흐름:**
```
시작 → 왼쪽에서 오른쪽으로 진행
ES → EF → 후행 활동의 ES → ...
```

**Step 2: 후진 계산 (Backward Pass) - LS, LF 계산**

**목적:** 각 활동의 최지 시작/완료일 산출

**규칙:**
1. 종료 활동: LF = EF (프로젝트 완료 목표일)
2. LS = LF - Duration
3. 선행 활동의 LF = 후행 활동들의 LS 중 **최소값** (min)
   - 예: 활동 A의 후행이 C(LS=10)와 D(LS=7)이면 → A의 LF = 7
4. 시작 활동까지 계산

**후진 계산 흐름:**
```
종료 ← 오른쪽에서 왼쪽으로 진행
LF ← LS ← 선행 활동의 LF ← ...
```

**Step 3: 여유 시간 계산**

각 활동별로:
```
TF = LS - ES (또는 LF - EF)
FF = (후행 활동의 최소 ES) - EF
```

**Step 4: 주공정 식별**

- TF = 0인 활동들을 연결
- 주공정은 1개 이상 존재 가능
- 주공정의 총 기간 = 프로젝트 최소 완료 기간

#### 5. 노드 다이어그램 표기법

**AON (Activity-On-Node) 표기:**

```
┌──────────────────┐
│ 활동 ID: A       │
├─────┬──────┬─────┤
│ ES  │ Dur  │ EF  │
│  0  │  5   │  5  │
├─────┼──────┼─────┤
│ LS  │ TF   │ LF  │
│  0  │  0   │  5  │
└─────┴──────┴─────┘
```

**9-Box 표기법:**

```
  ES  │ Dur │  EF
──────┼─────┼──────
 활동 │  TF │ 활동명
  ID  │     │
──────┼─────┼──────
  LS  │     │  LF
```

#### 6. 주공정이 2개 이상인 경우

**발생 원인:**
- 병렬 경로들의 기간이 동일
- 여러 경로 모두 TF = 0

**의미:**
- 모든 주공정 경로 관리 필요
- 리스크 더 높음 (관리 대상 증가)

**예시:**
```
경로 1: A → B → E → G (기간: 20일)
경로 2: A → C → F → G (기간: 20일)
경로 3: A → D → G (기간: 15일)

→ 경로 1과 2가 모두 주공정 (TF = 0)
→ 경로 3은 비주공정 (TF = 5일)
```

**관리 전략:**
- 두 주공정 모두 집중 관리
- 하나라도 지연되면 프로젝트 지연
- 자원 배분 시 두 경로 모두 고려

### 예시 (25분)

**예시 1: CPM 기본 계산 - 웹사이트 리뉴얼 프로젝트**

**활동 목록:**

| ID | 활동명 | 기간 | 선행 활동 |
|----|--------|------|----------|
| A | 현황 분석 | 3일 | - |
| B | UI 설계 | 4일 | A |
| C | DB 설계 | 2일 | A |
| D | 프론트 개발 | 6일 | B |
| E | 백엔드 개발 | 8일 | C |
| F | 통합 및 테스트 | 4일 | D, E |

**Step 1: 네트워크 다이어그램**

```
        [A, 3일]
        /      \
     FS        FS
      /          \
[B, 4일]      [C, 2일]
    |              |
    |FS            |FS
    |              |
[D, 6일]      [E, 8일]
    \              /
     \    FS    FS
      \          /
      [F, 4일]
```

**Step 2: 전진 계산 (ES, EF)**

| ID | Duration | ES | EF | 계산 근거 |
|----|----------|----|----|----------|
| A | 3 | 0 | 3 | 시작: ES=0, EF=0+3=3 |
| B | 4 | 3 | 7 | A의 EF=3, EF=3+4=7 |
| C | 2 | 3 | 5 | A의 EF=3, EF=3+2=5 |
| D | 6 | 7 | 13 | B의 EF=7, EF=7+6=13 |
| E | 8 | 5 | 13 | C의 EF=5, EF=5+8=13 |
| F | 4 | 13 | 17 | max(D의 EF=13, E의 EF=13)=13, EF=13+4=17 |

**프로젝트 최소 완료 기간: 17일**

**Step 3: 후진 계산 (LS, LF)**

| ID | Duration | LF | LS | 계산 근거 |
|----|----------|----|----|----------|
| F | 4 | 17 | 13 | 종료: LF=EF=17, LS=17-4=13 |
| E | 8 | 13 | 5 | F의 LS=13, LS=13-8=5 |
| D | 6 | 13 | 7 | F의 LS=13, LS=13-6=7 |
| C | 2 | 5 | 3 | E의 LS=5, LS=5-2=3 |
| B | 4 | 7 | 3 | D의 LS=7, LS=7-4=3 |
| A | 3 | 3 | 0 | min(B의 LS=3, C의 LS=3)=3, LS=3-3=0 |

**Step 4: 여유 시간 및 주공정**

| ID | ES | EF | LS | LF | TF = LS-ES | 주공정? |
|----|----|----|----|----|-----------|--------|
| A | 0 | 3 | 0 | 3 | 0 | ✓ |
| B | 3 | 7 | 3 | 7 | 0 | ✓ |
| C | 3 | 5 | 3 | 5 | 0 | ✓ |
| D | 7 | 13 | 7 | 13 | 0 | ✓ |
| E | 5 | 13 | 5 | 13 | 0 | ✓ |
| F | 13 | 17 | 13 | 17 | 0 | ✓ |

**주공정:**
- **경로 1: A → B → D → F (기간: 3+4+6+4 = 17일)** ✅
- **경로 2: A → C → E → F (기간: 3+2+8+4 = 17일)** ✅

**결론:**
- 주공정이 2개 존재 (모든 활동이 주공정)
- 모든 활동의 TF = 0 → 어느 활동이든 지연되면 프로젝트 지연
- 매우 타이트한 일정
- 관리 포인트: 모든 활동 집중 관리 필요

**주공정:** A → B → D → F (기간: 5+6+10+5 = 26일)

**분석:**
- **활동 C:** TF = 8일, FF = 8일
  - C를 8일 지연시켜도 프로젝트 완료일에 영향 없음
  - C를 8일 지연시켜도 후행 활동 E의 시작일에 영향 없음
  
- **활동 E:** TF = 9일, FF = 9일
  - E를 9일 지연시켜도 프로젝트 완료일에 영향 없음
  - E를 9일 지연시켜도 후행 활동 F의 시작일에 영향 없음 (D가 더 긴 경로)

**관리 전략:**
- 주공정 활동 (A, B, D, F): 집중 관리, 최우선 자원 배분
- 비주공정 활동 (C, E): 여유 시간 활용
  - 숙련도 낮은 인력 배치 가능
  - 주공정 문제 발생 시 자원 재배치 가능
  - 리스크 대응 버퍼로 활용

### 퀴즈 (10분)

**문제 1:**
CPM의 전진 계산(Forward Pass)과 후진 계산(Backward Pass)의 목적을 각각 설명하고, 계산 순서를 쓰시오.

**모범답안:**

**전진 계산 (Forward Pass):**
- 목적: 프로젝트 최소 완료 기간 산출, ES와 EF 계산
- 방향: 시작 → 종료 (왼쪽에서 오른쪽)
- 계산 순서:
  1. 시작 활동의 ES = 0
  2. EF = ES + Duration
  3. 후행 활동의 ES = 선행 활동들의 EF 중 **최대값** (max)
  4. 종료 활동의 EF = 프로젝트 완료 기간

**후진 계산 (Backward Pass):**
- 목적: 각 활동의 최지 시작/완료일 산출, LS와 LF 계산
- 방향: 종료 → 시작 (오른쪽에서 왼쪽)
- 계산 순서:
  1. 종료 활동의 LF = EF (프로젝트 목표 완료일)
  2. LS = LF - Duration
  3. 선행 활동의 LF = 후행 활동들의 LS 중 **최소값** (min)
  4. 시작 활동까지 역순 계산

**핵심 차이:**
- 전진: max 사용 (모든 선행 완료 대기)
- 후진: min 사용 (가장 빠른 후행에 맞춤)

**문제 2:**
총 여유(Total Float)와 자유 여유(Free Float)의 차이를 설명하고, 각각의 계산 공식을 쓰시오. 예를 들어 TF=5일, FF=2일인 활동의 의미를 설명하시오.

**모범답안:**

**총 여유 (Total Float - TF):**
- 정의: 활동을 지연시켜도 **프로젝트 완료일**에 영향을 주지 않는 최대 시간
- 계산: TF = LS - ES (또는 LF - EF)
- 의미: "프로젝트 차원의 여유 시간"

**자유 여유 (Free Float - FF):**
- 정의: 활동을 지연시켜도 **후행 활동의 ES**에 영향을 주지 않는 시간
- 계산: FF = (후행 활동의 최소 ES) - EF
- 의미: "내 활동만의 여유 시간"

**관계:**
- FF ≤ TF (항상)
- TF - FF = 후속 활동들의 여유에 영향을 주는 시간

**예시 해석: TF=5일, FF=2일**
- 이 활동을 **2일 지연**시키면:
  - 후행 활동 시작일: 영향 없음 ✓
  - 프로젝트 완료일: 영향 없음 ✓
  
- 이 활동을 **3~5일 지연**시키면:
  - 후행 활동이 늦게 시작함 (영향 있음) ✗
  - 하지만 프로젝트 완료일: 여전히 영향 없음 ✓
  - 후속 활동들의 TF가 감소함

- 이 활동을 **6일 이상 지연**시키면:
  - 후행 활동 지연 ✗
  - 프로젝트 완료일 지연 ✗

**활용:**
- FF만 사용하는 것이 안전 (후속에 영향 없음)
- TF 전체를 사용하려면 팀 조정 필요

**문제 3:**
주공정(Critical Path)의 정의와 특징 3가지를 쓰고, 주공정 활동이 지연되면 프로젝트에 어떤 영향이 있는지 설명하시오.

**모범답안:**

**주공정 (Critical Path) 정의:**
프로젝트 네트워크에서 여유 시간(TF)이 0인 활동들로 구성된, 가장 긴 경로. 프로젝트의 최소 완료 기간을 결정하는 경로.

**특징:**
1. **TF = 0:** 여유 시간이 없음 → 지연 즉시 프로젝트 지연
2. **가장 긴 경로:** 시작부터 종료까지의 경로 중 소요 시간이 가장 김
3. **집중 관리 대상:** 자원, 리스크, 일정 관리의 최우선 순위

**주공정 활동 지연 시 영향:**

1. **프로젝트 완료일 직접 영향:**
   - 주공정 활동이 1일 지연 → 프로젝트 1일 지연
   - 계약 위반, 위약금 발생 가능

2. **연쇄 지연:**
   - 후속 주공정 활동들도 연쇄적으로 지연
   - 전체 일정 재조정 필요

3. **자원 충돌:**
   - 지연 만회 위해 추가 자원 투입
   - 비용 증가
   - 다른 활동에서 자원 차출 → 2차 문제

4. **리스크 증폭:**
   - 압축 일정으로 만회 시도 → 품질 저하 리스크
   - 팀 사기 저하, 번아웃

**관리 전략:**
- 주공정 활동에 최우선 자원 배분
- 진척률 일일 모니터링
- 리스크 사전 식별 및 대응 계획
- 버퍼 확보 (우발 예비)

**문제 4:**
다음 상황에서 CPM을 재계산해야 하는가? 이유를 설명하시오.

**상황 A:** 비주공정 활동 C(TF=5일)가 2일 지연되었다.
**상황 B:** 주공정 활동 D가 3일 단축되었다.
**상황 C:** 새로운 요구사항으로 활동 X가 추가되었다.

**모범답안:**

**상황 A: 비주공정 활동 C(TF=5일)가 2일 지연**
- **재계산 필요: 선택적**
- 이유:
  - C의 남은 TF = 5 - 2 = 3일 (여전히 여유 있음)
  - 프로젝트 완료일에는 영향 없음
  - 하지만 **남은 TF 업데이트 필요** (리스크 관리)
  - C의 후속 활동들의 여유도 3일로 감소
- 권장: 간단히 TF 업데이트하고 모니터링 강화

**상황 B: 주공정 활동 D가 3일 단축**
- **재계산 필요: 필수 ✅**
- 이유:
  - 주공정 변경 가능성
  - 기존 비주공정이 새로운 주공정이 될 수 있음
  - 모든 활동의 TF 재계산 필요
  - 프로젝트 완료일 업데이트
- 예시:
  ```
  기존: 경로 1 (주공정, 20일), 경로 2 (비주공정, 18일, TF=2일)
  D 단축 후: 경로 1 (17일), 경로 2 (새 주공정, 18일, TF=0)
  → 주공정이 바뀜!
  ```

**상황 C: 새로운 활동 X 추가**
- **재계산 필요: 필수 ✅**
- 이유:
  - 네트워크 구조 변경
  - X가 주공정에 추가되면 프로젝트 기간 증가
  - X가 비주공정에 추가되어도 기존 TF 영향
  - 전체 CPM 재계산 필수

**일반 원칙:**
- 주공정 활동 변경: 항상 재계산
- 비주공정 활동 지연: TF 범위 내면 간단 업데이트, TF 초과 시 재계산
- 네트워크 구조 변경: 항상 재계산
- 정기 재계산: 주간/월간 단위 권장

## 5교시: 간트차트 및 일정 단축 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 간트차트 (Gantt Chart)

**정의:**
- 프로젝트 일정을 막대 그래프로 시각화한 도구
- 가로축: 시간 (일, 주, 월)
- 세로축: 활동 목록
- 막대 길이: 활동 기간

**구성 요소:**

1. **활동 막대:**
   - 시작일부터 완료일까지 수평 막대
   - 길이 = 기간
   - 색상으로 상태 구분 (계획/진행중/완료/지연)

2. **마일스톤:**
   - ◆ 다이아몬드 기호
   - 기간 = 0
   - 중요 시점 표시

3. **의존 관계:**
   - 화살표로 연결
   - FS, SS, FF, SF 관계 표시

4. **주공정:**
   - 다른 색상(빨강)으로 강조
   - 또는 굵은 선

5. **진척률:**
   - 막대 내부에 진행 상황 표시
   - 예: ■■■■□□ (50% 완료)

6. **자원 배정:**
   - 활동별 담당자/팀 표시
   - 자원 과부하 식별

7. **기준선 (Baseline):**
   - 원 계획 일정 (회색 막대)
   - 실제 일정과 비교

**장점:**
1. **직관적 이해:** 누구나 쉽게 이해
2. **진척 추적:** 계획 대비 실적 비교 용이
3. **의사소통:** 경영진, 고객 보고용 최적
4. **자원 관리:** 자원 배정 및 충돌 식별
5. **일정 조정:** What-if 시나리오 분석

**단점:**
1. **의존성 불명확:** 네트워크 다이어그램보다 관계 파악 어려움
2. **복잡도 한계:** 활동 수가 많으면 (100개+) 가독성 저하
3. **주공정 미표시:** 기본 간트차트는 주공정 자동 식별 안 함

**간트차트 vs 네트워크 다이어그램:**

| 특징 | 간트차트 | 네트워크 다이어그램 |
|------|---------|------------------|
| 시각화 | 막대 (시간 중심) | 노드와 화살표 (관계 중심) |
| 이해 용이성 | 매우 쉬움 | 기술적 |
| 의존 관계 | 화살표 (선택) | 명확함 |
| 주공정 식별 | 수동 강조 | CPM으로 자동 |
| 진척 추적 | 우수 | 미흡 |
| 보고용 | 최적 | 기술팀용 |

**실무 활용:**
- CPM으로 주공정 계산 → 간트차트로 시각화 → 보고/관리
- 도구: MS Project, Primavera, Smartsheet, Excel

#### 2. 일정 단축 (Schedule Compression)

**필요 상황:**
1. 고객이 완료일 앞당김 요청
2. 프로젝트 지연 발생 후 만회
3. 시장 출시 시점(Time-to-Market) 중요
4. 계약 인센티브 (조기 완료 시 보너스)

**주의사항:**
- 일정 단축 ≠ 무료
- Trade-off 존재: 비용 ↑, 리스크 ↑, 품질 ↓

**2.1 Crashing (일정 압축)**

**정의:**
- **비용을 투입**하여 활동 기간 단축
- 주로 **자원 추가 투입**
- "돈으로 시간을 사는 것"

**방법:**
1. **인력 추가:**
   - 개발자 2명 → 3명
   - 야근/주말 근무 (초과 수당)
   - 외부 용역 투입

2. **장비/도구 추가:**
   - 고성능 서버 추가
   - 자동화 도구 구매
   - 병렬 작업 환경 구축

3. **Fast-track 자재:**
   - 특급 배송 (추가 비용)
   - 프리미엄 재료 사용

**Crashing 대상 선정:**
✅ **주공정 활동만 단축**
- 비주공정 단축은 프로젝트 기간에 무의미
- 주공정 중에서도 비용 대비 효과가 큰 활동 선택

**비용 기울기 (Cost Slope):**
```
비용 기울기 = (Crash Cost - Normal Cost) / (Normal Duration - Crash Duration)
             = 추가 비용 / 단축 기간
```

**최적 Crashing 절차:**
1. 주공정 활동 식별
2. 각 활동의 비용 기울기 계산
3. 비용 기울기가 가장 낮은 활동부터 단축
4. 목표 기간 달성 또는 더 이상 단축 불가까지 반복
5. 주의: 단축 중 주공정 변경 가능 → 재계산

**예시:**

| 활동 | 정상 기간 | 정상 비용 | 단축 기간 | 단축 비용 | 비용 기울기 |
|------|----------|----------|----------|----------|------------|
| A | 10일 | 100만원 | 8일 | 120만원 | 10만원/일 |
| B | 15일 | 200만원 | 12일 | 230만원 | 10만원/일 |
| C | 8일 | 150만원 | 6일 | 190만원 | 20만원/일 |

- 주공정: A → B → C
- 프로젝트 기간: 33일
- 목표: 30일로 단축 (3일 필요)

**단축 전략:**
1. 1차: A를 2일 단축 (비용 기울기 낮음) → 31일, 추가 비용 20만원
2. 2차: B를 1일 단축 → 30일, 추가 비용 10만원
3. 총 추가 비용: 30만원

**장점:**
- 확실한 기간 단축
- 품질 유지 가능 (자원 추가)

**단점:**
- 비용 증가 (예산 초과 리스크)
- Brooks' Law: 인력 추가가 항상 효과적이지 않음
- 자원 과부하 리스크

**2.2 Fast Tracking (공정 중첩)**

**정의:**
- 원래 **순차적**으로 수행할 활동을 **병렬**로 수행
- 의존 관계 재검토 및 조정
- FS → SS 또는 FS with Lead

**방법:**
1. **임의적 의존성 제거:**
   - 강제적 의존성: 변경 불가
   - 임의적 의존성: 병렬화 검토

2. **병렬 작업:**
   - "설계 100% 완료 → 개발 시작" (FS)
   - → "설계 30% 완료 → 개발 시작" (SS + 래그)

3. **단계적 인도:**
   - 전체 완료 대기하지 않고 일부 완료 시 다음 단계 시작

**예시:**

**기존 일정 (FS 관계):**
```
설계 (10일) ─FS→ 개발 (15일) ─FS→ 테스트 (5일)
총 기간: 30일
```

**Fast Tracking 후 (SS 관계):**
```
설계 (10일)
   └─SS+3일→ 개발 (15일)
                └─SS+5일→ 테스트 (5일)
                
총 기간: 10 + 3 + 15 + 5 = 23일 (7일 단축)
```

**장점:**
- **추가 비용 없음** 또는 최소
- 상당한 기간 단축 가능

**단점 (리스크):**
1. **재작업 리스크 ↑:**
   - 설계 변경 시 이미 시작한 개발 수정 필요
   - 예: 설계 80% 시점에 개발 시작 → 설계 변경 → 개발 재작업

2. **품질 저하 리스크:**
   - 불완전한 정보로 작업 시작
   - 오류 발생 가능성

3. **커뮤니케이션 오버헤드:**
   - 설계팀과 개발팀 간 빈번한 동기화 필요
   - 회의 시간 증가

4. **팀 스트레스:**
   - 불확실한 상태로 작업
   - 잦은 변경으로 인한 피로도

**적용 가능 여부 판단:**

✅ **Fast Tracking 가능:**
- 임의적 의존성
- 재작업 비용 < 지연 비용
- 팀 간 커뮤니케이션 원활
- 병렬 작업 경험 있음

❌ **Fast Tracking 불가:**
- 강제적 의존성 (물리적, 법적)
- 재작업 리스크 매우 높음
- 팀 간 협업 어려움
- 안전/품질 크리티컬 분야

**2.3 Crashing vs Fast Tracking 비교**

| 구분 | Crashing | Fast Tracking |
|------|----------|---------------|
| 방법 | 자원 추가 투입 | 활동 병렬화 |
| 비용 | 증가 ↑ | 변동 없음 또는 소폭 증가 |
| 리스크 | 자원 과부하 | 재작업, 품질 저하 |
| 의존 관계 | 유지 | 변경 (FS→SS) |
| 품질 영향 | 낮음 | 높음 |
| 적용 대상 | 주공정 활동 | 임의적 의존성 |
| 단축 정도 | 제한적 | 상당함 (20-30%) |
| 관리 난이도 | 중간 | 높음 |

**실무 전략:**
1. 먼저 **Fast Tracking** 검토 (비용 ↓)
2. Fast Tracking 한계 도달 시 **Crashing** 적용
3. 두 기법 **조합** 사용 가능
4. 항상 **리스크 vs 이익** 분석

#### 3. 일정 단축 시 고려사항

**1. Triple Constraint 영향 분석**
```
일정 단축 (Schedule ↓)
    ↓
비용 증가 (Cost ↑) 또는 품질 저하 (Quality ↓)
```

**2. 고객 협의**
- 비용 증가 승인
- 품질 기준 재협의
- 범위 축소 가능성 검토

**3. 팀 역량 고려**
- 장시간 근무 지속 가능성
- 번아웃 리스크
- 사기 저하

**4. 리스크 관리**
- 일정 단축으로 인한 신규 리스크 식별
- 재작업 예비비 확보
- 품질 검증 절차 강화

**분석:**
- B 활동(주공정)이 지연 중 → 즉시 조치 필요
- 대응: 설계 인력 1명 추가 (Crashing)

**예시 2: Crashing 적용 - 소프트웨어 출시 앞당김**

**시나리오:**
경쟁사가 유사 제품을 3개월 후 출시 예정. 우리 제품을 먼저 출시해야 시장 선점 가능.

**현재 일정:**
- CPM 계산 결과: 120일
- 고객 요구: 90일로 단축 (30일 단축 필요)

**주공정 활동 및 Crashing 분석:**

| ID | 활동 | 정상 | Crash | 최대 단축 | 정상 비용 | Crash 비용 | 비용 기울기 |
|----|------|------|-------|----------|----------|-----------|------------|
| A | 요구분석 | 15일 | 12일 | 3일 | 500만원 | 560만원 | 20만원/일 |
| B | 개발 | 60일 | 45일 | 15일 | 3000만원 | 3600만원 | 40만원/일 |
| C | 테스트 | 30일 | 25일 | 5일 | 1000만원 | 1150만원 | 30만원/일 |
| D | 배포준비 | 15일 | 12일 | 3일 | 400만원 | 460만원 | 20만원/일 |

**Crashing 전략 (비용 최소화):**

1. **1단계: A 단축 3일** (비용 기울기 가장 낮음)
   - 기간: 120 → 117일
   - 추가 비용: 60만원

2. **2단계: D 단축 3일**
   - 기간: 117 → 114일
   - 추가 비용: 60만원

3. **3단계: C 단축 5일**
   - 기간: 114 → 109일
   - 추가 비용: 150만원

4. **4단계: B 단축 15일**
   - 기간: 109 → 94일
   - 추가 비용: 600만원

5. **5단계: 추가 4일 필요 → B에서 4일 더 단축 불가**
   - **Fast Tracking 검토 필요**

**Fast Tracking 추가 적용:**
- "테스트"를 "개발" 완료 전 시작 (FS → SS + 10일 래그)
- 리스크: 개발 변경 시 테스트 재수행
- 효과: 4일 추가 단축

**최종 결과:**
- 목표 달성: 90일 ✅
- 총 추가 비용: 870만원
- 리스크: 테스트 재작업 가능성

**경영진 보고:**
- "30일 단축 가능하나 870만원 추가 비용과 재작업 리스크 존재"
- "시장 선점 효과(예상 매출 5억) vs 추가 비용(870만원)" 비교
- 승인 획득

**예시 3: Fast Tracking 실패 사례 - 건설 프로젝트**

**프로젝트:** 20층 오피스텔 건설

**원래 일정:**
```
설계 완료 (2개월) ─FS→ 인허가 (1개월) ─FS→ 기초공사 (3개월)
```

**PM의 Fast Tracking 시도:**
"설계 완료 전에 인허가 신청 시작하면 1개월 단축 가능!"

```
설계 (2개월)
  └─SS+1개월→ 인허가 신청 (1개월)
```

**결과:**
- 설계 진행 중(60%) 인허가 신청 시작
- 설계 변경 발생 (80% 시점)
- 이미 제출한 인허가 서류 전면 수정 필요
- 인허가 재심사 → 3주 추가 소요
- **총 기간: 오히려 3주 증가**
- **추가 비용: 인허가 재신청 비용 500만원**

**교훈:**
- 강제적 의존성(법적)은 Fast Tracking 불가
- 설계 확정 전 인허가는 리스크 매우 높음
- Fast Tracking 전 의존성 유형 정확히 판단 필요

**올바른 접근:**
- 설계는 빠른 의사결정으로 기간 단축 (Crashing)
- 인허가는 담당 공무원과 사전 협의로 처리 기간 단축
- 기초공사 준비를 인허가와 병렬 진행 (자재 선발주)

**문제 2:**
Crashing과 Fast Tracking의 차이를 5가지 측면에서 비교하고, 각 기법을 적용하기 적합한 상황을 2가지씩 설명하시오.

**모범답안:**

**비교:**

| 측면 | Crashing | Fast Tracking |
|------|----------|---------------|
| **방법** | 자원 추가 투입 (인력, 장비, 야근) | 활동 병렬화 (FS→SS) |
| **비용** | 증가 ↑ (추가 인건비, 장비비) | 변동 없음 또는 최소 |
| **리스크** | 자원 과부하, Brooks' Law | 재작업, 품질 저하 |
| **의존 관계** | 유지 | 변경 (임의적 의존성) |
| **적용 대상** | 주공정 활동 (비주공정 단축 무의미) | 임의적 의존성이 있는 활동 |

**Crashing 적합 상황:**
1. **예산 여유, 시간 부족:** 추가 비용 투입 가능하고 확실한 단축 필요
2. **품질 Critical:** 품질을 유지하면서 기간 단축 필요 (자원 추가로 품질 유지)

**Fast Tracking 적합 상황:**
1. **예산 제약:** 추가 비용 투입 불가, 일정은 단축 필요
2. **임의적 의존성 多:** 모범 사례에 의한 순차 진행이지만 병렬화 가능한 경우

**부적합한 경우:**
- Crashing: 예산 한계, 자원 확보 불가, 병목 작업 (인력 추가 무의미)
- Fast Tracking: 강제적 의존성, 재작업 비용 > 지연 비용, 안전/품질 Critical

**문제 3:**
다음 상황에서 Crashing 계획을 수립하시오.

**주공정 활동:**

- B: 15일, 정상 비용 300만원, 최대 5일 단축 가능, 단축 비용 400만원

**질문:**

2. 비용을 최소화하는 Crashing 계획을 제시하시오.
3. 총 추가 비용은?

**모범답안:**

**1. 비용 기울기 계산:**

활동 A:
```

```

활동 B:
```
비용 기울기 = (400 - 300) / 5 = 100 / 5 = 20만원/일
```

활동 C: 단축 불가

**2. Crashing 계획:**

A와 B의 비용 기울기가 동일하므로, 둘 다 선택 가능. 효율을 위해:

**전략:**
- A를 3일 단축 (최대치)
- B를 2일 단축
- 총 5일 단축 달성 ✅

**또는:**
- B를 5일 단축 (최대치)
- 총 5일 단축 달성 ✅

**전략 1:**

- **총: 100만원**

**전략 2:**
- B 단축 비용: 20만원/일 × 5일 = 100만원
- **총: 100만원**

**결론:** 두 전략 모두 100만원으로 동일. 실무에서는:
- 전략 1 선택 (리스크 분산, 두 활동에 분산)
- 또는 B 활동이 단축하기 더 쉬우면 전략 2

**문제 4:**
Fast Tracking을 적용하면 안 되는 상황 3가지를 쓰고, 각 상황에서 Fast Tracking이 위험한 이유를 설명하시오.

**모범답안:**

**1. 강제적 의존성 (Mandatory Dependency)**
- 상황: "기초공사 → 골조공사" (건설), "설계 인허가 → 시공" (법적)
- 위험: 물리적/법적으로 불가능. 강행 시 안전사고, 법규 위반, 재작업
- 예: 기초 없이 골조 시공 → 건물 붕괴

**2. 재작업 비용 > 지연 비용**
- 상황: "상세 설계 미완료 상태로 제조 착수" (제조업)
- 위험: 설계 변경 시 이미 제작한 부품 폐기, 재제작 비용이 지연 비용보다 훨씬 큼
- 예: 금형 제작 후 설계 변경 → 금형 재제작 수천만원

**3. 안전/품질 Critical 분야**
- 상황: 의료기기, 항공, 원자력 등 안전이 최우선인 프로젝트
- 위험: 설계 검증 미완료 상태로 제조 → 안전 결함, 인명 사고
- 예: 의료기기 설계 검증 전 양산 → 결함 발견 → 리콜, 소송

**추가로 부적합한 경우:**
- 팀 간 협업 어려움 (지리적 분산, 조직 문화)
- 불확실성이 매우 높은 R&D 프로젝트 (설계 변경 빈번)
- 경험 부족 팀 (병렬 작업 코디네이션 역량 부족)

**결론:**
Fast Tracking은 "비용 없이 기간 단축"이라는 장점이 크지만, 적용 가능 여부를 신중히 판단해야 합니다. 의존성 유형, 재작업 리스크, 안전/품질 요구사항을 종합적으로 고려하여 결정해야 합니다.

**문제 3:**
프로젝트 일정 관리에서 범하기 쉬운 오류 5가지를 쓰고, 각각의 해결 방법을 제시하시오.

**모범답안:**

**1. 학생 증후군 (Student Syndrome)**
- 오류: 활동 시작을 최대한 늦춤 ("마감 직전에 시작해도 되지")
- 결과: 버퍼 소진, 문제 발생 시 대응 불가
- 해결: 
  - 조기 시작 문화 조성
  - 마일스톤을 짧게 (2주 이내)
  - 진척률 일일 추적

**2. 파킨슨의 법칙 (Parkinson's Law)**
- 오류: "일은 주어진 시간만큼 늘어난다" → 8일 작업을 10일 배정하면 10일 소요
- 결과: 불필요한 일정 증가, 비효율
- 해결:
  - 현실적이지만 타이트한 일정 설정
  - 완료 기준 명확히 (DoD - Definition of Done)
  - 조기 완료 시 보상

**3. 비주공정 무시**
- 오류: "TF가 있으니 나중에"라는 안일함
- 결과: 비주공정이 주공정으로 변함, 갑작스러운 위기
- 해결:
  - 비주공정도 정기 모니터링
  - 남은 TF 추적
  - TF 50% 소진 시 경고

**4. 리소스 과부하 간과**
- 오류: CPM상 병렬 가능하지만 자원이 부족하여 실제 불가능
- 예: 개발자 1명이 2개 활동에 동시 배정
- 결과: 일정 지연, 자원 갈등
- 해결:
  - 자원 평준화 (Resource Leveling) 수행
  - 자원 히스토그램으로 과부하 구간 식별
  - 자원 배정 현실성 검토

**5. 진척률 낙관적 보고**
- 오류: "90% 완료"가 3주째 지속 (마지막 10%가 실제 50%)
- 결과: 실제 진척 파악 불가, 갑작스러운 일정 위기
- 해결:
  - 객관적 진척 측정 기준 (Earned Value)
  - 0/100% 규칙 (완료 또는 미완료만)
  - 코드 리뷰, 테스트 통과 등 객관적 증거
  - "90% 완료"는 의미 없음 인식 공유

**추가 오류:**
- **버퍼 미확보:** 우발 예비 없이 일정 수립 → 첫 문제에 위기
- **의존성 오판:** 강제적 의존성을 임의적으로 착각 → Fast Tracking 실패
- **CPM 1회 계산 후 방치:** 상황 변화 시 재계산 안 함 → 주공정 변경 놓침
