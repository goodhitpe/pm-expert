## 1교시: 의사소통 관리 개요 (1.5시간)

### 이론 (50분)

#### 1. 의사소통의 중요성

**PM의 시간 배분:**
- 연구 결과: PM은 업무 시간의 **약 90%**를 의사소통에 사용
- 나머지 10%: 기획, 분석, 의사결정

**의사소통 실패의 영향:**
- 프로젝트 실패 원인의 **70% 이상**이 의사소통 문제
- 요구사항 오해, 기대치 불일치, 갈등 증폭
- 재작업, 일정 지연, 비용 초과

#### 2. 의사소통 관리의 3개 프로세스

**1) 의사소통 관리 계획 수립 (Plan Communications Management)**
- 프로세스 그룹: 기획
- 이해관계자의 정보 요구사항 결정
- 의사소통 접근 방법 정의
- 산출물: 의사소통 관리 계획서

**2) 의사소통 관리 (Manage Communications)**
- 프로세스 그룹: 실행
- 계획에 따라 정보 수집, 생성, 배포, 저장, 검색, 관리, 통제, 모니터링, 최종 처리
- 산출물: 프로젝트 의사소통

**3) 의사소통 감시 (Monitor Communications)**
- 프로세스 그룹: 감시 및 통제
- 의사소통이 이해관계자의 정보 요구를 충족하는지 확인
- 산출물: 작업 성과 정보, 변경 요청

#### 3. 의사소통 채널 수 공식

**공식:** 채널 수 = n(n-1)/2  
(n = 이해관계자 수)

**예시:**
```
3명: 3(3-1)/2 = 3개
5명: 5(5-1)/2 = 10개
10명: 10(10-1)/2 = 45개
15명: 15(15-1)/2 = 105개
20명: 20(20-1)/2 = 190개
```

**중요성:**
- 1명 추가 시 채널이 (n-1)개 증가
- 예: 10명 → 11명: +10개 채널 (45 → 55)
- 의사소통 복잡도가 기하급수적으로 증가
- PM은 의사소통 채널을 최소화해야 함

**IT 프로젝트 예시:**
```
초기 팀: PM 1명 + 개발자 4명 = 5명
→ 10개 채널 (관리 가능)

확대 팀: PM 1명 + 개발 5명 + QA 3명 + 디자이너 2명 + 현업 5명 = 16명
→ 120개 채널 (복잡함)

대응:
- 계층 구조 도입 (팀장 통해 의사소통)
- 정기 회의로 채널 통합
- 의사소통 도구 표준화
```

#### 4. 의사소통의 5C

**1) Correct (정확)**
- 사실에 기반한 정보
- 검증된 데이터

**2) Concise (간결)**
- 핵심만 전달
- 불필요한 내용 제거

**3) Clear (명확)**
- 오해의 여지 없음
- 구체적인 표현

**4) Coherent (일관)**
- 논리적 흐름
- 일관된 메시지

**5) Courteous (정중)**
- 존중하는 태도
- 긍정적 표현

### 예시 (25분)

#### 예시 1: 의사소통 채널 수 폭발적 증가

**프로젝트:** 금융 플랫폼 구축 프로젝트

**팀 규모 변화:**

```
1차 킥오프: 5명
채널 수 = 5×4/2 = 10개

2차 확대 (현업 5명 추가): 10명
채널 수 = 10×9/2 = 45개 (4.5배 증가!)

3차 확대 (협력사 5명 추가): 15명
채널 수 = 15×14/2 = 105개 (10.5배 증가!)
```

**문제 발생:**
- 회의가 하루에 6-7건
- 동일한 정보가 여러 채널에서 따로따로 전달됨
- 정보 일관성 불일치 → 의사결정 혼란

**해결책:**
```
3계층 의사소통 구조 도입:
Level 1: 스폰서 (월간 보고)
Level 2: PM + 팀장 계층 (주간 회의)
Level 3: 팀원 계층 (일일 스탠드업)

결과: 실질적 채널 수 = 20개로 관리
```

#### 예시 2: 의사소통 실패 사례

**시나리오:** 공공기관 시스템 전환 프로젝트

**문제:**
- PM이 범위 변경을 이메일로만 공지 (Push)
- 일부 팀원이 이메일 미확인
- 변경된 범위를 모른 채 개발 진행

**영향:**
- 기존 방식대로 개발한 모듈이 범위 밖으로 판명
- 2주치 개발 재작업
- 납기 지연 3주

**교훈:**
```
중요 변경 공지 시 올바른 방법:
1. 공식 회의에서 Interactive 방식으로 공지
2. Email Push로 서면 기록
3. 프로젝트 공유 드라이브 Pull로 문서 업데이트
4. 수신 확인 요청 (Read receipt)
5. 다음 회의에서 이해 여부 재확인
```

### 퀴즈 (15분)

**Q1. 팀원이 5명에서 10명으로 증가했을 때 의사소통 채널 수의 변화는?**

A) 5개 증가 (10→15)  
B) 35개 증가 (10→45)  
C) 25개 증가 (10→35)  
D) 2배 증가 (10→20)

**정답: B**

A) 프로젝트 위키 페이지 업로드

## 2교시: 의사소통 모델 및 방법 (1.5시간)

### 이론 (50분)

#### 1. 의사소통 모델 (Communication Model)

**기본 모델:**

```
발신자(Sender)
    ↓ 부호화(Encode) - 메시지를 언어/기호로 변환
메시지(Message)
    ↓ 매체(Medium/Channel) - 전달 경로
    ↓ 해독(Decode) - 수신자가 해석
수신자(Receiver)
    ↓ 피드백(Feedback) - 이해 여부 확인
```

**각 단계 상세:**

**발신자 역할:**
- 명확한 메시지 구성
- 적절한 부호화 선택 (언어, 그림, 수치)
- 수신자 수준에 맞게 조정

**부호화 주의사항:**
- 용어 수준 (전문용어 vs 일반 언어)
- 문화적 맥락 고려
- 메시지 구조 (결론-근거-행동 순서)

**매체 선택:**
- 긴급도, 민감도, 거리에 따라 결정
- 동기식(Synchronous): 실시간 (전화, 회의)
- 비동기식(Asynchronous): 시간 불일치 허용 (이메일, 게시판)

**잡음(Noise): 의사소통 방해 요인**

| 유형 | 예시 |
|------|------|
| 물리적 잡음 | 시끄러운 환경, 나쁜 연결 상태 |
| 심리적 잡음 | 편견, 선입견, 감정 |
| 언어적 잡음 | 전문 용어, 언어 장벽, 모호한 표현 |
| 문화적 잡음 | 다른 문화적 맥락, 관습 차이 |
| 기술적 잡음 | 시스템 오류, 파일 형식 불일치 |

**피드백의 중요성:**
- 발신자가 제대로 전달됐는지 확인하는 유일한 방법
- "알겠습니다"가 "동의합니다"와 다를 수 있음
- 확인 질문으로 진짜 이해 여부 파악

#### 2. 의사소통 유형

**A. 공식적 vs 비공식적**

| 유형 | 예시 | 장점 | 단점 |
|------|------|------|------|
| 공식 문서 | 보고서, 공문, 의사록 | 기록 유지, 법적 효력 | 시간 소요, 유연성 낮음 |
| 공식 구두 | 프레젠테이션, 공식회의 | 설득력, 즉각 피드백 | 기록 없음 |
| 비공식 문서 | 이메일, 메모 | 빠름, 편리 | 공식 효력 약함 |
| 비공식 구두 | 복도 대화, 메신저 | 즉각적, 자연스러움 | 기록 없음, 오해 가능 |

**PM에게 중요한 점:**
- 중요한 결정은 반드시 공식 문서로 기록
- "구두로만 들었습니다"는 분쟁 시 인정 안 됨
- 비공식 대화에서 합의 시 → 즉시 공식 문서로 확인

**B. 수직적 vs 수평적 vs 대각선**

- **수직적(Vertical):** 상하 계층 간 (PM↔스폰서, PM↔팀원)
- **수평적(Horizontal):** 같은 계층 간 (PM↔다른 PM, 팀원↔팀원)
- **대각선(Diagonal):** 다른 계층의 다른 부서 간

**C. 현재 vs 과거 vs 예측 트렌드**

- **현재 상태:** 지금 어떠한가? (상태 보고)
- **과거 실적:** 지금까지 무엇을 했나? (진행 보고)
- **미래 예측:** 향후 어떻게 될 것인가? (예측 보고)

#### 3. 의사소통 방법 상세

**Interactive (상호작용)**
- 양방향, 실시간, 즉각 피드백
- 예: 회의, 전화, 화상회의, 1:1 면담
- **사용 시점:**
  - 복잡하거나 민감한 이슈
  - 즉각적 의사결정 필요
  - 갈등 해결
  - 요구사항 수집

**Push (밀어내기)**
- 단방향, 발신자 주도
- 예: 이메일, 팩스, 공문, SMS, 보고서 배포
- **사용 시점:**
  - 다수에게 표준 정보 배포
  - 공식 통보 (기록 필요)
  - 수신자가 언제 읽을지 제어 불필요
- **주의:** 읽힘을 보장하지 못함

**Pull (끌어당기기)**
- 수신자가 필요할 때 검색
- 예: 인트라넷, 공유 드라이브, Wiki, 지식 저장소
- **사용 시점:**
  - 참고 자료, 절차서, 표준
  - 대용량 정보
  - 수신자 선택에 맡길 때

#### 4. 비언어적 의사소통

**55-38-7 법칙 (메라비안의 법칙):**
- **55%:** 바디 랭귀지 (표정, 자세, 제스처)
- **38%:** 목소리 (톤, 속도, 강세, 음량)
- **7%:** 말의 내용 (실제 단어)

**IT 프로젝트 적용:**
- 팀원이 말은 "괜찮습니다"라고 해도, 자세와 톤이 신경 쓰인다면 → 추가 확인 필요
- 화상회의에서는 표정을 더 적극적으로 활용
- 원격 팀원: 목소리 톤에 더 집중

**파라 언어(Paralingual)의 중요성:**
- "우리가 일정을 맞출 수 있다" (자신있는 톤) vs (불확실한 톤)
- 같은 말이지만 완전히 다른 의미 전달

### 예시 (25분)

#### 예시 1: 의사소통 모델 실패 케이스

**시나리오:** 인프라 팀이 서버 마이그레이션 일정을 공유

```
1단계 발신자: 인프라팀장 (발송)
"다음 주 토요일 02:00~06:00 서버 점검이 있습니다."
→ 이메일 발송

2단계 잡음: 
- 개발팀은 "다음 주 토요일"을 6일 후로 이해
- 인프라팀은 이번 주 주말(2일 후)을 의미

3단계 결과:
개발팀이 토요일 새벽에 긴급 배포 예약 (서버 점검 모름)
→ 배포 실패, 긴급 대응 발생

원인: 모호한 날짜 표현 ("다음 주" vs 구체적 날짜)
     + 피드백 미확인

교훈:
- 날짜는 "2월 22일(토)" 형태로 명확히 기재
- 중요 일정은 수신 확인 요청
- 영향받는 팀에게 Interactive 방식으로 사전 고지
```

#### 예시 2: 의사소통 방법 최적 선택

**프로젝트:** ERP 도입, 스테이크홀더 15명

**상황별 최적 방법:**

| 상황 | 방법 | 이유 |
|------|------|------|
| 범위 변경 승인 요청 | Interactive (회의) | 의사결정 필요 |
| 월간 진행 보고서 | Push (이메일) | 공식 기록 + 다수 수신 |
| 프로젝트 표준 문서 | Pull (Wiki) | 참고 자료, 최신 버전 유지 |
| 팀원 성과 피드백 | Interactive (1:1) | 민감한 내용, 즉각 질의응답 |
| 긴급 파일 수정 요청 | Push (메신저) | 빠른 전달 필요 |
| 고객 클레임 대응 | Interactive (전화) | 즉각 대응, 신뢰 구축 |

### 실습 (20분)

#### 실습 1: 의사소통 방법 결정 워크숍

**과제:** 다음 상황에 적합한 의사소통 방법과 이유를 작성하세요.

1. 프로젝트 요구사항 변경 (첫 번째 논의)
2. 최종 확정된 요구사항 공유

## 3교시: 의사소통 관리 계획 수립 (1.5시간)

### 이론 (50분)

#### 1. 의사소통 관리 계획서 (Communications Management Plan)

**목적:**
- 이해관계자 각자가 필요한 정보를 올바른 시점에 올바른 방법으로 받을 수 있도록 보장

**구성 요소:**

**A. 이해관계자 정보 요구사항 분석**

| 이해관계자 | 필요한 정보 | 이유 |
|-----------|-----------|------|
| 스폰서 | 일정/비용 현황, 주요 리스크 | 의사결정 권한자 |
| 현업 담당자 | 기능 구현 현황, 테스트 일정 | 업무 영향 사전 파악 |
| 팀원 | 상세 업무 지시, 피드백 | 실행 책임자 |
| IT 운영팀 | 기술 사양, 배포 일정 | 운영 준비 |
| 외부 감사 | 준수사항 현황, 문서 | 규정 준수 확인 |

**B. 의사소통 매트릭스 (Communication Matrix)**

| 정보 유형 | 수신자 | 발신자 | 방법 | 주기 | 형식 |
|---------|--------|--------|------|------|------|
| 프로젝트 상태 보고 | 스폰서, 임원 | PM | 이메일+회의 | 월간 | PDF 보고서 |
| 주간 진행 현황 | PM, 팀장 | 팀원 | 스탠드업 | 매일 | 구두 |
| 이슈/리스크 현황 | 스폰서 | PM | 이메일 | 즉시 | 이슈 리스트 |
| 릴리즈 공지 | 전 이해관계자 | PM | 이메일 | 배포 시 | 공고문 |
| 팀 내부 소통 | 팀원 | 팀원 | Slack | 수시 | 자유 형식 |

**C. 에스컬레이션 절차 (Escalation Procedure)**

**목적:** 이슈가 PM 수준에서 해결되지 않을 때 상위 관리자에게 보고하는 절차

**에스컬레이션 기준:**

| 이슈 등급 | 기준 | 에스컬레이션 대상 | 기한 |
|---------|------|---------------|------|
| Low | 일정 1주 이내 지연 | PM 자체 해결 | 3일 내 |
| Medium | 일정 2-4주 지연, 비용 5-10% 초과 | 담당 임원 | 48시간 내 |
| High | 일정 1개월+ 지연, 비용 10%+ 초과 | 프로젝트 오너 | 24시간 내 |
| Critical | 프로젝트 중단 위기 | 최고 의사결정권자 | 즉시 |

**에스컬레이션 5단계 프로세스:**
1. 이슈 정확한 문서화 (what, when, who, impact)
2. 직속 상위자에게 보고
3. 해결이 안 될 경우 다음 레벨
4. 해결책 이행
5. 이슈 종결 및 기록

**D. 의사소통 기술 선정**

**선택 기준:**
- **긴급성:** 즉각 처리 vs 지연 허용
- **기밀성:** 암호화 필요 여부
- **수신자 기술 능력:** 도구 사용 가능 여부
- **팀 분산도:** 지역 vs 원격
- **비용:** 도구 라이센스

**IT 프로젝트 도구 스택 예시:**

| 용도 | 도구 | 방법 |
|------|------|------|
| 팀 채팅 | Slack, Teams | Interactive/Push |
| 영상 회의 | Zoom, Teams | Interactive |
| 문서 협업 | Confluence, Notion | Pull |
| 이슈 관리 | Jira | Pull |
| 이메일 | Outlook, Gmail | Push |
| 파일 공유 | SharePoint, Drive | Pull |

#### 2. 의사소통 장벽과 극복 방법

**장벽 유형:**

**A. 물리적 장벽**
- 지리적 분산, 시간대 차이
- 극복: 비동기 방식 준비, 협업 도구

**B. 언어적 장벽**
- 전문 용어, 외국어, 표현 차이
- 극복: 용어 사전, 평이한 언어, 통역

**C. 문화적 장벽**
- 고맥락(High-context) vs 저맥락(Low-context) 문화
- 직접적 vs 간접적 의사소통 스타일
- 극복: 문화 인식 교육, 명확한 확인 과정

**D. 심리적 장벽**
- 권위에 대한 두려움, 실패 보고 꺼림
- 극복: 심리적 안전감(Psychological Safety) 조성

**PM이 해야 할 것:**
- "나쁜 소식을 빨리 가져오면 좋은 것"이라는 문화 구축
- 불이익 없는 문제 제기 보장

## 4교시: 의사소통 관리 실행 (1.5시간)

### 이론 (50분)

#### 1. 능동적 경청 (Active Listening)

**정의:** 상대방의 말을 판단 없이 완전히 이해하려는 의도적 노력

**일반적 경청 vs 능동적 경청:**

| 구분 | 일반 경청 | 능동적 경청 |
|------|---------|------------|
| 목적 | 대답 준비 | 이해 |
| 집중 | 내 생각 | 상대방 |
| 반응 | 즉각 반박 | 확인 질문 |
| 결과 | 오해 발생 | 명확한 이해 |
| 비언어 | 무시 | 주목 |

**능동적 경청 기술:**

**1) 주의 집중 (Attending)**
- 눈 맞춤, 메모
- 방해 요소 제거 (폰 내려놓기)
- 열린 자세 (팔 꼬지 않기)

**2) 반영 (Reflecting)**
- 상대방 말을 자신의 말로 바꿔 확인
- "제가 이해하기로는... 맞나요?"
- 오해를 조기에 발견

**3) 명확화 질문 (Clarifying)**
- "좀 더 구체적으로 말씀해 주시겠어요?"
- "그게 무엇을 의미하나요?"
- "예를 들어주실 수 있나요?"

**4) 요약 (Summarizing)**
- 중요 포인트를 정리하여 재확인
- "지금까지 논의된 내용을 요약하면..."
- 긴 논의 후 합의 사항 명확화

**5) 판단 보류 (Suspending Judgment)**
- 선입견 제거
- 말이 끝날 때까지 반박 준비 안 하기
- 감정적 반응 조절

#### 2. 효과적 회의 운영

**회의의 3단계:**

**A. 회의 전 준비**

```
체크리스트:
✓ 회의 목적 명확화 (정보 공유? 의사결정? 문제 해결?)
✓ 안건(Agenda) 사전 배포 (최소 24시간 전)
✓ 참석자 선정 (꼭 필요한 사람만)
✓ 필요 자료 사전 배포
✓ 예상 소요 시간 배분
✓ 장소/도구 준비 (회의실, Zoom 링크)
```

**안건 형식:**
```
회의 안건 예시:
---
회의명: 스프린트 3 계획 회의
일시: 2026.02.10 (화) 10:00-11:30
장소: 3층 회의실 A
참석: 개발팀 7명

안건:
1. 스프린트 2 완료 사항 확인 (10분)
2. 스프린트 3 우선순위 결정 (30분) - 의사결정 필요
3. 기술 이슈 공유 (20분)
4. 다음 스프린트 일정 확인 (10분)

준비물: Jira 백로그 업데이트, 기술 이슈 리스트
---
```

**B. 회의 중 운영**

**촉진자(Facilitator) 역할:**
- 안건 진행 관리 (타임키핑)
- 모든 참석자 참여 유도
- 주제 이탈 방지 ("주차장(Parking Lot)" 활용)
- 갈등 관리

**주차장(Parking Lot) 기법:**
- 회의 주제와 관련 없는 안건을 별도 기록
- 회의 후 별도 처리
- "좋은 의견인데, 오늘 안건이 아니므로 주차장에 기록해두겠습니다"

**결정 기록:**
- 모든 결정 사항 즉시 기록
- "결정: ..., 담당: ..., 기한: ..."

**C. 회의 후 관리**

```
체크리스트:
✓ 회의록(Minutes) 24시간 내 발송
✓ 결정 사항 명확히 기록
✓ 액션아이템 (Action Item) 명시
  - 무엇을: 구체적 행동
  - 누가: 담당자 (1인)
  - 언제: 기한 명시
✓ 다음 회의 일정 확인
✓ 액션아이템 진행 상황 추적
```

**회의록 템플릿:**
```
회의록
========================
일시: 2026.02.10 10:00-11:25
참석자: 김철수(PM), 이민우(개발팀장), 최지연(QA), 박영희(SA)
기록: 김철수

논의 사항:
1. [안건 1] API 성능 이슈
   - 현황: 응답 시간 2.5초 (목표: 1.5초)
   - 원인: DB 쿼리 최적화 미완
   - 결정: 이번 스프린트에서 최우선 처리

액션아이템:
| # | 내용 | 담당 | 기한 |
|---|------|------|------|
| 1 | DB 쿼리 분석 및 최적화 | 이민우 | 02.14 |
| 2 | 성능 테스트 재수행 | 최지연 | 02.17 |
| 3 | 결과 보고 | 이민우 | 02.18 |

다음 회의: 02.17 (화) 10:00 (성능 테스트 결과 검토)
========================
```

#### 3. 효과적 발표와 보고

**이해관계자 수준별 커뮤니케이션:**

| 수준 | 관심사 | 보고 방식 |
|------|--------|---------|
| 임원/스폰서 | ROI, 리스크, 결론 | 한 페이지 요약, 수치 중심 |
| 중간 관리자 | 팀별 현황, 이슈 | 상세 상태 보고서 |
| 팀원 | 업무 지시, 기술 사항 | 상세 기술 문서, 일정 |
| 외부 이해관계자 | 계약 이행, 납기 | 공식 보고서 |

**Pyramid Principle (피라미드 원칙):**
- 결론을 먼저 (Bottom-Line Up Front)
- 핵심 요점 3가지
- 각 요점에 대한 근거 제시
- 임원 보고에 특히 효과적

### 예시 (25분)

#### 예시 1: 비효율적 vs 효율적 회의

**비효율적 회의:**

#### 2. 보고서 유형

**A. 상태 보고서 (Status Report)**
- **목적:** 현재 시점의 프로젝트 상태 스냅샷
- **내용:** 일정, 비용, 품질, 리스크 현황

```
  - 대응: 팀원 추가 투입 검토

▷ 비용: 🟢 정상 (CPI 1.05, 5% 절감)

▷ 품질: 🟢 정상 (결함 0.3/FP)

▷ 리스크: 🟡 주의 (2건 High Risk 모니터링 중)

▷ 이번 주 달성:

  - API 연동 업체 일정 협의 스폰서 지원 필요
```

**B. 진행 보고서 (Progress Report)**
- **목적:** 기간 동안의 성과
- **내용:** 완료된 작업, 진행 중 작업, 다음 계획
- **비교 기준:** 이전 주기 대비

**C. 예측 보고서 (Forecasting Report)**
- **목적:** 향후 성과 예측
- **내용:** EAC, ETC, 완료 예상 날짜

**D. 편차 보고서 (Variance Report)**
- **목적:** 계획 대비 실적 차이 분석
- **내용:** SV, CV, SPI, CPI, 원인 분석
- **활용:** 근본 원인 파악

**E. 추세 보고서 (Trend Report)**
- **목적:** 시간 경과에 따른 패턴
- **내용:** 주간/월간 SPI, CPI 변화
- **활용:** 악화 추세 조기 감지

#### 3. RAG 보고 (Red-Amber-Green)

**신호등 방식 상태 표시:**

| 색상 | 의미 | 조치 |
|------|------|------|
| 🟢 Green | 계획대로 진행 | 모니터링 유지 |
| 🟡 Amber | 주의 필요, 작은 이슈 | 대응 계획 수립 |
| 🔴 Red | 즉각 조치 필요 | 에스컬레이션 고려 |

**RAG 기준 예시:**

| 지표 | Green | Amber | Red |
|------|-------|-------|-----|
| SPI | ≥ 0.95 | 0.85~0.95 | < 0.85 |
| CPI | ≥ 0.95 | 0.85~0.95 | < 0.85 |
| 품질(결함 밀도) | < 0.3 | 0.3~0.5 | > 0.5 |
| 리스크 | Low | Medium | High/Critical |

**RAG의 장점:**
- 임원이 한눈에 파악 가능
- 세부 지표 없이도 빠른 의사결정
- 추세 파악 용이 (이전 주와 비교)

#### 4. 대시보드와 시각적 보고

**번다운 차트 (Burndown Chart):**
- Agile 프로젝트에서 남은 작업량 시각화
- 실제 vs 이상적 번다운 비교
- 일정 지연을 직관적으로 파악

**번업 차트 (Burnup Chart):**
- 완료된 작업량 누적 시각화
- 범위 변경 영향도 파악 가능

**EVM 시각화:**
- S-Curve: PV, EV, AC 비교
- 현재 위치와 미래 예측 동시 표시

**Q3.** Push와 Pull 의사소통 방법의 차이를 예시를 들어 설명하시오. (10점)

**정답:** Push는 발신자가 수신자에게 직접 배포(이메일 발송, 보고서 배부), Pull은 수신자가 필요할 때 직접 검색(공유 드라이브, 인트라넷). Push는 공식 통보에, Pull은 참조 자료 배포에 적합합니다.
