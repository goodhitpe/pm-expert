# Day 15: 데이터베이스 - 상세 강의안

## 1교시: 데이터베이스 개요 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 데이터베이스와 DBMS

**데이터베이스(DB):** 구조화된 데이터의 집합  
**DBMS:** DB를 관리·조작하는 소프트웨어 (MySQL, Oracle, PostgreSQL)

**특성:**
- 데이터 독립성: 앱과 DB 분리
- 무결성: 유효한 데이터만 저장
- 보안: 접근 권한 제어
- 동시성: 여러 사용자 동시 접근

#### 2. RDBMS vs NoSQL

| 구분 | RDBMS | NoSQL |
|------|-------|-------|
| 모델 | 테이블 (관계형) | Document/Key-Value/Graph |
| 스키마 | 고정적 | 유연함 |
| 확장 | 수직 확장 (Scale-Up) | 수평 확장 (Scale-Out) |
| ACID | 강력 | 일부만 지원 |
| 사례 | 금융, 재고 | 로그, 실시간 분석 |

#### 3. PM의 DB 관점

- 데이터 마이그레이션 계획: 기존 DB → 신규 DB
- 성능 요구사항: TPS, 응답시간
- 백업/복구 전략: RPO/RTO
- 컴플라이언스: 개인정보보호

#### 4. 데이터 모델

**계층형:** 트리 구조 (1970년대)  
**관계형:** 테이블과 관계 (현재 주류)  
**NoSQL:** Document, Key-Value, Graph

## 2교시: RDBMS 핵심 개념 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 테이블 구조

**테이블:** 데이터를 저장하는 2차원 구조  
**행(Row/Tuple):** 하나의 레코드  
**열(Column/Attribute):** 속성

**예: 고객 테이블**
| 고객ID (PK) | 이름 | 이메일 | 가입일 |
|------------|------|--------|--------|
| 1 | 김철수 | kim@example.com | 2024-01-10 |
| 2 | 이영희 | lee@example.com | 2024-01-15 |

#### 2. 키(Key) 개념

**PK (Primary Key):** 행을 유일하게 식별  
- NOT NULL, UNIQUE
- 예: 고객ID, 주문ID

**FK (Foreign Key):** 다른 테이블의 PK 참조  
- 참조 무결성 보장
- 예: 주문 테이블의 고객ID → 고객 테이블의 고객ID

**복합키:** 2개 이상 컬럼 조합

#### 3. ACID 속성

**Atomicity (원자성):** 전부 성공 또는 전부 실패  
**Consistency (일관성):** 규칙 위반 불가  
**Isolation (격리성):** 트랜잭션 간 독립  
**Durability (영속성):** 완료 후 영구 저장

**예: 송금**
1. A 계좌 -10만원
2. B 계좌 +10만원
→ 중간 장애 시 롤백 (Atomicity)

#### 4. 트랜잭션

**정의:** 논리적 작업 단위

**명령:**
- BEGIN: 시작
- COMMIT: 확정
- ROLLBACK: 취소

#### 5. 인덱스

**목적:** 조회 속도 향상

**단점:** 쓰기 성능 저하, 저장 공간

**종류:**
- B-Tree: 범위 검색
- Hash: 동등 비교

### 예시 (25분)

#### 예시 1: PK/FK 설계

**주문 시스템:**

**고객 테이블:**
- 고객ID (PK)
- 이름
- 이메일

**주문 테이블:**
- 주문ID (PK)
- 고객ID (FK)
- 주문일
- 총액

**주문 상세 테이블:**
- 주문상세ID (PK)
- 주문ID (FK)
- 상품ID (FK)
- 수량

**관계:**
- 고객 1 : N 주문
- 주문 1 : N 주문상세

#### 예시 2: ACID 시나리오

**시나리오:** 온라인 결제

```sql
BEGIN;
UPDATE 고객 SET 포인트 = 포인트 - 5000 WHERE 고객ID=123;
INSERT INTO 주문 VALUES (...);
UPDATE 재고 SET 수량 = 수량 - 1 WHERE 상품ID=456;
COMMIT;
```

**장애 발생 시:**
- 포인트 차감 후 주문 실패 → ROLLBACK
- 3가지 작업 모두 취소 (원자성)

#### 예시 3: 인덱스 효과

**Before (인덱스 없음):**
```sql
SELECT * FROM 주문 WHERE 주문일='2024-01-10';
-- 전체 스캔: 100만 건 → 2초
```

**After (인덱스 생성):**
```sql
CREATE INDEX idx_주문일 ON 주문(주문일);
SELECT * FROM 주문 WHERE 주문일='2024-01-10';
-- 인덱스 스캔: 1,000건 → 0.02초
```

#### 실습 2: ACID 설명

**과제:** 항공 예약 시나리오로 ACID를 설명하세요.

<details>
<summary>해설</summary>

**시나리오 (서울→제주, 100번 좌석):**

```sql
BEGIN;
-- 1. 좌석 확인
SELECT status FROM 좌석 WHERE 항공편ID=123 AND 좌석번호=100;

-- 2. 예약 가능하면 좌석 점유
UPDATE 좌석 SET status='예약중' WHERE 항공편ID=123 AND 좌석번호=100;

-- 3. 예약 생성
INSERT INTO 예약 VALUES (예약ID, 고객ID, 항공편ID, 좌석번호, ...);

-- 4. 결제
UPDATE 고객 SET 포인트 = 포인트 - 50000 WHERE 고객ID=456;

COMMIT;
```

**ACID 적용:**

**Atomicity:** 결제 실패 시 좌석 점유도 취소 (전부 또는 전무)

**Consistency:** 
- 좌석은 1명만 예약 가능 (UNIQUE 제약)
- 포인트는 음수 불가 (CHECK 제약)

**Isolation:**  
- A와 B가 동시에 100번 예약 시도 → 한 명만 성공
- Lock으로 처리

**Durability:**  
- COMMIT 후 서버 장애 발생해도 예약 유지

**PM 관점:**
- 항공/티켓팅은 ACID 필수
- NoSQL 사용 불가 (이중 예약 위험)
- 트랜잭션 테스트 필수 (동시 예약 시나리오)

</details>

## 3교시: ERD와 데이터 모델링 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 데이터 모델링 3단계

**개념적 모델링 (Conceptual):**
- 업무 관점, 고수준
- 엔티티와 관계 정의
- ERD 작성

**논리적 모델링 (Logical):**
- 속성 추가, 정규화
- PK/FK 설정
- DBMS 독립적

**물리적 모델링 (Physical):**
- 테이블, 컬럼, 인덱스
- 성능 최적화
- DBMS 의존적

#### 2. ERD 구성 요소

**엔티티(Entity):** 관리 대상 (고객, 주문)  
**속성(Attribute):** 특성 (이름, 주문일)  
**관계(Relationship):** 연결 (고객이 주문)

**표기법:**
- 사각형: 엔티티
- 타원: 속성
- 다이아몬드: 관계

#### 3. 관계 카디널리티

**1:1 (일대일):** 사원 ↔ 사원증  
**1:N (일대다):** 부서 → 사원  
**M:N (다대다):** 학생 ↔ 강좌

**M:N 해소:** 중간 테이블 생성  
학생-수강-강좌 (M:N → 1:N + 1:N)

#### 4. 참여 제약

**필수 참여 (total):** 이중선, 반드시 참여  
**선택 참여 (partial):** 단일선, 선택적

**예:**
- 사원은 반드시 부서 소속 (필수)
- 부서는 사원 없을 수 있음 (선택)

### 예시 (25분)

#### 예시 1: 온라인 주문 ERD

**엔티티:**
- 고객 (고객ID, 이름, 이메일, 가입일)
- 주문 (주문ID, 주문일, 총액)
- 상품 (상품ID, 상품명, 가격)

**관계:**
- 고객 1 : N 주문 (한 고객이 여러 주문)
- 주문 M : N 상품 (한 주문에 여러 상품, 한 상품이 여러 주문)

**M:N 해소:**
- 주문 1 : N 주문상세
- 주문상세 N : 1 상품
- 주문상세 (주문ID, 상품ID, 수량, 단가)

#### 예시 2: 대학 수강신청 ERD

**학생 M : N 강좌**

**Before (M:N):**
```
학생 ↔ 강좌
```

**After (1:N + 1:N):**
```
학생 → 수강 ← 강좌
```

**수강 테이블:**
- 학생ID (FK)
- 강좌ID (FK)
- 수강년도
- 학기
- 성적

**복합 PK:** (학생ID, 강좌ID, 수강년도, 학기)

#### 예시 3: 참여 제약

**사원-부서:**
- 사원은 반드시 1개 부서 소속 (필수 참여)
- 부서는 사원 0명 이상 (선택 참여, 신설 부서)

**SQL:**
```sql
CREATE TABLE 사원 (
  사원ID INT PRIMARY KEY,
  부서ID INT NOT NULL, -- 필수
  FOREIGN KEY (부서ID) REFERENCES 부서(부서ID)
);
```

## 4교시: SQL 기초 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. SQL 분류

**DDL (Data Definition Language):** 구조 정의  
- CREATE, ALTER, DROP

**DML (Data Manipulation Language):** 데이터 조작  
- SELECT, INSERT, UPDATE, DELETE

**DCL (Data Control Language):** 권한 제어  
- GRANT, REVOKE

#### 2. SELECT 기본

```sql
SELECT 컬럼 FROM 테이블 WHERE 조건 ORDER BY 컬럼;
```

**WHERE 연산자:**
- =, !=, >, <, >=, <=
- BETWEEN, IN, LIKE
- AND, OR, NOT

**예:**
```sql
SELECT * FROM 주문 
WHERE 주문일 BETWEEN '2024-01-01' AND '2024-01-31'
AND 상태 IN ('결제', '배송');
```

#### 3. JOIN

**INNER JOIN:** 양쪽 모두 존재하는 행  
**LEFT JOIN:** 왼쪽 테이블 모두 + 오른쪽 매칭  
**RIGHT JOIN:** 오른쪽 테이블 모두 + 왼쪽 매칭  
**FULL OUTER JOIN:** 양쪽 모두

```sql
SELECT 고객.이름, 주문.주문일
FROM 고객
INNER JOIN 주문 ON 고객.고객ID = 주문.고객ID;
```

#### 4. 집계 함수

**COUNT, SUM, AVG, MAX, MIN**

**GROUP BY:** 그룹별 집계  
**HAVING:** 그룹 필터링

```sql
SELECT 카테고리, COUNT(*) AS 상품수
FROM 상품
GROUP BY 카테고리
HAVING COUNT(*) >= 10;
```

#### 5. 서브쿼리

**WHERE 절:**
```sql
SELECT * FROM 주문
WHERE 고객ID IN (SELECT 고객ID FROM 고객 WHERE 등급='VIP');
```

**FROM 절:**
```sql
SELECT AVG(일매출) FROM (
  SELECT 주문일, SUM(총액) AS 일매출
  FROM 주문
  GROUP BY 주문일
) AS 일별집계;
```

#### 예시 2: INNER vs LEFT JOIN

**INNER JOIN (주문한 고객만):**
```sql
SELECT 고객.이름, 주문.주문ID
FROM 고객
INNER JOIN 주문 ON 고객.고객ID = 주문.고객ID;
```
→ 주문 없는 고객 제외

**LEFT JOIN (모든 고객):**
```sql
SELECT 고객.이름, 주문.주문ID
FROM 고객
LEFT JOIN 주문 ON 고객.고객ID = 주문.고객ID;
```
→ 주문 없는 고객도 포함 (주문ID = NULL)

#### 예시 3: GROUP BY + HAVING

```sql
FROM 주문
GROUP BY 고객ID
HAVING COUNT(*) >= 2
ORDER BY 주문수 DESC;
```

**결과:**
| 고객ID | 주문수 |
|--------|-------|
| 123 | 5 |
| 456 | 3 |

```sql
SELECT 고객.이름, COUNT(주문.주문ID) AS 주문수
FROM 고객
INNER JOIN 주문 ON 고객.고객ID = 주문.고객ID
GROUP BY 고객.이름;
```

**결과:**
| 이름 | 주문수 |
|------|-------|
| 김철수 | 2 |
| 이영희 | 1 |

(박민수 제외, 주문 없음)

**LEFT JOIN:**
```sql
SELECT 고객.이름, COUNT(주문.주문ID) AS 주문수
FROM 고객
LEFT JOIN 주문 ON 고객.고객ID = 주문.고객ID
GROUP BY 고객.이름;
```

**결과:**
| 이름 | 주문수 |
|------|-------|
| 김철수 | 2 |
| 이영희 | 1 |
| 박민수 | 0 |

**PM 관점:**
- "모든 고객 포함" → LEFT JOIN
- "주문한 고객만" → INNER JOIN
- 요구사항 명확히 정의 필수

## 5교시: 정규화와 NoSQL (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. 정규화 목적

- 데이터 중복 최소화
- 이상(Anomaly) 방지
- 무결성 향상

**이상 현상:**
- 삽입 이상: 불필요한 데이터 강제 입력
- 삭제 이상: 원치 않는 데이터 삭제
- 갱신 이상: 일부만 수정되어 불일치

#### 2. 정규화 단계

**1NF (제1정규형):** 원자값, 반복 그룹 제거  
**2NF:** 1NF + 부분 함수 종속 제거  
**3NF:** 2NF + 이행적 함수 종속 제거

**함수 종속:** A → B (A가 B를 결정)

#### 3. 정규화 예시

**비정규 (주문):**
| 주문ID | 상품목록 | 고객명 | 고객우편번호 | 고객주소 |
|--------|---------|--------|------------|---------|
| 1 | 사과, 배 | 김철수 | 12345 | 서울 강남 |

**1NF (원자값):**
| 주문ID | 상품 | 고객명 | 고객우편번호 | 고객주소 |
|--------|------|--------|------------|---------|
| 1 | 사과 | 김철수 | 12345 | 서울 강남 |
| 1 | 배 | 김철수 | 12345 | 서울 강남 |

**3NF (분리):**
- 주문 (주문ID, 고객ID)
- 고객 (고객ID, 고객명, 고객우편번호, 고객주소)
- 주문상세 (주문ID, 상품ID)

#### 4. 역정규화

**목적:** 성능 향상

**방법:**
- 테이블 통합
- 컬럼 중복
- 파생 컬럼 추가

**예:** 주문 총액을 매번 계산 vs 주문 테이블에 총액 컬럼 추가

#### 5. NoSQL 특징

**스키마 유연성:** 필드 자유 추가  
**수평 확장:** 서버 추가로 확장  
**BASE:** Eventually Consistent

**유형:**
- Document: MongoDB (JSON)
- Key-Value: Redis (캐시)
- Column-Family: Cassandra (대용량)
- Graph: Neo4j (관계)

#### 6. RDBMS vs NoSQL 선택

| 기준 | RDBMS | NoSQL |
|------|-------|-------|
| 정합성 | 높음 (ACID) | 낮음 (BASE) |
| 스키마 | 고정 | 유연 |
| 확장 | 수직 | 수평 |
| 조인 | 효율적 | 비효율적 |
| 사례 | 금융, 재고 | 로그, 캐시, 빅데이터 |

### 예시 (25분)

#### 예시 1: 2NF 위반

**Before (2NF 위반):**
수강 (학생ID, 강좌ID, 학생이름, 학과, 강좌명)  
- 복합키: (학생ID, 강좌ID)
- 학생이름, 학과는 학생ID에만 종속 (부분 종속)

**After (2NF):**
- 학생 (학생ID, 학생이름, 학과)
- 강좌 (강좌ID, 강좌명)
- 수강 (학생ID, 강좌ID, 성적)

#### 예시 2: 3NF 위반

**Before (3NF 위반):**
사원 (사원ID, 이름, 부서ID, 부서명, 부서위치)  
- 부서명, 부서위치는 부서ID에 종속 (이행 종속)

**After (3NF):**
- 사원 (사원ID, 이름, 부서ID)
- 부서 (부서ID, 부서명, 부서위치)

#### 예시 3: 역정규화

**정규화 (조인 필요):**
```sql

FROM 주문
JOIN 주문상세 ON 주문.주문ID = 주문상세.주문ID
```

→ 매번 계산, 느림

**역정규화 (컬럼 추가):**

→ 조회 빠름, 단 주문상세 변경 시 총액도 UPDATE 필요

**정규화 (Normalization):**

**목적:** 데이터 중복 제거, 무결성 향상

**장점:**

**장점:**

- 쿼리 단순화

**단점:**
- 데이터 중복
- 갱신 비용 증가 (여러 곳 수정)

**PM 의사결정 기준:**

**1. 성능 요구사항:**
- 조회 빈도 >> 수정 빈도 → 역정규화 고려
- 예: 주문 총액 (조회 1만 회/일, 수정 100회/일)

**2. 데이터 정합성:**
- 금융, 재고: 정규화 유지 (정합성 우선)
- 로그, 통계: 역정규화 가능

**3. 규모:**
- 소규모: 정규화 (조인 성능 문제 없음)
- 대규모: 역정규화 필요 (수백만 건 조인 느림)

**4. 복잡도:**
- 조인 3개 이상: 역정규화 검토
- 예: 주문-고객-상품-카테고리 (4개 조인)

**의사결정 프로세스:**

**Step 1: 기본은 3NF**
- 설계 단계에서 3NF까지 정규화

**Step 2: 성능 테스트**
- 실제 데이터로 조회 성능 측정
- 목표: 응답시간 3초 이내

**Step 3: 병목 식별**
- 슬로우 쿼리 분석 (EXPLAIN)
- 조인이 병목인지 확인

**Step 4: 대안 검토**
1. 인덱스 추가 (먼저 시도)
2. 뷰(View) 생성
3. 마지막 수단: 역정규화

**Step 5: 역정규화 적용**
- 자주 조인하는 컬럼 중복
- 트리거로 일관성 유지
```sql
CREATE TRIGGER update_총액
AFTER INSERT ON 주문상세
FOR EACH ROW
UPDATE 주문 SET 총액 = 총액 + NEW.수량*NEW.단가
WHERE 주문ID = NEW.주문ID;
```

**예시 (전자상거래):**

**정규화 3NF:**
```sql
SELECT 
  주문.주문ID,
  고객.고객명,
  SUM(주문상세.수량 * 주문상세.단가) AS 총액
FROM 주문
JOIN 고객 ON 주문.고객ID = 고객.고객ID
JOIN 주문상세 ON 주문.주문ID = 주문상세.주문ID
GROUP BY 주문.주문ID, 고객.고객명;
```
→ 3개 테이블 조인, 주문 100만 건 시 5초

**역정규화 (주문에 고객명, 총액 추가):**
```sql
SELECT 주문ID, 고객명, 총액
FROM 주문;
```
→ 조인 없음, 0.1초

**트레이드오프:**
- 조회: 5초 → 0.1초 (50배 빠름)
- 갱신: 주문상세 추가 시 주문.총액도 UPDATE (복잡도 증가)
- 불일치 위험: 트리거로 관리

**PM 체크리스트:**
- [ ] 성능 요구사항 정의? (응답시간)
- [ ] 조회 vs 갱신 비율 분석?

- [ ] 일관성 유지 방법? (트리거)
- [ ] 테스트 시나리오 준비? (동시성)

**결론:**
- 기본: 정규화 (3NF)
- 병목: 인덱스 우선
- 최후: 역정규화 (성능 vs 복잡도 트레이드오프)

## 6교시: 데이터 관리 트렌드 및 PM 관점 (이론 + 예시 + 실습 + 퀴즈)

### 이론 (50분)

#### 1. Data Warehouse (DW)

**정의:** 분석용 데이터 저장소

**특징:**
- OLAP (분석 처리)
- 역사적 데이터 (수년치)
- 읽기 집중

**vs OLTP:**
- OLTP: 업무용 (주문, 결제), 쓰기 빈번
- OLAP: 분석용 (매출 추이), 읽기 집중

#### 2. Data Lake

**정의:** 원시 데이터 저장 (비정형 포함)

**특징:**
- 모든 형식 (CSV, JSON, 이미지, 로그)
- Schema-on-Read (읽을 때 스키마)
- 저렴한 저장 (S3, Azure Data Lake)

#### 3. ETL vs ELT

**ETL (Extract, Transform, Load):**
- 전통적 방식
- 변환 후 적재 (DW에 정제된 데이터)

**ELT:**
- 최신 방식 (Data Lake)
- 적재 후 변환 (Spark, Databricks)

#### 4. 데이터 거버넌스

**정의:** 데이터 관리 체계

**핵심:**
- 소유권: 데이터 책임자
- 품질: 정확성, 완전성
- 보안: 접근 통제, 암호화
- 컴플라이언스: GDPR, 개인정보보호법

#### 5. 백업/복구 전략

**RPO (Recovery Point Objective):** 복구 시점  
- 예: RPO=1시간 (1시간 전 데이터로 복구)

**RTO (Recovery Time Objective):** 복구 시간  
- 예: RTO=4시간 (4시간 내 복구)

**백업 전략:**
- Full: 전체 (주간)
- Incremental: 변경분 (일간)
- 3-2-1 규칙: 3벌 복사, 2개 매체, 1개 오프사이트

#### 6. 성능 지표

**TPS (Transactions Per Second):** 초당 트랜잭션  
**QPS (Queries Per Second):** 초당 쿼리  
**Latency:** 응답 시간

**목표 예:**
- 전자상거래: TPS 1,000, 응답시간 1초
- 금융: TPS 10,000, 응답시간 100ms

### 예시 (25분)

#### 예시 1: OLTP vs OLAP

**OLTP (주문 시스템):**
```sql
INSERT INTO 주문 VALUES (...);
SELECT * FROM 주문 WHERE 주문ID=123;
```
- 실시간, 짧은 트랜잭션
- PostgreSQL, MySQL

**OLAP (매출 분석):**
```sql
SELECT 
  YEAR(주문일), MONTH(주문일), 
  SUM(총액) AS 월매출
FROM 주문
WHERE 주문일 >= '2020-01-01'
GROUP BY YEAR(주문일), MONTH(주문일);
```
- 대량 집계, 복잡한 쿼리
- Redshift, BigQuery

#### 예시 2: Data Lake 활용

**시나리오:** 전자상거래 로그 분석

**수집:**
- 웹 로그 (Nginx) → S3
- 앱 로그 (JSON) → S3
- 클릭 스트림 → S3

**분석 (Spark):**
```python
df = spark.read.json("s3://logs/2024/*")
df.groupBy("page").count().show()
```

**적용:**
- 인기 상품 파악
- 추천 알고리즘 개선

#### 예시 3: 백업/복구 계획

**은행 시스템:**
- RPO: 0 (데이터 손실 불가)
- RTO: 1시간

**전략:**
- Replication (실시간 복제)
- Full Backup (일간, 새벽)

**테스트:**
