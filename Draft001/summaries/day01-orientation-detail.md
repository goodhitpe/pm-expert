# Day 1: 프로젝트 관리 오리엔테이션 - 상세 강의안

## 1교시: 프로젝트란 무엇인가? (1.5시간)

### 이론 (50분)

#### 1. 프로젝트의 정의

**PMBOK® 정의:**  
"고유한 제품, 서비스, 결과를 창출하기 위해 일시적으로 투입하는 노력"

#### 2. 프로젝트의 3대 핵심 특성

**가. 일시성(Temporary)**
- 명확한 시작일과 종료일이 존재합니다
- 종료 조건:
  - 프로젝트 목표가 달성됨
  - 목표 달성이 불가능하다고 판단됨
  - 프로젝트의 필요성이 사라짐
- 주의사항: "일시적"이 "짧은 기간"을 의미하는 것은 아닙니다
  - 예: 경부고속도로 건설(1968~1970, 2년 4개월)
  - 예: 신공항 건설(수년~십수년 소요)
- 중요: 산출물 자체는 영구적일 수 있습니다
  - 완성된 빌딩은 수십 년 사용
  - 개발된 소프트웨어는 지속적으로 운영

**나. 고유성(Unique)**
- 이전에 정확히 동일한 방식으로 수행된 적이 없는 작업
- 반복 요소가 있더라도 전체 조합은 고유함
- 불확실성을 내포하고 있어 리스크 관리 필요
- 예시 설명:
  - 100채의 동일 설계 아파트를 건설하더라도
  - 각 프로젝트의 위치, 팀 구성, 토질, 날씨, 규제 환경이 다름
  - IT 프로젝트: 동일한 ERP 패키지 도입도 회사마다 커스터마이징이 다름

**다. 점진적 구체화(Progressive Elaboration)**
- 프로젝트 초기: 상위 수준의 일반적 계획
- 진행됨에 따라: 더 많은 정보 수집, 더 정확한 추정 가능
- 롤링 웨이브 계획(Rolling Wave Planning):
  - 가까운 미래는 상세히 계획
  - 먼 미래는 상위 수준으로만 계획
- 예시:
  - 초기: "신규 CRM 시스템 구축, 예산 5억"
  - 1개월 후: "고객관리 30개 기능, 영업관리 20개 기능, 예산 4.8억"
  - 3개월 후: "각 기능별 상세 명세 150페이지, 예산 4.9억, 일정 조정"

#### 3. 프로젝트 vs 운영(Operation)

**프로젝트(Project)의 특징:**
- 시작과 끝이 명확히 정의됨
- 고유한 결과물 창출
- 변화를 추진하는 역할
- 임시 조직(프로젝트 팀)으로 구성
- 불확실성이 높음
- 예산과 자원이 한정적
- 성공 기준이 명확함

**운영(Operation)의 특징:**
- 반복적이고 지속적으로 수행
- 동일한 제품/서비스를 반복 생산
- 현상 유지 및 안정성 추구
- 상설 조직(부서)으로 운영
- 예측 가능성이 높음
- 지속적인 자원 배분
- 효율성과 일관성이 핵심

**프로젝트와 운영의 관계:**
- 프로젝트가 끝나면 운영으로 전환되는 경우가 많음
- 예: ERP 시스템 구축(프로젝트) → ERP 운영/유지보수(운영)
- 운영 중 개선 필요 시 프로젝트를 시작함
- 조직은 프로젝트와 운영을 동시에 수행

#### 4. 프로젝트 관리(Project Management)의 정의

"프로젝트 요구사항을 충족시키기 위해 지식(Knowledge), 기술(Skills), 도구(Tools), 기법(Techniques)을 프로젝트 활동에 적용하는 것"

**프로젝트 관리의 목표:**
1. 요구사항 충족
2. 이해관계자 만족
3. 제약 조건 내에서 목표 달성 (범위, 일정, 원가, 품질)
4. 비즈니스 가치 창출

#### 5. 프로젝트 관리자(PM)의 역할

**PM의 책임:**
- 프로젝트 목표 달성
- 이해관계자 기대사항 관리
- 범위, 일정, 원가, 품질, 리스크 통합 관리
- 팀 리더십 발휘
- 의사결정 및 문제 해결

**PMI Talent Triangle - PM의 3대 역량:**

1) **기술적 프로젝트 관리(Technical Project Management)**
   - 10대 지식영역 전문성
   - 프로젝트 관리 방법론 활용
   - PMBOK 프로세스 적용
   - WBS, CPM, EVM 등 도구 활용
   - 예: 일정 기준선 수립, 리스크 분석 수행

2) **리더십(Leadership)**
   - 팀원 동기부여
   - 갈등 조정 및 해결
   - 협상 및 영향력 발휘
   - 비전 제시 및 변화 촉진
   - 감성 지능(EQ) 활용
   - 예: 팀 사기 저하 시 격려, 부서 간 이견 조정

3) **전략 및 비즈니스 관리(Strategic and Business Management)**
   - 조직 전략과 프로젝트 연계
   - 비즈니스 가치 이해
   - 재무적 관점(ROI, NPV, IRR) 이해
   - 시장 및 경쟁 환경 파악
   - 예: 프로젝트가 회사 전략에 기여하는 방식 설명

### 예시 (30분)

#### 예시 1: IT 업계의 프로젝트 vs 운영

**프로젝트 사례:**
1. **신규 전자상거래 플랫폼 구축**
   - 기간: 2025.3.1 ~ 2025.12.31 (10개월)
   - 목표: 고객 직접 판매 채널 신규 구축
   - 산출물: 웹사이트, 모바일 앱, 결제 시스템, 물류 연동
   - 팀: 프로젝트 매니저 1명, 개발자 8명, 디자이너 2명, QA 2명
   - 예산: 8억원
   - 종료 후: 운영팀으로 이관

2. **클라우드 마이그레이션 프로젝트**
   - 기간: 6개월
   - 목표: 온프레미스 → AWS 전환
   - 고유성: 회사 고유의 레거시 시스템 구조
   - 불확실성: 마이그레이션 중 다운타임, 데이터 정합성 리스크

3. **모바일 앱 2.0 버전 출시**
   - 기간: 4개월
   - 목표: 사용자 경험 개선, 신기능 10개 추가
   - 점진적 구체화: 베타 테스트 피드백으로 기능 조정

**운영 사례:**
1. **기존 ERP 시스템 운영**
   - 매일 반복: 사용자 지원, 장애 대응, 백업 수행
   - 월간 반복: 패치 적용, 성능 모니터링 보고
   - 목표: 시스템 가용률 99.9% 유지

2. **IT 헬프데스크**
   - 지속적: 직원 PC/네트워크 문제 해결
   - 표준 절차: 티켓 등록 → 분류 → 해결 → 종료
   - 목표: 평균 응답 시간 4시간 이내

3. **월간 급여 처리 시스템 운영**
   - 매월 반복: 근태 집계 → 급여 계산 → 이체 처리
   - 동일한 프로세스 반복
   - 목표: 정확성 100%, 기한 준수

#### 예시 2: PM 역량의 실제 적용

**시나리오: 대규모 ERP 구축 프로젝트**

**상황:**  
제조업 기업의 노후화된 ERP를 차세대 시스템으로 전환하는 18개월 프로젝트. 예산 50억원, 팀원 30명, 전사 사용자 500명 영향.

**PM의 3대 역량 적용:**

1) **기술적 PM 역량 발휘:**
   - WBS 작성: 5단계까지 상세 분해, 작업 패키지 120개
   - 일정 관리: CPM으로 주공정 식별, 총 18개월 일정
   - 원가 관리: EVM으로 매주 SPI/CPI 추적
   - 리스크 관리: 45개 리스크 식별, 상위 10개 집중 관리
   - 결과: 데이터 기반 객관적 의사결정 가능

2) **리더십 역량 발휘:**
   - 팀 동기부여: 주요 마일스톤 달성 시 팀 회식, 칭찬
   - 갈등 조정: 현업 vs 개발팀 요구사항 충돌 시 협의 회의 주재
   - 변화 관리: 사용자 저항에 대한 교육 및 소통 강화
   - 결과: 팀 이탈률 0%, 현업 참여도 향상

3) **전략/비즈니스 역량 발휘:**
   - 전략 연계: CFO에게 ERP가 재무 투명성 제고에 기여함을 설명
   - ROI 계산: 운영비 절감 연 5억 → 10년 ROI 긍정적
   - 우선순위 조정: 비즈니스 영향도 높은 모듈 우선 개발
   - 결과: 경영진 지속 지원 확보

#### 예시 3: 프로젝트 실패 사례 - 3대 특성 무시

**사례: 공공기관 시스템 통합 프로젝트 실패**

**배경:**  
2023년 정부 부처 A의 5개 시스템을 통합하는 프로젝트. 예산 100억원, 기간 2년.

**실패 원인:**

1. **일시성 망각:**
   - 종료 기준이 불명확: "완벽할 때까지" 라는 모호한 목표
   - 결과: 2년 → 3년으로 연장, 그래도 종료 못함
   - 교훈: 명확한 인수 기준과 종료 조건 필요

2. **고유성 무시:**
   - 타 기관 성공 사례를 그대로 복제 시도
   - 현 조직의 고유한 프로세스와 문화를 간과
   - 결과: 현업 거부감 증가, 사용률 30%
   - 교훈: 조직 특성 분석과 커스터마이징 필수

3. **점진적 구체화 실패:**
   - 초기에 모든 요구사항을 확정하려 함
   - 변경 요청을 모두 거부
   - 결과: 실제 필요와 괴리, 재작업 50% 발생
   - 교훈: 반복적 기획과 유연한 변경 관리 필요

모범답안:

## 2교시: Triple Constraint와 품질 (1.5시간)

### 이론 (50분)

#### 1. Triple Constraint (삼중 제약) 개념

**정의:**  
프로젝트의 성공은 **범위(Scope), 일정(Time), 원가(Cost)** 세 가지 제약 조건의 균형에 의해 결정됩니다.

**핵심 원리:**
- 세 요소는 상호 의존적이며, 하나를 변경하면 나머지 두 개 이상에 영향을 줍니다
- "빠르게, 저렴하게, 품질 좋게" 세 가지를 동시에 달성하기는 매우 어렵습니다
- PM은 이들 간의 균형을 유지하고 트레이드오프를 관리해야 합니다

#### 2. 범위(Scope)

**의미:** "무엇을 만들 것인가?"  
프로젝트가 인도해야 할 모든 산출물과 수행해야 할 작업

**구성 요소:**
- 제품 범위: 기능, 특성
- 프로젝트 범위: 수행할 작업
- 인도물(Deliverables): 검증 가능한 결과물

**범위 관리의 핵심:**
- 범위 크리프(Scope Creep) 방지: 통제되지 않은 범위 확장
- 명확한 범위 정의와 변경 통제 프로세스 필요

#### 3. 일정(Time/Schedule)

**의미:** "언제까지 완료할 것인가?"  
프로젝트의 시작부터 종료까지의 기간

**구성 요소:**
- 마일스톤: 주요 체크포인트
- 활동 기간: 각 작업에 소요되는 시간
- 주공정(Critical Path): 프로젝트 최소 완료 기간을 결정하는 경로

**일정 관리의 핵심:**
- 현실적인 기간 산정
- 의존 관계 파악
- 진척 모니터링과 조기 경보

#### 4. 원가(Cost)

**의미:** "얼마의 비용이 드는가?"  
프로젝트 수행에 필요한 모든 자원의 금전적 가치

**구성 요소:**
- 인건비: 팀원 급여, 외부 컨설턴트 비용
- 장비/자재비: 하드웨어, 소프트웨어 라이선스
- 외주비: 용역 계약 비용
- 간접비: 관리비, 시설 사용료

**원가 관리의 핵심:**
- 정확한 원가 산정
- 예산 대비 실적 추적
- 원가 초과 조기 발견 및 대응

#### 5. 품질(Quality)과 Triple Constraint의 관계

**품질의 위치:**
- 품질은 삼중 제약의 "균형점"에 위치
- 4번째 제약이 아니라, 세 제약의 조화로 달성되는 결과

**상호 영향:**

1) **범위 증가 → 다른 제약에 미치는 영향**
   - 일정 지연 (더 많은 작업 시간 필요)
   - 원가 증가 (더 많은 자원 투입)
   - 품질 저하 가능 (일정/원가 유지 시)

2) **일정 단축 → 다른 제약에 미치는 영향**
   - 범위 축소 (일부 기능 포기)
   - 원가 증가 (추가 인력 투입, 야근 수당)
   - 품질 저하 (테스트 시간 부족, 재작업 증가)

3) **원가 절감 → 다른 제약에 미치는 영향**
   - 범위 축소 (저렴한 대안 선택)
   - 일정 연장 (적은 인력으로 작업)
   - 품질 저하 (숙련도 낮은 인력, 저가 자재)

#### 6. 기준선(Baseline)의 역할

**정의:**  
승인된 범위, 일정, 원가 계획으로, 실제 성과를 측정하는 기준

**3대 기준선:**
- 범위 기준선: 승인된 WBS, 범위 기술서
- 일정 기준선: 승인된 일정 계획
- 원가 기준선: 시간 배분된 승인 예산

**기준선의 중요성:**
- 성과 측정의 기준점
- 변경 영향 분석의 시작점
- 획득가치 관리(EVM)의 기반
- 공식적 변경 통제를 통해서만 수정 가능

### 예시 (30분)

#### 예시 1: 범위 크리프(Scope Creep)로 인한 프로젝트 실패

**프로젝트:** B2C 쇼핑몰 구축
**초기 계획:**
- 범위: 상품 조회, 장바구니, 결제 기능
- 일정: 6개월
- 원가: 3억원
- 팀: 개발자 5명

**진행 중 변화:**

**1개월차:**
- 마케팅팀 요청: "소셜 로그인 기능 추가"
- PM 판단: "간단하니 추가 가능" → 승인 (변경 통제 없이)
- 영향: +5일, +500만원

**2개월차:**
- 고객서비스팀 요청: "실시간 채팅 상담 기능"
- PM: "고객 만족도에 중요" → 승인
- 영향: +15일, +2,000만원

**3개월차:**
- 경영진 요청: "모바일 앱도 함께 개발"
- PM: 거절 불가 → 승인
- 영향: +2개월, +1억원

**4개월차:**
- 개발팀: "원래 일정에 추가 기능까지 불가능"
- 품질: 테스트 시간 부족으로 버그 다수 발견

**최종 결과:**
- 범위: 초기 대비 150% 증가
- 일정: 6개월 → 11개월 (83% 지연)
- 원가: 3억 → 5.2억 (73% 초과)
- 품질: 출시 후 장애 빈발, 고객 불만 증가
- 결과: 프로젝트 실패 판정

**교훈:**
- 모든 범위 변경은 공식 변경 통제 프로세스를 거쳐야 함
- 작은 변경도 축적되면 큰 영향
- 범위 증가 시 일정/원가 동시 조정 필요
- "안됩니다"라고 말할 수 있는 용기 필요

#### 예시 2: 일정 단축 압박과 품질 트레이드오프

**프로젝트:** 금융사 모바일 뱅킹 앱 업그레이드
**초기 계획:**
- 기간: 8개월
- 예산: 10억원
- 주요 기능: 생체 인증, 간편 송금, 자산 분석 대시보드

**4개월차 경영진 지시:**
- "경쟁사가 2개월 후 유사 기능 출시 예정"
- "우리도 2개월 후(6개월차) 출시하라"
- 요구: 일정 2개월 단축

**PM의 대응 옵션 분석:**

**Option 1: Crashing (인력 증원)**
- 개발자 5명 → 8명으로 증원
- 예상 원가 증가: +2억원
- 예상 일정 단축: 1개월 (6개월 → 7개월)
- 리스크: 신규 인력 적응 시간, 의사소통 복잡도 증가
- 품질 영향: 중간 수준

**Option 2: Fast-tracking (병렬 수행)**
- 개발과 테스트를 동시 진행
- 예상 원가 증가: +5천만원 (재작업 비용)
- 예상 일정 단축: 1.5개월
- 리스크: 재작업 증가, 통합 이슈
- 품질 영향: 높음 (테스트 부족)

**Option 3: 범위 축소**
- 자산 분석 대시보드 기능 제외 (v2.0으로 연기)
- 예상 원가 절감: -1억원
- 예상 일정 단축: 2개월
- 리스크: 차별화 약화
- 품질 영향: 낮음 (핵심 기능에 집중)

**PM의 선택:**
- Option 3 (범위 축소) 선택
- 이유: 금융 앱은 안정성이 최우선, 품질 타협 불가
- 경영진 설득: "버그 많은 빠른 출시 vs 안정적인 늦은 출시"
- 대안 제시: v1.0 핵심 기능 우선, v2.0에서 고급 기능 추가

**결과:**
- 6개월 만에 안정적 출시
- 초기 사용자 만족도 높음
- v2.0 출시(3개월 후)로 경쟁 우위 확보
- 품질 우선 전략 성공

**교훈:**
- 일정 단축 요구 시 반드시 트레이드오프 분석
- 품질이 중요한 프로젝트는 범위 조정이 최선
- 데이터 기반 의사결정과 대안 제시가 PM의 책임

#### 예시 3: 균형 잡힌 제약 관리 - 성공 사례

**프로젝트:** 스타트업의 신규 SaaS 제품 개발
**초기 계획:**
- 범위: MVP(최소 기능 제품) 10개 핵심 기능
- 일정: 4개월
- 원가: 2억원 (자체 개발팀)
- 품질 목표: 버그 발견율 < 5%

**PM의 균형 관리 전략:**

1) **범위 관리:**
   - MoSCoW 기법 적용: Must 5개, Should 3개, Could 2개
   - 매주 우선순위 리뷰
   - 변경 요청은 반드시 CCB(변경통제위원회) 승인
   - 결과: Scope Creep 0건

2) **일정 관리:**
   - 2주 스프린트로 반복 개발
   - 일일 스탠드업으로 장애물 조기 제거
   - 버퍼 타임 20% 확보 (리스크 대비)
   - 결과: 일정 준수율 95%

3) **원가 관리:**
   - 주간 실제 지출 추적
   - EVM으로 CPI 계산 (목표: CPI ≥ 0.95)
   - 외주 최소화, 오픈소스 활용
   - 결과: 예산 내 완료 (1.9억원, 5% 절감)

4) **품질 관리:**
   - 자동화 테스트 커버리지 80% 유지
   - 코드 리뷰 필수화
   - 매 스프린트 종료 시 회고
   - 결과: 버그 발견율 3.2%

**최종 결과:**
- 범위: MVP 10개 기능 모두 완성
- 일정: 4개월 정확히 준수
- 원가: 예산 대비 5% 절감
- 품질: 목표 초과 달성
- 결과: 성공적 출시, 초기 사용자 1,000명 확보

**성공 요인:**
- 초기부터 명확한 기준선 설정
- 지속적인 모니터링과 투명한 보고
- 작은 변경도 공식 프로세스 준수
- 데이터 기반 의사결정
- 팀 참여와 소통

### 퀴즈 (10분)

**Q1. 범위 증가가 일정과 원가에 미치는 영향을 설명하시오.**

모범답안:
범위가 증가하면 더 많은 작업을 수행해야 하므로, 일정이 연장되거나 원가가 증가합니다. 구체적으로:
- 일정 영향: 추가 기능 개발, 테스트에 더 많은 시간 소요 → 일정 지연
- 원가 영향: 추가 인력 투입, 연장된 기간의 인건비 증가 → 예산 초과
- 만약 일정과 예산을 고정한다면, 품질 저하나 팀 번아웃이 발생할 수 있습니다.

**Q2. 일정 단축 시 발생 가능한 품질 리스크를 2가지 쓰시오.**

모범답안:
1. 테스트 시간 부족: 충분한 테스트 없이 배포하여 버그가 많이 발생
2. 설계 부실: 급하게 개발하다 보니 기술 부채 증가, 유지보수 어려움
(추가 가능: 문서화 소홀, 팀원 피로누적으로 실수 증가, 코드 리뷰 생략 등)

**Q3. 기준선의 역할을 한 줄로 정의하시오.**

모범답안:
기준선은 승인된 계획으로, 프로젝트 실제 성과를 측정하고 비교하는 기준점 역할을 합니다.

## 3교시: 프로젝트 관리 프레임워크 개요 (1시간)

### 이론 (40분)

#### 1. PMBOK 프레임워크 개요

**PMBOK(Project Management Body of Knowledge)란?**
- PMI(Project Management Institute)에서 발행하는 프로젝트 관리 표준 가이드
- 업계 모범 사례(Best Practice)를 체계화한 지식 체계
- PMP 시험의 기준 문서
- 현재 7판까지 발행 (2021년)

**PMBOK의 구조:**
- 5개 프로세스 그룹 (Process Groups)
- 10개 지식 영역 (Knowledge Areas)  
- 49개 프로세스 (6판 기준)
- 상호 의존적이고 반복적으로 수행

#### 2. 5개 프로세스 그룹 (Process Groups)

**프로세스 그룹의 의미:**
논리적으로 그룹화된 프로젝트 관리 프로세스의 모음. 프로젝트 생애주기를 통해 반복 수행됨.

**1) 착수 프로세스 그룹 (Initiating)**
- **목적:** 새로운 프로젝트나 단계를 공식적으로 시작
- **핵심 활동:**
  - 프로젝트 헌장 개발
  - 이해관계자 식별
  - 프로젝트 목표와 정당성 수립
  - PM 임명 및 권한 부여
- **주요 산출물:** 프로젝트 헌장, 이해관계자 등록부
- **중요도:** 프로젝트의 공식적 승인이 없으면 PM 권한 없음

**2) 기획 프로세스 그룹 (Planning)**
- **목적:** 프로젝트 목표 달성을 위한 상세 계획 수립
- **핵심 활동:**
  - 범위, 일정, 원가, 품질 계획 수립
  - WBS 작성, 일정 개발, 예산 수립
  - 리스크 식별 및 대응 계획
  - 조달 및 의사소통 계획
- **주요 산출물:** 프로젝트 관리 계획서 (통합 문서)
- **특징:** 가장 많은 프로세스 포함 (24개)
- **중요도:** "계획에 실패하는 것은 실패를 계획하는 것"

**3) 실행 프로세스 그룹 (Executing)**
- **목적:** 계획된 작업을 실제로 수행하여 인도물 생산
- **핵심 활동:**
  - 프로젝트 작업 지시 및 관리
  - 팀 구성, 개발, 관리
  - 품질 관리 수행
  - 의사소통 관리
  - 조달 수행
- **주요 산출물:** 인도물(Deliverables), 작업 성과 데이터
- **소요 자원:** 전체 예산의 80~90% 소비
- **중요도:** 실제 가치가 창출되는 단계

**4) 감시 및 통제 프로세스 그룹 (Monitoring and Controlling)**
- **목적:** 계획 대비 실적을 추적, 검토, 조정
- **핵심 활동:**
  - 작업 성과 모니터링
  - 통합 변경 통제 수행
  - 범위, 일정, 원가 통제
  - 품질, 리스크 통제
- **주요 산출물:** 작업 성과 보고서, 변경 요청, 시정 조치
- **특징:** 프로젝트 전 기간 동안 지속적 수행
- **중요도:** 프로젝트를 정상 궤도에 유지

**5) 종료 프로세스 그룹 (Closing)**
- **목적:** 프로젝트나 단계를 공식적으로 완료
- **핵심 활동:**
  - 최종 인도물 인수
  - 교훈 수집 및 문서화
  - 계약 종료
  - 팀원 방출
  - 최종 보고서 작성
- **주요 산출물:** 최종 보고서, 교훈 문서, 인수 증명서
- **중요도:** 적절한 종료 없이는 법적/재무적 문제 발생 가능

**프로세스 그룹 간 관계:**
- 순차적이면서도 중첩적으로 수행
- 기획 → 실행 → 감시/통제는 반복적 순환
- 각 단계(Phase)마다 5개 그룹 모두 수행 가능

#### 3. 10개 지식 영역 (Knowledge Areas)

**지식 영역의 의미:**
프로젝트 관리를 구성하는 전문 지식 분야. 각 영역은 여러 프로세스를 포함.

**1) 프로젝트 통합 관리 (Integration Management)**
- **역할:** 모든 지식 영역을 조율하고 통합
- **프로세스:** 7개
- **핵심:** 프로젝트 헌장, 관리 계획서, 변경 통제
- **PM의 책임:** PM만이 수행하는 유일한 영역

**2) 프로젝트 범위 관리 (Scope Management)**
- **역할:** 프로젝트에 필요한 모든 작업 포함 및 불필요한 작업 제외
- **프로세스:** 6개
- **핵심:** WBS 작성, 범위 확인, Scope Creep 방지

**3) 프로젝트 일정 관리 (Schedule Management)**
- **역할:** 적시에 프로젝트 완료 보장
- **프로세스:** 6개
- **핵심:** CPM, 간트차트, 일정 단축 기법

**4) 프로젝트 원가 관리 (Cost Management)**
- **역할:** 승인된 예산 내에서 프로젝트 완료
- **프로세스:** 4개
- **핵심:** 원가 산정, 예산 수립, EVM

**5) 프로젝트 품질 관리 (Quality Management)**
- **역할:** 요구사항 충족 및 고객 만족
- **프로세스:** 3개
- **핵심:** 품질 계획, QA, QC, 7가지 품질 도구

**6) 프로젝트 자원 관리 (Resource Management)**
- **역할:** 프로젝트 성공을 위한 자원 확보 및 관리
- **프로세스:** 6개
- **핵심:** RACI 차트, 팀 개발, 갈등 관리

**7) 프로젝트 의사소통 관리 (Communications Management)**
- **역할:** 적시에 적절한 정보 생성, 배포, 관리
- **프로세스:** 3개
- **핵심:** 의사소통 채널 수, 보고 체계, 회의 관리

**8) 프로젝트 리스크 관리 (Risk Management)**
- **역할:** 위협 최소화, 기회 극대화
- **프로세스:** 7개
- **핵심:** 리스크 식별, P-I 매트릭스, EMV, 대응 전략

**9) 프로젝트 조달 관리 (Procurement Management)**
- **역할:** 외부에서 제품/서비스 획득
- **프로세스:** 3개
- **핵심:** Make or Buy, 계약 유형, RFP

**10) 프로젝트 이해관계자 관리 (Stakeholder Management)**
- **역할:** 이해관계자 기대 관리 및 참여 유도
- **프로세스:** 4개
- **핵심:** 권력-관심도 매트릭스, 참여 전략

#### 4. 49개 프로세스 맵 개요

**프로세스 분포:**
- 착수: 2개
- 기획: 24개 (가장 많음)
- 실행: 10개
- 감시/통제: 12개
- 종료: 1개

**프로세스 간 관계:**
- 입력(Input) - 도구/기법(Tools & Techniques) - 출력(Output)
- 한 프로세스의 출력이 다른 프로세스의 입력이 됨
- 순차적이면서도 반복적

#### 5. 이해관계자(Stakeholder)의 중요성

**정의:**
프로젝트에 영향을 주거나, 프로젝트로부터 영향을 받거나, 영향을 받는다고 인식하는 개인, 그룹, 조직

**이해관계자의 범위:**
- 내부: 스폰서, PM, 프로젝트 팀, PMO, 경영진, 현업 부서
- 외부: 고객, 최종 사용자, 공급업체, 규제 기관, 일반 대중

**이해관계자 관리가 중요한 이유:**
1. **프로젝트 성패 결정:** 연구에 따르면 프로젝트 실패의 70%가 이해관계자 관리 실패
2. **요구사항 충족:** 이해관계자의 요구와 기대를 이해해야 올바른 결과물 도출
3. **지원 확보:** 주요 이해관계자의 지지가 자원과 권한 확보에 필수
4. **저항 최소화:** 조기 참여와 소통으로 변화 저항 감소
5. **리스크 관리:** 이해관계자가 주요 리스크 원천이자 해결책

#### 예시 3: 이해관계자 관리 실패로 인한 프로젝트 위기

**프로젝트:** 병원 전자의무기록(EMR) 시스템 도입
**배경:** 5개 진료과, 의사 50명, 간호사 150명 영향

**문제 상황:**

**초기 (착수/기획):**
- PM은 병원장, IT팀장만을 주요 이해관계자로 파악
- 실제 사용자인 의사, 간호사는 요구사항 수집에서 제외
- 의사 1명만 형식적으로 인터뷰

**중간 (실행 6개월차):**
- 시스템 개발 80% 완료
- 파일럿 테스트에서 의사들이 "사용 불가" 판정
- 주요 불만:
  - 진료 중 입력 단계가 너무 많음 (기존 3단계 → 신규 8단계)
  - 의학 용어가 부정확함
  - 응급실 워크플로우가 실제와 다름

**위기 발생:**
- 의사협의회에서 시스템 도입 반대 결의안 통과
- 간호사들도 "현장 의견 무시" 불만 표출
- 병원장이 프로젝트 중단 검토 지시

**PM의 대응 (잘못된 접근):**
- "시스템은 완벽하니 교육으로 해결" 주장
- "의사들이 변화를 거부하는 것" 프레임
- 더 강력한 추진 요구

**결과:**
- 프로젝트 6개월 중단
- 추가 예산 5억원 투입하여 재설계
- PM 교체, 새 PM이 의사/간호사 참여 워킹그룹 구성
- 최종적으로 1년 지연, 예산 50% 초과

**교훈:**
1. **조기 식별:** 모든 이해관계자를 프로젝트 초기에 식별해야 함
2. **실사용자 참여:** 시스템 사용자의 참여가 가장 중요
3. **지속적 소통:** 일회성 인터뷰가 아닌 지속적 참여 필요
4. **권력 vs 영향력:** 병원장(권력 높음)보다 의사들(영향력 높음)이 더 중요할 수 있음
5. **저항 관리:** 기술적 완성도보다 사용자 수용도가 성공 요인

**Q1. 5개 프로세스 그룹을 순서대로 쓰시오.**

모범답안:

**Q2. 이해관계자 관리가 중요한 이유를 2가지 쓰시오.**

1. 프로젝트 성패를 결정: 연구에 따르면 프로젝트 실패의 약 70%가 이해관계자 관리 실패에 기인하며, 주요 이해관계자의 지지가 없으면 프로젝트가 중단될 수 있습니다.

2. 요구사항 충족: 이해관계자의 요구와 기대를 정확히 파악해야 올바른 결과물을 만들 수 있으며, 조기 참여와 지속적 소통으로 변화 저항을 최소화할 수 있습니다.

(추가 가능: 자원 확보, 리스크 관리, 의사결정 지원 등)

(추가 가능: 시정 조치, 예방 조치, 결함 수리 등)

**Q4. 기획 프로세스 그룹에 가장 많은 프로세스가 포함되는 이유는?**

모범답안:
프로젝트의 성공은 철저한 계획에 달려있기 때문입니다. 범위, 일정, 원가, 품질, 자원, 의사소통, 리스크, 조달 등 모든 지식 영역에 대한 상세한 계획을 수립해야 하므로, 49개 프로세스 중 24개(약 49%)가 기획 단계에 집중되어 있습니다. "계획에 실패하는 것은 실패를 계획하는 것"이라는 격언이 이를 잘 설명합니다.

## 4교시: PMP 6판 vs 7판 비교 (2시간)

### 이론 (70분)

#### 1. PMBOK 진화 배경

**PMBOK 발행 역사:**
- 1판 (1996): 130페이지, 기본 프레임워크 제시
- 2판 (2000): 지식영역 체계 확립
- 3판 (2004): 44개 프로세스
- 4판 (2008): 42개 프로세스 (9개 지식 영역 유지)
- 5판 (2013): **이해관계자 관리 영역 추가 (9개 → 10개)**, 47개 프로세스
- **6판 (2017): 49개 프로세스, 애자일 내용 추가**
- **7판 (2021): 원칙 기반 접근으로 전환**

**변화의 배경:**
- 프로젝트 환경의 급격한 변화 (디지털 트랜스포메이션)
- 예측형(Waterfall)과 적응형(Agile) 모두 수용 필요
- 복잡성, 불확실성, 변동성이 증가한 프로젝트 환경
- 프로젝트 관리의 보편적 원칙 강조 필요성

#### 2. PMBOK 6판의 특징

**구조:**
- **프로세스 기반 접근(Process-Based Approach)**
- 5개 프로세스 그룹 × 10개 지식 영역 = 49개 프로세스
- ITTO 구조: Input(Inputs) - Tools & Techniques - Output(Outputs)

**핵심 콘텐츠:**
1. **49개 프로세스 상세 설명**
   - 각 프로세스의 ITTO를 매우 상세히 기술
   - 예: 범위 정의 프로세스의 Input 8개, Tools 5개, Output 3개

2. **예측형(Predictive) 중심**
   - Waterfall 모델을 기본 가정
   - 순차적 프로세스 진행 강조
   - 상세 사전 계획 강조

3. **표준화된 프로세스**
   - 모든 프로젝트에 49개 프로세스를 일관되게 적용
   - 프로젝트 특성에 따른 조정 언급은 제한적

**장점:**
- 체계적이고 명확한 가이드 제공
- PMP 시험 준비에 적합 (명확한 학습 범위)
- 표준화된 방법론으로 팀 간 의사소통 용이
- 조직 프로세스 표준화 기반

**한계:**
- 경직된 프로세스 중심으로 유연성 부족
- 애자일 환경에서 적용 어려움
- 프로세스 수행이 목적이 되는 형식주의 위험
- 변화하는 비즈니스 환경 대응 부족

#### 3. PMBOK 7판의 특징

**구조:**
- **원칙 기반 접근(Principle-Based Approach)**
- 12개 프로젝트 관리 원칙(Principles)
- 8개 성과 도메인(Performance Domains)
- 프로세스 목록 삭제, 원칙과 성과 중심

**A. 12개 프로젝트 관리 원칙**

프로젝트 관리의 보편적인 가치와 기본 원칙:

1. **성실한 스튜어드(Stewardship)**
   - 의미: 책임감 있게 행동, 윤리적 행동
   - 적용: 조직과 사회에 대한 책임성 고려

2. **팀(Team)**
   - 의미: 협력적인 프로젝트 팀 환경 조성
   - 적용: 팀 문화, 다양성, 협업 중시

3. **이해관계자(Stakeholders)**
   - 의미: 이해관계자와 효과적으로 관계 맺기
   - 적용: 적극적 참여 유도, 기대 관리

4. **가치(Value)**
   - 의미: 비즈니스 가치 창출에 집중
   - 적용: 단순한 산출물이 아닌 비즈니스 가치를 목표로

5. **시스템 사고(Systems Thinking)**
   - 의미: 전체론적 관점, 상호작용 이해
   - 적용: 프로젝트를 고립된 실체가 아닌 시스템의 일부로 인식

6. **리더십(Leadership)**
   - 의미: PM의 리더십 발휘
   - 적용: 영향력, 동기부여, 비전 제시

7. **테일러링(Tailoring)**
   - 의미: 프로젝트 특성에 맞게 조정
   - 적용: 하나의 방법이 모든 프로젝트에 맞지 않음

8. **품질(Quality)**
   - 의미: 품질을 프로세스와 인도물에 내장
   - 적용: 일회성 검사가 아닌 지속적 품질 관리

9. **복잡성(Complexity)**
   - 의미: 복잡성을 처리하는 능력
   - 적용: 복잡한 환경에서 적응적 접근

10. **리스크(Risk)**
    - 의미: 불확실성을 적극적으로 관리
    - 적용: 위협과 기회 모두 고려

11. **적응성과 회복력(Adaptability and Resiliency)**
    - 의미: 변화에 빠르게 대응하고 회복
    - 적용: 변화를 수용하고 실패로부터 학습

12. **변화 관리(Change Management)**
    - 의미: 변화를 체계적으로 관리
    - 적용: 조직 변화 관리와 통합

**B. 8개 성과 도메인(Performance Domains)**

프로젝트 활동의 주요 영역:

1. **이해관계자(Stakeholders)**
   - 활동: 식별, 분석, 참여, 감시
   - 결과: 효과적인 이해관계자 참여

2. **팀(Team)**
   - 활동: 팀 문화 구축, 리더십, 동기부여
   - 결과: 고성과 팀 형성

3. **개발 접근법 및 생애주기(Development Approach & Life Cycle)**
   - 활동: 예측형/적응형/하이브리드 선택
   - 결과: 프로젝트에 적합한 접근법 적용

4. **기획(Planning)**
   - 활동: 초기 계획, 진화적 정교화
   - 결과: 효과적인 계획과 적응성 균형

5. **프로젝트 작업(Project Work)**
   - 활동: 작업 실행, 프로세스 관리, 학습
   - 결과: 효율적인 프로젝트 진행

6. **인도(Delivery)**
   - 활동: 인도물 범위, 품질, 수정
   - 결과: 기대치를 충족하는 가치 제공

7. **측정(Measurement)**
   - 활동: 성과 측정, 데이터 분석, 보고
   - 결과: 데이터 기반 의사결정

8. **불확실성(Uncertainty)**
   - 활동: 리스크와 불확실성 관리
   - 결과: 변동성에 효과적으로 대응

**7판의 장점:**
- 유연한 적용: Waterfall과 Agile 모두 수용
- 원칙 중심: 보편적 원칙 적용, 표준 프로세스에 종속되지 않음
- 테일러링 강조: 프로젝트별 맞춤형 접근
- 가치 중심: 프로세스 준수보다 가치 창출
- 현대적 프로젝트 환경에 적합

**7판의 한계:**
- PMP 시험 준비 어려움: 명확한 프로세스 목록 없음
- 초보자에게 어려움: 추상적 원칙을 실무에 적용하는 법 학습 필요
- 조직 표준화 어려움: 명확한 프로세스 없어 표준 정립이 어려울 수 있음

#### 4. 6판 vs 7판 주요 차이점 비교

| 구분 | 6판 | 7판 |
|------|------|------|
| **접근 방식** | 프로세스 기반 (Process-Based) | 원칙 기반 (Principle-Based) |
| **핵심 구조** | 5개 프로세스 그룹, 10개 지식영역, 49개 프로세스 | 12개 원칙, 8개 성과 도메인 |
| **ITTO** | 모든 프로세스에 ITTO 상세 기술 | ITTO 개념 삭제 |
| **방법론** | 예측형(Waterfall) 중심 | 예측형 + 적응형 모두 포괄 |
| **테일러링** | 제한적 언급 | 12가지 원칙 중 하나로 강조 |
| **가치 관점** | 산출물 완성 | 비즈니스 가치 창출 |
| **유연성** | 표준 프로세스 준수 | 상황별 적절한 조정 |
| **페이지 수** | 약 750페이지 | 약 250페이지 |
| **학습 난이도** | 초보자 접근 용이 | 경험자에게 적합 |
| **적용 대상** | 명확한 프로세스 필요 시 | 복잡/불확실 환경 |

#### 5. 현실적 적용 가이드

**주무신** 6판과 7판은 대체 관계가 아니라 **보완 관계**입니다.

**6판 활용이 적합한 경우:**
- 조직 프로세스 표준화가 필요한 경우
- PMP 자격증 취득을 목표로 하는 경우
- 예측형 프로젝트가 대부분인 조직
- 신입 PM 교육 (Step-by-step 가이드 필요)

**7판 활용이 적합한 경우:**
- Agile/하이브리드 프로젝트가 많은 조직
- 빠르게 변화하는 산업/비즈니스 환경
- 복잡성과 불확실성이 높은 프로젝트
- 경험 많은 PM의 전략적 사고 개발

**통합 접근:**
- 6판의 프로세스를 기본 프레임워크로
- 7판의 원칙을 의사결정 가이드로
- 프로젝트 특성에 따라 테일러링

### 예시 (40분)

#### 예시 1: 동일 프로젝트에 6판/7판 적용 비교

**프로젝트:** 전자상거래 플랫폼 신규 개발
**상황:** 급변하는 시장, 경쟁 심화, 고객 요구 불명확

**A. 6판 접근법 적용 (Waterfall)**

**1단계: 기획 (2개월)**
- 49개 프로세스 중 관련 프로세스 모두 수행
- 범위 관리: 6개 프로세스 (범위 계획 → 요구사항 수집 → 범위 정의 → WBS 작성)
- 일정 관리: 6개 프로세스 (일정 계획 → 활동 정의 → 순서 배열 → 기간 산정 → 일정 개발)
- 교훈: 150페이지 상세 계획서

**2단계: 실행 (6개월)**
- 계획대로 순차적 개발
- 변경 요청은 공식 CCB를 통해 승인
- 문제: 3개월차 시장 트렌드 변화 발견 → 변경에 2개월 소요

**3단계: 출시 (8개월차)**
- 출시 시점에 시장 환경이 또 변화
- 경쟁사가 먼저 신기능 출시

**6판 적용 결과:**
- 장점: 체계적인 계획, 모든 프로세스 문서화
- 단점: 시장 변화에 느린 대응, 2개월 지연, 경쟁력 저하
- 결론: **단단하나 유연하지 못함**

**B. 7판 접근법 적용 (Agile/하이브리드)**

**1단계: 초기 계획 (2주)**
- 12가지 원칙 검토: 특히 테일러링, 적응성, 가치 중시
- 8개 성과 도메인 정의
  - 개발 접근법: 스크럼(2주 스프린트)
  - 기획: 목표 수립, MVP 정의, 과도한 상세 계획 회피
  - 측정: 주간 검토, 빠른 피드백
- 교훈: 20페이지 초기 론칭 계획

**2단계: 반복 개발 (6개월, 12개 스프린트)**
- 스프린트 1-2: 핵심 기능 MVP
- 스프린트 3-4: 사용자 피드백으로 피봇 출시
- 스프린트 5: 시장 트렌드 변화 발견 → 현재 스프린트에 반영 (1주 만에 조정)
- 스프린트 6-10: 새로운 기능 추가
- 스프린트 11: 경쟁사 신기능 파악 → 차별화 기능 2주 만에 추가
- 스프린트 12: 안정화 및 출시

**7판 적용 결과:**
- 장점: 빠른 변화 대응, 지속적 가치 제공, 경쟁력 유지
- 단점: 초기 모호함, 문서화 부족
- 결론: **유연하지만 구조가 덜 명확함**

**결론:**
- 6판 접근: 안정적 환경에 적합 (8개월 소요, 유연성 부족)
- 7판 접근: 불확실 환경에 적합 (6개월 소요, 신속 대응)
- 이 프로젝트에는 **7판(Agile) 접근이 적합**

#### 예시 2: 테일러링의 실제 적용 (7판 핵심)

**배경:** 동일 회사에서 2개 프로젝트를 동시 진행

**프로젝트 A: 규제 대응 시스템**
- 특성: 모든 요구사항 명확, 규제 준수 필수, 변경 불가
- 테일러링 결정:
  - 6판 프로세스 모델 채택 (Waterfall)
  - 49개 프로세스 중 관련 프로세스 모두 수행
  - 상세한 문서화 필수 (ITTO 엄격 적용)
  - 강력한 변경 통제
- 결과: 성공적 완료, 감사 통과

**프로젝트 B: 신규 서비스 개발**
- 특성: 요구사항 불명확, 시장 반응 필요, 빠른 출시 중요
- 테일러링 결정:
  - 7판 원칙 기반 접근 (Scrum)
  - 2주 스프린트, 증분 배포
  - 최소 필요 문서화
  - 유연한 변경 수용
- 결과: 성공적 완료, 신속한 시장 대응

**테일러링의 교훈:**
- "하나의 방법이 모든 프로젝트에 맞지 않는다"
- 프로젝트 특성(복잡도, 불확실성, 변동성)에 따라 선택
- 6판과 7판은 도구상자, PM이 선택해야 함

#### 예시 3: 7판 12가지 원칙의 실무적 적용

**상황:** COVID-19 팬데믹으로 인한 재택근무 전환 프로젝트

**원칙 1 - 적응성과 회복력:**
- 상황: 갑자기 사무실 폐쇄, 1주 내 재택근무 전환
- 원칙 적용: 기존 계획 폐기, 신속 대응 계획 수립
- 결과: 3일 만에 처음 VPN 100명 구축, 2주 만에 전직원 재택근무

**원칙 2 - 팀:**
- 상황: 팀원이 물리적으로 분산
- 원칙 적용: 가상 팀 문화 구축, 일일 화상 회의, 온라인 팀 빌딩
- 결과: 팀 응집력 유지, 오히려 의사소통 효율성 증가

**원칙 3 - 테일러링:**
- 상황: 기존 대면 회의 중심 프로세스가 불가능
- 원칙 적용: 비동기 의사결정, 문서 기반 협업 도구 강화
- 결과: 새로운 협업 프로세스 정착

**원칙 4 - 시스템 사고:**
- 상황: 재택근무가 보안, 성과 관리, HR 모두에 영향
- 원칙 적용: 보안/HR/법무팀과 조기 협력 체계 구축
- 결과: 통합 솔루션 도출

**교훈:**
- 6판 프로세스만 따라갔다면 대응 불가능했을 상황
- 7판 원칙(적응성, 시스템 사고)이 위기 관리에 필수
- 원칙은 "무엇을"보다 "왜"과 "어떻게"에 중점

### 퀴즈 (10분)

**Q1. 6판과 7판의 핵심 차이를 2가지 쓰시오.**

모범답안:
1. **접근 방식:** 6판은 프로세스 기반(Process-Based) 접근으로 49개의 명확한 프로세스를 제시하는 반면, 7판은 원칙 기반(Principle-Based) 접근으로 12개 원칙과 8개 성과 도메인을 중심으로 합니다.

2. **유연성:** 6판은 예측형(Waterfall) 방법론을 주로 다루는 반면, 7판은 예측형과 적응형(Agile) 모두를 포괄하며 테일러링(구체적 상황에 맞게 조정)을 강조합니다.

(추가 가능: ITTO 구조, 가치 관점, 표준화 vs 유연성 등)

**Q2. 테일러링의 목적은 무엇인가?**

모범답안:
테일러링은 프로젝트의 고유한 특성(크기, 복잡도, 중요도, 불확실성 등)에 맞게 프로젝트 관리 접근법, 프로세스, 도구를 조정하는 것입니다. 7판에서는 "하나의 방법이 모든 프로젝트에 맞지 않는다"는 인식 하에 12가지 원칙 중 하나로 강조되고 있습니다.

**Q3. 7판의 성과 도메인 중 3개를 쓰시오.**

모범답안: (다음 중 3개)
1. 이해관계자(Stakeholders)
2. 팀(Team)
3. 개발 접근법 및 생애주기(Development Approach & Life Cycle)
4. 기획(Planning)
5. 프로젝트 작업(Project Work)
6. 인도(Delivery)
7. 측정(Measurement)
8. 불확실성(Uncertainty)

**Q4. 6판이 적합한 프로젝트 유형과 7판이 적합한 프로젝트 유형을 각각 1가지 예시로 드시오.**

모범답안:
- **6판 적합:** 규제 준수 시스템 구축 (모든 요구사항이 명확하고 변경이 적으며 상세한 문서화가 필수인 경우)
- **7판 적합:** 신규 스타트업 서비스 개발 (요구사항이 불명확하고 빠른 시장 변화 대응이 필요하며 반복적 개선이 필요한 경우)

## 5교시: PM이 IT 기술을 배워야 하는 이유 (1.5시간)

### 이론 (50분)

#### 1. IT 프로젝트의 특수성

**전통적 프로젝트 vs IT 프로젝트:**
- 전통적 프로젝트(건설, 제조): 물리적 산출물, 명확한 요구사항, 예측 가능한 기술
- IT 프로젝트: 무형의 산출물, 요구사항 변동, 기술 급변, 복잡한 기술 의존성

**IT 프로젝트의 고유 특성:**
1. **빠른 기술 변화**
   - 신기술이 1-2년마다 등장 (클라우드, AI, 컨테이너 등)
   - 레거시 기술과 신기술의 통합 필요
   
2. **무형의 산출물**
   - 소프트웨어는 보이지 않아 진척 파악이 어려움
   - "90% 완료" 증후군: 개발 초반 90%는 빠르고, 나중 10%에 시간이 더 소요

3. **복잡한 기술 의존성**
   - 프론트엔드, 백엔드, DB, 네트워크, 보안 등 레이어 간 의존성
   - 한 영역의 문제가 전체 시스템에 영향
   
4. **높은 불확실성**
   - 기술 선택의 리스크 (성능, 호환성, 보안)
   - 사용자 요구사항이 개발 중 변경

#### 2. PM이 기술을 이해해야 하는 5가지 이유

**이유 1: 정확한 의사결정**

- **상황:** 개발팀이 "마이크로서비스로 전환해야 한다" 제안
- **기술 이해 있을 때:** 
  - 현재 시스템 규모, 트래픽, 팀 역량 고려
  - 마이크로서비스의 장점(확장성, 독립 배포)과 단점(복잡성, 인프라 비용) 파악
  - 데이터 기반 의사결정 (예: 현재 모놀리식도 충분하다면 불필요한 전환 회피)
- **기술 이해 없을 때:**
  - 기술 용어에 압도되어 개발팀 의견을 무조건 수용
  - 또는 반대로 리스크를 이해 못해 무조건 거부
  - 일정 3개월, 비용 5천만원 추가 영향을 예측 못함

**이유 2: 효과적인 의사소통**

- **상황:** 개발팀과 비즈니스 팀 사이의 통역자 역할
- **기술 이해 있을 때:**
  - 개발팀: "API 호출 레이턴시가 300ms라 UX가 나빠요"
  - PM이 비즈니스에 전달: "사용자가 버튼 클릭해서 결과 나오기까지 0.3초인데, 목표는 0.1초입니다. 캐싱 전략으로 개선하면 1주 소요합니다."
- **기술 이해 없을 때:**
  - 개발팀 용어를 비즈니스 팀에 그대로 전달 → 혼란
  - 또는 기술 세부 내용을 묻지 못해 요구사항 누락

**이유 3: 현실적인 일정 산정**

- **상황:** "사용자 인증 기능 추가"
- **기술 이해 있을 때:**
  - 질문: "자체 인증? 소셜 로그인? 2FA 필요?"
  - 세부 분석: OAuth 통합 (2일), UI 개발 (2일), 테스트 (1일)
  - 추가 고려: 세션 관리, 보안 취약점 검토
  - **산정: 5일**
- **기술 이해 없을 때:**
  - "로그인 화면 하나 추가하는 거 아닌가요? 1일이면 충분"
  - **산정: 5배 차이 → 일정 지연**

**이유 4: 기술 리스크 조기 식별**

- **상황:** DB에 1억 건 데이터 저장 예정
- **기술 이해 있을 때:**
  - 질문: "인덱스 전략은? 샤딩 필요? 백업/복구 시간은?"
  - 리스크 식별: 쿼리 성능 저하, 백업 시간 8시간 소요
  - 완화 전략: 파티셔닝, 증분 백업 설계
- **기술 이해 없을 때:**
  - 개발 후반에 성능 문제 발견
  - 근본적 재설계 필요 → 3개월 지연

**이유 5: 이해관계자 신뢰 확보**

- **상황:** 경영진이 "왜 3개월이나 걸리나?"라고 질문
- **기술 이해 있을 때:**
  - "백엔드 API 개발 1개월, 프론트엔드 통합 1개월, 성능 테스트와 보안 감사 1개월입니다. 특히 보안 감사는 금융권 규제 준수 필수입니다."
  - 경영진 납득 → 신뢰 확보
- **기술 이해 없을 때:**
  - "개발팀이 그렇게 말했습니다" → 전문성 의심

#### 3. 본 교육에서 다룰 IT 기술 영역과 PM 관점 중요성

**Day 12: 소프트웨어 공학 (Software Engineering)**
- **내용:** SDLC, 설계 원칙, 테스트 전략, 형상 관리, DevOps
- **PM 관점 중요성:**
  - SDLC 이해 → 프로젝트 생애주기 선택 (Waterfall vs Agile)
  - 테스트 전략 → 품질 계획, 결함 관리
  - DevOps → CI/CD 파이프라인, 배포 빈도와 안정성

**Day 13: 네트워크 (Network)**
- **내용:** OSI 7계층, TCP/IP, HTTP/HTTPS, DNS, 로드밸런서
- **PM 관점 중요성:**
  - 네트워크 아키텍처 → 인프라 비용, 확장성 계획
  - 대역폭/레이턴시 → 성능 요구사항 검증
  - DNS, CDN → 글로벌 서비스 전략

**Day 14: 보안 (Security)**
- **내용:** 인증/인가, 암호화, SQL Injection, XSS, OWASP Top 10
- **PM 관점 중요성:**
  - 보안 요구사항 → 규제 준수, 감사 대응
  - 취약점 관리 → 리스크 관리 프로세스
  - 보안 테스트 → 일정 계획 (보안 감사 2-3주 소요)

**Day 15: 데이터베이스 (Database)**
- **내용:** RDBMS, SQL, 정규화, 인덱스, 트랜잭션, NoSQL
- **PM 관점 중요성:**
  - DB 설계 → 성능, 확장성 영향
  - 인덱스/쿼리 최적화 → 대용량 데이터 성능 리스크
  - RDBMS vs NoSQL 선택 → 비용, 기술 스택

**Day 16: 신기술 (New Technologies)**
- **내용:** 클라우드, 컨테이너, MSA, AI/ML, 블록체인
- **PM 관점 중요성:**
  - 클라우드 마이그레이션 → 비용 모델 변화 (CapEx → OpEx)
  - 컨테이너/MSA → 아키텍처 복잡성, 팀 구조
  - AI/ML → 데이터 품질, 모델 학습 시간

#### 4. PM 학습의 균형

**오해: PM은 개발자 수준으로 기술을 알 필요 없다?**
- 맞습니다. PM은 코드를 작성할 필요는 없습니다.

**핵심: PM은 "의사결정에 필요한 수준"의 기술 이해가 필요**

**학습 수준 비교:**

| 영역 | 개발자 수준 | PM 필요 수준 |
|------|------------|-------------|
| **프로그래밍** | Python, Java 능숙 | 주요 언어의 장단점, 생태계 파악 |
| **네트워크** | 패킷 분석, 라우팅 설정 | HTTP/HTTPS 동작, 병목 지점 파악 |
| **DB** | 복잡한 쿼리 작성, 튜닝 | 인덱스 전략, 정규화 영향 이해 |
| **보안** | 취약점 패치 코드 작성 | OWASP Top 10, 보안 감사 프로세스 |
| **클라우드** | Kubernetes 운영 | 클라우드 비용 모델, 서비스 종류 |

**PM의 기술 학습 전략:**
1. **넓게 알기:** 모든 기술 영역의 기본 개념
2. **맥락 이해:** 기술 선택이 일정/비용/품질에 미치는 영향
3. **질문 능력:** 개발팀에 올바른 질문을 던져 리스크 파악
4. **트레이드오프 판단:** 기술 결정의 장단점 비교

### 예시 (25분)

#### 예시 1: 기술 이해 부족으로 인한 프로젝트 실패

**프로젝트:** 온라인 쇼핑몰 리뉴얼
**문제 상황:**

**Week 1:**
- 개발팀: "백엔드는 Node.js로 하겠습니다."
- PM (기술 이해 없음): "네, 좋습니다." (근거 없이 승인)

**Week 8:**
- 개발팀: "동시 접속 1만 명 처리가 어렵습니다. 서버를 10대 더 추가해야 합니다."
- PM: "왜요? 처음에는 가능하다고 했잖아요?"
- 개발팀: "Node.js는 싱글 스레드라 CPU 집약 작업에 약합니다. 이미지 처리 때문에 병목입니다."

**결과:**
- 일정 2개월 지연 (아키텍처 재설계)
- 비용 3천만원 추가 (서버 증설)
- PM의 신뢰도 하락

**만약 PM이 기술을 이해했다면:**
- Week 1에 질문: "Node.js를 선택한 이유는? 이미지 처리 같은 CPU 작업이 많은데 괜찮나?"
- 대안 검토: Python + Celery (비동기 작업), Go (고성능), 또는 이미지 처리는 별도 서비스로 분리
- 초기 아키텍처 결정 단계에서 리스크 완화

#### 예시 2: 기술 이해로 일정 단축과 비용 절감

**프로젝트:** 고객 데이터 분석 대시보드
**문제 상황:**

**초기 제안:**
- 개발팀: "실시간 대시보드를 만들려면 Kafka + Spark Streaming + Redis가 필요합니다. 구축에 3개월, 비용 1억원입니다."

**PM의 기술 이해 기반 질문:**
- PM: "데이터 갱신이 실시간이어야 하나요? 사용자가 기대하는 '실시간'은 몇 초 단위인가요?"
- 비즈니스 팀: "사실 5분 지연은 괜찮습니다. 중요한 건 어제와 오늘 비교입니다."

**대안 제시:**
- PM: "그렇다면 배치 처리 (5분마다 집계)로도 충분합니다. 기존 DB + 캐싱으로 가능합니다."
- 개발팀: "그러면 2주 만에 구축 가능하고, 추가 비용은 거의 없습니다."

**결과:**
- 일정 단축: 3개월 → 2주
- 비용 절감: 1억원 → 0원

### 퀴즈 (10분)

**Q1. 기술 이해가 일정 산정에 도움이 되는 이유를 구체적 예시와 함께 설명하시오.**

모범답안:
PM이 기술을 이해하면 세부 작업을 정확히 분해하고 각 작업의 난이도를 평가할 수 있습니다.

예시: "사용자 인증 기능 추가" 요구사항이 있을 때,
- 기술 이해 없음: "로그인 화면 하나 추가 = 1일"
- 기술 이해 있음:
  - 자체 인증 vs OAuth 소셜 로그인 선택에 따라 2일 vs 3일
  - 2FA(Two-Factor Authentication) 추가 시 +2일
  - 세션 관리 (Redis 캐시 사용) +1일
  - 보안 테스트 (SQL Injection, XSS 방어) +1일
  - **실제 산정: 5-7일**

결과적으로 기술 이해가 있으면 일정 산정 정확도가 5배 이상 향상되어 프로젝트 지연을 방지할 수 있습니다.

**Q2. PM이 알아야 할 IT 영역 3가지를 쓰고, 각 영역이 프로젝트 관리에 미치는 영향을 1가지씩 설명하시오.**

모범답안: (다음 중 3개 선택)

1. **소프트웨어 공학**
   - 영향: SDLC 이해 → 프로젝트 생애주기 선택 (Waterfall vs Agile), 테스트 전략 수립, 결함 관리 프로세스

2. **네트워크**
   - 영향: 대역폭/레이턴시 이해 → 성능 요구사항 검증, 글로벌 서비스 시 CDN/멀티 리전 필요성 판단

3. **보안**
   - 영향: 보안 감사 프로세스 이해 → 일정 계획 (감사에 2-3주 소요), 규제 준수 요구사항 반영

4. **데이터베이스**
   - 영향: 인덱스 전략, 정규화 이해 → 대용량 데이터 성능 리스크 조기 발견, 백업/복구 시간 계획

5. **신기술 (클라우드, 컨테이너 등)**
   - 영향: 클라우드 비용 모델 이해 → 예산 계획 (CapEx vs OpEx), 확장성 전략

**Q3. PM의 기술 학습 수준과 개발자의 기술 수준은 어떻게 다른가?**

모범답안:
- **개발자:** 특정 기술을 깊이 있게 학습하여 실제 코드를 작성하고 시스템을 구현하는 수준
- **PM:** 여러 기술 영역을 넓게 이해하여 기술 선택이 일정/비용/품질에 미치는 영향을 판단하고 올바른 질문을 던질 수 있는 수준

예시:
- 개발자: Python으로 복잡한 알고리즘 구현 가능
- PM: Python vs Java vs Go의 장단점과 프로젝트 특성에 맞는 선택 기준 파악

핵심: PM은 "의사결정에 필요한 수준"의 기술 이해가 목표입니다.

- 예산: 3억원
- 팀: 개발 10명, 디자인 2명, QA 2명

**제공되는 정보:**
- 비즈니스 요구사항 문서 (5페이지)
- 조직 구조 및 이해관계자 명단
- 기존 시스템 현황
- 주요 제약 조건 (일정, 예산, 기술 스택)

**여러분이 작성할 산출물:**

1. **프로젝트 헌장 (Day 2 학습 내용)**
   - 프로젝트 목적 및 정당성
   - 고수준 요구사항 5가지
   - 주요 이해관계자 명단
   - 성공 기준 3가지

   - 3단계 분해 (프로젝트 → 단계 → 작업 패키지)

   - 각 작업에 담당자 배정

3. **일정표 (Day 4 학습 내용)**
   - Gantt Chart 형식
   - 주공정선 식별
   - 마일스톤 5개 설정

4. **예산 (Day 5 학습 내용)**
   - 인건비, 인프라 비용, 라이선스 비용 분리
   - 예비비 10% 포함
   - 월별 예산 배분

5. **리스크 등록부 (Day 9 학습 내용)**
   - 최소 10개 리스크 식별
   - 확률 × 영향도 계산
   - 상위 5개 리스크에 대응 전략

- 완성도: 모든 항목이 빠짐없이 작성되었는가?
- 논리성: 일정과 예산이 현실적인가? 리스크 대응이 적절한가?
- 통합성: WBS-일정-예산-리스크가 일관성 있게 연결되는가?

**교훈:**
- 종합 실습을 통해 11일간 학습한 내용이 하나의 프로젝트로 통합됩니다.

- 예: "EVM의 PV, EV, AC"를 "여행 비유"로

  - EV (획득 가치): 실제로는 대전까지 도착 (300km)

**Q2. 실습 중심 학습의 장점을 2가지 쓰시오.**

모범답안:
1. **능동적 학습:** 단순히 이론을 듣는 수동적 학습이 아니라, 직접 산출물을 작성하면서 개념을 체화할 수 있습니다. 예를 들어 WBS를 직접 만들어보면서 작업 분해의 원리를 깊이 이해하게 됩니다.

2. **실무 적용 준비:** 실제 프로젝트 시나리오로 실습하므로, 교육 종료 후 바로 실무에 적용할 수 있습니다. 종합 실습의 헌장, WBS, 일정, 예산, 리스크 통합 경험은 실제 PM 업무의 축소판입니다.

**Q3. Day 2부터 학습 효과를 높이기 위한 방법 3가지를 제시하시오.**

모범답안:
1. **사전 준비:** 전날 학습 내용을 10분간 복습하고, 다음 날 주제의 주요 용어를 미리 파악하여 수업 중 이해도를 높입니다.
