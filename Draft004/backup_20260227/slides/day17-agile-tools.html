애자일 개요 & 선언문

왜 애자일이 등장했는가?

❌ 워터폴의 함정 — 실제 사례

🏦 모바일 뱅킹 앱 재구축 프로젝트 (2018)

12개월 워터폴 → 런칭 당일 "이게 아닌데요…"
1,500개 버그 + 6개월 지연 + 예산 150%

📉 워터폴 12개월
Month 1–3: 요구사항 (200페이지)
Month 4–9: 개발 (고객 참여 없음)
Month 10: "이게 아닌데요!" 발견
Month 11–12: 긴급 재작업
→ 고객이 처음 본 건 10개월째
✅ 애자일 2주 스프린트
Sprint 1: 로그인 → 고객 시연 → 피드백
Sprint 2: 계좌 조회 → "이게 맞아요!"
Sprint 3: 이체 기능…
→ 매 2주마다 고객이 확인

📜 애자일 선언문 (Agile Manifesto, 2001)

4가지 핵심 가치

오른쪽도 가치 있지만, 왼쪽을 더 가치 있게 여긴다

✅ 더 가치 있는 것 vs 덜 강조하는 것
개인과 상호작용 > 프로세스와 도구
작동하는 소프트웨어 > 포괄적인 문서
고객과의 협력 > 계약 협상
변화에 대응 > 계획을 따르기

⚖️ 워터폴 vs 애자일 — 언제 무엇을?

구분 워터폴 애자일
요구사항 초기 확정 점진적 발견
변경 통제 대상 (비용↑) 환영 대상
인도 주기 프로젝트 종료 시 매 2주
고객 참여 초기·종료 시 매 스프린트
적합 상황 요건 명확·규제 산업 불확실·혁신 프로젝트
💡 40대 PM 인사이트

결재 문화가 있어도 괜찮습니다. 팀 내부는 스크럼으로 개발하고, 경영진 보고는 워터폴 마일스톤 형식으로. 이것이 현실적인 하이브리드 접근입니다.

스크럼 프레임워크

3역할 · 5이벤트 · 3산출물

👥 스크럼 3가지 역할

🎯
Product Owner (PO)
무엇을 만들까?
• 제품 비전 제시
• 백로그 관리·우선순위
• Sprint Review 인수 판단
• ROI 최대화

⚠️ PO ≠ PM: 역할 충돌 주의
🛡️
Scrum Master (SM)
어떻게 일하나?
• 스크럼 프로세스 코치
• 장애물(Impediment) 제거
• 팀 외부 방해 차단
• 서번트 리더십

명령이 아닌 섬김과 지원
⚙️
개발 팀
어떻게 만드나?
• 교차기능(설계·개발·테스트)
• 자기조직화
• 3–9명 권장
• 스프린트 목표 달성

10명↑ → 팀 분리 고려

📅 스크럼 5가지 이벤트

🏃 스프린트 (Sprint)
1–4주 타임박스. 일반적으로 2주 권장
📋 스프린트 계획 (Sprint Planning)
스프린트 목표 설정 + 백로그 선정 + 작업 분해
2주 스프린트 기준 4시간
☀️ 일일 스크럼 (Daily Scrum)
15분 스탠딩
①어제? ②오늘? ③블로커?
⚠️ 상태보고 회의가 아님
🎉 스프린트 리뷰 (Sprint Review)
완성 증분 직접 시연 (PPT 금지!)
이해관계자 피드백 → 백로그 업데이트
2주 기준 2시간
🔄 스프린트 회고 (Sprint Retrospective)
Start / Stop / ✅ Continue
프로세스 개선 액션 아이템 도출
2주 기준 1.5시간

📝 사용자 스토리 & INVEST 기준

사용자 스토리 형식

As a [사용자 유형],
I want [기능],
So that [가치/이유].

I — Independent
다른 스토리에 독립적
N — Negotiable
구현 방법은 협상 가능
V — Valuable
사용자/비즈니스 가치
E — Estimable
크기 추정 가능
S — Small
스프린트 내 완료 가능
T — Testable
인수 기준 명확
스토리 포인트 (피보나치)

1 · 2 · 3 · 5 · 8 · 13 · 21 — 기준 스토리(3pt): "로그인 기능 구현"
13pt 이상이면 사용자 스토리를 더 작게 분해하세요.

칸반 & 하이브리드

흐름 최적화 · WIP 제한 · 현실적 방법론

📊 칸반 보드 & WIP 제한

┌──────────┬──────────┬────────────┬───────────┬──────────┐
│ Backlog  │  To Do   │ In Progress│  Review   │  Done    │
│          │          │  (WIP: 3)  │  (WIP: 2) │          │
├──────────┼──────────┼────────────┼───────────┼──────────┤
│ [고객관리]│ [결제#1] │ [검색#3]   │ [로그인#1]│ [회원#2] │
│ [리포트] │ [배송#2] │ [알림#4]   │ [장바구니]│ [상품#5] │
│ [통계]   │          │ ~~WIP꽉~~  │           │ ...15개  │
└──────────┴──────────┴────────────┴───────────┴──────────┘
❌ WIP 제한 없을 때
10개 작업 병렬 진행
→ 컨텍스트 스위칭 비용 누적
→ 완료된 것이 없음
→ 고객은 아무것도 못 받음
✅ WIP 제한 있을 때
최대 3개만 진행
→ 1개 완료 → Done → 다음 시작
→ 리드타임 단축
→ 예측 가능성 향상

⚖️ 스크럼 vs 칸반 — 언제 무엇을?

구분 스크럼 칸반
역할 PO / SM / 팀 정의 기존 역할 유지
주기 스프린트 고정 연속적 (없음)
변경 허용 스프린트 중 제한 언제든 가능
성과 지표 벨로시티 리드타임, 사이클타임
적합 상황 반복적 신기능 개발 운영·유지보수·지원
하이브리드 (Water-Scrum-Fall) — 가장 현실적

Phase1 계획·설계 (워터폴) → Phase2 개발 (스크럼 스프린트) → Phase3 인수·오픈 (워터폴)

Jira 실습

스크럼 보드 · 번다운 차트 · JQL

🔧 Jira 핵심 개념

이슈 유형 계층

Epic (에픽 — 큰 기능)
  └─ Story (사용자 스토리)
      └─ Sub-task (작업)
  └─ Bug (버그)
  └─ Task (기타 작업)

스크럼 보드 설정 단계

  1. 프로젝트 생성 (소프트웨어 → 스크럼)
  2. 제품 백로그 입력 (사용자 스토리 형식)
  3. 스프린트 생성 (2주 기간 설정)
  4. 백로그 → 스프린트 이동
  5. 스프린트 시작 → 보드 확인
  6. 번다운 차트 자동 생성 확인

🔍 JQL & 보고서 활용

자주 쓰는 JQL

# 내 작업 (오픈 스프린트)
assignee = currentUser()
AND sprint in openSprints()

# 블로킹된 이슈 (3일 이상 방치)
status = "In Progress"
AND updated < -3d

# 이번 주 완료
status = Done
AND resolved >= startOfWeek()

보고서 종류 & PM 활용

번다운 차트
스프린트 진행률 → 지연 조기 발견
번업 차트
누적 완료 vs 범위 → 범위 크리프 감지
속도(Velocity) 차트
스프린트별 완료 pt → 향후 계획 기준
스프린트 보고서
완료/미완료 항목 → 스폰서 보고

MS Project & Trello

간트차트 · EVM · 칸반 도구

📊 MS Project — 간트차트 & EVM

핵심 기능

  • 간트차트: 작업·기간·의존성 시각화
  • 크리티컬 패스: 빨간색으로 자동 표시
  • 자원 배정: 과부하(빨간) 자동 경고
  • 기준선(Baseline): 원래 계획 저장 → 비교
  • EVM 보고서: PV·EV·AC 자동 계산
EVM 핵심 지표 복습

CPI = EV/AC  (1 이상 = 예산 절감)
SPI = EV/PV  (1 이상 = 일정 순조)

실습 순서

  1. 작업 목록 입력 (7개 단계)
  2. 의존성 설정 (FS 관계)
  3. 크리티컬 패스 표시 활성화
  4. 자원 시트 생성 + 배정
  5. 과부하 확인 → 자원 평준화
  6. 기준선 0 저장
  7. 진척도 입력 → EVM 보고서

🗺️ PM 도구 선택 가이드

상황 추천 도구 이유
스크럼 팀 (5–50명) Jira 스프린트/번다운/백로그 통합
워터폴·대형 프로젝트 MS Project 간트/EVM/자원 관리 강력
소규모·개인 업무 (≤5명) Trello 단순·직관적 칸반, 무료
문서+PM 통합 필요 Jira + Confluence 이슈↔문서 완전 연동
유연한 도구 (비IT 부서) Notion / Asana 직관적 UI, 다목적
결론: 도구는 방법론을 따라야 한다

먼저 방법론(스크럼/칸반/워터폴)을 결정하고, 그에 맞는 도구를 선택하세요.
도구가 방법론을 결정하는 역방향이 되면 안 됩니다.
