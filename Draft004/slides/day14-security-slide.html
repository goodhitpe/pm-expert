정보보안 개요 & CIA Triad

기밀성 · 무결성 · 가용성 · 추가 보안 속성

⏱ 1.5시간

정보보안 3요소 — CIA Triad

🔒 기밀성 (Confidentiality)

인가된 사용자만 정보에 접근 가능

구현 방법:

• 암호화 (AES, RSA)
• 접근 통제 (ACL)
• 정보 분류 체계
• 사용자 인증

침해 사례: 데이터 유출
미승인 데이터베이스 접근

✅ 무결성 (Integrity)

비인가된 변경 없이 정확하고 완전하게 유지

구현 방법:

• 해시 함수 (SHA-256)
• 디지털 서명
• 체크섬
• 감사 로그

침해 사례: 데이터 변조
SQL Injection으로 DB 수정

⚡ 가용성 (Availability)

인가된 사용자가 필요할 때 접근 가능

구현 방법:

• 이중화 (HA Cluster)
• 백업 및 복구
• DR(재해복구)
• DDoS 방어

침해 사례: DDoS 공격
서버 다운으로 서비스 불가

추가 보안 속성: 인증(Authentication — 신원확인) / 인가(Authorization — 권한부여) / 부인방지(Non-repudiation) — 행위 부인 방지 / 책임추적성(Accountability)

암호화 방식 비교 & 인증 기술

암호화 3가지 방식

방식특징알고리즘용도
대칭키
Symmetric		 동일 키로
암호화·복호화
빠름		 AES, DES 대용량
데이터 암호화
비대칭키
Asymmetric		 공개키·개인키 쌍
느림
키 배포 용이		 RSA, ECC 키 교환
디지털 서명

Hash		 단방향 변환

패스워드 저장
HTTPS 동작 원리
① 서버가 공개키(인증서) 제공
② 클라이언트가 대칭 세션키를 공개키로 암호화 전송
③ 이후 세션키(대칭키)로 빠르게 통신
→ 비대칭키의 안전성 + 대칭키의 속도 결합
PKI (Public Key Infrastructure)
CA(인증기관): 공개키 인증서 발급·관리
인증서 = 공개키 + 소유자 정보 + CA 서명
HTTPS 자물쇠 = 인증서가 유효함을 의미

인증 기술 3요소 (MFA)

인증 요소예시
지식 기반 (What you know)패스워드, PIN, 보안 질문
소유 기반 (What you have)OTP, 보안카드, 스마트카드
생체 기반 (What you are)지문, 홍채, 안면 인식
MFA (다중 인증): 2가지 이상 인증 요소 조합
예: 패스워드 + OTP = 2요소 인증(2FA)
→ 패스워드 유출이 되어도 추가 방어선 확보
PM: 중요 시스템은 MFA 필수 요구사항으로 명시

보안 관리 체계 & 접근 통제 정책

ISMS (정보보호 관리체계)
한국 인증 제도 (KISA 주관)
ISMS: IT 보안 중심
ISMS-P: 개인정보보호까지 포함

관리적 · 기술적 · 물리적 보안 3분야
ISO 27001
국제 정보보안 관리 표준 (PDCA 기반)
14개 도메인, 114개 통제 항목

보안 정책 계층:
정책(Policy) → 표준(Standard) → 지침(Guideline) → 절차(Procedure)

접근 통제 3방식

방식결정 주체특징

임의적 접근통제		 소유자 재량 유연하나 보안 취약
일반 파일 시스템
MAC
강제적 접근통제		 시스템(정책) 엄격한 레벨 기반
군사·정부 시스템
RBAC
역할 기반 접근통제		 역할(Role) 실무에서 가장 일반적
직책별 권한 부여
보안 핵심 원칙 2가지
① 최소 권한 원칙(Least Privilege): 업무에 필요한 최소한의 권한만 부여
② 직무 분리(Separation of Duties): 중요 작업을 한 사람이 혼자 수행 금지
예: 개발자와 배포 담당자 분리

BCP/DR — RPO & RTO

RPO (Recovery Point Objective)
데이터 손실 허용 시점
"몇 시간 전 데이터까지 잃어도 되는가?"
RPO = 1시간 → 1시간마다 백업 필요

RTO (Recovery Time Objective)
서비스 복구 목표 시간
"장애 후 몇 시간 내 복구가 목표인가?"
RTO = 4시간 → 4시간 내 서비스 재개 목표
보안 사고 대응 6단계
준비 → 식별 → 억제 → 제거 → 복구 → 교훈
Preparation → Identification → Containment
→ Eradication → Recovery → Lessons Learned

학습 확인 퀴즈

Q1. CIA Triad의 각 요소를 정의하고 IT 시스템 침해 사례를 하나씩 예시로 드시오.

▶ 모범 답안

기밀성(C): 인가된 사용자만 접근 가능 / 침해 사례: 해킹으로 고객 DB 유출
무결성(I): 비인가된 변경 없이 정확·완전 유지 / 침해 사례: SQL Injection으로 DB 데이터 변조
가용성(A): 인가된 사용자가 필요할 때 접근 가능 / 침해 사례: DDoS 공격으로 서비스 마비

Q2. 대칭키 암호화와 비대칭키 암호화의 차이점과 HTTPS에서 두 방식이 어떻게 함께 사용되는지 설명하시오.

▶ 모범 답안

대칭키: 동일 키로 암호화·복호화. 빠르지만 키 배포 문제 (AES)
비대칭키: 공개키로 암호화, 개인키로 복호화. 느리지만 키 안전 배포 가능 (RSA)

HTTPS 결합 방식:
① 서버가 비대칭키 공개키(인증서) 제공
② 클라이언트가 대칭 세션키를 공개키로 암호화 전송 (비대칭키 = 키 교환용)
③ 이후 세션키(대칭키 AES)로 빠른 통신
→ 비대칭키의 안전성 + 대칭키의 속도를 동시에 확보

Q3. RPO와 RTO의 차이를 설명하고, PM이 프로젝트 기획 단계에서 이를 어떻게 요구사항으로 정의해야 하는지 서술하시오.

▶ 모범 답안

RPO(Recovery Point Objective): 데이터 손실 허용 시점 — "몇 시간 전 데이터까지 잃어도 되는가?"
RTO(Recovery Time Objective): 서비스 복구 목표 시간 — "장애 후 몇 시간 내 복구가 목표인가?"

PM 요구사항 정의 방법:
① 이해관계자(비즈니스 담당자, CIO)와 RPO/RTO 목표값 합의 (0시간 RPO는 매우 비용이 높음)
② NFR(비기능 요구사항)으로 명시: "시스템은 RPO 1시간, RTO 4시간을 보장해야 한다"
③ SLA에 의무적으로 포함하고 위반 시 패널티 조항 추가
④ RFP에 DR 구축 요구사항으로 명시 (Day10 조달 연계)
