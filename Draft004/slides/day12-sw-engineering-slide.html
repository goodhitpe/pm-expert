요구사항 분석

User Story · MoSCoW · Kano 모델 · 유스케이스

⏱ 1시간

요구사항 분석 기법

User Story 형식
"As a [사용자 유형],
I want [기능/목표],
so that [가치/이유]"

예: "As a 구매 담당자,
I want 견적서를 PDF로 출력하기,
so that 협력사에 즉시 공유할 수 있다"

MoSCoW 우선순위

분류의미
Must Have반드시 포함 — 없으면 프로젝트 실패
Should Have포함해야 하지만 없어도 최소 운영 가능
Could Have있으면 좋음 — 시간 여유 시 포함
Won't Have (this time)이번 버전 제외 — 다음 릴리스로 이연

Kano 모델 — 요구사항 3유형

매력적 요구사항 (Delighter)
없어도 불만 없지만, 있으면 만족도 폭등
예: AI 추천 기능, 자동 완성
성능 요구사항 (One-Dimensional)
많을수록 만족도 상승 (선형 관계)
예: 처리 속도, 정확도
기본 요구사항 (Must-Be)
없으면 극단적 불만족, 있어도 기대치 이상 아님
예: 로그인 기능, 데이터 저장
💡 PM을 위한 활용법:
• Must-Be 먼저 확보 (기본 충족)
• One-Dim 최적화 (경쟁력)
• Delighter 1~2개 발굴 (차별화)

설계 원칙 & 소프트웨어 테스팅

결합도·응집도 · 아키텍처 패턴 · 테스트 4수준 · V-모델

⏱ 2.5시간

소프트웨어 설계 원칙

높은 결합도 (Bad)

모듈 A가 바뀌면
모듈 B도 변경 필요

유지보수 어려움

테스트 복잡도 증가

낮은 결합도 (Good)

각 모듈 독립적 변경

유지보수 용이

재사용성 높음

설계 목표: 낮은 결합도 + 높은 응집도
결합도(Coupling): 모듈 간 의존도 → ↓ 낮아야 좋음
응집도(Cohesion): 모듈 내부 관련성 → ↑ 높아야 좋음

아키텍처 패턴 (PM 관점)

복잡도·비용 증가
3계층 표현-비즈니스-데이터 표준적, 역할 분리 명확
MVC Model-View-Controller UI와 로직 분리

Q1. 워터폴과 애자일의 차이점 4가지를 비교하시오. 각 방법론이 적합한 프로젝트 유형도 함께 설명하시오.

▶ 모범 답안

①진행 방식: 워터폴=순차(선형), 애자일=반복적 증분
②요구사항: 워터폴=초기 확정 후 변경 어려움, 애자일=변경 환영
③고객 참여: 워터폴=초기·종료만, 애자일=Sprint마다 참여
④결과물: 워터폴=완성 후 납품, 애자일=매 Sprint 동작 SW
워터폴 적합: 요구사항 명확하고 변경 없는 곳 (국방·금융 규제 프로젝트)
애자일 적합: 요구사항 불명확하고 시장 변화 빠른 곳 (스타트업·앱 개발)

Q2. 기능 요구사항(FR)과 비기능 요구사항(NFR)을 각 3가지 예시와 함께 설명하고, PM이 NFR을 계약에 명시해야 하는 이유를 설명하시오.

▶ 모범 답안

FR 예시: 사용자 로그인 / 주문 처리 / 보고서 생성
NFR 예시: 응답시간 2초 이내 / 99.9% 가용성 / HTTPS 통신 필수
NFR을 계약에 명시해야 하는 이유:
NFR은 측정 가능한 기준이 없으면 "빠르게"처럼 모호한 언어가 됩니다.
계약에 "응답시간 2초 이내" 명시 → 인수 테스트 시 객관적 Pass/Fail 기준 제공
NFR 미명시 시 분쟁의 원인이 되며, Day 06 품질 관리의 품질 기준과 연계됩니다.

Q3. V-모델에서 요구분석 단계와 대응하는 테스트 수준은 무엇이며, PM 관점에서 어떤 의미를 갖는가?

▶ 모범 답안

요구분석 단계 ↔ 인수 테스트(Acceptance Test)
PM 관점 의미:
① 인수 테스트 기준은 요구사항 분석 단계에서 이미 정의되어야 합니다
② UAT(User Acceptance Test) 일정을 WBS 초기부터 계획에 포함
③ 인수 기준이 불명확하면 프로젝트 종료 시 고객 분쟁 발생
→ PM은 프로젝트 시작 직후 이해관계자와 인수 기준을 합의·문서화해야 합니다
