품질의 기본 개념

품질(Quality) 정의
고유 특성들이 요구사항을 충족시키는 정도
낮은 품질 = 항상 문제 ❌

품질 (Quality)

요구사항 충족도
낮으면 항상 문제!

예: "버그가 많다"
= 품질 낮음

등급 (Grade)

기능 범주의 수준
낮아도 OK (설계 의도)

예: 이코노미석
= 낮은 등급이지만
품질 불량 아님

정확도 (Accuracy)

측정값이
참값에 가까운 정도

예: 목표 근처에 맞힘

정밀도 (Precision)

반복 측정의
일관성/재현성

예: 같은 곳에 맞히지만
목표와 멀 수도

→ 품질 관리 계획서

프로세스가 올바른가? 감사, 개선 활동
→ 품질 보고서, 테스트·평가 문서
8.3 품질 통제 (Control Quality)

2교시

품질 관리 계획 수립 & COQ

⏱ 1.5시간

COQ — 품질 비용 (Cost of Quality)

✅ 적합 비용 (Cost of Conformance)
품질 달성을 위해 투자하는 비용
예방 비용
(Prevention)
• 교육 훈련
• 프로세스 문서화
• 장비 교정
• 품질 계획 활동
평가 비용
(Appraisal)
• 검사·테스팅
• 품질 감사
• 동료 검토
• 표본 추출
❌ 부적합 비용 (Cost of Nonconformance)
품질 실패로 발생하는 비용
내부 실패
(Internal Failure)
• 재작업 (Rework)
• 폐기 (Scrap)
• 불량 수정
• 재검사 비용
외부 실패
(External Failure)
• 고객 보증 수리
• 소송/벌금
• 신뢰 상실
• 브랜드 손상
💡 핵심 원칙
예방 비용 ↑ → 내·외부 실패 비용 ↓↓
"예방 1원 = 수정 10원 절감"

크로스비: "Quality is Free"
예방에 투자한 비용 < 실패 비용
∴ 처음부터 올바르게 (Do it right the first time)

💡 IT 프로젝트 COQ 예시

비용 유형IT 사례
예방코드 리뷰 문화, 자동화 테스트 구축
평가단위/통합 테스트, 보안 감사
내부 실패프로덕션 배포 전 발견된 버그 수정
외부 실패고객 환경 장애, 데이터 유출 보상
3교시

품질 보증(QA) vs 품질 통제(QC)

프로세스 중심 vs 제품 중심 · 예방적 vs 탐지적 · 품질 감사 · 검사

⏱ 1.5시간

품질 보증(QA) vs 품질 통제(QC)

품질 보증
(QA / Quality Assurance)

예방적 (Proactive)
  • 초점: 프로세스
  • 질문: "올바른 방법으로 하고 있나?"
  • 시기: 실행 중 지속적으로
  • 활동: 품질 감사, 프로세스 분석, 표준 준수 검토
  • 책임: QA 팀 / 외부 감사
IT 예시:
• CI/CD 파이프라인 점검
• 코드 리뷰 프로세스 감사
• ISO 9001 인증 심사

품질 통제
(QC / Quality Control)

탐지적 (Reactive)
  • 초점: 제품/인도물
  • 질문: "결과물이 기준을 충족하나?"
  • 시기: 인도물 완성 후 검사
  • 활동: 검사(Inspection), 테스팅, 통계적 표본추출
  • 책임: QC 팀 / 테스터
IT 예시:
• 기능 테스트, 회귀 테스트
• 성능/보안 취약점 테스트
• UAT (사용자 인수 테스트)
🔗 연결 관계: QC에서 나온 검증된 인도물(Verified Deliverables) → 5.5 범위 확인(Validate Scope)에서 고객 공식 인수 → 인수된 인도물(Accepted Deliverables)

#도구목적언제 사용?

2 흐름도

핵심 도구: 파레토 차트 & 관리도

📊 파레토 차트 (80/20 원칙)

파레토 원칙: 20%의 원인이 80%의 결함 유발
→ 소수의 주요 원인에 집중해서 해결!

📈 관리도 (Control Chart)

Line: UCL (상한 관리한계) / 평균 / LCL (하한 관리한계)
한계 이탈 = 특수 원인 分 → 즉각 조사
Rule of Seven: 7개 연속 데이터가 평균의 한쪽 → 비정상!

인과도 (Fishbone / Ishikawa Diagram)

목적: 결함/문제의 근본 원인(Root Cause)을 6M 카테고리로 체계화 — "왜 발생했나?"

6M 카테고리: 사람(Man), 기계(Machine), 방법(Method), 재료(Material), 환경(Milieu), 측정(Measurement)

지속적 개선 프레임워크

🔄 PDCA 사이클 (데밍)

Plan: 개선 목표 설정, 방법 계획
Do: 소규모 시범 실행
Check: 결과 측정·분석
Act: 표준화 또는 재계획

📊 6시그마 (DMAIC)

목표: 100만개 중 결함 3.4개 이하
D: Define (문제 정의)
M: Measure (현 수준 측정)
A: Analyze (원인 분석)
I: Improve (개선 방안 실행)
C: Control (지속 관리)

📈 CMMI 5단계 성숙도

5 최적화 — 지속적 프로세스 개선
4 정량적 관리 — 통계 기반 성과 관리
3 정의됨 — 표준화된 프로세스 존재
2 관리됨 — 프로젝트별 계획·추적
1 초기 — 혼돈, 임기응변, 예측 불가
린 (Lean)
낭비(Muda) 7가지 제거
가치 흐름 최적화
예: JIT, 칸반 보드
카이젠 (Kaizen)
소규모 지속적 개선
전 구성원 참여
예: 매일 15분 개선 회의

학습 확인 퀴즈

Q1. 품질(Quality)과 등급(Grade)의 차이를 IT 프로젝트 예시로 설명하시오.

▶ 모범 답안

품질: 요구사항 충족도. 낮은 품질은 항상 문제 (예: 버그로 인한 시스템 오류).
등급: 기능적 범주. 낮은 등급 자체는 문제가 아님 (예: 기본형 SaaS는 이코노미석처럼 등급은 낮지만, 명시된 기능이 모두 작동하면 품질은 높음).

Q2. COQ에서 예방 비용을 투자하면 전체 품질 비용이 줄어드는 이유는?

▶ 모범 답안

예방 비용(교육·프로세스 개선)을 투자하면 결함 발생 자체가 줄어듧니다.
결함이 줄면 내부 실패(재작업)·외부 실패(고객 보상)가 감소합니다.
크로스비의 "Quality is Free": 예방에 투자한 비용 < 실패로 인한 손실.
"처음부터 올바르게(Do it right the first time)"가 가장 저렴한 품질 관리입니다.

Q3. QA와 QC의 차이를 설명하고, IT 프로젝트에서의 각 활동 사례를 드시오.

▶ 모범 답안

QA: 프로세스 중심, 예방적. "올바른 방법으로 일하고 있나?"
IT 사례: CI/CD 파이프라인 점검, 코드 리뷰 프로세스 감사, ISO 9001 심사
QC: 제품 중심, 탐지적. "인도물이 기준을 충족하나?"
IT 사례: 단위 테스트, 통합 테스트, 보안 취약점 스캔, UAT

Q4. 관리도의 Rule of Seven이란 무엇인가? 발견 시 어떤 조치를 취해야 하는가?

▶ 모범 답안

7개의 연속 데이터 포인트가 평균의 한쪽(위 또는 아래)에만 위치하는 현상.
UCL/LCL을 이탈하지 않았지만 비정상적인 패턴이므로 프로세스에 문제가 있음을 시사.
조치: 프로세스 원인 분석 착수, 인과도 작성, 특수 원인 조사. QA 팀에 프로세스 검토 요청.
