RDBMS 핵심 용어 & 키 체계

핵심 용어

개념동의어설명
테이블릴레이션(Relation)데이터의 2차원 구조
행(Row)레코드·튜플하나의 데이터 인스턴스
열(Column)필드·속성데이터의 속성 유형
도메인-속성이 가질 수 있는 값의 범위

관계 유형 3가지

관계예시비고
1:1 (일대일)직원:사원증드문 케이스
1:N (일대다) 고객:주문
부서:직원		 가장 일반적
M:N (다대다) 학생:강의
상품:주문		 중간 테이블로 해소
(e.g. 수강 테이블)

키(Key) 체계

키 종류역할
기본키 (PK) 각 행을 고유하게 식별 — NULL 불가
외래키 (FK) 다른 테이블의 PK를 참조 — 참조 무결성 보장
후보키 PK가 될 수 있는 후보 (고유+NOT NULL)
복합키 2개 이상 컬럼 조합으로 식별
무결성 제약 3가지
• 개체 무결성: PK는 NULL·중복 불가
• 참조 무결성: FK는 참조 대상이 반드시 존재
• 도메인 무결성: 정의된 값 범위 내 데이터만 허용

트랜잭션 & ACID 속성

트랜잭션(Transaction): 하나의 논리적 작업 단위. "모두 성공 or 모두 취소" 예: 계좌 이체 = 출금 + 입금이 하나의 트랜잭션
⚛️

Atomicity (원자성)

트랜잭션의 모든 연산이 전부 수행되거나 전부 취소

이체 중 오류 → 출금도 취소

✅

Consistency (일관성)

트랜잭션 전후에 DB가 항상 일관된 상태 유지

잔액이 음수가 되면 안 됨

🔀

Isolation (독립성)

동시 실행되는 트랜잭션이 서로 영향을 미치지 않음

동시에 같은 계좌 이체 시 충돌 방지

💾

Durability (영속성)

완료된 트랜잭션은 시스템 오류에도 영구 반영

COMMIT 후 정전이 나도 데이터 유지

SQL 명령: COMMIT (확정 영구저장) / ROLLBACK (이전 상태로 되돌리기) / SAVEPOINT (부분 저장점)

ERD (Entity-Relationship Diagram)

ERD 3요소 + 데이터 모델링 3단계

ERD 요소표기예시
개체(Entity)사각형 □고객, 주문, 상품
속성(Attribute)타원 ○고객명, 이메일

• Chen: 학술용 원조 표기법
• Crow's Foot (까마귀발): 실무 표준, 카디널리티를 선 끝으로 표현

SELECT 문 구조 (실행 순서)

SELECT 컬럼명
FROM 테이블명
WHERE 조건 (행 필터)
GROUP BY 그룹 기준
HAVING 그룹 조건
ORDER BY 정렬 기준;
예시 (부서별 평균 연봉)
SELECT 부서, AVG(연봉) AS 평균연봉
FROM 직원
GROUP BY 부서
HAVING AVG(연봉) > 5000
ORDER BY 평균연봉 DESC;

정규화 & 반정규화

이상(Anomaly) 현상 — 정규화 전
• 삽입 이상: 불필요한 데이터 함께 삽입
• 갱신 이상: 일부만 수정 시 불일치 발생
• 삭제 이상: 필요한 데이터까지 삭제됨

정규화 3단계

1NF (제1정규형)
원자값: 반복 그룹·다중값 속성 제거
예: 취미[독서,영화] → 별도 행으로 분리
2NF (제2정규형)
부분 함수 종속 제거 (복합키일 때 적용)
일부 키에만 종속된 속성 분리
3NF (제3정규형)
이행 함수 종속 제거 (A→B→C 연쇄 종속)
예: 부서번호→부서명 분리
반정규화 (Denormalization)
정규화를 거꾸로 — 의도적 중복 허용
목적: 조회 성능(JOIN 비용 절감)
시기: 정규화 후 성능 문제 발생 시
⚠️ 데이터 일관성 관리 부담 증가
항목RDBMSNoSQL
스키마고정 (엄격)유연 (동적)
트랜잭션ACIDBASE (Eventually Consistent)
확장수직 Scale-up수평 Scale-out
데이터 유형정형 데이터비정형·반정형
대표 제품MySQL, PostgreSQLMongoDB, Redis, Neo4j
적합 케이스금융·ERP·정확성↑SNS·IoT·빅데이터

학습 확인 퀴즈

Q1. ACID 속성의 각 요소를 은행 계좌 이체(A계좌→B계좌) 사례로 설명하시오.

▶ 모범 답안

Atomicity(원자성): A계좌 출금과 B계좌 입금이 모두 성공하거나 모두 취소됨. 출금만 되고 입금 실패 시 전체 ROLLBACK
Consistency(일관성): 이체 전후 A+B 계좌 합계는 변하지 않아야 함. 잔액이 음수가 되는 비정상 상태 불가
Isolation(독립성): 동시에 두 명이 같은 계좌에서 이체해도 서로 간섭하지 않음. 중간 상태가 외부에 노출되지 않음
Durability(영속성): COMMIT 완료 후 정전·시스템 오류가 발생해도 이체 데이터는 영구 저장됨

Q2. 데이터베이스에서 M:N(다대다) 관계가 왜 문제이며, 어떻게 해결하는지 예시와 함께 설명하시오.

▶ 모범 답안

M:N 관계는 직접 구현 시 데이터 중복과 무결성 문제가 발생합니다.

문제: 학생 테이블에 수강 강의 목록을 넣으면 한 학생이 여러 강의를 들을 때 학생 정보가 반복 저장됨 (갱신 이상)

해결 — 중간(연결) 테이블 생성:
학생(학번, 이름) + 강의(강의코드, 강의명) + 수강(학번 FK, 강의코드 FK)
→ '수강' 테이블이 M:N 관계를 두 개의 1:N 관계로 분해
→ 데이터 중복 제거 + 각 테이블 독립성 보장

Q3. PM 관점에서 RDBMS와 NoSQL 중 하나를 선택해야 할 때 기준이 되는 요소 4가지를 설명하시오.

▶ 모범 답안

① 데이터 구조의 고정 여부: 스키마가 고정적이고 복잡한 관계가 있으면 RDBMS, 스키마가 자주 바뀌거나 비정형이면 NoSQL
② 트랜잭션 정확성 요구: 금융·인사·ERP처럼 데이터 일관성이 절대적이면 ACID(RDBMS), SNS 피드·실시간 로그처럼 약간의 불일치를 허용하면 BASE(NoSQL)
③ 확장성 요구: 수직 확장(고성능 서버 1대)이면 RDBMS, 수평 확장(서버 여러 대 분산)이 필요한 대규모 서비스면 NoSQL
④ 데이터 볼륨과 유형: 정형 데이터(테이블) → RDBMS, 빅데이터·JSON·그래프 등 비정형 대용량 → NoSQL
