## 📚 강의 구성

### 오전 세션 (4시간)

#### 1교시: 소프트웨어공학 개요 및 SDLC (1.5시간)
- **소프트웨어공학의 정의**: 소프트웨어의 개발, 운용, 유지보수에 대한 체계적 접근
- **소프트웨어 위기(Software Crisis)**: 일정 지연, 예산 초과, 품질 저하
- **SDLC(Software Development Life Cycle)**
  1. **요구사항 분석(Requirements Analysis)**
     - 요구사항 수집, 분석, 명세, 검증
     - 기능 요구사항(FR): 시스템이 수행해야 할 기능
     - 비기능 요구사항(NFR): 성능, 보안, 가용성, 확장성 등
  2. **설계(Design)**
     - 아키텍처 설계 (상위 설계)
     - 상세 설계 (하위 설계)
  3. **구현(Implementation/Coding)**
     - 코딩 표준, 코드 리뷰
  4. **테스트(Testing)**
     - 단위, 통합, 시스템, 인수 테스트
  5. **배포(Deployment)**
     - 릴리스 관리, 배포 전략
  6. **유지보수(Maintenance)**
     - 수정적, 적응적, 완전적, 예방적 유지보수

#### 2교시: 개발 방법론 비교 (1.5시간)
- **워터폴(Waterfall) 모델**
  - 순차적 진행, 이전 단계 완료 후 다음 단계
  - 문서 중심, 변경 비용 높음
  - 요구사항이 명확하고 변경이 적은 프로젝트에 적합
- **V-모델(V-Model)**
  - 각 개발 단계에 대응하는 테스트 단계
  - 요구분석↔인수시험, 설계↔시스템시험, 상세설계↔통합시험, 코딩↔단위시험
- **프로토타입 모델**
  - 시제품을 먼저 만들어 요구사항 확인
  - 요구사항이 불명확할 때 활용
- **나선형(Spiral) 모델**
  - 반복적 개발 + 리스크 분석
  - 계획 → 리스크분석 → 개발 → 평가 반복
- **애자일(Agile)**
  - 애자일 선언문(Agile Manifesto) 4대 가치
    1. 프로세스/도구보다 **개인과 상호작용**
    2. 포괄적 문서보다 **작동 소프트웨어**
    3. 계약 협상보다 **고객과의 협력**
    4. 계획 추종보다 **변화에 대한 대응**
  - 12개 원칙 주요 내용
  - **스크럼(Scrum)**
    - 역할: Product Owner, Scrum Master, Development Team
    - 산출물: Product Backlog, Sprint Backlog, Increment
    - 이벤트: Sprint(2-4주), Sprint Planning, Daily Standup, Sprint Review, Sprint Retrospective
    - 번다운/번업 차트
  - **칸반(Kanban)**: 시각화, WIP(Work in Progress) 제한
  - **XP(eXtreme Programming)**: 페어 프로그래밍, TDD, CI

#### 3교시: 요구사항 분석 상세 (1시간)
- **요구사항 분석 기법**
  - 유스케이스(Use Case) 다이어그램
  - 사용자 스토리(User Story): "As a [사용자], I want [기능], so that [가치]"
  - 페르소나(Persona)
  - 요구사항 명세서(SRS: Software Requirements Specification)
- **기능 요구사항 vs 비기능 요구사항**
  - FR 예시: 사용자 로그인, 주문 처리, 보고서 생성
  - NFR 예시: 응답시간 2초 이내, 99.9% 가용성, HTTPS 통신
- **요구사항 우선순위 기법**
  - MoSCoW: Must/Should/Could/Won't
  - Kano 모델: 기본/성능/매력적 요구사항

### 오후 세션 (4시간)

#### 4교시: 소프트웨어 설계 원칙 (1시간)
- **설계 원칙**
  - 모듈화(Modularity): 독립적 모듈로 분리
  - 결합도(Coupling): 모듈 간 의존도 → 낮을수록 좋음
  - 응집도(Cohesion): 모듈 내부 요소의 관련성 → 높을수록 좋음
  - 추상화(Abstraction): 복잡성 숨기기
  - 캡슐화(Encapsulation): 정보 은닉
- **아키텍처 패턴 (PM이 알아야 할 수준)**
  - 모놀리식(Monolithic) vs 마이크로서비스(MSA)
  - 클라이언트-서버(Client-Server)
  - MVC (Model-View-Controller)
  - 3계층 아키텍처 (프레젠테이션-비즈니스로직-데이터)
- **UML 다이어그램 개요**
  - 클래스, 시퀀스, 유스케이스, 활동 다이어그램

#### 5교시: 소프트웨어 테스팅 (1.5시간)
- **테스트 수준**
  - **단위 테스트(Unit Test)**: 개별 모듈/함수 테스트
  - **통합 테스트(Integration Test)**: 모듈 간 인터페이스 테스트
    - 빅뱅, 하향식, 상향식, 샌드위치
  - **시스템 테스트(System Test)**: 전체 시스템의 요구사항 충족 확인
  - **인수 테스트(Acceptance Test)**: 고객의 요구사항 충족 확인
    - 알파 테스트: 개발자 환경에서 사용자가 수행
    - 베타 테스트: 실제 환경에서 사용자가 수행
- **테스트 기법**
  - 블랙박스 테스트: 내부 구조 모르는 상태에서 (동치분할, 경계값)
  - 화이트박스 테스트: 내부 구조를 알고 (구문 커버리지, 분기 커버리지)
  - 회귀 테스트(Regression Test): 변경 후 기존 기능 정상 동작 확인
- **테스트 산출물**: 테스트 계획, 테스트 케이스, 테스트 결과 보고서

#### 6교시: 형상관리 및 PM 관점 정리 (1.5시간)
- **형상관리(Configuration Management)**
  - 버전 관리 (Git 개요): commit, branch, merge, pull request
  - 형상 항목 식별 및 통제
  - 변경 관리와의 연계
- **PM이 소프트웨어공학을 알아야 하는 이유 정리**
  - 현실적인 일정 산정을 위한 SDLC 단계별 공수 이해
  - 개발팀과의 효과적 의사소통
  - 기술적 리스크 조기 식별
  - 적절한 개발 방법론 선택 지원
  - 인수 테스트 및 품질 검증의 이해
- **Q&A 및 토론**
