| 시간 프레임 | 단기/중기/장기 편익 분류 | 즉시(1년), 중기(3년), 장기(5년+) |

### 6-2. 프로젝트 헌장 vs 제안서 vs SOW 비교

| 구분 | 작성자 | 목적 | 승인자 | 내용 |
|------|------|------|------|------|
| **제안서 (Proposal)** | 판매자(공급사) | 수주 획득 | 구매자 | 기술 방식, 비용, 일정 제안 |
| **SOW (작업기술서)** | 구매자 | 요구사항 명확화 | 양측 계약 | 인도물, 기술 사양, 표준 |
| **헌장 (Charter)** | 스폰서/PMO | 프로젝트 공식 승인 | 스폰서 | 목적, 예산 한도, PM 권한 부여 |

> PMP 시험 포인트: "프로젝트 헌장이 없으면 PM에게 권한이 없다" — 헌장은 PM의 법적 위임 문서.

### 6-3. 형상 관리 (Configuration Management) 개념

| 항목 | 내용 |
|------|------|
| **정의** | 프로젝트 산출물(문서·소프트웨어·시스템)의 버전, 변경, 이력을 일관되게 관리 |
| **형상 항목 (CI)** | 관리 대상: 요구사항 문서, 설계서, 소스코드, 테스트 케이스 |
| **형상 감사** | 최신 기준선 문서와 실제 인도물의 일치 여부 확인 |
| **형상 상태 계정** | 변경 이력 추적 (언제, 누가, 왜 변경했는가) |
| **도구 예시** | Git (소스코드), Confluence (문서), Jira (이슈 추적) |

> 변경 통제 vs 형상 관리: 변경 통제는 변경 승인 프로세스, 형상 관리는 변경 후 결과물 버전 관리.

### 6-4. 점진적 구체화 (Progressive Elaboration) 원칙

**정의**: 프로젝트 초반에는 高수준 정보만 존재, 시간이 지나면서 상세화

```
프로젝트 시작 → 헌장 (개략 예산/일정) → 계획서 (상세 WBS/CPM) → 실행 (실제 데이터)
             High-Level                 Detailed                  Actual
```

- **핵심**: 초기에 모든 것을 알 수 없음 → 기획 단계에서 Baseline 수립 후 점차 구체화
- **PMP 함의**: Rolling Wave Planning — 가까운 미래는 상세히, 먼 미래는 개략적으로 계획
- **리스크**: 점진적 구체화 ≠ Scope Creep. 기준선 대비 변경은 반드시 CCB 통과 필요

---

### 6-5. CoP (Community of Practice) — 암묵적 지식 공유

| 특성 | 내용 |
|------|------|
| 정의 | 공통 관심사나 전문성을 가진 사람들이 자발적으로 형성한 학습 커뮤니티 |
| PM 관련성 | PM들이 교훈(Lessons Learned)을 비공식적으로 공유하는 채널 |
| 예시 | 사내 PM 길드, 아키텍처 챕터, QA 커뮤니티 |
| 교훈 등록부와 차이 | 교훈 등록부 = 공식 문서. CoP = 비공식 지식 교류 채널 |
