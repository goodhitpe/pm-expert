## 4. 보완 자료 (슬라이드 미포함 핵심 내용)

### 4-1. Kano 모델 3유형 상세

| 유형 | 정의 | 구현 시 고객 반응 | 미구현 시 | IT 예시 |
|------|------|----------------|---------|--------|
| **기본 요구 (Must-be)** | 당연히 있어야 할 기능 | 반응 미미 | 강한 불만 | 로그인 기능, 데이터 저장 |
| **성능 요구 (Performance)** | 많을수록 만족 증가 | 비례 증가 | 비례 감소 | 검색 속도, 배터리 수명 |
| **매력 요구 (Excitement)** | 기대하지 않은 기능 | 큰 만족 | 반응 없음 | AI 추천, 다크모드 자동 전환 |

> **PM 활용 포인트**: MoSCoW와 결합 — Must-be=M, Performance=S, Excitement=C/W. Kano 분석으로 고객 만족 대비 투자 효율 극대화.

### 4-2. WBS 작성 방법 — 상향식 vs 하향식

| 구분 | 방법 | 장점 | 단점 | 적합 상황 |
|------|------|------|------|---------|
| **하향식 (Top-Down)** | 최종 인도물 → 하위 분해 | 전체 조망 용이, 빠른 초안 | 하위 세부 누락 위험 | 유사 프로젝트, 경험 있는 PM |
| **상향식 (Bottom-Up)** | 개별 작업 식별 → 집합 | 세부 사항 완전, 팀 참여 | 시간 소요, 중복 위험 | 신규 영역, 복잡 기술 |
| **혼합형** | 상위: 하향식, 하위: 상향식 | 두 방식 장점 결합 | 조정 필요 | 대규모 복잡 프로젝트 |

### 4-3. 인도물 기반 vs 단계 기반 WBS

**인도물 기반 WBS (PMBOK 권장)**:
```
소핑몰 구축 프로젝트
├── 1. 사용자 인터페이스
│   ├── 1.1 메인 페이지
│   ├── 1.2 상품 목록 페이지
│   └── 1.3 결제 페이지
├── 2. 백엔드 API
│   ├── 2.1 회원 관리 API
│   ├── 2.2 상품 관리 API
│   └── 2.3 주문 처리 API
└── 3. 인프라 & 배포
```

**단계 기반 WBS (비권장)**:
```
소핑몰 구축 프로젝트
├── 1. 요구사항 분석
├── 2. 설계
├── 3. 개발
└── 4. 테스트
```
> 단계 기반은 100% 규칙 위반 위험 높음 — 같은 인도물이 여러 단계에 분산될 수 있음.

### 4-5. ERP 범위 확장(Scope Creep) 실제 사례

**상황**: 국내 제조사 ERP 구축 프로젝트 (계획 6개월, ₩5억)

| 단계 | 발생 요구 | 결과 |
|------|---------|------|
| 2개월 차 | "모바일 앱도 붙여주세요" (현업 요청) | 범위 기준선 미등록, 비공식 수락 |
| 3개월 차 | "영업팀 대시보드 커스텀 필요" | 또 다른 추가 요구 발생 |
| 5개월 차 | "기존 레거시 연동도 해주세요" | 개발 70% 재작업 |
| 결과 | 11개월 완료, ₩11억 지출 | 원가 초과율 120%, 일정 83% 지연 |

**교훈**: 변경 요청은 반드시 CCB → WBS·기준선 업데이트 → 재승인 절차 거쳐야 함.
