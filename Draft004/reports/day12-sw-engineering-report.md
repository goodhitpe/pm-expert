## 시험 포인트 (PMP/자격증 관련)

### 🔴 고빈도 출제 항목

1. **V-모델 대응 관계**
   - 요구분석 ↔ 인수 테스트
   - 설계 ↔ 시스템 테스트
   - 상세설계 ↔ 통합 테스트
   - 코딩 ↔ 단위 테스트

2. **애자일 선언문 핵심**
   - 개인과 상호작용 > 프로세스와 도구
   - 작동 SW > 포괄적 문서
   - 고객 협력 > 계약 협상
   - 변화 대응 > 계획 추종

3. **스크럼 역할 구분**
   - PO: 비즈니스 가치, 백로그 우선순위 (PM과 다름)
   - SM: 프로세스 가이드, 장애 제거 (PM과 다름)
   - PM은 스크럼 외부에 존재하거나 PO 역할 겸임

4. **알파 vs 베타 테스트**
   - 알파: 개발자 통제 환경에서 사용자 수행
   - 베타: 실제 환경에서 사용자 수행

5. **NFR 계약 명시 중요성**
   - "응답시간 2초 이내" = 측정 가능 → 인수 기준
   - "빠르게" = 모호 → 분쟁 원인

6. **MoSCoW 순서**
   - Must > Should > Could > Won't (이번 버전 제외)

## 보완 자료 (슬라이드 미포함 핵심 내용)

### B-1. XP (eXtreme Programming) 5대 핵심 관행

| 관행 | 설명 | PM 관점 이점 |
|------|------|------------|
| TDD (테스트 주도 개발) | 코드 작성 전 테스트 먼저 → Red-Green-Refactor 사이클 | 결함 비용 조기 절감 |
| 페어 프로그래밍 | 개발자 2인 1조, 실시간 코드 리뷰 | WIP 감소, 지식 공유 |
| 리팩토링 | 기능 변경 없이 내부 코드 개선 | 기술 부채 관리 |
| CI (지속적 통합) | 하루 수차례 메인 브랜치 병합 | 통합 리스크 조기 발견 |
| 소규모 릴리즈 | 2주 이내 사이클로 배포 | 피드백 루프 단축 |

### B-2. UML 다이어그램 4종 — PM 필수 이해

| 다이어그램 | 목적 | 요소 | 시험 포인트 |
|----------|------|------|-----------|
| Use Case Diagram | 기능 요구사항 시각화 | 액터, 유스케이스, <<include>>/<<extend>> | 요구사항 수집 산출물 |
| Class Diagram | 데이터 구조 정적 표현 | 클래스, 속성, 메서드, 관계(상속/집합/의존) | 설계 단계 |
| Sequence Diagram | 시간 흐름에 따른 메시지 교환 | 생명선, 활성박스, 동기/비동기 메시지 | 인터페이스 설계 |
| Activity Diagram | 프로세스 흐름 표현 | 시작/끝 노드, 포크/조인, 판단 노드 | 프로세스 개선 |

### B-3. 배포 전략 비교 — Blue-Green vs Canary vs Rolling

| 전략 | 방법 | 장점 | 단점 | 적합 상황 |
|------|------|------|------|---------|
| Blue-Green | 구버전(Blue)·신버전(Green) 동시 운영, DNS 전환 | 즉시 롤백 가능 | 서버 2배 비용 | 금융·의료 핵심 시스템 |
| Canary | 소수(1~5%) 사용자에게 먼저 배포 후 확대 | 실제 트래픽으로 검증 | 모니터링 복잡 | 사용자 기반 대형 서비스 |
| Rolling Update | 인스턴스 순차적 교체 | 서버 추가 불필요 | 도중 구·신 혼재 | Kubernetes 기본 전략 |

### B-4. SRS (Software Requirements Specification) 구조 — IEEE 830

| 섹션 | 내용 |
|------|------|
| 1. 목적 | 시스템 목적, 범위, 약어 정의 |
| 2. 전체 설명 | 제품 퍼스펙티브, 기능, 사용자 특성, 제약사항 |
| 3. 기능 요구사항 | Use Case 또는 기능 명세 (SHALL 표현) |
| 4. 비기능 요구사항 | 성능, 보안, 가용성, 유지보수성, 이식성 |
| 5. 인터페이스 요구사항 | UI, 하드웨어, 소프트웨어, 통신 인터페이스 |
